# 02 - schema 版本与生命周期规则

版本：v0.1.0

生命周期：Draft

日期：2026-06-20

适用范围：`02 - schema`

## 1. 定位

本规则用于约束 `02 - schema` 中规则文件、自动化脚本和规则变更日志的版本写法。

它只解决三件事：

1. 规则文件版本号怎么写。
2. 规则生命周期怎么标记。
3. 规则、脚本、变更日志如何保持同步。

它不规定 raw、wiki、GBrain DB、GitHub、Supabase 的业务流程。

## 2. 版本号格式

schema 层规则版本必须使用三段式语义化版本：

```text
vMAJOR.MINOR.PATCH
```

允许示例：

```text
v0.1.0
v0.1.1
v0.2.0
v1.0.0
```

禁止示例：

```text
vMAJOR.MINOR
MAJOR.MINOR.PATCH
```

## 3. 生命周期

规则文件必须声明生命周期：

```text
Draft
Stable
Deprecated
```

含义：

| 生命周期 | 含义 |
| --- | --- |
| Draft | 仍在整理和试运行，可以继续修改当前版本 |
| Stable | 已确认可长期引用，后续实质修改必须升版本 |
| Deprecated | 已废弃，只保留追溯，不再作为活动规则 |

生命周期为 Draft 的规则文件使用：

```text
生命周期：Draft
```

## 4. 活动规则文件

`02 - schema` 根目录中的 Markdown 规则文件是活动规则文件。

`规则变更日志.md` 只记录规则契约变化，不作为活动规则文件。

`02 - schema/自动化脚本` 下被活动规则文件引用的脚本是活动脚本。

活动规则文件必须声明版本号和生命周期。

活动脚本必须声明依赖规则和规则版本。

## 5. 规则文件头

规则文件头必须包含：

```text
# 规则名称

版本：v0.1.0

生命周期：Draft

日期：YYYY-MM-DD

适用范围：`适用目录或对象`
```

## 6. 脚本依赖版本

脚本头部必须写明依赖规则和规则版本。

示例：

```text
依赖规则：02 - schema/《00 - inbox》路由分发规则.md
规则版本：v0.1.0
```

脚本输出中的 `RuleVersion` 或 `规则版本` 必须与脚本头部一致。

修改规则文件时，必须检查相关脚本是否需要同步。

修改脚本判断逻辑时，必须检查相关规则文件和变更日志是否需要同步。

## 7. 规则变更日志

规则变更日志文件固定为：

```text
02 - schema/规则变更日志.md
```

变更日志从当前状态重新登记，第一条正式基线为：

```text
v0.1.0 - 2026-06-20
```

本次基线建立前的旧 changelog 条目和临时报告不进入正式版本链；需要追溯时，以 Git 历史或旧提交为准。

### 7.1 写入原则

规则变更日志只记录规则契约变化，不记录执行过程。

每条变更必须包含具体影响范围、兼容影响、规则契约、决策原因和迁移要求。

### 7.2 写入条件

只有变化影响以下对象时，才写入规则变更日志：

1. 用户如何写入知识。
2. Agent 如何判断知识。
3. 脚本如何读取、校验、输出。
4. 文件或目录的目标位置。
5. 旧数据是否需要迁移。
6. 后续同步、编译、路由是否改变行为。

以下内容不写入规则变更日志：

1. 跑了一次审计。
2. 修了多少个文件。
3. 删除一次性报告或副产物脚本。
4. 某次手工整理过程。
5. Git / GitHub / Supabase 操作记录。
6. inbox 当前是否为空。

### 7.3 条目结构

每条变更固定使用以下字段：

```markdown
## vX.Y.Z - YYYY-MM-DD

变更类型：
影响范围：
兼容影响：
涉及规则：
规则契约：
决策原因：
迁移要求：
```

### 7.4 字段取值

`变更类型` 只允许：

```text
Baseline
Added
Changed
Deprecated
Removed
Breaking
```

`兼容影响` 只允许：

```text
无
向后兼容
需要迁移
破坏兼容
```

`影响范围` 只允许：

```text
frontmatter
inbox 路由
raw 目录命名
脚本输入输出
版本生命周期
wiki 编译规则
```

新增影响范围前，必须先修改本规则。

### 7.5 好例子

```markdown
## v0.2.0 - 2026-06-20

变更类型：Changed
影响范围：frontmatter
兼容影响：需要迁移
涉及规则：
- 《00 - inbox》路由分发规则.md

规则契约：
`source_type` 只允许 `用户写入 / Agent回写 / 网页剪藏 / unknown`。缺失、空值、非法值统一为 `unknown`，禁止 Agent 根据正文自动推断。

决策原因：
`source_type` 表示知识进入系统的通道，该信息依赖真实操作场景，无法从正文稳定推断。

迁移要求：
已有 raw Markdown 中非法 `source_type` 需要改为 `unknown`。
```

### 7.6 坏例子

```markdown
## v0.2.0 - 2026-06-20

今天扫描了 raw，发现 12 个问题，然后修复了 tags，又删除了一个脚本。
```

错误原因：

1. 记录的是执行过程，不是规则契约。
2. 没有说明影响范围。
3. 没有说明兼容影响。
4. 没有说明以后系统必须遵守什么规则。
5. 后续用户、Agent、脚本无法据此判断行为变化。

## 8. 版本升级规则

Draft 阶段：

1. 修错别字、路径、示例、格式：可以继续保持当前版本。
2. 修改字段、枚举、输出结构、脚本判断逻辑：必须在变更日志中登记。
3. 影响自动化行为的变更：必须升级 MINOR 并登记。

Stable 阶段：

1. 修正文档错别字：升级 PATCH。
2. 新增兼容能力、字段、流程：升级 MINOR。
3. 删除旧流程、改变核心输入输出、破坏兼容：升级 MAJOR。

## 9. Stable 归档

当前暂不建立 `versions` 目录。

当某个规则进入 Stable 后，必须冻结当时版本。

后续如果还要修改，应复制到版本归档目录：

```text
02 - schema/versions/<规则名>/vX.Y.Z.md
```

活动文件保留 latest 文件名。

## 10. status 与生命周期

知识文件 frontmatter 中的：

```yaml
status: raw
```

表示知识材料在 raw 层的位置状态。

它不等于 schema 规则的生命周期。

schema 规则生命周期必须单独写：

```text
生命周期：Draft
```

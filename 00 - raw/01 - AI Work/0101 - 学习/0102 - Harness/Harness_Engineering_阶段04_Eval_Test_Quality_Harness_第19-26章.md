# Harness Engineering｜阶段四：Eval / Test / Quality Harness（第 19–26 章）

> 适用范围：Agent Harness Engineering，不是 Harness.io 产品教程。
>
> 本阶段目标：建立 Agent 工程里的测试、评估、回归保护与质量门禁体系，让 Agent 不只是“能生成”，而是“能被验证、能被改进、能稳定交付”。

---

## 阶段四总览

| 章 | 主题 | 核心问题 | 一句话理解 |
|---:|---|---|---|
| 19 | Test Harness 基础 | 如何用传统测试思想约束 Agent | 测试先解决“有没有做对” |
| 20 | Eval Harness 基础 | 如何评估非确定性输出质量 | 评估解决“做得好不好” |
| 21 | 输入测试 | Agent 该不该被触发 | 防止误触发、漏触发 |
| 22 | 输出测试 | Agent 输出是否符合契约 | 防止少字段、乱格式、缺内容 |
| 23 | 过程评估 | Agent 是否按流程执行 | 防止绕流程、跳检查、假完成 |
| 24 | 结果评估 | Agent 是否真正解决问题 | 防止形式合格但业务无效 |
| 25 | Regression Harness | 如何防止改好一个、弄坏三个 | 失败案例要变成回归保护 |
| 26 | Quality Gate | 什么时候允许交付 | 未过门禁，不得宣称完成 |

# 第 19 章：Test Harness 基础｜fixture、stub、driver

## 19.1 本章核心

> Test Harness 的作用，是给被测对象提供一个可控、可重复、可判断的测试环境。

迁移到 Agent 工程里：

```text
Test Harness = 用一组固定输入、固定环境、固定检查规则，判断 Agent 是否按约定做对。
```

它更偏“确定性检查”，适合回答：

```text
有没有触发？
有没有漏字段？
有没有按 schema 输出？
有没有跑测试？
有没有修改错目录？
有没有违反禁止规则？
```

---

## 19.2 Test Harness 的传统组成

| 组成 | 简单理解 | 在 Agent 中的迁移 |
|---|---|---|
| Fixture | 准备好的测试数据和环境 | 测试输入、示例文件、模拟用户请求 |
| Driver | 驱动被测对象运行 | 调用 Agent 的测试脚本 / workflow runner |
| Stub | 替代真实依赖的简化模块 | 模拟 API、模拟工具返回、假文件系统 |
| Mock | 可检查交互的替身对象 | 检查 Agent 是否调用了正确工具 |
| Assertion | 断言结果是否符合预期 | 检查输出字段、文件变更、工具调用 |
| Runner | 执行测试套件 | pytest、脚本、eval runner、CI |
| Report | 测试报告 | pass / fail、失败原因、日志 |

---

## 19.3 Agent 为什么需要 Test Harness

Agent 的输出有不确定性，但不是所有东西都只能主观评价。

| 可确定测试的内容 | 示例 |
|---|---|
| 格式 | 是否输出 JSON / Markdown / 表格 |
| 字段 | 是否包含 title、summary、risks、next_steps |
| 触发 | 是否在正确场景启动 skill |
| 非触发 | 是否拒绝无关任务 |
| 工具使用 | 是否先读文件再写文件 |
| 文件变更 | 是否只修改允许目录 |
| 安全约束 | 是否避免删除、发送、部署等高风险动作 |
| 执行结果 | 测试是否通过、构建是否成功 |

原则：

```text
能用程序判断的，不要交给模型主观判断。
```

---

## 19.4 Test Harness 与 Eval Harness 的区别

| 维度 | Test Harness | Eval Harness |
|---|---|---|
| 核心问题 | 有没有做对 | 做得好不好 |
| 判断方式 | 程序断言、规则检查 | 评分器、模型评审、人工评审、指标 |
| 适合对象 | 格式、字段、触发、流程、工具调用 | 质量、相关性、完整性、业务价值 |
| 输出 | pass / fail | 分数、等级、理由、对比 |
| 稳定性 | 更高 | 相对更低 |
| 典型例子 | JSON schema 校验 | 判断一篇分析是否有洞察 |

---

## 19.5 Agent Test Harness 的基本结构

```text
测试用例
→ 准备 fixture
→ 调用 Agent / Skill
→ 捕获输出、工具调用、文件变更
→ 执行断言
→ 输出测试报告
```

---

## 19.6 示例：Skill 触发测试

| 测试项 | 输入 | 期望 |
|---|---|---|
| 正确触发 | “帮我评估这个 SKILL.md 的质量” | 调用 skill-quality-reviewer |
| 错误触发 | “帮我翻译这段英文” | 不调用 skill-quality-reviewer |
| 近似场景 | “这个 skill 好不好？” | 可触发，但需先确认目标 |
| 边界场景 | “什么是 skill？” | 不触发评估型 skill，只做概念解释 |

---

## 19.7 示例：llm-wiki 文件输出测试

| 测试项 | 规则 |
|---|---|
| 文件存在 | 必须生成 .md 文件 |
| 文件名规范 | 包含主题、阶段、章节范围 |
| 结构完整 | 必须有标题、总览、章节、自检、总结 |
| 内容范围 | 只包含当前阶段，不混入下一阶段 |
| 格式规范 | 标题层级不能混乱 |
| 可下载 | 必须提供文件链接 |

---

## 19.8 常见误区

| 误区 | 问题 | 正确做法 |
|---|---|---|
| Agent 输出无法测试 | 很多结构性要求可以测试 | 先把可确定部分测起来 |
| Eval 可以替代 Test | Eval 不适合做硬约束 | Test 和 Eval 分工 |
| 只做人工检查 | 成本高且不稳定 | 程序测试优先 |
| 测试越多越好 | 无关测试会拖慢迭代 | 围绕关键失败设计 |
| 只测成功场景 | 容易误触发 | 必须测反例和边界 |

---

## 19.9 Feynman 解释

Test Harness 像考试里的客观题。

```text
姓名有没有写？
题号有没有答？
选择题有没有按格式填？
答案是不是 A/B/C/D？
```

它不判断作文写得多好，但能先保证基本规则没有错。

---

## 19.10 本章掌握标准

| 你需要能做到 | 判断方式 |
|---|---|
| 区分 Test 和 Eval | 能说出哪些适合程序检查，哪些适合评分 |
| 设计 fixture | 能准备固定输入和测试环境 |
| 设计 assertion | 能把要求转成 pass / fail 规则 |
| 设计反例测试 | 能防止误触发和越权 |
| 迁移到 Skill / llm-wiki | 能写出基础测试清单 |

# 第 20 章：Eval Harness 基础｜测试集、评分器、基准线

## 20.1 本章核心

> Eval Harness 的作用，是评估 Agent 在非确定性任务上的质量表现。

OpenAI 的 evals 文档将 evaluations 描述为测试模型输出是否满足你指定的风格和内容标准；其 evaluation best practices 也强调，生成式 AI 具有变量性，同样输入可能产生不同输出，因此传统测试不足以覆盖 AI 系统质量。LangSmith 也把 evaluation 用于从开发前测试到生产监控的完整生命周期，包括 benchmark、unit tests、regression tests、online monitoring 等。

---

## 20.2 Eval Harness 解决什么问题

Test 更适合判断：

```text
有没有
对不对
是否符合格式
是否违反硬规则
```

Eval 更适合判断：

```text
是否相关
是否完整
是否有洞察
是否符合用户偏好
是否可执行
是否有商业价值
是否比旧版本更好
```

---

## 20.3 Eval Harness 的基本组成

| 组成 | 作用 |
|---|---|
| Dataset | 测试样本集 |
| Input | 输入任务 |
| Expected / Reference | 参考答案或期望行为 |
| Evaluator | 评分器 |
| Rubric | 评分标准 |
| Baseline | 当前基准版本 |
| Candidate | 新版本输出 |
| Score | 分数或等级 |
| Rationale | 评分理由 |
| Report | 评估报告 |

---

## 20.4 Dataset：测试集

测试集不是越大越好，关键是覆盖真实场景。

| 样本类型 | 作用 |
|---|---|
| Golden Cases | 最核心、最常见的成功场景 |
| Edge Cases | 边界场景 |
| Negative Cases | 不该触发、不该执行的场景 |
| Failure Cases | 历史失败案例 |
| Regression Cases | 防止旧问题复发 |
| Stress Cases | 长上下文、复杂约束、多步骤任务 |
| Business Cases | 真实业务输入 |

---

## 20.5 Evaluator：评分器类型

OpenAI cookbook 中把评估大致分为两类：用代码逻辑检查答案，或用模型本身检查答案。工程实践中可以进一步拆成下面几类。

| 评分器 | 适合判断 | 风险 |
|---|---|---|
| Code Evaluator | 格式、字段、精确匹配、数值 | 难判断语义质量 |
| Rule Evaluator | 是否满足明确规则 | 规则写不好会漏判 |
| LLM-as-Judge | 语义质量、完整性、风格 | 评分可能不稳定 |
| Human Evaluator | 高价值判断、最终验收 | 成本高 |
| Pairwise Evaluator | A 版本和 B 版本哪个好 | 需要对照输出 |
| Composite Evaluator | 多指标综合评分 | 设计复杂 |

---

## 20.6 Rubric：评分标准

Eval 的核心不是“让模型打分”，而是把“好”拆成可评分维度。

### 示例：知识沉淀 Agent Rubric

| 维度 | 评分问题 | 权重 |
|---|---|---:|
| 范围准确 | 是否只覆盖指定阶段 | 20% |
| 结构清晰 | 是否层级合理、方便消化 | 20% |
| 信息密度 | 是否避免大段水文 | 15% |
| 可沉淀性 | 是否适合直接放入知识库 | 20% |
| 可迁移性 | 是否能指导未来实践 | 15% |
| 格式规范 | Markdown 是否规范 | 10% |

---

## 20.7 Baseline：基准线

没有基准线，就无法判断新版本是否更好。

| 基准线类型 | 说明 |
|---|---|
| 当前线上版本 | 和现有 Agent 比 |
| 人工专家版本 | 和人工高质量答案比 |
| 旧模型版本 | 和旧模型输出比 |
| 旧 prompt 版本 | 和旧提示词比 |
| 历史最佳版本 | 和过去最高分版本比 |

---

## 20.8 Offline Eval 与 Online Eval

| 类型 | 使用时机 | 作用 |
|---|---|---|
| Offline Eval | 上线前 | 测试版本质量、比较方案、防止回归 |
| Online Eval | 上线后 | 监控真实用户交互、发现异常、观察漂移 |

LangSmith 官方文档也将 offline evaluation 用于开发期测试和回归捕获，将 online evaluation 用于生产环境真实交互监控。

---

## 20.9 常见误区

| 误区 | 问题 | 正确做法 |
|---|---|---|
| Eval 就是让模型打分 | LLM-as-Judge 只是评分器之一 | 混合代码、规则、模型、人工 |
| 分数高就一定好 | Rubric 可能有缺陷 | 看错误案例和业务效果 |
| 测试集越大越好 | 低质量样本会污染评估 | 先做小而精测试集 |
| 只测平均分 | 平均分会掩盖关键失败 | 关注失败样本和红线指标 |
| 没有 baseline | 无法判断进步还是退步 | 每次评估都要有对照 |

---

## 20.10 Feynman 解释

Eval Harness 像作文评分。

客观题可以直接判对错；作文不能只看有没有写满字数，还要看：

```text
是否切题
结构是否清楚
论据是否充分
表达是否准确
有没有洞察
是否符合读者需求
```

这就是 Eval。

---

## 20.11 本章掌握标准

| 你需要能做到 | 判断方式 |
|---|---|
| 设计测试集 | 能覆盖成功、失败、边界、历史错误 |
| 设计评分标准 | 能把“好”拆成具体维度 |
| 选择评分器 | 能决定代码、规则、模型、人工各自负责什么 |
| 建立基准线 | 能做版本对比 |
| 解释评估结果 | 不只看分数，还能分析失败样本 |

# 第 21 章：输入测试｜正确触发、错误触发、近似场景

## 21.1 本章核心

> 输入测试解决的是：Agent / Skill 在什么情况下应该启动，什么情况下不应该启动。

这是 Skill 和 Agent 系统里非常关键的一层。

如果输入测试没做好，会出现两类问题：

| 问题 | 表现 |
|---|---|
| 漏触发 | 该启动时没启动 |
| 误触发 | 不该启动时乱启动 |

---

## 21.2 为什么输入测试重要

Agent 系统不是单个 prompt，而是多个能力模块组合。

如果路由错了：

```text
后面执行再好也没用。
```

例子：

| 用户请求 | 正确行为 | 错误行为 |
|---|---|---|
| “评估这个 SKILL.md” | 触发 skill-quality-reviewer | 普通解释 skill 概念 |
| “帮我翻译这段话” | 触发翻译能力 | 误触发知识沉淀 skill |
| “把这轮对话沉淀成 md” | 触发 llm-wiki 沉淀流程 | 只在聊天里总结 |
| “这个需求有点模糊，帮我理清楚” | 触发需求澄清 Agent | 直接开始执行 |

---

## 21.3 输入测试的四类样本

| 类型 | 目的 | 示例 |
|---|---|---|
| Positive Case | 测正确触发 | “请评估这个 SKILL.md 的质量” |
| Negative Case | 测不要误触发 | “什么是 SKILL.md？” |
| Near-miss Case | 测近似场景判断 | “这个 skill 感觉不太好” |
| Ambiguous Case | 测模糊输入处理 | “帮我看看这个文件” |

---

## 21.4 正确触发测试

正确触发要覆盖不同表达方式。

以“知识沉淀 Agent”为例：

| 输入 | 期望 |
|---|---|
| “把这次对话整理成 md” | 触发 |
| “沉淀到我的 llm-wiki” | 触发 |
| “生成知识库文件” | 触发 |
| “导出 Markdown 文件” | 触发 |
| “整理成 Obsidian 笔记” | 触发 |

关键是不要只测一种关键词。

---

## 21.5 错误触发测试

错误触发测试更重要。

| 输入 | 期望 |
|---|---|
| “解释一下 Harness 是什么” | 不触发文件沉淀，只讲解 |
| “帮我写一封邮件” | 不触发知识沉淀 |
| “翻译这段文字” | 不触发知识沉淀 |
| “给我一个标题” | 不触发完整课程输出 |
| “这个概念和 Prompt 有什么区别” | 不触发 md 导出 |

---

## 21.6 近似场景测试

近似场景不是简单 yes/no，而是测试 Agent 能不能判断边界。

| 输入 | 合理行为 |
|---|---|
| “这个内容能不能沉淀？” | 先分析是否值得沉淀 |
| “帮我整理一下” | 可能需要判断是聊天总结还是文件输出 |
| “这个 skill 怎么样？” | 如果有文件，触发评估；没有文件，先问或说明需要输入 |
| “这轮内容做个图谱” | 触发图谱生成，不一定触发 md 沉淀 |

---

## 21.7 输入测试的评价指标

| 指标 | 说明 |
|---|---|
| Precision | 触发的场景中，有多少是真该触发 |
| Recall | 真该触发的场景中，有多少被触发 |
| False Positive | 不该触发但触发了 |
| False Negative | 该触发但没触发 |
| Ambiguity Handling | 模糊场景是否处理得当 |

简单理解：

```text
Precision 防止乱调用；
Recall 防止该用不用。
```

---

## 21.8 Skill description 的测试

Skill 是否能正确触发，很大程度取决于 description。

高质量 description 应该包含：

```text
适用场景
不适用场景
输入类型
任务边界
触发关键词但不依赖关键词
```

测试方式：

| 测试 | 问题 |
|---|---|
| 正例测试 | description 是否能覆盖核心任务 |
| 反例测试 | description 是否避免误触发 |
| 近似测试 | description 是否能处理相似但不同任务 |
| 缺失测试 | 输入不完整时是否先澄清 |

---

## 21.9 常见误区

| 误区 | 问题 | 正确做法 |
|---|---|---|
| 靠关键词触发 | 容易误判 | 按意图触发 |
| 只测正例 | 会严重误触发 | 正例、反例、近似都测 |
| description 写得越宽越好 | 触发泛滥 | 明确边界 |
| 输入模糊就直接执行 | 容易跑偏 | 低风险假设，高风险澄清 |
| 漏触发比误触发更可怕 | 不一定 | 高风险工具误触发更危险 |

---

## 21.10 Feynman 解释

输入测试像医院分诊。

病人来了，不是所有人都进急诊，也不是所有人都去牙科。分诊错了，后面医生再专业也会浪费资源，甚至出事故。

Agent 输入测试就是给任务分诊。

---

## 21.11 本章掌握标准

| 你需要能做到 | 判断方式 |
|---|---|
| 设计正例 | 能覆盖多种正确表达 |
| 设计反例 | 能防止误触发 |
| 设计近似场景 | 能判断边界 |
| 设计模糊处理 | 能决定澄清、假设执行或拒绝 |
| 优化 description | 能通过测试反推触发描述是否清楚 |

---

# 第 22 章：输出测试｜格式、字段、完整性、约束

## 22.1 本章核心

> 输出测试解决的是：Agent 的交付物是否符合输出契约。

输出测试不负责判断内容“是否有深度”，而是先判断：

```text
结构是否完整
字段是否齐全
格式是否正确
约束是否遵守
产物是否存在
```

---

## 22.2 输出契约是什么

输出契约就是 Agent 交付结果必须满足的结构约定。

| 契约类型 | 例子 |
|---|---|
| 格式契约 | 必须输出 Markdown |
| 字段契约 | 必须包含 summary、issues、recommendations |
| 层级契约 | H1/H2/H3 层级不能乱 |
| 内容契约 | 必须包含背景、核心概念、误区、自检 |
| 文件契约 | 必须生成 .md 文件 |
| 命名契约 | 文件名包含主题和章节范围 |
| 语言契约 | 必须中文 |
| 风格契约 | 信息密度高，少长段落，多表格 |
| 禁止契约 | 不混入未讲章节 |

---

## 22.3 输出测试的常见检查项

| 检查项 | 例子 |
|---|---|
| Required Sections | 是否有总览、章节、总结、自检 |
| Required Fields | JSON 字段是否齐全 |
| Type Check | 字段类型是否正确 |
| Length Check | 文案是否超过字符限制 |
| Language Check | 是否使用指定语言 |
| Format Check | Markdown 是否规范 |
| Link Check | 文件链接是否存在 |
| Scope Check | 是否超出范围 |
| Constraint Check | 是否违反禁止项 |
| Artifact Check | 是否真的生成文件 |

---

## 22.4 结构化输出测试

如果输出是 JSON，可以做 schema 测试：

```json
{
  "title": "string",
  "summary": "string",
  "risks": ["string"],
  "recommendations": ["string"],
  "confidence": "low | medium | high"
}
```

测试规则：

| 测试 | 说明 |
|---|---|
| 字段存在 | title、summary、risks 必须存在 |
| 类型正确 | risks 必须是数组 |
| 枚举合法 | confidence 只能是 low / medium / high |
| 非空 | title 不能为空 |
| 可解析 | 必须是合法 JSON |

---

## 22.5 Markdown 输出测试

以 llm-wiki 沉淀文件为例：

| 测试项 | 要求 |
|---|---|
| H1 标题 | 只有一个 H1 |
| H2 章节 | 每章有独立 H2 |
| 表格规范 | 表格列数一致 |
| 代码块闭合 | 所有代码块必须闭合 |
| 文件名 | 包含主题、阶段、章节 |
| 链接 | 下载链接可访问 |
| 范围 | 不包含下一阶段内容 |
| 结尾 | 有阶段总结和掌握标准 |

---

## 22.6 输出完整性测试

完整性不是越长越好，而是该有的部分不能少。

| 任务 | 完整性要求 |
|---|---|
| 课程章节 | 概念、原理、案例、误区、自检 |
| Skill 评估 | 格式、触发、执行、资源、测试、改进 |
| 广告分析 | 数据、现象、原因、假设、动作、风险 |
| 文案输出 | 英文标题、英文正文、中文标题、中文正文、FABE |
| 图片提示词 | 主体、场景、构图、风格、比例、负面约束 |

---

## 22.7 输出约束测试

约束测试用于防止 Agent 忽略用户要求。

| 约束 | 测试方式 |
|---|---|
| “不要太长” | 检查段落长度和总字数 |
| “必须中文” | 检查语言 |
| “输出 3 个版本” | 检查数量 |
| “不要解释” | 检查是否包含额外说明 |
| “不要混入下一阶段” | 检查标题和内容范围 |
| “先输出课程再生成 md” | 检查执行顺序 |

---

## 22.8 常见误区

| 误区 | 问题 | 正确做法 |
|---|---|---|
| 输出看起来完整就行 | 可能漏字段 | 用契约检查 |
| Markdown 无法测试 | 很多结构可测 | 检查标题、表格、链接、代码块 |
| 只测最终文本 | 文件产物可能没生成 | 同时测 artifact |
| 输出测试等于质量评估 | 只解决形式合规 | 质量还要 Eval |
| 用户约束靠模型记住 | 模型可能漏 | 约束要变成检查项 |

---

## 22.9 Feynman 解释

输出测试像收货验货。

你订了 5 件商品，收货时先检查：

```text
数量对不对
型号对不对
包装有没有破
配件有没有少
发票有没有给
```

这不代表商品质量一定最好，但能先确认基本交付没有缺失。

---

## 22.10 本章掌握标准

| 你需要能做到 | 判断方式 |
|---|---|
| 定义输出契约 | 能列出格式、字段、文件、语言、范围 |
| 写 schema 测试 | 能检查结构化输出 |
| 写 Markdown 测试 | 能检查标题、表格、链接、代码块 |
| 检查完整性 | 能判断该有的部分是否缺失 |
| 检查用户约束 | 能把用户要求转成测试项 |

---

# 第 23 章：过程评估｜是否绕流程、是否遗漏检查

## 23.1 本章核心

> 过程评估解决的是：Agent 是否按正确流程完成任务，而不是只看最终答案像不像完成了。

很多 Agent 的问题是：

```text
最终输出看起来不错，但过程是错的。
```

例如：

| 场景 | 表面结果 | 真实问题 |
|---|---|---|
| Coding Agent | 输出“已完成” | 没跑测试 |
| Skill Agent | 生成 SKILL.md | 没做触发测试 |
| 知识沉淀 Agent | 输出总结 | 没生成文件 |
| 研究 Agent | 给出结论 | 没查最新资料 |
| 客服 Agent | 写了回复 | 没核对 case 事实 |

---

## 23.2 为什么要评估过程

Agent 的失败常常发生在过程层：

| 过程缺陷 | 后果 |
|---|---|
| 没读关键文件 | 输出基于错误上下文 |
| 没先计划 | 执行范围失控 |
| 没跑测试 | 假完成 |
| 没看 diff | 改错文件不可见 |
| 没做风险检查 | 高风险动作失控 |
| 没处理失败 | 错误被掩盖 |
| 没做回归 | 老问题复发 |

---

## 23.3 过程评估看什么

| 维度 | 评估问题 |
|---|---|
| 任务入口 | 是否形成 Task Brief |
| 上下文审计 | 是否读取必要资料 |
| 计划 | 是否先拆解步骤 |
| 工具调用 | 是否用对工具、顺序是否正确 |
| 文件操作 | 是否先读后写、是否限制目录 |
| 测试 | 是否运行必要测试 |
| 失败处理 | 是否处理错误，而不是忽略 |
| 质量门禁 | 是否通过验收再交付 |
| 变更审查 | 是否输出 diff 或变更摘要 |
| 沉淀 | 是否把失败和经验写入规则 |

---

## 23.4 Trace 是过程评估的基础

没有 trace，就只能看最终答案。

过程评估需要记录：

```text
读了哪些上下文
调用了哪些工具
每次工具参数是什么
工具返回了什么
执行到哪个步骤
哪些检查通过 / 失败
Agent 如何处理失败
最终为什么判断完成
```

OpenAI Agents 文档把 tracing、evals、guardrails、human review 都作为 Agent 工作流的重要构件；guardrails 可以自动检查输入、输出或工具行为，human review 可以让运行暂停并等待批准或拒绝。对质量工程来说，这些机制共同构成过程控制与过程审查的基础。

---

## 23.5 过程评估 Rubric

| 维度 | 0 分 | 1 分 | 2 分 |
|---|---|---|---|
| 任务理解 | 未澄清目标 | 部分理解 | 明确目标和范围 |
| 上下文读取 | 未读取 | 读取不完整 | 读取必要上下文 |
| 执行计划 | 无计划 | 计划粗略 | 步骤清晰、可验证 |
| 工具使用 | 乱用 / 不用 | 部分合理 | 顺序正确、参数正确 |
| 测试验证 | 未验证 | 简单自检 | 外部测试 / eval 通过 |
| 失败处理 | 忽略失败 | 简单重试 | 分类、修复、重验 |
| 风险控制 | 无控制 | 提到风险 | 有审批 / 回滚设计 |
| 交付判断 | 自称完成 | 部分满足 | 按 DoD 完成 |

---

## 23.6 过程评估与输出测试的区别

| 维度 | 输出测试 | 过程评估 |
|---|---|---|
| 看什么 | 最终产物 | 任务执行路径 |
| 目标 | 检查交付物格式和完整性 | 检查行为是否可靠 |
| 例子 | Markdown 文件是否存在 | 是否先讲完再生成文件 |
| 风险 | 形式合格但过程错误 | 过程正确但输出仍需质量评估 |

---

## 23.7 迁移到 Codex 场景

你之前提到：明明有 PRD，Codex 执行时仍然降低标准、绕过流程、未完成就宣传完成。

过程评估应该检查：

| 检查项 | 目的 |
|---|---|
| 是否完整读取 PRD | 防止漏需求 |
| 是否生成执行计划 | 防止直接乱改 |
| 是否列出验收标准 | 防止假完成 |
| 是否限制修改范围 | 防止改错文件 |
| 是否运行测试 | 防止没验证 |
| 是否输出 diff | 防止不可审查 |
| 是否说明未完成项 | 防止宣传完成 |
| 是否更新 changelog | 防止不可追踪 |

---

## 23.8 常见误区

| 误区 | 问题 | 正确做法 |
|---|---|---|
| 最终答案好就行 | 过程错误会导致不可复现 | 同时看 trace |
| Agent 自称跑了测试就信 | 可能虚构 | 要看测试输出 |
| 过程评估太麻烦 | 复杂任务必须评估过程 | 只对关键流程做评估 |
| 只评估成功路径 | 失败处理更重要 | 评估异常分支 |
| 看不到过程就算了 | 那就无法工程化 | 必须设计日志和 trace |

---

## 23.9 Feynman 解释

过程评估像检查厨师做菜流程。

菜端上来好像能吃，但你还要知道：

```text
食材有没有过期
有没有洗干净
有没有按温度加热
有没有交叉污染
有没有试毒式品控
```

只看成品，不看过程，很难保证稳定质量。

---

## 23.10 本章掌握标准

| 你需要能做到 | 判断方式 |
|---|---|
| 设计过程检查点 | 能列出关键步骤是否执行 |
| 使用 trace 思维 | 能要求记录工具、步骤、失败 |
| 判断绕流程 | 能识别跳过计划、测试、diff、验收 |
| 设计过程 rubric | 能给执行过程评分 |
| 迁移到 Codex | 能防止“未完成就宣称完成” |

# 第 24 章：结果评估｜是否真的解决问题

## 24.1 本章核心

> 结果评估解决的是：Agent 的最终产物是否真正满足用户目标和业务目标。

输出测试看“形式是否合格”。

过程评估看“是否按流程做”。

结果评估看：

```text
这件事是否真的被解决了？
```

---

## 24.2 结果评估和输出测试的区别

| 维度 | 输出测试 | 结果评估 |
|---|---|---|
| 问题 | 交付物是否符合结构 | 交付物是否有用 |
| 判断 | 字段、格式、完整性 | 目标达成、决策价值、用户收益 |
| 示例 | 文案有标题和正文 | 文案是否能提高转化可能性 |
| 示例 | md 文件已生成 | md 是否适合长期沉淀和复用 |
| 示例 | 报告有数据表 | 报告是否给出可执行诊断 |

---

## 24.3 结果评估的四层

| 层 | 评估问题 |
|---|---|
| 目标层 | 是否回答了用户真正的问题 |
| 任务层 | 是否完成了本次任务要求 |
| 业务层 | 是否能产生实际价值 |
| 迁移层 | 是否能复用到下次任务 |

---

## 24.4 结果评估 Rubric 示例：知识课程输出

| 维度 | 评分问题 | 权重 |
|---|---|---:|
| 准确性 | 概念是否讲对 | 20% |
| 完整性 | 是否覆盖本章必要内容 | 20% |
| 可理解性 | 是否符合费曼教学法 | 15% |
| 结构性 | 是否 MECE、层级清楚 | 15% |
| 可迁移性 | 是否能迁移到 Agent 工程 | 15% |
| 可沉淀性 | 是否能放入 llm-wiki | 15% |

---

## 24.5 结果评估 Rubric 示例：A+ 文案 Agent

| 维度 | 评分问题 |
|---|---|
| 差异化 | 是否体现竞品没有的卖点 |
| 用户收益 | 是否站在父母购买理由上 |
| 转化力度 | 是否短、直、强 |
| 准确性 | 是否没有夸大和错误承诺 |
| 语言自然度 | 是否符合美国市场表达 |
| 结构合规 | 是否包含标题、正文、FABE |

---

## 24.6 结果评估 Rubric 示例：广告分析 Agent

| 维度 | 评分问题 |
|---|---|
| 数据正确 | 是否使用正确时间范围和指标 |
| 诊断清楚 | 是否区分 CTR、CVR、CPC、ACOS 原因 |
| 假设合理 | 是否列出可验证假设 |
| 动作可执行 | 是否给出具体 bid / placement / keyword 操作 |
| 风险明确 | 是否说明动作副作用 |
| 复盘闭环 | 是否定义观察周期和判断标准 |

---

## 24.7 结果评估的数据来源

| 来源 | 适用 |
|---|---|
| 用户反馈 | 主观满意度、是否解决问题 |
| 专家评审 | 专业质量判断 |
| 业务指标 | CTR、CVR、ACOS、转化率 |
| A/B 测试 | 比较两个方案效果 |
| 回归测试 | 是否破坏旧能力 |
| 线上监控 | 真实使用表现 |
| 人工抽检 | 高价值任务最终验收 |

---

## 24.8 常见误区

| 误区 | 问题 | 正确做法 |
|---|---|---|
| 形式完整等于结果好 | 可能只是漂亮废话 | 用业务目标评估 |
| 用户没批评就算好 | 用户可能没发现问题 | 建立 rubric |
| 结果评估只靠主观感觉 | 不稳定 | 结合评分、样本、指标 |
| 所有任务都用一个 rubric | 不同任务质量标准不同 | 按任务类型设计 |
| 一次评估就定论 | 结果可能随场景变化 | 持续评估 |

---

## 24.9 Feynman 解释

结果评估像判断一把刀好不好。

输出测试看：

```text
刀有没有刀柄、刀刃、包装。
```

过程评估看：

```text
刀是不是按正确工艺制造。
```

结果评估看：

```text
这把刀到底锋不锋利、耐不耐用、适不适合用户场景。
```

---

## 24.10 本章掌握标准

| 你需要能做到 | 判断方式 |
|---|---|
| 区分形式合格和结果有效 | 能指出“看起来完整但没价值”的输出 |
| 设计结果 rubric | 能按任务类型拆质量维度 |
| 引入业务指标 | 能用真实指标验证 Agent 价值 |
| 结合人工评审 | 能知道哪些判断需要专家 |
| 设计持续评估 | 能让结果评估进入迭代循环 |

# 第 25 章：Regression Harness｜防止改好一个、弄坏三个

## 25.1 本章核心

> Regression Harness 的作用，是防止 Agent、Prompt、Skill、Workflow、模型或工具升级后，旧能力退化。

Agent 系统会不断变化：

```text
换模型
改 prompt
加工具
改 workflow
更新 memory
增加 skill
修改 schema
优化输出格式
```

每次变化都有可能：

```text
新问题解决了，旧能力坏了。
```

这就是回归风险。

---

## 25.2 为什么 Agent 特别需要回归保护

| 原因 | 说明 |
|---|---|
| 输出非确定 | 同一输入可能有不同输出 |
| Prompt 牵一发动全身 | 改一条规则影响多个场景 |
| 工具链复杂 | 一个工具变化影响 workflow |
| Memory 会污染 | 新记忆可能影响旧任务 |
| 模型升级不可控 | 新模型风格和判断可能变化 |
| 多 Skill 易冲突 | 新 skill 可能抢占旧 skill 触发 |

---

## 25.3 Regression Harness 的组成

| 组成 | 作用 |
|---|---|
| Regression Dataset | 历史关键样本和失败样本 |
| Baseline Outputs | 旧版本表现 |
| Candidate Outputs | 新版本表现 |
| Comparison Evaluator | 比较是否退化 |
| Failure Tracker | 记录回归失败 |
| CI Trigger | 每次变更自动运行 |
| Approval Gate | 退化时阻止合并 |

LangSmith 的 evaluation 文档明确将 regression tests 作为 pre-deployment testing 的一类，并支持在开发中比较版本、benchmark 和捕获 regressions。这类机制正是 Agent Regression Harness 的工程化形态。

---

## 25.4 哪些内容应该进入回归集

| 样本类型 | 原因 |
|---|---|
| 最常用任务 | 不能因为优化边缘场景破坏核心能力 |
| 高价值任务 | 失败成本高 |
| 历史失败案例 | 防止同样错误复发 |
| 用户明确纠正过的案例 | 代表真实偏好 |
| 边界触发案例 | 防止误触发 / 漏触发 |
| 高风险工具案例 | 防止越权和副作用 |
| 格式严格案例 | 防止输出结构退化 |

---

## 25.5 回归测试的层级

| 层级 | 测什么 |
|---|---|
| Input Regression | 触发和路由是否退化 |
| Output Regression | 输出格式和字段是否退化 |
| Process Regression | 流程是否仍然执行 |
| Tool Regression | 工具调用是否仍然正确 |
| Quality Regression | 内容质量是否下降 |
| Safety Regression | 安全约束是否失效 |
| Business Regression | 业务效果是否变差 |

---

## 25.6 回归比较方式

| 方式 | 适用场景 |
|---|---|
| Exact Match | 固定答案、字段、格式 |
| Schema Match | JSON / Markdown 结构 |
| Rule Check | 禁止项、长度、语言 |
| Pairwise Comparison | 新旧版本谁更好 |
| Score Delta | 新版本分数是否下降 |
| Human Review | 高价值样本 |
| Business Metric | 线上真实表现 |

---

## 25.7 失败案例如何变成回归测试

| 失败 | 回归测试 |
|---|---|
| Agent 把概念讲太泛 | 加“必须包含边界、区别、案例”检查 |
| Skill 误触发翻译任务 | 加 negative trigger case |
| Codex 没跑测试就说完成 | 加 process gate 测试 |
| md 文件漏掉自检部分 | 加 required section 测试 |
| A+ 文案太长 | 加字符限制测试 |
| 广告分析没给动作 | 加 actionable recommendation 检查 |

---

## 25.8 回归保护的工作流

```text
发现失败
→ 记录失败输入
→ 记录错误输出
→ 分析失败原因
→ 修复 Agent / Prompt / Tool / Workflow
→ 将失败样本加入回归集
→ 每次变更自动运行
→ 如果失败，禁止交付
```

---

## 25.9 常见误区

| 误区 | 问题 | 正确做法 |
|---|---|---|
| 修完当前问题就结束 | 下次还会复发 | 加入回归集 |
| 只保留成功样本 | 看不到旧问题 | 失败样本更重要 |
| 回归集越大越好 | 维护成本高 | 先保留高价值样本 |
| 只比平均分 | 关键样本退化会被掩盖 | 看红线案例 |
| 模型升级不用回归 | 风格和行为会变化 | 升级必须跑回归 |

---

## 25.10 Feynman 解释

Regression Harness 像汽车维修后的复检。

修好了刹车，不代表方向盘、油门、灯光没坏。每次改动后，都要检查核心功能是否仍然正常。

Agent 也是一样。

---

## 25.11 本章掌握标准

| 你需要能做到 | 判断方式 |
|---|---|
| 理解回归风险 | 能解释为什么改 prompt 可能破坏旧能力 |
| 建立回归集 | 能选择核心、失败、边界、高风险样本 |
| 设计回归指标 | 能比较新旧版本是否退化 |
| 把失败沉淀为测试 | 能让错误不再重复发生 |
| 接入交付流程 | 能让回归失败阻止发布或合并 |

# 第 26 章：Quality Gate｜任务完成前的质量门禁

## 26.1 本章核心

> Quality Gate 的作用，是在 Agent 宣称完成、交付、写入、发布、合并之前，强制检查是否满足完成标准。

没有 Quality Gate，Agent 很容易：

```text
没做完就说做完
没测试就说通过
没生成文件就说已生成
没核对事实就给结论
没处理失败就进入下一步
```

OpenAI Agents 文档中的 guardrails 可以自动验证输入、输出或工具行为，human review 可以让敏感动作暂停等待审批；这类机制本质上就是质量门禁的一部分：决定一次 run 是继续、暂停还是停止。

---

## 26.2 Quality Gate 和 Test / Eval 的关系

| 模块 | 作用 |
|---|---|
| Test | 检查硬性规则 |
| Eval | 评估质量表现 |
| Regression | 防止旧能力退化 |
| Guardrail | 阻止违规输入、输出或工具行为 |
| Human Review | 审批高风险动作 |
| Quality Gate | 汇总上述结果，决定能否进入下一步 |

简单说：

```text
Test / Eval / Regression / Guardrail 是检查器；
Quality Gate 是放行规则。
```

---

## 26.3 Quality Gate 的位置

| Gate | 位置 | 检查内容 |
|---|---|---|
| Entry Gate | 任务开始前 | 目标、范围、输入是否清楚 |
| Plan Gate | 执行前 | 计划是否合理、风险是否识别 |
| Context Gate | 推理前 | 是否读取必要上下文 |
| Tool Gate | 工具调用前 | 工具是否必要、参数是否安全 |
| Write Gate | 写文件前 | 路径、权限、覆盖风险 |
| Test Gate | 交付前 | 测试是否通过 |
| Eval Gate | 交付前 | 质量是否达标 |
| Regression Gate | 合并前 | 是否破坏旧能力 |
| Approval Gate | 高风险动作前 | 是否有人批准 |
| Delivery Gate | 最终交付前 | 是否满足 Definition of Done |

---

## 26.4 Definition of Done

Quality Gate 必须依赖 Definition of Done。

没有 DoD，就不知道门禁该检查什么。

### 示例：阶段课程输出 DoD

| 条件 | 是否必须 |
|---|---:|
| 覆盖指定章节 | 必须 |
| 不混入下一阶段 | 必须 |
| 每章有核心概念、误区、自检 | 必须 |
| 输出结构清晰 | 必须 |
| 生成 .md 文件 | 必须 |
| 提供下载链接 | 必须 |
| 语言为中文 | 必须 |
| 适合 llm-wiki 沉淀 | 必须 |

---

## 26.5 Quality Gate 的放行策略

| 结果 | 动作 |
|---|---|
| 全部通过 | 放行 |
| 非关键问题 | 放行但记录 |
| 关键问题失败 | 阻止交付 |
| 高风险不确定 | 暂停，进入人工审批 |
| 测试缺失 | 不允许宣称完成 |
| 评估低于阈值 | 回到修复流程 |
| 回归失败 | 不允许合并或发布 |

---

## 26.6 红线门禁

红线门禁是绝对不能放行的条件。

| 场景 | 红线 |
|---|---|
| 文件任务 | 文件未生成，不得说已生成 |
| Coding Agent | 测试未运行，不得说测试通过 |
| 平台 case | 事实未核对，不得发送 |
| 广告操作 | 未确认，不得自动修改预算 |
| 知识沉淀 | 范围错误，不得归档 |
| Skill 创建 | 触发测试失败，不得标记完成 |
| 高风险工具 | 未审批，不得执行 |

---

## 26.7 Quality Gate 模板

```text
Quality Gate Checklist

1. Task Gate
- [ ] 任务目标清楚
- [ ] 范围边界清楚
- [ ] 输出产物清楚

2. Context Gate
- [ ] 必要上下文已读取
- [ ] 上下文没有明显冲突
- [ ] 过期资料已排除

3. Execution Gate
- [ ] 已按流程执行
- [ ] 工具调用合理
- [ ] 文件写入范围正确

4. Test Gate
- [ ] 输入测试通过
- [ ] 输出测试通过
- [ ] 必要脚本 / 测试已运行

5. Eval Gate
- [ ] 结果满足 rubric
- [ ] 关键样本未退化
- [ ] 风险已说明

6. Delivery Gate
- [ ] 满足 Definition of Done
- [ ] 未完成项已明确说明
- [ ] 可交付产物已生成
```

## 26.9 常见误区

| 误区 | 问题 | 正确做法 |
|---|---|---|
| Quality Gate 只是 checklist | checklist 不执行就没用 | 必须绑定放行规则 |
| 自检通过就能交付 | 自检可能幻觉 | 需要外部测试 / eval |
| 门禁越多越好 | 过重会降低效率 | 按风险分级 |
| 非关键问题也阻塞 | 影响迭代速度 | 区分 blocker 和 warning |
| 没通过也可以先说完成 | 破坏可信度 | 明确未完成项 |

---

## 26.10 Feynman 解释

Quality Gate 像工厂出货口。

产品在生产线上做了很多步骤，但出货前必须检查：

```text
型号对不对
功能是否正常
质检是否通过
包装是否完整
是否有召回风险
能不能发给客户
```

没过门禁，就不能出货。

Agent 也是一样：没过质量门禁，就不能说完成。

| 你需要能做到 | 判断方式 |
|---|---|
| 设计 DoD | 能定义什么叫完成 |
| 设计多层 Gate | 能在入口、计划、工具、测试、交付前设门禁 |
| 区分 blocker / warning | 能按风险决定是否放行 |
| 接入 Test / Eval / Regression | 能把检查结果转成放行规则 |
| 防止假完成 | 能明确“未通过不得宣称完成” |

# 阶段四总结

## 1. 用一句话总结

> 阶段四的核心是：把 Agent 的“看起来做了”变成“经过测试、评估、回归保护和质量门禁后，确认可以交付”。

---

## 2. Test / Eval / Quality Gate 总关系

```text
Test Harness
负责硬规则：有没有做对

Eval Harness
负责软质量：做得好不好

Regression Harness
负责稳定性：新版本有没有破坏旧能力

Quality Gate
负责放行：是否允许进入下一步或交付
```

---

## 3. 八章关系图

```text
19 Test Harness 基础
→ 建立可测试环境

→ 建立质量评估体系

→ 判断该不该触发

22 输出测试

→ 判断是否按流程执行

→ 防止旧能力退化

26 Quality Gate

```

---

## 4. 阶段四最重要的判断

```text
Agent 工程里的质量，不是靠“模型自觉”，也不是靠“看起来不错”。

必须拆成：

输入是否该触发
输出是否合规
过程是否正确
结果是否有效
旧能力是否退化
是否满足完成标准
```

## 5. 阶段四掌握标准

|---|---|

| 过程评估 | 能识别跳步骤、漏检查、假完成 |
| 结果评估 | 能判断任务是否真正解决业务问题 |
| Regression Harness | 能把历史失败转成回归测试 |
| Quality Gate | 能设定未通过不得交付的门禁规则 |

# Codex 产品方案工作流

日期：2026-06-16

页面类型：Agent Playbook

适用范围：NexGaios 内部所有由 Codex 参与的产品定位、PRD、功能方案、用户流程、路线图、竞品分析、需求拆解、产品评审材料和产品决策沉淀任务。

## 1. 这页是什么

这页定义 Codex 在 NexGaios 做产品方案时如何使用 GBrain 作为第二大脑。

Codex 做产品方案不能只依赖通用产品方法论。它必须先理解 NexGaios 的公司阶段、Terry 的判断口径、目标用户、已有产品能力、历史决策、业务约束和当前项目状态，再输出方案。

本工作流的目标是让 Codex 产出的产品方案不只是“看起来完整”，而是能真正服务 NexGaios 的 AI 转型、业务落地和知识库产品化。

## 2. 适用场景

只要任务符合下面任一条件，Codex 都应该进入本工作流：

1. 写产品方案、PRD、功能说明、页面流程或用户故事。
2. 设计 GBrain、权限后台、MCP 接入、知识库内容层、同步流程等产品能力。
3. 讨论 NexGaios AI 转型方案、Agent 落地方式、第二大脑架构。
4. 做竞品、市场、用户需求、商业化路径、SaaS 成熟度分析。
5. 把 Terry 的想法整理成产品决策、路线图、里程碑或执行计划。

## 3. 做产品方案前必须检索的第二大脑内容

Codex 开始做产品方案前，应优先从 GBrain 检索以下页面或目录。若页面还不存在，应在方案中标记“缺少该类长期记忆”，并建议后续补齐。

### 3.1 公司背景和战略

优先检索：

1. `company/context`
2. `company/strategy`
3. `company/business-model`
4. `company/terry-principles`

目的：

1. 理解 NexGaios 当前做什么、为什么做、优先级是什么。
2. 理解 Terry 对效率、落地、商业化、AI 转型的判断标准。
3. 避免产出泛泛而谈的产品方案。

### 3.2 目标用户和使用场景

优先检索：

1. `product/users`
2. `product/personas`
3. `product/use-cases`
4. `projects/current`

目的：

1. 明确产品是给 Terry、内部团队、外部客户、运营人员还是工程成员使用。
2. 明确用户在什么场景下使用，真实痛点是什么。
3. 区分内部效率工具、商业 SaaS 产品、客户交付方案和临时脚本。

### 3.3 当前产品能力

优先检索：

1. `product/current-capabilities`
2. `engineering/architecture`
3. `engineering/permissions`
4. `agent-playbooks/codex-coding-workflow`

目的：

1. 确认现在已经有什么，不重复设计已经完成的东西。
2. 确认哪些能力还只是文档，哪些已经真实部署。
3. 明确产品方案是否会影响架构、权限、部署或内容同步。

### 3.4 历史产品决策

优先检索：

1. `decisions/product`
2. `decisions/gbrain`
3. `decisions/knowledge-architecture`
4. `aiwork/gbrain/permission-system-handoff-2026-06-11`

目的：

1. 理解为什么 source 按公司层级设计，而不是按部门拆散。
2. 理解为什么先做权限系统，再做知识库内容层。
3. 理解 GBrain 的定位是公司级 Agent Memory OS，而不是普通文档仓库。

### 3.5 当前项目状态和下一步

优先检索：

1. `projects/gbrain/current-status`
2. `projects/gbrain/next-steps`
3. `agent-playbooks/*`

目的：

1. 确认当前阶段重点。
2. 避免产品方案脱离当前资源和执行节奏。
3. 把方案拆成可执行下一步，而不是只输出远景蓝图。

### 3.6 外部市场和竞品

当涉及最新竞品、价格、政策、市场动态、模型能力、平台功能时，Codex 必须联网验证，优先引用官网、官方文档、公开产品页面或可信来源。

目的：

1. 避免用过时信息做产品判断。
2. 区分事实、推断和 Terry 的战略选择。
3. 让产品方案能服务真实商业判断。

## 4. 产品方案必须回答的问题

Codex 输出产品方案时，至少要回答：

1. 这个方案解决什么问题？
2. 目标用户是谁？
3. 用户现在为什么痛？
4. 为什么现在要做？
5. 这件事对 Terry 和 NexGaios 的价值是什么？
6. 最小可行版本是什么？
7. 哪些能力必须真实落地，哪些只是后续增强？
8. 需要哪些工程、内容、权限、运营资源？
9. 成功标准是什么？
10. 最大风险和验证方式是什么？

如果这些问题无法回答，Codex 应明确指出缺少哪些输入，而不是强行补完。

## 5. 产品方案的推荐结构

Codex 默认使用以下结构输出产品方案：

1. 背景
2. 问题
3. 目标用户
4. 核心洞察
5. 方案目标
6. MVP 范围
7. 用户流程
8. 功能模块
9. 权限和数据边界
10. 技术/内容依赖
11. 验收标准
12. 风险
13. 后续路线图
14. 需要 Terry 决策的问题

## 6. 做方案时必须遵守的判断原则

### 6.1 先场景，后功能

产品方案不能先堆功能列表。必须先明确用户场景和任务链路。

错误方式：

1. 先列大量页面、按钮、模块。
2. 把商业 SaaS 的通用能力直接套进当前系统。
3. 为了“完整”设计一堆当前无法维护的功能。

正确方式：

1. 先定义用户要完成的工作。
2. 再定义系统如何减少阻力。
3. 最后才定义页面、字段、按钮和接口。

### 6.2 先 MVP，后成熟形态

Codex 应把方案拆成：

1. 当前可落地版本
2. 小团队试点版本
3. 成熟商业 SaaS 版本

这样 Terry 能判断现在做什么、后面做什么，而不是被完整蓝图拖慢。

### 6.3 区分事实、判断和建议

产品方案中必须区分：

1. 已确认事实
2. Terry 已表达的判断
3. Codex 的推断
4. 需要验证的假设
5. 推荐下一步

这可以避免把模型的推测误当成公司决策。

### 6.4 保留 Terry 的决策权

涉及方向、商业化、组织流程、权限边界、成本、长期产品定位时，Codex 应给出清晰选项和取舍，不应替 Terry 做最终战略决定。

## 7. 做完产品方案后必须写回 GBrain 的内容

如果方案产生长期价值，Codex 必须写回 GBrain。

### 7.1 必须写回的内容

1. 新产品定位
2. 新功能方案
3. 新用户流程
4. 新 PRD 或产品 memo
5. Terry 明确拍板的决策
6. 被否决的方案和原因
7. MVP 范围
8. 验收标准
9. 后续路线图
10. 需要工程或内容团队执行的事项

### 7.2 推荐写回位置

根据内容类型写回：

1. 产品定位：`product/positioning`
2. 用户和场景：`product/users`、`product/use-cases`
3. 产品方案：`product/specs`
4. 路线图：`product/roadmap`
5. 决策记录：`decisions/product`
6. GBrain 项目状态：`projects/gbrain/current-status`
7. 模板：`templates/product`
8. Agent 工作规则：`agent-playbooks/product-planning-workflow`

## 8. 什么情况必须问 Terry

以下情况不得默认执行，必须先问 Terry：

1. 会改变产品长期定位。
2. 会改变目标用户。
3. 会把内部工具包装成外部商业 SaaS。
4. 会新增团队长期维护成本。
5. 会引入新的第三方平台或付费服务。
6. 会改变权限系统、成员接入或数据边界。
7. 会改变知识库顶层目录结构。
8. 会把未验证想法写成确定战略。
9. 会对外发布、销售或承诺某个产品能力。

## 9. 标准执行模板

Codex 做产品方案时，应按这个节奏：

1. 识别产品任务类型。
2. 检索 GBrain 中的公司、产品、项目和历史决策页面。
3. 若涉及市场或竞品，联网验证最新信息。
4. 复述当前已知事实和缺口。
5. 给出方案结构。
6. 输出 MVP 和成熟形态。
7. 标出需要 Terry 决策的问题。
8. 得到 Terry 反馈后更新方案。
9. 把最终方案、决策和后续行动写回 GBrain。

## 10. 当前第一版的边界

这份 playbook 是 Codex 产品方案场景的第一版。它目前服务于 GBrain 第二大脑、知识库内容层、权限后台、MCP 接入和 NexGaios AI 转型方案。

后续需要补充：

1. 产品 PRD 模板。
2. 产品 memo 模板。
3. 竞品分析模板。
4. 路线图模板。
5. Terry 产品判断原则页面。
6. GBrain 商业化定位页面。

## 11. 本页的维护规则

当 Terry 明确改变产品方向、目标用户、商业化路径或判断原则时，Codex 应更新本页。

当某个产品方案被 Terry 拍板、调整或否决时，Codex 应把决策写入 `decisions/product`，并在相关产品页面中同步当前状态。

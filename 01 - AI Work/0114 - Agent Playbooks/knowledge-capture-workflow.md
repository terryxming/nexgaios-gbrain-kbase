# Codex 内容沉淀工作流

日期：2026-06-16

页面类型：Agent Playbook

适用范围：NexGaios 内部所有由 Codex 参与的知识沉淀、会议纪要整理、业务判断记录、SOP 更新、项目复盘、产品决策、Amazon 运营经验、代码交付经验和第二大脑维护任务。

## 1. 这页是什么

这页定义 Codex 如何把一次对话、一次执行、一次判断、一次复盘，沉淀成 GBrain 里的长期企业知识。

企业知识库真正有价值的地方，不是把所有内容都塞进去，而是把“以后会反复用到的判断、流程、事实、模板、经验和上下文”整理成可检索、可复用、可更新的页面。

本工作流的目标是让 GBrain 从文件仓库变成 NexGaios 的第二大脑：Codex 每次工作前能检索它，每次工作后能补充它，让团队的经验不断累积，而不是散落在聊天记录、临时文件和个人记忆里。

## 2. 适用场景

只要任务符合下面任一条件，Codex 都应该进入本工作流：

1. Terry 做出了新的业务判断。
2. 团队完成了一次重要执行。
3. 某个流程被验证有效或无效。
4. 出现重复问题，需要沉淀解决方式。
5. 生成了可复用模板、SOP、清单或标准。
6. 完成了产品方案、代码交付、Amazon 运营、权限管理或客户交付任务。
7. 发现现有知识库页面过时、缺失或冲突。
8. 用户明确要求“写入知识库”“沉淀一下”“记下来”“以后照这个执行”。

## 3. 什么内容应该进入 GBrain

Codex 不应该把所有聊天内容都写进 GBrain。只有具备长期复用价值的内容才应该沉淀。

### 3.1 必须沉淀的内容

以下内容必须优先沉淀：

1. Terry 明确确认过的决策。
2. 公司战略、业务方向、阶段目标。
3. 组织角色、权限设计、职责分工。
4. 已验证的 SOP 和执行流程。
5. 重要项目交接文档。
6. 产品方案、PRD、路线图和评审结论。
7. Amazon 业务资料、SKU、Listing、广告、库存、复盘。
8. 代码架构、部署流程、运维规则、故障处理。
9. 客户交付标准、沟通模板、复盘记录。
10. Codex 和其他 Agent 的工作规则。

### 3.2 应该沉淀但需要整理的内容

以下内容可以沉淀，但必须先整理：

1. 临时讨论中的有效结论。
2. 模糊想法中的方向性判断。
3. 一次执行中的经验教训。
4. 多轮对话中逐步形成的方案。
5. 多个文件中的重复信息。
6. 外部资料中的关键摘录和启发。

整理时必须把原始表达转成结构化页面，不能直接粘贴长篇聊天记录。

### 3.3 不应该沉淀的内容

以下内容默认不写入 GBrain：

1. 明文密码、token、密钥、Cookie。
2. 临时错误日志中的敏感信息。
3. 未确认的猜测。
4. 与公司无关的闲聊。
5. 一次性命令输出。
6. 重复、过期、没有上下文的片段。
7. 无来源的外部事实。

如果某条信息有价值但包含敏感内容，必须脱敏后再写入。

## 4. 内容沉淀前必须回答的问题

Codex 在写入 GBrain 前，至少要回答：

1. 这条知识以后会被谁用到？
2. 它属于事实、决策、流程、模板、复盘、经验还是参考资料？
3. 它应该新建页面，还是更新已有页面？
4. 它应该放在哪个目录和 slug？
5. 它的来源是什么？
6. 它是否已经被 Terry 或负责人确认？
7. 它是否包含敏感信息？
8. 它是否会和已有页面冲突？
9. 它什么时候需要更新？
10. 它应该被哪些 playbook 或 SOP 引用？

如果无法回答这些问题，Codex 应先标记为草稿或待确认，而不是直接写成正式规则。

## 5. GBrain 内容类型

### 5.1 事实

事实是相对稳定的信息，例如公司背景、产品资料、SKU 信息、系统地址、架构说明。

写事实时必须：

1. 写清楚来源。
2. 写清楚更新时间。
3. 避免混入未经确认的判断。
4. 信息变化后及时更新。

### 5.2 决策

决策是 Terry 或负责人明确拍板的内容，例如采用某种权限模型、确定某个产品方向、决定某个流程。

写决策时必须包含：

1. 决策内容。
2. 决策人。
3. 决策日期。
4. 背景。
5. 为什么这样决定。
6. 影响范围。
7. 后续动作。

### 5.3 流程

流程是可以反复执行的步骤，例如 MCP 接入、权限开通、SOP 生成、Amazon Listing 优化。

写流程时必须包含：

1. 适用范围。
2. 触发条件。
3. 执行人。
4. 输入资料。
5. 操作步骤。
6. 检查清单。
7. 异常处理。
8. 输出物。

### 5.4 模板

模板是以后可以复用的结构，例如 PRD 模板、SOP 模板、复盘模板、Amazon Listing 模板。

写模板时必须：

1. 明确适用场景。
2. 保留可替换字段。
3. 给出填写说明。
4. 避免写死某个项目的临时信息。

### 5.5 复盘

复盘用于记录一次执行后的经验。

写复盘时必须包含：

1. 发生了什么。
2. 原计划是什么。
3. 实际结果是什么。
4. 做对了什么。
5. 做错了什么。
6. 下次怎么改。
7. 哪些规则或 SOP 需要更新。

## 6. 推荐目录和 slug 规则

### 6.1 推荐目录

根据内容类型写入：

1. 公司背景：`company`
2. 产品方案：`product`
3. Amazon 业务：`amazon`
4. 工程和代码：`engineering`
5. GBrain 自身：`aiwork/gbrain`
6. 权限系统：`aiwork/gbrain/permission-system`
7. 决策记录：`decisions`
8. SOP：`sop`
9. 模板：`templates`
10. 复盘：`retrospectives`
11. 交接：`handoffs`
12. Agent 工作流：`agent-playbooks`

### 6.2 slug 命名规则

slug 应该：

1. 使用小写英文、数字和连字符。
2. 目录表达领域，文件名表达主题。
3. 不使用空格、中文标点和随机编号。
4. 对长期页面使用稳定名称。
5. 对日期型记录使用 `yyyy-mm-dd`。

示例：

1. `company/context`
2. `product/gbrain-permission-admin`
3. `amazon/listing-optimization-template`
4. `aiwork/gbrain/permission-system-handoff-2026-06-11`
5. `retrospectives/gbrain-mcp-onboarding-2026-06-16`

## 7. 内容页面的标准结构

除非内容类型有专门模板，Codex 新建知识页时默认使用下面结构。

### 7.1 标题和元信息

必须包含：

1. 页面标题
2. 日期
3. 页面类型
4. 负责人或来源
5. 状态：草稿、待确认、已确认、已废弃

### 7.2 背景

说明为什么需要这页，解决什么问题。

### 7.3 核心内容

写清楚事实、规则、流程、方案或判断。

### 7.4 使用方式

说明 Codex 或团队成员在什么场景下应该检索和使用这页。

### 7.5 待确认事项

列出尚未确认的信息，避免把不确定内容混入正式知识。

### 7.6 关联页面

列出相关的 SOP、决策、模板、复盘或 playbook。

### 7.7 维护规则

说明什么时候更新、由谁确认、哪些变化会影响本页。

## 8. 内容沉淀的质量标准

### 8.1 可检索

标题、slug 和正文必须包含真实场景中的关键词。团队成员或 Codex 搜索时应该能找到。

### 8.2 可复用

页面不应该只服务一次对话。它应该能在未来相似任务中再次使用。

### 8.3 可更新

页面应有更新时间、状态和维护规则。过期内容不能悄悄留在知识库里。

### 8.4 可追溯

重要事实和决策必须能追溯来源。至少要说明来自 Terry、业务负责人、系统验证、项目执行还是外部资料。

### 8.5 低噪音

GBrain 不是垃圾桶。Codex 应避免把过程性闲聊、重复内容、未确认猜测写进去。

## 9. 写入 GBrain 前的检查清单

写入前必须确认：

1. 页面有清晰标题。
2. slug 符合命名规则。
3. 内容属于正确目录。
4. 页面类型明确。
5. 已标记来源和日期。
6. 敏感信息已脱敏。
7. 待确认事项没有伪装成事实。
8. 与已有页面没有明显冲突。
9. 关联页面已列出。
10. 写入后可以通过 `get_page` 读回。

## 10. 写入 GBrain 后必须做什么

Codex 写入 GBrain 后必须：

1. 执行读回验证。
2. 确认页面标题存在。
3. 确认关键章节存在。
4. 汇报 source、slug 和本地文件路径。
5. 如果是重要规则或 SOP，应说明它会影响哪些后续任务。
6. 如果发现旧页面冲突，应建议合并、更新或废弃。

## 11. 与其他 Agent Playbook 的关系

内容沉淀工作流是所有其他 playbook 的收口动作。

### 11.1 写代码后

应沉淀：

1. 架构变化。
2. 部署流程。
3. 重要 bug 和修复方式。
4. 新增配置和权限。
5. 可复用工程经验。

### 11.2 做产品方案后

应沉淀：

1. 产品决策。
2. 用户场景。
3. PRD。
4. 路线图。
5. 被 Terry 确认的判断。

### 11.3 做 Amazon 业务后

应沉淀：

1. SKU 资料。
2. Listing 规则。
3. 关键词池。
4. 广告复盘。
5. 库存和售后经验。

### 11.4 生成 SOP 后

应沉淀：

1. SOP 正式版本。
2. 检查清单。
3. 异常处理。
4. 维护规则。
5. 执行后的复盘。

## 12. 什么情况必须问 Terry 或负责人

以下情况不得直接写成正式知识：

1. 内容会影响公司战略。
2. 内容会改变组织权限或职责。
3. 内容涉及商业承诺。
4. 内容涉及 Amazon 价格、广告预算、库存计划或对外 Listing。
5. 内容涉及财务、法务、合规或平台政策。
6. 内容来自未经确认的外部信息。
7. 内容会覆盖已有重要决策。
8. 内容包含团队成员个人评价或敏感信息。

这类内容可以先写成草稿，并标记“待 Terry 或负责人确认”。

## 13. 标准执行模板

Codex 做内容沉淀时，应按这个节奏：

1. 判断这次工作是否产生可复用知识。
2. 判断内容类型：事实、决策、流程、模板、复盘或参考资料。
3. 检索 GBrain，确认是否已有相关页面。
4. 决定新建、更新、合并或标记废弃。
5. 选择目录和 slug。
6. 整理内容，去掉噪音和敏感信息。
7. 标记来源、日期、状态和待确认事项。
8. 写入 GBrain。
9. 读回验证。
10. 向 Terry 汇报写入结果和后续影响。

## 14. 当前第一版的边界

这份 playbook 是 Codex 内容沉淀场景的第一版。它目前主要解决什么时候沉淀、沉淀什么、放在哪里、怎么保证质量的问题。

它还不能替代：

1. Terry 对关键知识的最终确认。
2. 团队对业务事实的维护。
3. 定期知识库清理。
4. 自动化知识抽取和去重。
5. 完整的知识生命周期管理。

后续需要补充：

1. 标准知识页模板。
2. 决策记录模板。
3. 复盘模板。
4. 知识库清理 SOP。
5. 页面废弃和合并规则。
6. GBrain 内容审核机制。
7. Agent 自动写回策略。

## 15. 本页的维护规则

当 NexGaios 的知识库结构、source 设计、目录规范、权限模型或 Agent 使用方式发生变化时，Codex 应更新本页。

当团队发现某类知识反复找不到、写错位置、难以复用或噪音过多时，Codex 应把问题写入复盘，并更新本页或相关模板。

# Agent 资料提炼与 raw 回写 SOP

版本：v0.1.0

生命周期：Draft

日期：2026-06-22

适用范围：Agent 处理外部资料、对话结论、经验教训、业务判断和图片表格材料时的 raw 回写质量要求、结构要求、写后验证要求

## 目标

这份 SOP 用来约束 Agent 在处理外部资料、日常 Agent 产物、业务判断、经验教训和知识库材料时的回写 workflow。

回写不是把一次对话或一次输出保存下来，而是把有长期复用价值的信息整理成可检索、可复核、可编辑、可演化的知识资产。

这份 SOP 要避免以下问题：

- 没有充分阅读和思考就直接回写。
- 把多篇资料粗暴合并成流水账。
- 把日常 Agent 使用中产生的经验、方案、知识点、踩坑记录和决策遗漏掉。
- 生成多个副产物，污染 inbox。
- 本地 Markdown 与 GBrain DB raw 不一致。
- 标题、文件名和 frontmatter title 不一致。
- 把图片内容生成为不可编辑图片，而不是转成可维护的信息表格。
- 笔记格式粗糙，缺少阅读欲望。
- 信息组织没有章法，后续无法复用。
- 把未编译 raw 误当作正式知识。
- 回写失败后仍然向用户宣称已经完成。
- 异步路由或定时任务移动文件后，仍把旧 inbox 路径当作查看入口。
- 本地文件已经路由到领域 raw 目录，但 GBrain DB 仍停留在旧 inbox slug，未向用户说明同步状态。

## 核心理念：让知识产生复利

用户要求回写，本质上是希望知识不要“一次使用就结束”。

一条合格的回写笔记应该让未来的自己或未来的 Agent 更快进入上下文、更少重复犯错、更容易复用判断和流程。

因此，Agent 写回的不是聊天记录，而是经过整理的知识单元。

| 维度 | 要求 | 反例 |
|---|---|---|
| 客观中立 | 区分事实、来源、推理、建议和待确认事项 | 把未经确认的猜测写成结论 |
| 可持续进化 | 保留来源、适用边界、后续可更新点 | 写成一次性的执行流水账 |
| 可复用 | 未来遇到相似问题时能直接使用 | 只记录“今天做了什么” |
| 可检索 | 标题、标签、结构能支持搜索和路由 | 标题含糊、关键词缺失 |
| 可阅读 | 版式清晰、重点突出、表格可扫读 | 大段堆砌、粗糙标题、无层次 |

Agent 必须避免“泛泛而谈”。如果一条内容不能回答“未来谁会在什么场景下复用它”，就不应直接回写，或应先整理成更明确的知识单元。

## 适用范围

适用于 Agent 执行以下任务时：

1. 用户明确要求“回写”“沉淀”“记下来”“写入知识库”。
2. 用户认为某个 Agent 使用过程中产生的信息值得保留。
3. 用户要求提炼文章、网页、图片、表格、对话或执行经验。
4. 用户要求生成 SOP、流程、检查清单、复盘或知识笔记。
5. Agent 在任务中发现需要长期复用的经验、规则、决策或错误教训。

回写材料不局限于外部资料。以下内容只要具备长期复用价值，也必须按本 SOP 执行：

| 来源类型 | 示例 | 回写重点 |
|---|---|---|
| 外部资料 | 微信公众号文章、网页、PDF、截图、视频摘要 | 来源复核、关键事实、结构化提炼 |
| Agent 经验 | 某次任务中验证有效的操作方式 | 适用场景、步骤、注意事项 |
| 方案 | 产品方案、技术方案、运营打法 | 目标、约束、路径、风险 |
| 知识点 | 新学到的概念、规则、判断框架 | 定义、边界、使用方式 |
| 踩坑记录 | 工具调用失败、DB 与文件不一致、格式校验失败 | 触发条件、根因、修复流程、预防措施 |
| 决策 | 用户确认的业务判断、流程选择、取舍 | 决策背景、结论、理由、影响范围 |
| 偏好 | 用户明确表达的工作方式偏好 | 应如何影响后续 Agent 行为 |

不适用于直接修改 active wiki、直接修改 schema、直接标记 canonical 的场景；这些场景必须先生成建议或草案，并等待用户确认。

## 总原则

Agent 回写必须遵守六条底线：

1. 先理解，再组织，再回写。
2. 新知识默认先进入 GBrain DB raw，再物化到 `00 - raw/00 - inbox`。
3. 未编译 raw 只能作为即时记忆和证据补充，不能伪装成正式知识。
4. 回写笔记必须客观中立，不能把猜测、情绪或过程噪音包装成事实。
5. 笔记格式和内容设计必须有阅读欲望，充分使用 Markdown 与 Obsidian 能力。
6. 每次回写必须能被复核、能被编辑、能被路由、能被读回验证。

## 标准流程

### 1. 明确要沉淀的知识对象

回写前先判断：这次到底要沉淀什么知识对象。

不要先写正文。先给内容定性。

应回写的内容包括：

| 类型 | 判断标准 |
|---|---|
| 外部资料提炼 | 以后会复用，且有明确来源 |
| 业务规则 | 会影响后续执行判断 |
| SOP / 流程 | 可重复执行，能降低沟通成本 |
| 经验 | 某次执行中被验证有效 |
| 方案 | 未来可能复用或继续演化 |
| 知识点 | 能解释一个概念、规则或判断框架 |
| 复盘教训 | 能避免重复犯错 |
| 踩坑记录 | 能帮助未来快速定位和修复问题 |
| 决策 | 用户已确认，会影响后续行动 |
| 用户确认的偏好 | 会影响后续 Agent 工作方式 |
| 结构化表格 | 比原始截图更适合 Obsidian 保存和二次编辑 |

不应回写的内容包括：

| 类型 | 处理方式 |
|---|---|
| 临时闲聊 | 不回写 |
| 未确认猜测 | 不写成事实；必要时标记待确认 |
| 明文密钥、token、Cookie | 不回写；如有价值，先脱敏 |
| 一次性命令输出 | 除非形成可复用排障经验，否则不回写 |
| 无来源外部事实 | 不写成事实 |

### 2. 判断为什么值得回写

每次回写前，Agent 必须回答三个问题：

1. 这条知识未来会在什么场景下再次使用？
2. 它能减少哪类重复劳动、重复沟通或重复踩坑？
3. 它应该以事实、规则、流程、案例、决策还是检查清单的形式存在？

如果答案不清楚，先向用户确认，或把内容标为“待整理草稿”，不要伪装成成熟笔记。

### 3. 记录来源并建立上下文

Agent 必须能说明信息从哪里来。

来源可能是外部资料，也可能是当前对话、任务执行过程、工具输出、用户确认或失败复盘。

必须记录：

| 字段 | 要求 |
|---|---|
| 来源类型 | 外部文章、当前对话、用户确认、工具执行、截图、PDF、系统输出等 |
| 来源位置 | URL、本地路径、对话说明或任务上下文 |
| 作者 / 账号 | 来源中存在才写，不编造 |
| 时间 | 来源中存在才写；对话沉淀写当前日期 |
| 复核说明 | 明确哪些是原始事实，哪些是 Agent 归纳 |
| 待确认事项 | 不能复核或用户未确认的内容必须标出 |

如果来源中有图片、表格、截图，Agent 应优先把其中的关键信息转写为可编辑 Markdown 表格，而不是生成图片。

### 4. 先思考信息结构，不直接照搬

Agent 不应把来源顺序简单压缩成流水账。提炼前必须判断信息之间的关系：

1. 这条知识的主对象是什么。
2. 多篇资料或多个经验是否应拆成多篇笔记。
3. 是否存在前置概念、因果关系、对比关系或执行流程。
4. 哪些内容是核心结论，哪些是规则明细，哪些是操作建议。
5. 哪些内容只适合作为来源证据，不应升级为判断。
6. 哪些内容应该做成表格、清单、流程图或 callout。

推荐的信息组织顺序：

```text
问题 / 背景
-> 核心结论
-> 证据 / 来源 / 规则
-> 结构化展开
-> 适用场景
-> 操作清单
-> 风险 / 待确认
-> 后续演化
```

不同知识对象采用不同章法：

| 知识对象 | 推荐结构 |
|---|---|
| 外部资料提炼 | 来源复核 -> 读这篇要解决的问题 -> 核心结论 -> 规则/对比/清单 |
| 经验 | 场景 -> 做法 -> 为什么有效 -> 适用边界 -> 下次怎么用 |
| 方案 | 目标 -> 约束 -> 方案结构 -> 执行路径 -> 风险与取舍 |
| 知识点 | 定义 -> 解决什么问题 -> 与相近概念区别 -> 使用方式 |
| 踩坑记录 | 现象 -> 根因 -> 修复步骤 -> 预防清单 |
| 决策 | 背景 -> 决策 -> 理由 -> 影响范围 -> 复核点 |
| SOP | 目标 -> 适用范围 -> 输入输出 -> 步骤 -> 检查清单 -> 异常处理 |

多篇资料处理规则：

| 场景 | 处理方式 |
|---|---|
| 两篇文章主题独立 | 分成两篇 raw 笔记 |
| 两篇文章存在依赖关系 | 分开写，并用“关联关系”说明依赖 |
| 一篇是规则，一篇是应用案例 | 规则和案例分开沉淀 |
| 用户明确要求合并 | 可以合并，但必须保留来源分区和逻辑关系 |

### 5. 设计笔记，而不是机械套标题

用户提出“出处、正文”等结构要求时，Agent 应理解其意图是“来源可复核、正文有组织”，不应机械写成 `出处`、`正文` 两个粗糙标题。

推荐结构：

```markdown
# 清晰语义标题

> [!info] 来源复核
> 原文：...
> 链接：...
> 说明：本笔记只依据原文，不补充外部判断。

## 读这篇要解决的问题

## 核心结论

## 关键规则 / 对比 / 流程

## 可执行检查清单

## 适用边界与待确认事项

## 待复核或不确定事项
```

标题要求：

1. 标题必须是语义标题，不使用日期/hash 作为主标题。
2. Obsidian 文件名应与标题一致或高度一致。
3. frontmatter `title`、正文 H1、文件名三者应保持一致。
4. 自动生成的日期/hash slug 只能作为系统内部过渡，不应作为最终用户可见文件名。

### 6. 用 Obsidian 和 Markdown 提升可读性

笔记应该让人愿意读，而不是只满足“有内容”。

Agent 应在不丢失信息的前提下，使用 Obsidian 友好的 Markdown 结构：

| 能力 | 使用场景 | 示例 |
|---|---|---|
| Callout | 来源、关键结论、风险、提示 | `> [!info]`、`> [!important]`、`> [!warning]` |
| 表格 | 对比、规则、字段、检查项 | 促销规则、方案取舍、错误处理 |
| Checklist | 执行前后自查 | `- [ ] 读回 GBrain DB` |
| Mermaid | 流程、状态流转、依赖关系 | raw -> wiki -> active |
| 代码块 | 命令、配置、模板 | ```yaml / ```text |
| Wikilink | 关联已有知识页 | `[[Agent 回写链路验收测试]]` |
| 分节标题 | 按逻辑拆块 | `## 核心结论`、`## 操作清单` |

格式设计原则：

1. 重点信息优先出现。
2. 长段落拆成短段落。
3. 列表用于步骤，表格用于比较，callout 用于提醒。
4. 不为了“好看”添加无意义装饰。
5. 不让格式压过信息本身。
6. 同一类笔记尽量保持结构一致，方便以后批量阅读和编译。

### 7. 图片和表格处理

如果原文图片包含关键内容，Agent 应优先复刻为可编辑信息表格。

| 原始材料 | 首选沉淀方式 |
|---|---|
| 规则对比图 | Markdown 对比表 |
| 流程图 | Mermaid 或分步骤列表 |
| 检查清单截图 | Markdown checklist 或表格 |
| 组织结构图 | Mermaid graph 或层级列表 |
| 数据图表 | 表格 + 结论；必要时注明无法复核原始数据 |

禁止行为：

1. 用户要求可编辑时，仍用生图替代结构化表格。
2. 图中没有的信息，不得为了补齐表格而编造。
3. 看不清的内容必须标记为“原图未清晰显示”，不得猜测。

### 8. 回写前自审

写入前必须检查：

| 检查项 | 标准 |
|---|---|
| 来源 | 能追溯到原始文章、对话、工具执行或用户确认 |
| 内容 | 没有编造来源不存在的信息 |
| 结构 | 不是流水账，有清晰逻辑层级 |
| 复利价值 | 未来能复用、复核或继续演化 |
| 客观中立 | 区分事实、推理、建议和待确认 |
| 标题 | 用户可读，适合 Obsidian 浏览 |
| 格式 | 使用 Markdown / Obsidian 能力提升阅读体验 |
| 表格 | 可编辑、可复用、无明显错列或漏列 |
| 风险 | 涉及政策、财务、客户、价格等内容时标记未验证 |
| 路由 | 新材料默认仍在 inbox，不直接绕过路由 |
| 唯一性 | 同一材料不生成多条 raw 记录 |

### 9. 执行 raw-first 回写

常规新增材料必须先写入 GBrain DB raw，再物化到 `00 - raw/00 - inbox`。

默认状态必须保持：

```yaml
knowledge_layer: raw
lifecycle_status: inbox
domain: unknown
tags: []
wiki_page_type: not_applicable
compile_status: 未编译
compiled_to: []
trust_level: raw
gbrain_db_sync_status: synced
gbrain_db_sync_error: not_applicable
```

对 Agent 回写 raw，`gbrain_db_sync_status: synced` 只表示已写入 GBrain DB raw 即时记忆，不表示已经完成路由、wiki 编译或正式同步。

如果使用 `agent_raw_writeback` 创建 raw，必须记住：

1. 它会先写 GBrain DB raw。
2. 它会物化到本地 inbox。
3. 它可能生成日期/hash 风格的 slug 或文件名。
4. 如果用户要求可读文件名，创建后必须把同一条记录改名并重新导入，而不是再创建一条新 raw。

### 10. 修正已有回写时，禁止新建副本

如果用户指出已回写内容存在问题，Agent 必须优先修正同一条 raw 记录。

禁止为了修正问题再次调用新增回写，导致多条副本。

正确补救流程：

1. 找到现有 raw 的 `knowledge_id`、slug、本地文件路径。
2. 判断问题属于内容、标题、文件名、frontmatter、DB 同步还是副产物。
3. 修改本地 Markdown。
4. 如改了文件名或 slug，必须同步更新 GBrain DB 里的 slug。
5. 用现有文件重新导入同一条 GBrain raw 记录。
6. 读回验证新 slug 可读，旧 slug 不再活跃。
7. 清理多余副产物。

### 11. 写后验证

回写后必须验证，不得只凭工具返回就向用户宣称完成。

必须验证：

| 验证项 | 方法 |
|---|---|
| 本地文件存在 | 检查当前实际位置；新增时通常是 `00 - raw/00 - inbox`，路由后应检查领域 raw 目录 |
| inbox 无结构错误 | 新增或仍在 inbox 时运行 inbox dry-run `-NoRepair` |
| 路由后路径变化 | 如果定时任务或路由任务可能运行过，必须用 `knowledge_id`、标题或文件名重新定位当前文件 |
| GBrain DB 可读 | 使用 `get_page` 读回当前 slug；如果只能读到旧 slug，必须说明 DB 同步滞后 |
| 标题一致 | 检查文件名、frontmatter title、正文 H1 |
| 内容一致 | 检查 GBrain compiled_truth 是否为新版正文 |
| 无旧副本 | 如果发生改名，确认旧 slug 查不到或已删除 |
| 无副产物 | 检查 inbox 是否只留下预期文件 |
| 当前状态一致 | 检查本地 `lifecycle_status` / `domain` 与汇报状态一致 |

异步路由或定时任务后的复验规则：

1. 不得假设最初写入的 inbox 路径仍然有效。
2. 必须先按 `knowledge_id`、标题或文件名在知识库中重新定位当前文件。
3. 如果文件已经从 inbox 移到领域 raw 目录，完成汇报必须给出领域 raw 路径。
4. 如果 GBrain DB 仍停留在旧 inbox slug，完成汇报必须标记为“本地已路由，DB 同步待完成”。
5. 如果本地和 DB 同时存在新旧 slug，必须判断是否是重复副本；不清楚时先报告，不擅自删除。

### 12. 完成汇报

验证完成后，Agent 必须向用户交代回写结果，不能只说“已完成”。

完成汇报必须回答四件事：

| 汇报项 | 必须说明 |
|---|---|
| 回写状态 | 已完成、部分完成、失败、已修正、已清理副本等 |
| 完成证据 | inbox dry-run 结果、GBrain `get_page` 读回结果、旧 slug 清理结果、文件列表检查结果 |
| 如何查看 | 当前实际 Obsidian 文件路径、GBrain slug、必要时说明所在目录 |
| 当前边界 | 是否仍是 inbox、是否已路由到领域 raw、是否未编译、是否 pending DB sync、是否需要后续路由/编译/审核 |

汇报路径时必须区分两类路径：

| 路径类型 | 含义 | 汇报要求 |
|---|---|---|
| 初始写入路径 | Agent 刚写入 raw 时的位置，通常是 inbox | 只能作为写入证据，不得默认作为长期查看入口 |
| 当前实际路径 | 文件此刻在 Obsidian 中的位置 | 必须作为用户查看入口 |

如果回写成功，推荐汇报格式：

```text
已回写：标题
当前本地文件：...
GBrain slug：...
状态：raw / inbox 或 已路由到领域 raw / pending DB sync / 未编译
完成证据：文件存在；inbox dry-run PASS 或不适用；get_page 可读或 DB 同步待完成；旧 slug 已清理 / 无旧 slug；无多余副产物
如何查看：在 Obsidian 打开当前本地文件，或用当前可读 GBrain slug 读回
```

如果只完成了部分步骤，必须明确剩余问题：

```text
已完成：...
未完成：...
原因：...
下一步：...
用户可查看：...
```

汇报时不得省略“raw 未编译”这一状态边界，避免用户误以为该内容已经进入正式知识层。

### 13. 失败处理

| 失败场景 | 必须处理 |
|---|---|
| GBrain DB raw 写入失败 | 不得说已回写；说明失败原因，必要时生成本地草稿并标记未入库 |
| 本地物化失败 | 保留 DB raw，标记同步失败，说明需要补偿物化 |
| 读回失败 | 不得宣称完成；先排查 slug/source/同步状态 |
| 用户指出内容质量不合格 | 修改同一条 raw，不新增副本 |
| 产生多条副本 | 清理副本，只保留核心记录 |
| 文件名不符合 Obsidian 使用 | 改成语义文件名，并同步 DB slug |
| 外部资料无法复核 | 标记来源不可复核，不写成确定事实 |
| 本地已路由但 DB 仍停留旧 slug | 不得说完全同步；说明“本地已路由，DB 同步待完成”，并给出当前本地路径和当前可读旧 slug |
| 定时任务移动了文件导致用户找不到 | 重新定位文件，说明移动原因、当前路径、状态边界和下一步同步要求 |

## 质量标准

一条合格的 Agent 回写 raw 应满足：

1. 用户能从标题看懂这条笔记是什么。
2. 日后能通过来源、对话说明或执行记录复核。
3. 正文不是流水账，而是有逻辑组织。
4. 信息客观中立，区分事实、推理、建议和待确认。
5. 内容能产生知识复利，未来可复用或可继续演化。
6. 格式美观，充分使用 Markdown / Obsidian 能力，但不牺牲信息准确性。
7. 图片中的关键信息被转成可编辑表格或 Mermaid。
8. 来源没有的信息没有被编造。
9. 本地 Markdown 与 GBrain DB raw 一致。
10. inbox 中没有多余副产物。
11. 未编译状态被明确保留。
12. 完成汇报能让用户看到状态、证据和当前实际查看入口。
13. 异步路由或定时任务改变路径后，Agent 能重新定位并说明当前状态。
14. 后续可以被路由、编译、验收和晋升。

## 禁止清单

Agent 回写时禁止：

1. 只写本地 Markdown，不写 GBrain DB raw。
2. 只写 GBrain DB raw，不物化本地 Markdown。
3. 为修正同一材料重复新建 raw。
4. 让日期/hash 文件名成为最终 Obsidian 标题。
5. 把用户说的结构词机械变成粗糙标题。
6. 把多篇文章强行合并成一篇流水账。
7. 编造来源不存在的事实。
8. 用图片替代用户要求的可编辑图表。
9. 把 Agent 日常使用中产生的经验、方案、踩坑和决策当作“不算资料”而忽略。
10. 把笔记写成粗糙记录，不考虑未来阅读和复用。
11. 绕过 inbox 直接放入领域目录。
12. 绕过 raw 到 wiki 编译，直接写 active wiki。
13. 未验证就标记 canonical。
14. 没有读回验证就汇报完成。
15. 回写完成时只说“完成了”，不告诉用户状态、证据和查看方式。
16. 定时任务或路由任务移动文件后，仍把旧 inbox 路径当作查看入口。
17. 本地路径和 GBrain DB slug 不一致时，仍宣称完全同步。

## 本 SOP 的维护规则

当 Agent 回写再次出现以下问题时，必须更新本 SOP：

1. 产生副产物。
2. DB 与本地 Markdown 不一致。
3. 标题、文件名、slug 不一致导致用户困惑。
4. 用户指出笔记不像精选笔记。
5. 图片或表格信息处理失败。
6. 日常 Agent 产物没有被正确纳入回写流程。
7. 修正过程生成了重复 raw。
8. 回写后无法被 inbox 路由或 GBrain 检索。
9. 定时任务或异步路由移动文件后，用户无法根据完成汇报找到当前文件。
10. 本地已路由但 GBrain DB 仍停留旧 slug，造成查看入口不一致。

本 SOP 是 raw 草案，后续应经过 inbox 路由、raw 到 wiki 编译和质量验收后，再晋升为正式的 Agent 执行规范。

## v0.2.0 - 2026-06-20

变更类型：Changed

影响范围：

- inbox 路由
- 脚本输入输出

兼容影响：需要迁移

涉及规则：

- 《00 - inbox》路由分发规则.md
- 自动化脚本/00 - inbox 校验与路由输入 dry-run.ps1

规则契约：

inbox 脚本只执行 frontmatter 结构校验、结构修复、材料包检查和路由输入输出。脚本不得根据关键词判断目标目录、置信度、分发后 `domain_hint`、`material_type` 或 `tags`。Codex 负责阅读正文主体并输出语义路由结果。

决策原因：

关键词匹配会把正文中的局部词误判为目标目录证据，不能满足“读正文主体、确认主对象、判断领域和用途”的路由规则。

迁移要求：

调用 inbox 脚本的流程必须在脚本之后增加 Codex 语义路由步骤；不得再把脚本输出当作目标目录建议。

涉及规则：

- 《00 - inbox》路由分发规则.md

inbox 语义路由结果必须生成 Excel 审核表。审核表固定保存到 `C:\Users\terry\Downloads`，文件名为 `YYYY-MM-DD - NN - inbox 路由审核表.xlsx`，字段固定为：文件、当前路径、source_type、当前 material_type、建议 material_type、当前 tags、建议 tags、建议目标目录、候选目标目录、判断依据、置信度、目标目录是否存在、是否需要修改文件名、动作建议、用户决策、用户备注、执行状态。

## v0.4.0 - 2026-06-20

变更类型：Changed

影响范围：

- inbox 路由
- 脚本输入输出

兼容影响：需要迁移

涉及规则：

- 《00 - inbox》路由分发规则.md
- 自动化脚本/00 - inbox 校验与路由输入 dry-run.ps1

规则契约：

inbox 路由先直接执行无需用户决策的高置信度材料；Excel 审核表只包含需要用户决策的材料。审核表不再包含 `当前路径`、`source_type`、`当前 material_type`、`当前 tags`、`目标目录是否存在`、`动作建议` 等执行噪音字段。执行器必须用固定 inbox 路径和 `文件` 字段重建源路径，源文件不在 inbox 时必须跳过。

决策原因：

Excel 的交互价值是承载用户决策，不是展示执行器已经知道的路径和状态。无需用户判断的材料进入 Excel 会增加确认成本，降低批量处理效率。

迁移要求：

生成审核表的流程必须过滤掉可直接执行材料；执行审核表的流程必须按 `文件` 字段从 inbox 定位源文件，不再读取 `当前路径` 字段。

## v0.8.0 - 2026-06-20

变更类型：Added

影响范围：wiki 编译规则

兼容影响：向后兼容

涉及规则：

- 《01 - wiki》目录与索引规则.md

规则契约：

新增 `01 - wiki` 目录与索引规则。wiki 目录只表达主知识领域，不表达 `candidate`、`active`、`deprecated`、`rejected` 等生命周期状态。`_index.md` 是总导航，`_meta` 只存放编译日志、冲突索引、晋升记录和待处理索引。治理索引是派生视图，不作为业务事实来源。

决策原因：

Karpathy LLM Wiki 的 wiki 层是持续维护的知识网络，GBrain 落地依赖 schema 和页面元数据解释知识对象。目录承担领域定位即可，状态和页面形态必须由 frontmatter 表达，避免文件移动破坏链接和 Agent 引用。

迁移要求：

初始化 `01 - wiki` 的 `_index.md`、`_meta` 和顶层领域目录。当前 `01 - wiki` 为空，无需迁移旧页面。

## v0.10.0 - 2026-06-20

变更类型：Changed

影响范围：wiki 编译规则

兼容影响：向后兼容

涉及规则：

- 《01 - wiki》目录与索引规则.md
- 《01 - wiki》知识编译规则.md

规则契约：

Wiki 写入后必须执行落盘命名、页面结构、补丁污染和索引同步验收。`_index.md`、`_meta` 及其治理文件必须按规则命名；发现 `index.md` 或 `meta` 时必须立即修正。Markdown 正文中禁止残留补丁标记。新增 candidate 页面后必须同步 `_index.md`、`_meta/编译日志.md` 和 `_meta/待处理索引.md`。

决策原因：

首次 wiki 初始化和下位页面补建暴露了两个执行风险：实际落盘名称可能与规则名称不一致；多文件补丁失败时，补丁标记可能进入 Markdown 正文。必须把运行验收固化为规则，避免后续 Agent 写入 wiki 时污染知识页或破坏索引一致性。

迁移要求：

已修正当前 `01 - wiki` 中的 `_index.md`、`_meta` 命名和补丁污染。后续所有 wiki 写入都必须执行本次新增验收规则。

变更类型：Added

影响范围：wiki 编译规则

兼容影响：向后兼容

涉及规则：

- 《01 - wiki》知识编译规则.md

规则契约：

candidate 晋升为 active 后必须执行 active 可用性验收。验收必须检查生命周期表述一致性、active 页面链接、索引页下位 active 覆盖、断链数量和真实使用问题测试。验收结果只能触发页面修复、待处理登记或冲突登记，禁止把可用性测试结果写成新的业务结论。

迁移要求：

后续所有 candidate 晋升 active 后必须执行 active 可用性验收。已有 active 页面在下次触达时按本规则补验。

## v0.13.0 - 2026-06-20

变更类型：Added

影响范围：

- frontmatter
- inbox 路由
- wiki 编译规则

兼容影响：需要迁移

涉及规则：

- 《00 - raw》编译状态规则.md
- 《00 - raw》原始材料目录命名规则.md
- 《00 - inbox》路由分发规则.md
- 《01 - wiki》知识编译规则.md

规则契约：

raw Markdown frontmatter 新增 `compile_status` 字段，用于表达 raw 是否已经参与 wiki 编译。允许值固定为：`未编译`、`已编译`、`部分编译`、`跳过编译`、`暂缓编译`、`已过期`。材料从 inbox 分发到 raw 领域目录后，默认写入 `compile_status: 未编译`。raw 完成 wiki 编译后，必须同步 `compiled_at` 和 `compiled_to`。后续 raw 覆盖率统计必须输出已编译、部分编译、未编译、跳过编译、暂缓编译、已过期和缺少 `compile_status` 的数量。

决策原因：

仅从 wiki `source_refs` 反查 raw 编译覆盖率不稳定，无法可靠回答“哪些 raw 已经被编译、哪些仍待处理”。raw 层需要直接记录编译生命周期，才能支持 Karpathy LLM Wiki 式的持续 ingest、compile、update 工作流。

迁移要求：

现有 raw Markdown 需要补充 `compile_status`。已被 wiki `source_refs` 引用的 raw 应标记为 `已编译` 或 `部分编译`，其余 raw 默认标记为 `未编译`，但跳过、暂缓和过期材料必须由用户或后续审计明确原因后再标记。

## v0.14.0 - 2026-06-21

变更类型：Changed

影响范围：frontmatter

兼容影响：向后兼容

涉及规则：

- 《00 - raw》编译状态规则.md

规则契约：

raw frontmatter 批量迁移后必须执行落盘验收。验收必须确认 Markdown raw 存在开头和结尾两个独立的 `---` 分隔符，正文内容不得进入 frontmatter，`compile_status` 必须位于 frontmatter 内。Obsidian 显示“无效属性”时，优先检查 frontmatter 结束分隔符是否缺失。

决策原因：

批量补充 `compile_status` 时，如果遗漏 frontmatter 结束分隔符，Obsidian 会把正文内容解析为属性区，导致页面显示“无效属性”。该错误会影响所有被迁移的 raw Markdown，需要把分隔符验收固化为规则。

迁移要求：

已修复当前 83 个 raw Markdown 的 frontmatter 结束分隔符。后续批量修改 raw frontmatter 后必须运行分隔符完整性校验。

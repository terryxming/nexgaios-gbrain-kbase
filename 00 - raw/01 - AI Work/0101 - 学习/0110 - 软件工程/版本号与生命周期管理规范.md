# 版本号与生命周期管理规范

> 适用于 Skill、GPT 配置、Prompt 框架、知识库模板、脚本、报告模板、工作流配置等可持续迭代资产。

---

## 1. 核心目标

版本号管理不是命名习惯，而是为了让资产具备以下能力：

- **可追溯**：知道每个版本改了什么。
- **可回滚**：出现问题时可以回到旧版本。
- **可测试**：每个版本可以对应测试结果。
- **可复盘**：能够判断某次修改是否带来正向效果。
- **可协作**：多人使用时不会出现“同名不同内容”的混乱。

---

## 2. 版本号基本格式

推荐采用语义化版本号：

```text
vMAJOR.MINOR.PATCH
```

示例：

```text
v0.1.0
v0.2.3
v1.0.0
v1.4.2
v2.0.0
```

三位数字含义如下：

| 位置 | 名称 | 含义 | 示例 |
|---|---|---|---|
| `MAJOR` | 主版本号 | 破坏性变化、不兼容旧版本 | `v1.4.2 → v2.0.0` |
| `MINOR` | 次版本号 | 新增功能，但兼容旧版本 | `v1.2.3 → v1.3.0` |
| `PATCH` | 修订号 | 修复问题、小优化、格式修正 | `v1.2.3 → v1.2.4` |

---

## 3. 版本号升级规则

### 3.1 Patch：修订号升级

适用于小修小补，不改变核心逻辑和使用方式。

```text
v0.2.0 → v0.2.1
```

常见场景：

- 修复错别字。
- 修复格式问题。
- 修复提示词歧义。
- 修复 bug。
- 补充少量说明。
- 优化表达，但不改变结构。

---

### 3.2 Minor：次版本号升级

适用于新增能力，但旧版本使用方式仍然有效。

```text
v0.2.1 → v0.3.0
```

常见场景：

- 新增模块。
- 新增流程阶段。
- 新增输出格式。
- 新增测试规则。
- 新增安全规则。
- 新增异常处理机制。

升级 `MINOR` 时，`PATCH` 必须归零。

正确：

```text
v0.2.5 → v0.3.0
```

错误：

```text
v0.2.5 → v0.3.6
```

---

### 3.3 Major：主版本号升级

适用于破坏性变化，即旧版本的使用方式、输出结构、接口或核心流程不再兼容。

```text
v1.4.2 → v2.0.0
```

常见场景：

- 重构整体工作流。
- 删除旧功能。
- 替换核心结构。
- 修改关键输入字段。
- 修改标准输出格式。
- 旧版本测试用例无法继续复用。

升级 `MAJOR` 时，`MINOR` 和 `PATCH` 必须同时归零。

正确：

```text
v1.8.6 → v2.0.0
```

错误：

```text
v1.8.6 → v2.9.7
```

---

## 4. 已发布版本不可修改

一旦某个版本已经确认、提交、交付、被测试引用或被正式使用，该版本就应视为冻结。

后续无论大改还是小改，都必须发布新版本。

正确做法：

```text
v0.2.0 已发布
发现小问题
发布 v0.2.1
```

错误做法：

```text
v0.2.0 已发布
偷偷修改 v0.2.0 的内容
仍然叫 v0.2.0
```

错误做法会导致：

- 无法判断哪个才是真正的 `v0.2.0`。
- 测试报告与实际文件无法对应。
- 问题定位困难。
- 无法稳定回滚。
- 协作时出现同名不同内容。

---

## 5. 草稿状态与正式发布的区别

版本号是否冻结，取决于它是否已经正式确认。

| 状态 | 是否可以继续修改同一版本 | 说明 |
|---|---:|---|
| 草稿中 | 可以 | 尚未发布，仍在编辑和验证 |
| 已发布 | 不可以 | 已经被引用、测试、交付或正式使用 |
| 已被替代 | 不建议修改 | 应保留用于追溯 |

示例：

```text
v0.3.0-draft   可以继续修改
v0.3.0         正式发布后不可直接修改
v0.3.1         用于修复 v0.3.0 的问题
```

---

## 6. 生命周期状态规范

生命周期状态用于说明某个版本当前处于什么阶段。它不是版本号本身，而是版本状态标记。

推荐使用三种基础状态：

```text
Draft
Stable
Deprecated
```

注意：

```text
status 和 lifecycle 不能混用。
```

在 Nexgaios GBrain 知识库里：

| 字段 | 含义 | 示例 |
|---|---|---|
| `status` | 知识材料在 raw 流程中的状态 | `inbox` / `raw` |
| `lifecycle` | 某个版本资产的生命周期状态 | `Draft` / `Stable` / `Deprecated` |

因此，知识库文件 frontmatter 里的 `status: raw` 只说明“这份材料已经进入 raw 层”，不表示这个版本已经 Stable。

---

### 6.1 Draft：草稿版

表示版本仍在编辑、测试或验证中，不能作为正式版本长期引用。

```markdown
version: v0.3.0
lifecycle: Draft
```

适用场景：

- 结构还在调整。
- 功能还在补充。
- 测试尚未完成。
- CHANGELOG 尚未完善。
- 输出效果尚未稳定。

---

### 6.2 Stable：稳定版

表示版本已经确认，可以正式使用、引用、测试、交付。

```markdown
version: v0.3.0
lifecycle: Stable
```

进入 `Stable` 后，该版本应被冻结，不再直接修改。

后续发现问题时，应发布新版本：

```text
v0.3.0 Stable
发现问题
v0.3.1 Draft
v0.3.1 Stable
```

---

### 6.3 Deprecated：已废弃

表示版本已经过时，不推荐新项目继续使用，但应保留用于追溯、回滚或对比。

```markdown
version: v0.2.0
lifecycle: Deprecated
replaced_by: v0.3.0
```

适用场景：

- 新版本已经替代旧版本。
- 旧版本存在明显缺陷。
- 旧流程不再推荐。
- 旧版本与新系统不兼容。

---

## 7. 生命周期流转

推荐的生命周期流转如下：

```text
Draft → Stable → Deprecated
```

含义：

| 阶段 | 含义 | 操作规则 |
|---|---|---|
| `Draft` | 开发中 | 可以继续修改 |
| `Stable` | 已发布 | 冻结内容，不再直接修改 |
| `Deprecated` | 已废弃 | 保留历史，不建议继续使用 |

示例：

```text
v0.2.0 Draft
↓
v0.2.0 Stable
↓
v0.2.0 Deprecated，被 v0.3.0 替代
```

---

## 8. 版本文件头推荐格式

每个版本文件建议在开头保留元信息，便于识别和管理。

```markdown
# 项目名称 / Skill 名称

version: v0.3.0
lifecycle: Stable
updated: 2026-06-16
owner: Terry
compatible_with: v0.2.x
replaced_by: N/A
```

字段说明：

| 字段 | 含义 |
|---|---|
| `version` | 当前版本号 |
| `lifecycle` | 生命周期状态 |
| `updated` | 最近更新时间 |
| `owner` | 维护人 |
| `compatible_with` | 兼容的旧版本范围 |
| `replaced_by` | 被哪个新版本替代 |

---

## 9. 文件命名规范

文件名应包含版本号，避免只靠文件内容识别版本。

推荐：

```text
画面导演_v0.3.0.md
画面导演_v0.3.1.md
画面导演_v1.0.0.md
```

不推荐：

```text
画面导演.md
画面导演_最新版.md
画面导演_最终版.md
画面导演_最终最终版.md
```

如果需要快捷入口，可以额外保留：

```text
latest.md
```

但 `latest.md` 只能作为当前推荐版本的入口，不能替代正式版本号。

---

## 10. 目录结构推荐

```text
skill-name/
├── README.md
├── CHANGELOG.md
├── latest.md
└── versions/
    ├── v0.1.0.md
    ├── v0.2.0.md
    ├── v0.2.1.md
    └── v0.3.0.md
```

说明：

| 文件 / 目录 | 作用 |
|---|---|
| `README.md` | 说明项目用途、使用方法、当前推荐版本 |
| `CHANGELOG.md` | 记录每个版本的变化 |
| `latest.md` | 指向当前推荐版本 |
| `versions/` | 保存所有历史版本 |

---

## 11. CHANGELOG 规范

每个正式版本都必须记录变更内容。

推荐格式：

```markdown
# Changelog

## v0.3.0

### Added
- 新增反例测试规则。
- 新增缺失输入处理机制。

### Changed
- 优化任务边界判断逻辑。

### Fixed
- 修复输出中出现无意义表达的问题。

## v0.2.1

### Fixed
- 修复部分格式不一致的问题。
```

常用分类：

| 分类 | 含义 |
|---|---|
| `Added` | 新增内容 |
| `Changed` | 修改已有内容 |
| `Fixed` | 修复问题 |
| `Removed` | 删除内容 |
| `Deprecated` | 标记废弃 |
| `Security` | 安全相关修改 |
| `Breaking Changes` | 破坏性变化 |

---

## 12. 破坏性变化必须单独标注

如果新版本会导致旧用法失效，必须明确写出破坏性变化。

推荐写法：

```markdown
## v1.0.0

### Breaking Changes
- 删除旧的三阶段输出流程。
- 改为“信息登记 → 提案 → 冻结提示词 → 生成图片”四阶段流程。
- 旧版 v0.4.x 的调用方式不再兼容。
```

不推荐写法：

```markdown
## v1.0.0

### Changed
- 优化流程。
```

“优化流程”过于模糊，无法判断是否兼容旧版本。

---

## 13. 日常版本发布流程

推荐每次修改按以下流程执行：

```text
1. 确认当前版本
2. 判断修改类型：Patch / Minor / Major
3. 复制旧版本，创建新版本文件
4. 修改新版本内容
5. 更新文件头元信息
6. 更新 CHANGELOG
7. 执行测试或人工检查
8. 确认后将状态从 Draft 改为 Stable
9. 更新 latest.md 指向新版本
10. 如有必要，将旧版本标记为 Deprecated
```

---

## 14. 修改类型判断表

| 修改内容 | 推荐版本变化 | 示例 |
|---|---|---|
| 修错别字 | Patch | `v0.2.0 → v0.2.1` |
| 修格式 | Patch | `v0.2.0 → v0.2.1` |
| 修复提示词歧义 | Patch | `v0.2.0 → v0.2.1` |
| 新增反例测试 | Minor | `v0.2.1 → v0.3.0` |
| 新增安全规则 | Minor | `v0.2.1 → v0.3.0` |
| 新增输出格式 | Minor | `v0.2.1 → v0.3.0` |
| 删除旧流程 | Major | `v1.4.2 → v2.0.0` |
| 重构整体架构 | Major | `v1.4.2 → v2.0.0` |
| 修改核心输入输出结构 | Major | `v1.4.2 → v2.0.0` |

---

## 15. 最小必须遵守清单

日常管理至少要遵守以下规则：

```text
1. 已发布版本不可修改。
2. 任何后续修改都必须发布新版本。
3. 修复问题升级 PATCH。
4. 新增功能升级 MINOR。
5. 破坏性变化升级 MAJOR。
6. 升级高位时，低位必须归零。
7. 每个正式版本必须写 CHANGELOG。
8. 每个版本必须标记生命周期状态。
9. 保留旧版本，不要只保留 latest。
10. 破坏性变化必须明确标注。
```

## 16. 推荐实践示例

```text
v0.1.0 Draft
初版可运行，但还在验证中。

v0.1.0 Stable
初版确认，可正式使用。

v0.2.0 Draft
新增测试模块。

v0.2.0 Stable
测试模块确认有效。

v0.2.1 Stable
修复 v0.2.0 的格式问题。

v0.3.0 Stable
新增安全规则和反例库。

v1.0.0 Stable
整体流程稳定，进入正式版。

v2.0.0 Stable
重构核心工作流，旧版本不兼容。
```

---

## 17. 一句话总结

版本号用于回答“这是哪一个版本”，生命周期状态用于回答“这个版本现在处于什么阶段”。

版本管理的核心原则是：

> 已发布版本必须冻结；后续任何修改都通过新版本承载；每个版本都要可追溯、可回滚、可测试、可复盘。

# wiki 编译与页面治理规则

版本：v0.1.1

生命周期：Draft

日期：2026-06-22

适用范围：wiki 目录、索引、治理记录、raw 到 wiki 编译、wiki 页面形态、生命周期、链接规则

## 1. wiki 定位

`01 - wiki` 是知识网络层，不复制 `00 - raw` 的碎片目录。

`raw` 保存原始材料和证据；`wiki` 保存可复用、可追溯、可更新的企业知识网络。

## 2. wiki 目录原则

`01 - wiki` 目录只表达主知识领域，不表达生命周期。

禁止创建：

```text
01 - wiki/active
01 - wiki/candidate
01 - wiki/deprecated
01 - wiki/rejected
```

第一版结构：

```text
01 - wiki
  _index.md
  _meta
    编译日志.md
    冲突索引.md
    晋升记录.md
    待处理索引.md
  AI Work
  Amazon
  认知管理
  财务投资
  设计
  产品
```

## 3. 页面放置

Wiki 页面放在主知识领域目录下。

主领域按以下顺序判断：

1. 页面当前结论主要服务哪个业务领域。
2. 页面主要被哪个项目、流程或 Agent 使用。
3. 页面证据来源主要集中在哪个领域。
4. 页面和哪个已有 active 页面关系最强。

跨领域页面必须选择一个主领域，其他领域关系通过 `相关链接` 表达。

无法判断主领域时，不创建正式 wiki 页面，进入 `_meta/待处理索引.md`。

禁止创建：

```text
其他
未分类
杂项
临时
```

## 4. _index.md

`_index.md` 是 wiki 总入口，只承担导航职责：

1. 列出主要领域入口。
2. 列出核心 active 页面。
3. 列出待处理 candidate 摘要。
4. 列出最近编译变化。

`_index.md` 不是事实来源。

## 5. _meta

`_meta` 只放治理索引和过程记录，不放业务知识正文。

| 文件 | 职责 |
| --- | --- |
| `编译日志.md` | 记录 raw 到 wiki 的编译动作时间线。 |
| `冲突索引.md` | 集中列出未解决冲突。 |
| `晋升记录.md` | 记录 candidate 晋升 active 的依据。 |
| `待处理索引.md` | 列出需要补证据、补链接、用户确认或暂缓处理的事项。 |

治理索引必须能追溯到具体 wiki 页面或 raw。

如果治理索引和具体 wiki 页面内容不一致，以具体 wiki 页面及其 raw 证据为准。

## 6. raw 进入 wiki 的准入规则

Raw 满足以下条件中的两项及以上，才进入 wiki 编译：

1. 后续会被重复使用。
2. 可以更新已有 wiki 页面。
3. 可以形成稳定知识对象。
4. 会影响业务、项目或 Agent 执行。
5. 可以连接多个 raw、主题或页面。
6. 具备可追溯证据。

以下 raw 不进入 wiki：

1. 重复剪藏。
2. 一次性参考。
3. 没有结论的临时对话。
4. 没有复用价值的过程记录。
5. 已过期且没有追溯价值的材料。
6. 信息过碎，不能形成概念、流程、方案、决策、案例或主题。

进入 wiki 不是复制 raw，而是触发以下动作之一：

1. 新建 wiki 页面。
2. 更新已有 wiki 页面。
3. 补充证据。
4. 标记冲突。
5. 降级旧知识。
6. 仅保留 raw。

## 7. Wiki 页面形态

第一版 wiki 页面收敛为 10 类：

| 页面形态 | 用途 |
| --- | --- |
| 索引页 | 导航入口和知识地图。 |
| 对象页 | 概念、实体、工具、系统、产品、站点、SKU、政策对象。 |
| 主题页 | 跨资料综合一个主题。 |
| 项目页 | 项目主页和长期沉淀。 |
| 流程页 | 可重复执行流程，正文中区分指南或 SOP。 |
| 决策页 | 方案取舍和理由。 |
| 政策页 | 外部政策、内部规范、平台规则。 |
| 模板页 | 可复用格式、提示词、文档模板。 |
| 案例页 | 成功案例、失败案例、复盘。 |
| 对比页 | 方案、工具、路径、产品对比。 |

已有 wiki 页面可以承载时，必须优先更新已有页面。

只有已有 wiki 无法承载时，才新建页面。

## 8. Wiki 生命周期

Wiki 使用以下状态：

```text
candidate
active
deprecated
rejected
```

Agent 可以自动生成 `candidate`。

以下内容禁止自动进入 `active`：

1. 证据不足。
2. 与已有 active 结论冲突。
3. 涉及权限、安全、财务、合规、商业决策或组织规则。
4. 会改变已有流程、规则或业务判断。
5. 无法追溯到 raw。

`candidate` 晋升为 `active` 必须满足：

1. raw 证据存在。
2. 知识对象清晰。
3. 不存在未解决冲突。
4. 不覆盖 active 结论。
5. 至少连接一个已有 active 页面。
6. 页面形态合法。
7. 证据来源可追溯。

## 9. 编译输出契约

每次编译必须输出固定判定单：

```text
原始材料
编译决策
受影响 Wiki 页面
Wiki 页面形态
执行动作
目标状态
证据来源
是否冲突
判断依据
风险等级
是否需要用户确认
```

风险等级：

| 风险等级 | 判断标准 |
| --- | --- |
| 低 | 补充事实、补充来源、优化表达，不改变结论。 |
| 中 | 形成新结论、更新流程、影响项目理解。 |
| 高 | 影响权限、安全、财务、合规、商业决策或组织规则。 |

高风险、冲突、降级 active、重大业务判断，必须设置：

```text
是否需要用户确认：是
```

## 10. Wiki 页面写入

Wiki 写结论、结构、关系、适用范围和证据。

禁止把 raw 简单摘要成 wiki。

Wiki 页面最小结构：

```markdown
## 当前结论

## 适用范围

## 依据来源

## 相关链接
```

存在冲突时，增加：

```markdown
## 冲突与待确认
```

存在结论演化时，增加：

```markdown
## 变更记录
```

`依据来源` 必须链接到 raw。

`相关链接` 只链接真实知识关系，不为凑数量建立链接。

## 11. Wiki 链接

有效关系包括：

```text
上位知识
下位知识
前置流程
后续流程
同一项目
同一系统
同一决策
替代关系
冲突关系
证据来源
```

禁止因为标题相似、目录相同、同一天创建或都提到 AI 而建立链接。

孤立页面只能保持 `candidate`，不能直接进入 `active`。

## 12. 工程执行契约

raw 到 wiki 编译必须留下可复核的本地落盘结果，不能只停留在对话回答中。

每次编译至少执行以下顺序：

1. 解析 raw 当前实际路径，确认 raw 不在 `00 - raw/00 - inbox` 的未路由临时状态中；如果仍在 inbox，只允许生成编译建议，不直接写 active wiki。
2. 读取候选 wiki 页面，优先判断是否更新已有页面。
3. 生成本规则第 9 节的固定判定单。
4. 按风险等级判断是否需要用户确认。
5. 新建或更新 wiki 页面，默认写入 `lifecycle_status: candidate`。
6. 更新 raw 的 `compiled_to` 和 `compile_status`，或写入待处理 / 冲突索引。
7. 更新 `_meta/编译日志.md`；涉及待处理或冲突时，同步更新 `_meta/待处理索引.md` 或 `_meta/冲突索引.md`。
8. 将被修改的本地 Markdown 的 `gbrain_db_sync_status` 置为 `pending`。
9. 进入 `05 - wiki 质量验收规则.md`。

固定判定单必须写入 `_meta/编译日志.md` 或当前任务的明确交接记录。

只在聊天中展示判定单、但没有落盘记录时，本次编译不得视为完成。

## 13. 页面命名与唯一性

新建 wiki 页面前必须检查：

1. 同领域目录下是否已有同主题页面。
2. 全 wiki 是否已有同标题、同义标题或清洗后 slug 相同的页面。
3. frontmatter `title`、正文 H1、文件名是否一致或高度一致。
4. 页面路径是否位于 `01 - wiki` 的合法主领域目录下。

出现同主题或 slug 冲突时，必须优先更新已有页面。

确需新建页面时，编译日志必须说明为什么已有页面不能承载。

## 14. 生命周期写入边界

Agent 编译 raw 时默认只能生成或更新 `candidate` wiki。

以下动作必须先经过 `05 - wiki 质量验收规则.md`，并记录晋升依据：

1. 将 `candidate` 改为 `active`。
2. 将 `trust_level` 改为 `canonical`。
3. 降级、废弃或拒绝已有 active 页面。
4. 删除 active 页面正文结论。

只修复错别字、断链、来源补全或模板残留时，可以直接修改 active 页面，但修改后仍必须执行质量验收，并将 `gbrain_db_sync_status` 置为 `pending`。

## 15. 证据来源解析规则

`依据来源` 必须指向当前可定位的 raw，而不是已经失效的旧 inbox 路径或旧 GBrain slug。

如果本地 raw 已路由但 GBrain DB 仍停留在旧 slug，本地 wiki 页面优先使用当前本地 raw 路径或 Obsidian 链接，并在编译日志中标记：

```text
DB 同步待完成，旧 slug 仅作为临时读回入口。
```

如果 raw 只存在于 GBrain DB raw 即时记忆、本地尚未物化或无法定位，禁止直接生成 active wiki；只能生成 candidate 草案或待处理记录。

## 16. 类型化链接格式

`相关链接` 必须使用可被人和脚本识别的类型化格式。

推荐格式：

```markdown
- 上位知识：[[页面标题]] - 关系说明
- 同一项目：[[页面标题]] - 关系说明
```

每条链接必须同时包含关系类型和目标页面。

无目标页面时保留空项没有意义，应删除该关系类型或写入待处理索引。

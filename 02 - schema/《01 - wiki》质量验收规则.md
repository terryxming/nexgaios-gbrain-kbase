# 01 - wiki 质量验收规则

版本：v0.1.0

生命周期：Draft

日期：2026-06-20

适用范围：`01 - wiki`

## 1. 规则边界

本规则只做：

1. 规定 wiki 页面和知识簇的质量验收标准。
2. 规定 candidate 晋升 active 后必须执行的可用性验收。
3. 规定 active 页面被修改后的质量复验。
4. 规定验收失败时应进入修复、待处理或冲突索引。

本规则不做：

1. 不判断 raw 是否值得进入 wiki。
2. 不决定 wiki 页面形态。
3. 不生成新的业务结论。
4. 不替代 `《01 - wiki》知识编译规则.md`。
5. 不执行 GitHub、Supabase 或 GBrain DB 同步。

质量验收的目标是确认 wiki 是否可被用户和 Agent 默认使用。

## 2. 触发条件

以下情况必须执行 wiki 质量验收：

1. candidate 晋升为 active 后。
2. active 页面新增、删除或修改正文结论后。
3. active 页面新增、删除或修改 `相关链接` 后。
4. 索引页新增或移除下位 active 页面后。
5. 发现断链、冲突、生命周期表述不一致或证据缺失时。

只修改错别字、空行或纯排版时，不需要执行完整质量验收。

## 3. 验收对象

验收对象分为两类：

| 对象 | 含义 |
| --- | --- |
| 单页面验收 | 检查一个 wiki 页面是否结构完整、证据可追溯、链接有效。 |
| 知识簇验收 | 检查一组相关 active 页面是否能形成可导航、可复用、可回答问题的知识网络。 |

索引页必须执行知识簇验收。

对象页、主题页、流程页、政策页、模板页、案例页、对比页至少执行单页面验收。

如果页面是某个知识簇的主入口，必须同时执行知识簇验收。

## 4. 单页面验收标准

单页面必须同时满足以下条件：

1. frontmatter 必填字段齐全。
2. `status`、`wiki_page_type`、`risk_level` 使用允许值。
3. `status: active` 页面必须包含 `promoted_at` 和 `promotion_basis`。
4. 流程页必须包含 `process_level`。
5. `source_refs` 必须指向 raw 来源。
6. 正文必须包含 `当前结论`、`适用范围`、`依据来源`、`相关链接`。
7. `当前结论` 必须是可执行判断，不是空泛摘要。
8. `适用范围` 必须同时说明适用和不适用场景。
9. `依据来源` 必须能追溯到 raw。
10. `相关链接` 必须使用类型化链接。
11. 页面内不得残留模板说明文字。
12. 页面内不得残留与当前生命周期冲突的表述。

生命周期冲突示例：

```text
页面 frontmatter 是 status: active，但正文写“当前状态仍为 candidate”。
```

## 5. 链接与图谱验收标准

链接必须满足以下条件：

1. 断链数量为 0。
2. active 页面必须至少连接一个已有 active 页面。
3. 相关链接必须写明关系类型。
4. 关系类型必须表达真实知识关系。
5. 索引页必须覆盖其直接下位 active 页面。
6. 知识簇主入口必须能导航到该知识簇的核心页面。

允许的关系类型包括：

```text
上位知识
下位知识
前置概念
前置知识
前置流程
前置政策
后续流程
后续知识
下位模板
下位案例
相关知识
相关政策
相关案例
同一项目
证据来源
替代
已被替代
冲突页面
```

禁止因为标题相似、目录相同、同一天创建或都提到 AI 而建立链接。

## 6. 知识簇可用性测试

知识簇验收必须使用真实问题测试。

每次知识簇验收至少提出 2 个真实使用问题。

真实问题必须满足：

1. 来自用户或业务场景。
2. 能验证该知识簇是否可导航。
3. 能验证页面结论是否可执行。
4. 能验证相关链接是否能支撑下一步判断。

测试结果只判断 wiki 是否可用，不写入新的业务结论。

可用性测试必须输出：

| 字段 | 含义 |
| --- | --- |
| 测试问题 | 本次用于验证的真实问题。 |
| 命中页面 | 回答该问题需要调用的 wiki 页面。 |
| 可执行路径 | 从入口页到具体页面的导航路径。 |
| 验收结论 | 通过、修复后通过、不通过。 |
| 处理动作 | 无、直接修复、进入待处理、进入冲突索引。 |

## 7. 验收结论

验收结论只允许三种：

| 结论 | 含义 |
| --- | --- |
| 通过 | 页面或知识簇可被用户和 Agent 默认使用。 |
| 修复后通过 | 存在可直接修复的问题，修复后可使用。 |
| 不通过 | 存在证据缺失、冲突、断链或用户判断缺口，不能默认使用。 |

## 8. 处理动作

验收发现问题时，只允许执行以下动作：

1. 直接修复页面正文、frontmatter、链接或索引。
2. 在 `_meta/待处理索引.md` 记录不能直接修复的问题。
3. 在 `_meta/冲突索引.md` 记录知识冲突。
4. 将不应默认使用的页面降级为 `candidate` 或 `deprecated`。

禁止行为：

1. 禁止为了通过验收而编造 raw 证据。
2. 禁止把可用性测试答案写成新的业务结论。
3. 禁止用弱相关链接凑图谱密度。
4. 禁止在发现 active 冲突时直接覆盖旧结论。

## 9. 治理同步

质量验收后必须检查以下文件是否需要同步：

1. `_index.md`
2. `_meta/编译日志.md`
3. `_meta/冲突索引.md`
4. `_meta/晋升记录.md`
5. `_meta/待处理索引.md`

如果只是修复页面内部链接、生命周期残留表述或索引覆盖，不需要新增编译日志。

如果验收导致页面状态变化，必须更新晋升记录、待处理索引或冲突索引。

## 10. 后续同步要求

修改以下内容时，必须同步更新本规则和规则变更日志：

1. 质量验收触发条件。
2. 单页面验收标准。
3. 链接与图谱验收标准。
4. 可用性测试输出字段。
5. 验收结论取值。
6. 验收失败处理动作。
7. 治理同步要求。

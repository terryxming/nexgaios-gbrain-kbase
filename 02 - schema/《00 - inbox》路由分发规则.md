# 00 - inbox 路由分发规则

版本：v0.6.0

生命周期：Draft

日期：2026-06-20

适用范围：`00 - raw/00 - inbox/*`

## 1. 规则边界

本规则只做：

1. 检查、补全、修正 inbox Markdown frontmatter。
2. 为非 Markdown 原文件检查同名 Markdown 说明卡。
3. 输出待 Codex 语义路由的材料清单。
4. 规定 Codex 语义路由的判断方法和输出字段。
5. 输出无法判断时的候选清单和理由。
6. 生成 inbox 路由审核表。

本规则不做：

1. 不改写正文内容。
2. 不执行 wiki 编译。
3. 不自动移动文件。
4. 不删除用户材料。
5. 不新建目录。
6. 不执行 Git / GitHub / Supabase 同步。

移动文件必须等用户确认。

职责分工：

1. 脚本只做结构校验、结构修复、材料包检查和路由输入输出。
2. 脚本不得根据关键词判断目标目录、置信度或分发后 `domain_hint`。
3. Codex 负责阅读正文主体，推断 `material_type`、`tags`、目标目录、判断依据、置信度和分发后 `domain_hint`。
4. 用户明确执行路由后，Codex 才能移动文件或材料包。

## 2. 基本入口

所有新知识材料先进入：

```text
00 - raw/00 - inbox
```

进入来源包括：网页剪藏、Agent 回写、用户手动维护、文件导入。

材料格式包括：Markdown、PDF、图片、截图、Office 文档。

例外：

```text
02 - schema 下的规则文件、脚本、变更日志不进入 inbox。
```

`00 - inbox` 是待处理入口，不是长期归档目录。能判断目标目录的材料，必须在用户确认后分发到 raw 领域目录；不能判断的材料，必须留下失败原因。

## 3. 文件类型

### Markdown

Markdown 文件直接在文件开头维护 frontmatter。

### 非 Markdown

非 Markdown 文件必须使用同名 Markdown 说明卡承载 frontmatter。

示例：

```text
亚马逊价格政策截图.png
亚马逊价格政策截图.md
```

说明卡正文第一行：

```markdown
关联文件：亚马逊价格政策截图.png
```

处理规则：

1. 原文件和同名说明卡组成一个材料包。
2. 材料包必须一起分发到同一个目标目录。
3. 缺少同名说明卡时，不进入分发，输出失败原因。
4. 非 Markdown 原文件不得因为不能写 frontmatter 而长期保留在 inbox。

## 4. Frontmatter 模板

inbox Markdown 必须使用以下字段，字段顺序固定：

```yaml
---
title:
status: inbox
created:
source_type:
material_type:
domain_hint: unknown
tags: []
---
```

字段只允许这 7 个：

```text
title
status
created
source_type
material_type
domain_hint
tags
```

多余字段删除。

## 5. 字段规则

### title

补全顺序：

1. 已有 `title`。
2. 正文第一个一级标题。
3. 文件名去掉扩展名。

标题和正文冲突指：标题主对象与正文主体对象不同，或标题包含的材料类型词与正文推断的 `material_type` 不一致。

材料类型词包括：

```text
方案
报告
模板
指南
```

发生冲突时，以正文为准，并在 dry-run 中输出“是否需要修改文件名：是”。

### status

inbox 阶段固定为：

```yaml
status: inbox
```

分发完成后改为：

```yaml
status: raw
```

分发完成后必须新增：

```yaml
compile_status: 未编译
```

`compile_status` 的后续状态变化以以下规则为准：

```text
02 - schema/《00 - raw》编译状态规则.md
```

### created

补全顺序：

1. 已有 `created`。
2. 文件最后修改时间。

格式：

```text
YYYY-MM-DD HH:mm
```

### source_type

允许值：

```text
用户写入
Agent回写
网页剪藏
unknown
```

修正规则：

1. 已有值属于允许值：沿用。
2. 缺失、空值、非法值：`unknown`。

禁止根据正文、标题、文件名推断 `source_type`。

### material_type

允许值：

```text
普通笔记
对话沉淀
网页文章
研究资料
模板
指南
方案
报告
unknown
```

判断证据优先级：

1. 正文主体。
2. 正文结构。
3. 一级标题。
4. 文件名。
5. 原 frontmatter。

正文主体是最高优先级。标题写“方案”，但正文是在解释概念、总结知识、记录学习过程时，`material_type` 判为 `普通笔记`，并提示修改文件名。

推断规则：

| 正文证据 | material_type |
| --- | --- |
| 解释概念、总结知识、学习讲义、知识点整理 | 普通笔记 |
| ChatGPT / Codex / 用户对话整理 | 对话沉淀 |
| 网页正文、网页转载、网页剪藏正文 | 网页文章 |
| 调研清单、资料汇总、论文列表、参考链接集合 | 研究资料 |
| 可复用格式、占位结构、提示词模板、笔记模板 | 模板 |
| 操作步骤、使用说明、教程、配置步骤 | 指南 |
| 目标、约束、流程、规则、架构、实施路径、运行方案 | 方案 |
| 分析结论、发现、评估、阶段结果、复盘报告 | 报告 |
| 信息不足或不属于以上类型 | unknown |

### domain_hint

inbox 阶段固定为：

```yaml
domain_hint: unknown
```

分发确认后，改为目标领域：

```text
AI Work
Amazon
认知管理
财务投资
设计
产品
```

### tags

用途：

```text
补充正文主题，辅助横向检索。
```

生成规则：

1. 只从正文内容或说明卡正文提取。
2. 优先提取正文中用于判断目标领域、材料类型或目标目录的词。
3. 不从标题、文件名、路径、来源类型、状态、领域名中提取。
4. 不确定时使用 `tags: []`。
5. 单个 tag 不允许空格。
6. 英文多词 tag 使用连字符。
7. 删除重复 tag。
8. 最多保留 8 个 tag。

合格示例：

```yaml
tags:
  - GBrain
  - agent-memory
  - Garry-Tan
```

不合格示例：

```yaml
tags:
  - agent memory
  - 网页剪藏
  - AI Work
```

## 6. Frontmatter 修复顺序

对 inbox Markdown 执行以下结构修复顺序：

1. 读取 frontmatter。
2. 删除 7 个字段外的多余字段。
3. 按固定顺序重排字段。
4. 补全 `title`。
5. 补全 `status: inbox`。
6. 补全 `created`。
7. 修正 `source_type`。
8. 缺失、空值、非法 `material_type` 改为 `unknown`。
9. 设置 `domain_hint: unknown`。
10. 缺失或结构非法的 `tags` 改为 `tags: []`。
11. 再次校验。

校验失败时停止分发，输出失败字段、失败原因和需要用户判断的事项。

`material_type` 和 `tags` 的正文语义推断由 Codex 在语义路由阶段执行。

## 7. 分发前置条件

Markdown 文件必须满足：

1. frontmatter 校验通过。
2. `status: inbox`。
3. `domain_hint: unknown`。

非 Markdown 材料包必须满足：

1. 原文件存在。
2. 同名 Markdown 说明卡存在。
3. 说明卡 frontmatter 校验通过。
4. 说明卡 `status: inbox`。
5. 说明卡 `domain_hint: unknown`。

不满足前置条件时，建议目标目录为 `待定`。

## 8. Dry-run 输出

脚本 dry-run 只输出路由输入，不输出目标目录建议。

脚本 dry-run 必须包含：

```text
文件
校验结果
路由状态
语义判断责任方
判断要求
```

脚本输出规则：

1. 校验通过的材料，路由状态为 `待 Codex 语义判断`。
2. 校验失败的材料，路由状态为 `不可路由`。
3. 缺少同名说明卡的非 Markdown 文件，路由状态为 `不可路由`。
4. 脚本不得输出 `高 / 中 / 低` 置信度。
5. 脚本不得输出具体目标目录。

Codex 语义路由输出必须包含：

```text
文件
建议 material_type
建议 tags
建议目标目录
候选目标目录
判断依据
置信度
目标目录是否存在
是否需要修改文件名
动作建议
```

Codex 语义路由示例：

```text
文件：亚马逊价格政策截图.png; 亚马逊价格政策截图.md
建议目标目录：02 - Amazon/0202 - 政策
判断依据：说明卡正文出现“亚马逊”“参考价格”“美国站/英国站”，内容是平台政策更新。
置信度：高
分发后 domain_hint：Amazon
目标目录是否存在：是
是否需要修改文件名：否
```

Codex 无法判断示例：

```text
文件：孤立截图.png
建议目标目录：待定
判断依据：缺少同名 Markdown 说明卡；无法读取正文、主题、来源和用途。请用户判断：1. 该图属于哪个领域；2. 是否属于某个项目；3. 是否需要新增说明卡。
置信度：低
分发后 domain_hint：unknown
目标目录是否存在：否
是否需要修改文件名：待定
```

## 9. 自动执行与 Excel 审核表

Excel 审核表只承载需要用户决策的材料，不承载无需用户决策的材料。

无需用户决策的材料由 Codex 直接执行移动。

直接执行必须同时满足：

1. `动作建议` 为 `可移动`。
2. 置信度为 `高`。
3. `建议目标目录` 是唯一目录。
4. 目标目录存在。
5. 源文件存在于 `00 - raw/00 - inbox`。
6. 目标文件不存在。
7. 当前 `status` 是 `inbox`。
8. 当前 `domain_hint` 是 `unknown`。
9. `是否需要修改文件名` 为 `否`。

直接执行后，必须输出数量摘要和移动结果。

直接执行移动后，必须将目标 raw Markdown 或同名说明卡 frontmatter 更新为：

```yaml
status: raw
compile_status: 未编译
```

以下材料进入 Excel 审核表：

1. 置信度为 `中` 或 `低`。
2. `动作建议` 为 `待用户确认` 或 `不可路由`。
3. `建议目标目录` 为 `待定` 或存在多个候选目录。
4. 需要用户补充目标目录、用途、项目归属或文件名处理意见。

Excel 审核表是用户决策入口，不是 GBrain 知识材料。

保存位置固定为：

```text
C:\Users\terry\Downloads
```

文件名固定为：

```text
YYYY-MM-DD - NN - inbox 路由审核表.xlsx
```

`NN` 为当天两位数序号，从 `01` 开始，已有同名文件时递增。

审核表字段顺序固定为：

```text
文件
问题类型
建议 material_type
建议 tags
建议目标目录
候选目标目录
判断依据
置信度
是否需要修改文件名
用户决策
用户指定目标目录
用户备注
执行状态
```

字段规则：

1. `文件` 只写文件名，不写完整路径。
2. 执行器用固定 inbox 路径和 `文件` 字段重建源路径。
3. 如果源文件不在 inbox，执行器必须跳过，不得猜测路径。
4. `问题类型` 只允许：`多候选目录`、`信息不足`、`目标目录缺失`、`需要改名`、`不可路由`。
5. `建议 material_type` 由 Codex 阅读正文主体后判断。
6. `建议 tags` 由 Codex 从正文内容或说明卡正文提取。
7. `候选目标目录` 只在两个以上目录满足证据或无法唯一判断时填写。
8. `用户指定目标目录` 只有在用户不同意 `建议目标目录` 或 `建议目标目录` 为 `待定` 时填写。
9. 未执行移动时，`执行状态` 填 `未执行`。

`用户决策` 只允许：

```text
同意移动
改目标目录
暂不处理
需要补充信息
不进入知识库
```

夜间自动化必须先直接执行无需用户决策的材料，再为需要用户决策的材料生成 Excel 审核表。

如果没有需要用户决策的材料，不生成 Excel 审核表。

对话只输出自动执行数量、待用户决策数量、审核表路径。

## 10. Codex 目标目录判断方法

目标目录不是靠固定关键词表硬匹配，而是按以下顺序判断。

主对象指正文中用于判断目标领域或目标目录的业务对象、项目名、产品名、技术名或政策名。

正文存在多个对象时，以正文主体段落中反复展开说明的对象为主对象。

1. 读正文主体，确认主对象。
2. 判断主对象属于哪个领域：AI Work、Amazon、认知管理、财务投资、设计、产品。
3. 判断材料用途：学习、项目、行业资讯、政策、工具、方法。
4. 在现有 raw 目录中选择最具体且已经存在的目录。
5. 如果两个以上目录同时满足证据，且正文没有明确主项目或主用途，输出 `待定`，并在判断依据里列出候选目录和理由。
6. 如果没有合适目录，输出 `待定`，并说明是信息不足还是现有目录缺失。

正确示例：

```text
正文：解释 PR、CI、Pull Request、GitHub Actions 是什么。
建议目标目录：01 - AI Work/0101 - 学习/0109 - GitHub 手册
理由：正文主体是 GitHub 基础概念学习，不是项目方案。
```

```text
正文：亚马逊美国站和英国站参考价格政策更新。
建议目标目录：02 - Amazon/0202 - 政策
理由：正文主体是 Amazon 平台政策。
```

```text
正文：Obsidian Web Clipper 模板字段配置。
建议目标目录：06 - 产品/0602 - 工具/Obsidian/插件
理由：正文主体是工具插件配置。
```

错误示例：

```text
标题：GBrain 运行方案
正文：解释 PR 和 CI 是什么。
错误目录：01 - AI Work/0102 - 项目/Nexgaios GBrain
错误原因：只看标题，忽略正文主体。
```

```text
source_type：网页剪藏
正文：亚马逊政策更新。
错误目录：01 - AI Work/0103 - 行业资讯
错误原因：把来源类型当成目录依据。
```

```text
tags：Amazon
正文：没有有效内容。
错误目录：02 - Amazon/0202 - 政策
错误原因：只看 tag，正文证据不足。
```

## 11. 无法判断时的处理

出现以下情况，建议目标目录必须为 `待定`：

1. 正文为空或只有链接。
2. 非 Markdown 文件缺少同名说明卡。
3. 正文同时指向多个领域，无法判断主领域。
4. 正文同时指向多个项目，无法判断主项目。
5. 现有 raw 目录没有合适目标目录。
6. 标题、正文、说明卡互相矛盾。

输出要求：

1. 在判断依据中列出无法判断的原因。
2. 在判断依据中列出候选目录和各自理由。
3. 明确请求用户判断需要补充的信息。

## 12. 置信度

置信度用于控制后续动作风险，不代表知识真假。

### 高

用途：

```text
用户确认后，可以进入移动候选。
```

条件：

1. 正文主体明确。
2. 目标领域明确。
3. 目标目录已存在。
4. 判断依据不少于 2 条。
5. 不需要新建目录。
6. 不需要用户补充关键信息。

### 中

用途：

```text
不能自动移动，需要用户在候选目录中选择。
```

条件：

1. 目标领域明确。
2. 两个以上现有目录同时满足目标目录证据。
3. 正文没有明确主项目或主用途。

### 低

用途：

```text
不移动，要求用户补充信息。
```

条件：

1. 目标领域不明确。
2. 判断依据不足。
3. 缺少同名说明卡。
4. 目标目录不存在。
5. 标题、正文、说明卡冲突。

## 13. 规则与脚本同步

本规则依赖脚本：

```text
02 - schema/自动化脚本/00 - inbox 校验与路由输入 dry-run.ps1
```

同步要求：

1. 修改本规则时，必须检查脚本是否需要同步。
2. 修改脚本路由逻辑时，必须同步更新本规则。
3. 修改字段、枚举、目标目录、置信度、输出字段时，必须同步更新本规则、脚本和变更日志。
4. 脚本文件头必须标明依赖的规则文件和规则版本。
5. 每次规则或脚本变更后，必须至少验证空 inbox、Markdown 样本、非 Markdown 材料包、缺少说明卡样本、无法判断样本。

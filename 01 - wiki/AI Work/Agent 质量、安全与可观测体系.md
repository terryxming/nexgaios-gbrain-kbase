# Agent 质量、安全与可观测体系

## 当前结论

1. Agent 质量不能只看最终答案是否像样，还必须同时检查输入识别、上下文、工具调用、执行过程、结果、风险和运营状态。
2. Agent 质量体系至少包含四层：输入质量、过程质量、结果质量、运营质量。
3. Test 和 Eval 不能混用：Test 检查是否按规则运行，Eval 判断结果质量是否足够好。
4. Trace 是产品级 Agent 的基础设施；没有 Trace，就无法还原过程、定位失败、评估工具调用和做回归治理。
5. Guardrail 必须覆盖输入、输出、工具、权限和安全边界，不能只在最终回答阶段检查。
6. 成本、延迟、稳定性、失败率和回滚能力是产品化指标，不是上线后的附加优化。

## 适用范围

### 适用

- 设计 Agent、Skill、MCP 工具链的质量体系。
- 为 Agent 建立 Test、Eval、Trace、Guardrail 和 AgentOps。
- 判断某个 Agent 是否达到产品级可上线标准。
- 复盘 Agent 执行失败、误触发、越权、输出不合格或成本失控问题。

### 不适用

- 不替代具体行业任务的业务评分标准。
- 不替代具体观测平台、日志系统或 eval 框架实现文档。
- 不适用于一次性实验 Demo 的最低验证。

## 质量四层模型

| 层级 | 关注点 | 示例 |
| --- | --- | --- |
| L1 输入质量 | 任务是否识别正确 | 用户要诊断广告，不是学习 ACOS 概念。 |
| L2 过程质量 | 步骤、上下文、工具是否正确 | 是否读取正确报表、调用正确工具。 |
| L3 结果质量 | 输出是否准确、完整、可执行 | 是否给出证据、根因和动作。 |
| L4 运营质量 | 成本、延迟、稳定性、安全 | 是否超预算、是否有日志、是否可回滚。 |

## Test 与 Eval 分工

| 对比项 | Test | Eval |
| --- | --- | --- |
| 核心问题 | 有没有按规则运行？ | 做得好不好？ |
| 判断方式 | pass / fail | score / grade / review |
| 关注点 | 触发、格式、流程、边界 | 有用性、准确性、完整性、业务价值 |
| 示例 | 缺少数据时是否停止 | 广告诊断是否抓住主因 |

## 最小质量清单

| 模块 | 检查问题 |
| --- | --- |
| 输入测试 | 是否正确触发、拒绝误触发、处理缺失信息。 |
| 输出测试 | 是否符合结构、字段、约束、引用和风险要求。 |
| 过程评估 | 是否按正确步骤读取上下文和调用工具。 |
| 结果评估 | 是否准确、完整、可执行、有证据。 |
| Trace | 是否能还原每一步输入、上下文、工具、输出和错误。 |
| Guardrail | 是否拦截 Prompt Injection、越权、泄露和危险动作。 |
| AgentOps | 是否监控成本、延迟、失败率、重试率、版本和回滚。 |

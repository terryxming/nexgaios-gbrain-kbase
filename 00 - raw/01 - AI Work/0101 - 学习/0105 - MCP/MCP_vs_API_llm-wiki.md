# MCP 与 API 的区别、联系、适用边界

## 0. 一句话结论

| 概念 | 一句话理解 |
|---|---|
| API | 软件系统之间调用能力的接口 |
| MCP | AI Agent 连接外部工具、资源、提示词和工作流的标准协议 |

更直白地说：

> **API 是软件世界的“办事窗口”；MCP 是 AI Agent 的“统一办事通道”。**

---

## 1. API 是什么

API，全称 **Application Programming Interface**。

它的本质是：

> **一个软件系统把自己的能力，以明确规则暴露给另一个软件系统调用。**

### 1.1 简单理解

| 维度 | API |
|---|---|
| 本质 | 软件能力的调用入口 |
| 使用者 | 开发者、前端、后端、其他系统 |
| 调用方式 | HTTP、SDK、函数、CLI、数据库接口等 |
| 典型例子 | 查询订单、创建支付、获取库存、登录验证 |
| 核心目标 | 让系统之间稳定、明确、可控地交互 |

### 1.2 API 的典型结构

```text
请求方 → API → 被调用系统
```

例如：

```text
前端页面 → 查询订单 API → 订单系统
```

用户点击“查看订单”，前端调用订单 API，订单系统返回订单信息。

---

## 2. MCP 是什么

MCP，全称 **Model Context Protocol**。

它的本质是：

> **让 AI 应用 / Agent 以标准方式连接外部工具、数据、资源和工作流的协议。**

MCP 不是普通 API 的替代品，而是面向 Agent 的连接层。

### 2.1 简单理解

| 维度 | MCP |
|---|---|
| 本质 | Agent 连接外部世界的标准协议 |
| 使用者 | AI 应用、LLM、Agent、AI IDE |
| 连接对象 | 工具、文件、数据库、API、Prompt、工作流 |
| 典型例子 | GitHub MCP、Filesystem MCP、Postgres MCP、Slack MCP |
| 核心目标 | 让 Agent 能发现、理解、读取并调用外部能力 |

### 2.2 MCP 暴露的核心能力

| MCP 能力 | 含义 | 例子 |
|---|---|---|
| Tools | 可执行动作 | 查询订单、发邮件、创建 Issue、调用搜索 |
| Resources | 可读取上下文 | 文件、数据库 schema、知识库、业务文档 |
| Prompts | 可复用提示词模板 | 客服回复模板、代码审查模板、分析流程模板 |

---

## 3. API 与 MCP 的核心区别

| 对比项 | API | MCP |
|---|---|---|
| 面向对象 | 软件系统、开发者 | AI 应用、LLM、Agent |
| 解决问题 | 软件如何调用软件 | Agent 如何连接外部世界 |
| 主要形态 | Endpoint、Method、SDK、Function | Tool、Resource、Prompt、Server |
| 是否专为 LLM 设计 | 否 | 是 |
| 是否强调上下文 | 不一定 | 强调上下文供给 |
| 是否支持工具发现 | 通常依赖文档 | Client 可发现 Server 暴露的能力 |
| 调用决策者 | 程序逻辑明确调用 | Agent 可能根据任务自主选择 |
| 适合场景 | 稳定系统集成 | AI 工作流、Agent 工具调用 |
| 底层关系 | 原始能力接口 | 可封装 API、数据库、文件系统和工具 |

核心区别：

> **API 是“能力入口”；MCP 是“把能力包装成 Agent 可理解、可发现、可调用的上下文协议”。**

---

## 4. 生活化理解

## 4.1 API 像“点餐窗口”

你去麦当劳点餐：

```text
我要一个巨无霸套餐，少冰可乐，不要洋葱。
```

这个过程很像 API。

| API 特点 | 点餐窗口里的对应 |
|---|---|
| 接口明确 | 柜台就是点餐入口 |
| 参数明确 | 套餐、饮料、是否加冰、是否去洋葱 |
| 结果明确 | 返回一份餐 |
| 调用者要懂规则 | 你要知道菜单上有什么，怎么点 |
| 不负责帮你规划 | 你不说，柜台不会主动安排晚餐 |

所以：

> **API = 你明确知道要什么，然后按规则告诉系统，系统返回结果。**

---

## 4.2 MCP 像“私人助理”

你不是去点餐窗口，而是对私人助理说：

```text
我今晚有朋友来家里聚会，帮我安排一下晚餐，别太贵，最好有小孩也能吃的东西。
```

私人助理可能会：

1. 看你的日历，确认朋友几点来；
2. 看冰箱里还有什么食材；
3. 查附近餐厅；
4. 比较配送时间；
5. 排除太辣、不适合小孩的菜；
6. 给你推荐几个方案；
7. 最后帮你下单。

这个过程更像 MCP。

| MCP 特点 | 私人助理里的对应 |
|---|---|
| 面向目标 | 你说的是“帮我安排晚餐” |
| 连接多个工具 | 日历、外卖、地图、支付、通讯录 |
| 读取上下文 | 谁来、几点来、预算多少、有没有小孩 |
| 选择工具 | 助理判断先查什么、后做什么 |
| 执行动作 | 推荐、下单、发消息、总结 |

所以：

> **MCP = 你告诉 AI 目标，AI 通过 MCP 去发现工具、读取资料、调用能力、完成任务。**

---

## 4.3 API 像“一个个窗口”，MCP 像“助理的统一办事系统”

### API 模式

```text
外卖窗口：点餐
地图窗口：查路线
日历窗口：看时间
支付窗口：付款
短信窗口：发消息
```

你要自己知道：

- 找哪个窗口；
- 怎么说；
- 传什么参数；
- 下一步找谁；
- 出错后怎么处理。

### MCP 模式

```text
你 → AI 助理 → MCP → 外卖 / 地图 / 日历 / 支付 / 短信
```

你只说目标：

```text
帮我安排今晚聚会的晚餐。
```

AI 助理通过 MCP 知道：

- 有哪些工具可用；
- 每个工具能做什么；
- 需要读取哪些上下文；
- 需要调用哪些动作；
- 哪些动作需要你确认。

## 5. API 与 MCP 的关系

## 5.1 MCP 经常包在 API 外面

很多 MCP Server 本质上是一个 **Agent 适配器**。

例如 GitHub：

```text
AI Agent
  ↓
MCP Client
  ↓
GitHub MCP Server
  ↓
GitHub API
  ↓
GitHub 数据与操作
```

这里：

| 层级 | 作用 |
|---|---|
| GitHub API | 真正执行读取 PR、创建 Issue、查询仓库等动作 |
| GitHub MCP Server | 把 GitHub API 包装成 Agent 能理解和调用的工具 |
| MCP Client | 让 AI 应用连接 MCP Server |
| Agent | 根据任务判断是否调用这些工具 |

所以：

> **MCP 不替代 API。很多 MCP Server 背后仍然是在调用 API。**

---

## 5.2 API 是底层能力，MCP 是 Agent 连接层

传统软件系统：

```text
前端 / 后端服务 → API → 业务系统
```

Agent 系统：

```text
AI Agent → MCP → API / 数据库 / 文件 / 工具 / 工作流
```

可以理解为：

| 层级 | 作用 |
|---|---|
| 业务系统 | 订单系统、GitHub、数据库、文件系统 |
| 能力接口 | API、SQL、SDK、CLI、文件读写 |
| Agent 连接协议 | MCP Server 把能力包装成 Tools / Resources / Prompts |

最终关系：

```text
API = 系统能力怎么被调用
MCP = Agent 怎么发现并使用这些能力
```

---

## 6. 适用边界

## 6.1 什么时候用 API

适合 API 的场景：

| 场景 | 原因 |
|---|---|
| 前端调用后端 | 调用逻辑明确 |
| 后端服务之间通信 | 稳定、低延迟、可测试 |
| 对外开放平台能力 | 给开发者清晰接口 |
| 高频、大规模、确定性调用 | API 更轻、更直接 |
| 业务核心交易链路 | 支付、订单、库存等需要强约束 |
| 系统同步数据 | 路径清晰、结果明确 |

一句话：

> **动作明确、规则固定、结果确定，用 API。**

### 示例

```text
用户点击“下单”
→ 前端调用 createOrder API
→ 后端生成订单
```

这个过程不需要 MCP，因为动作明确、路径固定、结果可预期。

---

## 6.2 什么时候用 MCP

适合 MCP 的场景：

| 场景 | 原因 |
|---|---|
| AI 助手需要连接多个工具 | MCP 可以统一工具接入 |
| Agent 需要读取文件、数据库、知识库 | MCP Resources 可以暴露上下文 |
| AI IDE 需要理解代码仓库 | MCP 可以连接 Git、GitHub、文件系统 |
| 企业内部系统要接给 AI | 避免每个 AI 应用单独集成 |
| Agent 需要工具发现 | MCP Server 可暴露可用工具列表 |
| 需要把现有 API 包装给 LLM 使用 | MCP Server 可作为 API Adapter |
| 任务需要多步骤组合 | Agent 可以根据目标选择工具 |

一句话：

> **目标复杂、需要 AI 判断、需要上下文、需要连接多个工具，用 MCP。**

### 示例

```text
用户说：帮我看一下这个仓库最近的 PR，并总结风险
→ Agent 通过 GitHub MCP 读取 PR
→ 通过文件系统 MCP 查看代码
→ 通过数据库 MCP 查询相关记录
→ 最后生成分析
```

---

## 6.3 不适合用 MCP 的场景

| 不适合场景 | 原因 |
|---|---|
| 没有 AI / Agent 参与 | MCP 的价值不明显 |
| 只是普通软件调用 | API 更简单 |
| 极高频、低延迟调用 | MCP 层增加复杂度 |
| 强确定性交易链路 | 不应让 Agent 直接决定关键动作 |
| 权限边界不清晰 | 暴露工具后风险更高 |
| 没有审计、鉴权、确认机制 | 容易误调用或越权调用 |

边界提醒：

> **MCP 适合辅助 Agent 使用外部能力，但不代表所有关键业务操作都应该交给 Agent 自动执行。**

## 7. 用订单系统举例

## 7.1 API 方式

订单系统提供 API：

```text
GET /orders/123
POST /orders/123/refund
PUT /orders/123/address
```

开发者在代码里明确写：

```text
调用查询订单 API
调用退款 API
调用修改地址 API
```

特点：

| 特点 | 说明 |
|---|---|
| 程序员控制调用逻辑 | 哪一步调用哪个 API 是固定的 |
| 接口固定 | 参数和返回结构明确 |
| 行为确定 | 适合业务系统 |
| 可测试 | 输入输出容易验证 |

---

## 7.2 MCP 方式

你做一个 `Order MCP Server`，向 AI 暴露：

```text
Tool: search_order
Tool: summarize_order_issue
Tool: request_refund
Resource: order_policy.md
Resource: refund_rules.md
Prompt: customer_service_reply_template
```

用户对 AI 说：

```text
帮我看一下这个客户为什么要求退款，并起草回复。
```

Agent 可以：

1. 调用 `search_order`；
2. 读取 `refund_rules.md`；
3. 判断是否符合退款条件；
4. 使用 `customer_service_reply_template`；
5. 起草客服回复；
6. 必要时请求人工确认后提交退款。

特点：

| 特点 | 说明 |
|---|---|
| Agent 根据目标选择工具 | 用户不需要指定每个接口 |
| MCP 提供上下文 | 规则、文档、模板可作为 Resources / Prompts |
| 背后仍可调用 API | MCP Server 可封装订单 API |
| 适合 AI 工作流 | 尤其适合分析、判断、生成、组合动作 |

## 8. 用 GitHub 项目举例

## 8.1 只用 API

你需要明确告诉系统：

```text
调用 GitHub API
获取 repo 列表
获取 issue 列表
读取 PR diff
查询 workflow 状态
创建 comment
```

这要求你知道 GitHub API 怎么用。

---

## 8.2 用 MCP

你可以对 AI 说：

```text
帮我检查这个仓库最近的 PR 有没有风险，并给出 review 建议。
```

AI 通过 GitHub MCP 可以：

1. 找到仓库；
2. 读取 PR；
3. 查看代码变更；
4. 读取 issue 背景；
5. 分析潜在风险；
6. 输出 review 建议；
7. 必要时创建评论。

区别：

| 方式 | 用户体感 |
|---|---|
| API | 我在操作系统 |
| MCP | 我在指挥助理 |
| Agent + MCP | 我在分配任务 |

## 9. 选型判断框架

只需要问一个核心问题：

> **这个能力是给普通软件调用，还是给 AI Agent 理解和使用？**

| 判断问题 | 更适合 |
|---|---|
| 只是让前端或后端调用一个功能？ | API |
| 需要 Agent 自己发现、选择、组合工具？ | MCP |
| 只是做一个稳定业务接口？ | API |
| 要把内部系统接入 ChatGPT / Claude / Cursor？ | MCP |
| 需要高频、确定性、低延迟调用？ | API |
| 需要让 AI 读取上下文并执行任务？ | MCP |
| 已有 API，想让 Agent 使用它？ | API + MCP Server |
| 涉及支付、退款、删除等高风险动作？ | API + 权限控制 + MCP 人工确认 |

---

## 10. 常见误区

| 误区 | 正确认知 |
|---|---|
| MCP 会替代 API | 不会。MCP 经常封装 API |
| MCP 就是 API | 不准确。MCP 是面向 Agent 的协议层 |
| 有 API 就不需要 MCP | 如果要给 Agent 用，仍可能需要 MCP |
| 有 MCP 就不需要权限控制 | 错。MCP 更需要鉴权、审计和确认 |
| 所有系统都应该 MCP 化 | 错。普通软件调用仍然适合 API |
| Agent 可以直接操作所有工具 | 高风险动作应设置确认机制 |

## 11. 最终记忆模型

### 11.1 最短版

```text
API = 一个办事窗口
MCP = AI 助理连接很多办事窗口的标准方式
```

### 11.2 稍完整版

```text
API：你亲自去窗口办事
MCP：AI 助理知道有哪些窗口，并替你去办事
```

### 11.3 工程版

```text
API = 系统到系统的连接方式
MCP = Agent 到外部世界的连接方式
```

### 11.4 架构版

```text
业务系统
  ↓
API / SQL / SDK / CLI / 文件读写
  ↓
MCP Server
  ↓
MCP Client
  ↓
AI Agent
  ↓
用户自然语言目标
```

## 12. 总结

| 维度 | API | MCP |
|---|---|---|
| 本质 | 软件能力接口 | Agent 连接协议 |
| 面向谁 | 开发者和软件系统 | AI 应用和 Agent |
| 用户表达 | 明确参数和调用路径 | 自然语言目标 |
| 核心价值 | 系统稳定互调 | Agent 使用外部能力 |
| 适合任务 | 确定性、单点、固定流程 | 多步骤、上下文、工具组合 |
| 与对方关系 | 底层能力 | 可封装 API 的上层协议 |

最终结论：

> **API 解决“系统怎么被调用”；MCP 解决“Agent 怎么使用系统”。**  
> **API 是底层能力入口；MCP 是把这些能力交给 Agent 使用的标准连接层。**

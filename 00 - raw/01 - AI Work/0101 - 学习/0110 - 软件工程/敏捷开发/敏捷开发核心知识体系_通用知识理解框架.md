# 敏捷开发核心知识体系｜Scrum 与 User Story

## 0. 这份笔记的定位

| 项目 | 内容 |
|---|---|
| 知识主题 | 敏捷开发核心概念：Scrum 与 User Story 体系 |
| 整理框架 | 通用知识理解框架 12 层 |
| 核心目标 | 把多个敏捷术语从“零散概念”整理成“从需求产生到价值交付的顺序链路” |
| 主线 | Story Mapping → User Story → Template → INVEST → Story Splitting → Definition of Ready → Sprint Planning → Sprint → Definition of Done → Velocity / Review / Retrospective |
| 核心理解 | Scrum 是迭代交付框架；User Story 是需求价值单元；INVEST、DoR、DoD 是质量门禁；Velocity 是交付速率度量 |

---

# 1. 知识识别层：它属于什么类型的知识？

| 知识点 | 类型 | 简单理解 |
|---|---|---|
| Scrum | 框架类 / 流程类 / 系统类 | 管理复杂产品开发的迭代协作框架 |
| User Story | 概念类 / 需求单元 | 以用户价值为中心的轻量需求单元 |
| User Story Template | 方法类 / 表达模板 | 用 “作为谁 / 想做什么 / 为了什么” 写故事 |
| Story Mapping | 方法类 / 规划工具 | 用二维地图组织用户故事和版本路线 |
| Story Splitting | 方法类 / 拆解技术 | 把过大的故事拆成更小的价值切片 |
| INVEST | 原则类 / 检查清单 | 判断用户故事质量是否合格 |
| Definition of Ready | 规范类 / 入口门禁 | 判断故事能不能进入 Sprint |
| Definition of Done | 规范类 / 出口门禁 | 判断故事是否真正完成 |
| Velocity | 指标类 / 度量工具 | 统计团队每轮真正完成的故事点 |

## 1.1 一句话总判断

敏捷开发这一组概念不是并列堆叠，而是一条顺序链：

```text
先看全局 → 写出需求 → 检查质量 → 拆小需求 → 准备进入迭代 → 执行交付 → 判断完成 → 统计速率 → 复盘改进
```

# 2. 知识定位层：它们在整个敏捷体系中的位置

## 2.1 上位概念

| 上位概念 | 说明 |
|---|---|
| Agile / 敏捷 | 一组价值观和原则，强调快速反馈、持续交付、适应变化 |
| Scrum | 敏捷中最常见的过程框架之一 |
| Product Development | Scrum 常用于产品开发和复杂知识工作 |
| Requirements Management | User Story 体系属于敏捷需求管理方式 |

## 2.2 核心链路定位

| 顺序 | 概念 | 在链路中的作用 |
|---:|---|---|
| 1 | Story Mapping | 看全局，按用户流程组织需求 |
| 2 | User Story | 把需求表达成用户价值单元 |
| 3 | User Story Template | 用标准句式写清 Who / What / Why |
| 4 | INVEST | 检查用户故事质量 |
| 5 | Story Splitting | 把过大的用户故事拆小 |
| 6 | Definition of Ready | 判断故事是否准备好进入 Sprint |
| 7 | Sprint Planning | 选择本轮 Sprint 要做的故事 |
| 8 | Sprint | 在固定时间盒中完成产品增量 |
| 9 | Definition of Done | 判断故事 / 增量是否真正完成 |
| 10 | Velocity | 统计本轮真正完成的故事点 |
| 11 | Sprint Review | 展示增量，获取反馈 |
| 12 | Sprint Retrospective | 回顾流程，改进下一轮协作 |

## 2.3 相邻概念关系

| 概念 | 容易混淆对象 | 核心区别 |
|---|---|---|
| User Story | 传统需求文档 | User Story 是轻量对话入口，不是完整文档 |
| User Story | 技术任务 | User Story 以用户价值为中心，技术任务以实现工作为中心 |
| Story Mapping | 普通 Backlog | Story Mapping 有用户流程维度，Backlog 通常只是列表 |
| Story Splitting | 任务拆分 | Story Splitting 拆的是价值切片，不是前端 / 后端 / 数据库 |
| DoR | DoD | DoR 管“能不能开始”，DoD 管“是否完成” |
| Velocity | KPI | Velocity 是团队预测和规划工具，不是个人绩效指标 |

# 3. 问题背景层：为什么会出现这些概念？

## 3.1 传统开发常见问题

| 问题 | 表现 | 敏捷概念如何应对 |
|---|---|---|
| 需求太大 | 一个需求要做很久，看不到阶段成果 | User Story + Story Splitting |
| 需求太散 | 只看到功能清单，看不到用户完整路径 | Story Mapping |
| 需求不清 | 开发过程中反复追问、返工 | User Story Template + DoR |
| 需求质量差 | 故事不可估算、不可测试、没有用户价值 | INVEST |
| 完成定义不一致 | 开发说完成，测试说不能验，业务说不能用 | DoD |
| 交付不可预测 | 不知道团队每轮能完成多少 | Velocity |
| 计划僵化 | 一开始定死全部范围，后续变化成本高 | Scrum 的短周期 Sprint |
| 反馈太晚 | 产品做完才发现方向错 | Sprint Review + Increment |

## 3.2 这些概念共同解决的问题

```text
把模糊的大需求，
转成小而清晰的用户价值单元，
再通过短周期迭代交付，
持续检查结果和过程，
最终降低方向错误、返工和交付不可控。
```

# 4. 本体定义层：核心概念的一句话定义

| 概念 | 一句话定义 |
|---|---|
| Scrum | Scrum 是一种用于管理产品开发和知识工作的经验主义过程框架，通过 Sprint、角色、事件和工件让团队持续交付、检查和调整。 |
| User Story | User Story 是以用户价值为中心的轻量需求单元，用来承载团队围绕用户目标进行讨论、排期、实现和验收。 |
| User Story Template | User Story Template 是一种用户故事表达格式，用 “As a / I want to / So that” 写清用户角色、用户目标和用户价值。 |
| Story Mapping | Story Mapping 是把用户故事按“用户流程横轴”和“优先级 / 成熟度纵轴”组织成二维地图的方法。 |
| Story Splitting | Story Splitting 是把过大的用户故事拆成更小的、仍然保留独立业务价值的用户故事。 |
| INVEST | INVEST 是判断用户故事质量的六项原则：独立、可协商、有价值、可估算、足够小、可测试。 |
| Definition of Ready | DoR 是团队约定的故事进入 Sprint 前必须满足的准备标准。 |
| Definition of Done | DoD 是团队约定的故事或增量被认为真正完成前必须满足的完成标准。 |
| Velocity | Velocity 是团队在一个 Sprint 中真正完成的故事点总量，用于辅助预测和容量规划。 |

# 5. 结构组成层：敏捷开发核心结构

## 5.1 Scrum 的三大结构

| 结构 | 组成 | 作用 |
|---|---|---|
| Roles / 角色 | Product Owner、Scrum Master、Development Team | 明确谁负责价值排序、流程保障和产品交付 |
| Events / 事件 | Sprint、Sprint Planning、Daily Scrum、Sprint Review、Sprint Retrospective | 形成固定检查点和协作节奏 |
| Artifacts / 工件 | Product Backlog、Sprint Backlog、Increment、Definition of Done | 管理需求、计划、交付成果和完成标准 |

## 5.2 User Story 体系结构

| 层级 | 概念 | 作用 |
|---|---|---|
| 需求全局 | Story Mapping | 先看用户完整路径和版本结构 |
| 需求单元 | User Story | 把需求转成小的用户价值单元 |
| 表达模板 | User Story Template | 写清 Who / What / Why |
| 质量检查 | INVEST | 检查故事是否像一个好故事 |
| 拆分方法 | Story Splitting | 把过大的故事拆成小故事 |
| 入口门禁 | Definition of Ready | 判断故事是否可以进入 Sprint |
| 出口门禁 | Definition of Done | 判断故事是否真正完成 |
| 交付度量 | Velocity | 统计真正完成的故事点 |

## 5.3 顺序主线结构

```text
1. Story Mapping：看全局，组织需求
2. User Story：形成单个价值单元
3. User Story Template：写清角色、目标、价值
4. INVEST：检查故事质量
5. Story Splitting：把大故事拆小
6. DoR：判断是否准备好进入 Sprint
7. Sprint Planning：选择本轮要做什么
8. Sprint：固定时间盒内执行
9. DoD：判断是否真正完成
10. Velocity：统计完成量
11. Review：获取产品反馈
12. Retrospective：改进团队过程
```

# 6. 运行机制层：它如何发挥作用？

## 6.1 敏捷交付的因果链

| 起点 | 机制 | 结果 |
|---|---|---|
| 需求不确定 | 用 Story Mapping 看全局 | 避免只堆功能、忽略完整用户路径 |
| 需求太抽象 | 用 User Story 表达用户目标 | 让团队围绕用户价值讨论 |
| 故事质量不稳定 | 用 INVEST 检查 | 过滤不可估算、不可测试、无价值的故事 |
| 故事太大 | 用 Story Splitting 拆小 | 让故事能在 Sprint 内完成 |
| 开发前不清楚 | 用 DoR 作为入口门禁 | 减少中途返工 |
| 执行中容易失焦 | 用 Sprint 固定目标和时间盒 | 保护团队专注交付 |
| 完成定义不一致 | 用 DoD 作为出口门禁 | 保证完成意味着可交付 |
| 交付能力不清楚 | 用 Velocity 统计 Done 的故事点 | 辅助后续预测和计划 |
| 产品方向需要验证 | 用 Review 获取反馈 | 把反馈放回 Product Backlog |
| 团队协作需要改善 | 用 Retrospective 复盘过程 | 下一轮持续优化 |

## 6.2 最小闭环

```text
Product Backlog
↓
Story Mapping / Splitting / INVEST / DoR
↓
Sprint Planning
↓
Sprint Execution
↓
Definition of Done
↓
Increment
↓
Velocity
↓
Review / Retrospective
↓
Backlog 调整
```

# 7. 边界关系层：它不是什么？

## 7.1 Scrum 不是什么

| 错误理解 | 正确理解 |
|---|---|
| Scrum = 每天开站会 | Daily Scrum 只是 Scrum 的一个事件 |
| Scrum = 项目经理驱动 | Scrum 强调团队自组织和角色分工 |
| Scrum = 完整产品方法论 | Scrum 从优先级 Backlog 开始，但不规定 Backlog 如何产生 |
| Scrum = 保证项目成功 | Scrum 提供检查和调整机制，不替代产品判断和工程能力 |

## 7.2 User Story 不是什么

| 错误理解 | 正确理解 |
|---|---|
| User Story = 需求文档 | 它是轻量需求单元和对话入口 |
| User Story = 功能清单 | 它要表达用户目标和业务价值 |
| User Story = 技术任务 | 它不应该只是数据库、接口、页面 |
| User Story = Use Case | Use Case 更偏完整交互流程，User Story 更偏小增量 |

## 7.3 Story Splitting 不是什么

| 错误拆法 | 问题 |
|---|---|
| 前端 / 后端 / 数据库 | 这是技术层拆分，不是用户价值拆分 |
| 页面 / 按钮 / 弹窗 | 这是 UI 组件拆分，不一定产生独立价值 |
| 把句子机械切碎 | 会丢失业务价值和上下文 |

## 7.4 Velocity 不是什么

| 错误理解 | 正确理解 |
|---|---|
| Velocity 是团队 KPI | Velocity 是预测和容量规划工具 |
| Velocity 越高越好 | 盲目追高会导致估算膨胀和质量下降 |
| 可以比较不同团队 Velocity | 不同团队估算基准不同，不能直接比较 |
| 可以计算个人 Velocity | Velocity 是团队级指标 |

# 8. 方法应用层：怎么实际使用？

## 8.1 从模糊需求到 Sprint 的执行步骤

| 步骤 | 动作 | 输出 |
|---:|---|---|
| 1 | 识别用户目标和主流程 | 用户活动流 |
| 2 | 做 Story Mapping | 用户故事地图 |
| 3 | 找到 Walking Skeleton / MVP | 第一层可用产品骨架 |
| 4 | 写 User Story | 用户故事条目 |
| 5 | 使用 User Story Template | 清晰表达 Who / What / Why |
| 6 | 用 INVEST 检查 | 高质量故事 |
| 7 | 对大故事做 Story Splitting | 更小的价值切片 |
| 8 | 用 DoR 判断是否 Ready | READY-ready 故事 |
| 9 | Sprint Planning 选择故事 | Sprint Backlog |
| 10 | Sprint 中执行 | 产品增量 |
| 11 | 用 DoD 验收 | DONE-done 增量 |
| 12 | 统计 Velocity | 本轮完成量 |
| 13 | Sprint Review 获取反馈 | 新增 / 调整 Backlog |
| 14 | Retrospective 复盘流程 | 下一轮改进行动 |

## 8.2 User Story 写法模板

```text
作为一个【用户角色】，
我想要【完成某个动作 / 获得某个能力】，
以便【达成某个目标 / 获得某种价值】。
```

## 8.3 INVEST 检查表

| 字母 | 英文 | 中文 | 检查问题 |
|---|---|---|---|
| I | Independent | 独立的 | 这个故事能否尽量独立交付？ |
| N | Negotiable | 可协商的 | 它是否是讨论入口，而不是写死的合同？ |
| V | Valuable / Vertical | 有价值的 / 垂直切片 | 它能否产生用户或业务价值？ |
| E | Estimable | 可估算的 | 团队能否大致估算工作量？ |
| S | Small | 足够小的 | 它能否在一个 Sprint 内完成？ |
| T | Testable | 可测试的 | 是否能验证完成 / 未完成？ |

## 8.4 DoR 检查表

| 检查项 | 判断问题 |
|---|---|
| 用户角色清楚 | 谁需要这个故事？ |
| 用户价值清楚 | 做完后有什么价值？ |
| 范围清楚 | 包含什么，不包含什么？ |
| 验收标准清楚 | 怎么判断完成？ |
| 足够小 | 能否进入一个 Sprint？ |
| 可估算 | 团队是否能估算？ |
| 依赖清楚 | 是否依赖设计、接口、数据、其他故事？ |
| 风险可见 | 是否存在需要提前 Spike 的技术风险？ |

## 8.5 DoD 检查表

| 检查项 | 判断问题 |
|---|---|
| 功能实现 | 用户故事描述的能力是否完成？ |
| 验收标准 | 所有 Acceptance Criteria 是否通过？ |
| 测试 | 单元测试、集成测试、回归测试是否通过？ |
| Code Review | 代码是否经过审查？ |
| 部署准备 | 是否可以部署到目标环境？ |
| 文档 | 用户文档 / 技术文档 / 变更说明是否更新？ |
| 质量要求 | 性能、安全、权限等是否满足必要标准？ |
| PO 验收 | Product Owner 是否确认符合预期？ |

# 9. 案例验证层：用 Skill 项目跑一遍

## 9.1 背景

假设要做一个 Skill：

```text
complex-task-clarifier：当用户提出复杂、模糊、早期想法时，先进行需求澄清、任务拆解、评估标准设计，而不是直接执行。
```

## 9.2 Story Mapping

| 用户流程 | 识别复杂需求 | 澄清目标 | 盘点约束 | 拆解任务 | 设计验收 | 输出任务包 |
|---|---|---|---|---|---|---|
| 第一层：MVP | 判断是否模糊复杂 | 追问目标 | 识别关键约束 | 拆成阶段任务 | 给基础验收标准 | 输出简版任务说明 |
| 第二层：增强 | 识别任务类型 | 追问背景 | 识别资源限制 | 拆成可执行清单 | 加质量门禁 | 输出 Codex 任务包 |
| 第三层：高级 | 自动识别风险 | 多轮澄清策略 | 识别冲突需求 | 生成路线图 | 加 evals | 输出完整 PRD |

## 9.3 User Story

```text
作为一个提出模糊复杂想法的用户，
我希望系统能先识别我的目标、背景、约束和交付物是否缺失，
以便在进入执行前补齐关键信息，避免方向错误。
```

## 9.4 INVEST 检查

| 标准 | 是否满足 | 说明 |
|---|---|---|
| Independent | 基本满足 | 可以作为单独能力开发 |
| Negotiable | 满足 | 具体追问方式可协商 |
| Valuable | 满足 | 直接减少错误执行 |
| Estimable | 基本满足 | 范围比较清楚 |
| Small | 满足 | 可在一个小迭代中实现 |
| Testable | 满足 | 可用测试输入验证是否追问缺失信息 |

## 9.5 Story Splitting

| 大故事 | 拆分后的小故事 |
|---|---|
| Skill 帮用户澄清复杂需求 | 系统能识别模糊复杂任务 |
| Skill 帮用户澄清复杂需求 | 系统能检查目标、背景、约束、交付物是否缺失 |
| Skill 帮用户澄清复杂需求 | 系统能输出不超过 5 个关键澄清问题 |
| Skill 帮用户澄清复杂需求 | 系统能根据补充信息生成结构化任务说明 |
| Skill 帮用户澄清复杂需求 | 系统能给出验收标准和风险提示 |

## 9.6 DoR

| 检查项 | 结果 |
|---|---|
| 用户角色清楚 | 是：提出模糊复杂想法的用户 |
| 用户价值清楚 | 是：避免方向错误 |
| 范围清楚 | 是：目标、背景、约束、交付物识别 |
| 验收标准清楚 | 是：必须先识别缺失信息，不能直接执行 |
| 足够小 | 是 |
| 可估算 | 是 |
| 依赖清楚 | 是：依赖 Skill 标准结构和输出模板 |
| 风险可见 | 是：可能过度追问、误触发或漏触发 |

## 9.7 DoD

| 检查项 | 标准 |
|---|---|
| SKILL.md | 已写清触发条件、不触发条件、执行步骤 |
| 输出模板 | 已稳定输出需求澄清结果 |
| 测试 | 至少 3 个正向触发、3 个误触发、3 个边界案例 |
| 评估 | 结果能判断是否避免直接执行 |
| 文档 | README / 使用说明已更新 |
| 可交付 | 可以被放入 `.agents/skills/complex-task-clarifier/` 使用 |

# 10. 评估判断层：怎么判断是否用得好？

## 10.1 User Story 质量判断

| 判断问题 | 合格标准 |
|---|---|
| 是否表达用户价值？ | 不只是功能或技术任务 |
| 是否足够小？ | 能在一个 Sprint 内完成 |
| 是否可测试？ | 有明确验收标准 |
| 是否可估算？ | 团队能大致判断规模 |
| 是否可协商？ | 不是死板的功能合同 |
| 是否能独立交付？ | 不严重依赖其他故事 |

## 10.2 Scrum 执行质量判断

| 判断问题 | 合格标准 |
|---|---|
| Sprint 是否有明确目标？ | 不只是塞满任务 |
| Sprint Backlog 是否稳定？ | Sprint 内不随意加范围 |
| Daily Scrum 是否用于协调？ | 不是向领导汇报状态 |
| Review 是否展示真实增量？ | 不是 PPT 进度汇报 |
| Retrospective 是否产生改进行动？ | 不是空泛聊天 |
| Increment 是否满足 DoD？ | 不是半成品 |

## 10.3 失败信号

| 失败信号 | 可能原因 |
|---|---|
| Sprint 经常做不完 | 故事太大、DoR 不清、估算偏差 |
| Velocity 忽高忽低 | 故事粒度不稳定、DoD 不一致 |
| 开发中频繁返工 | DoR 缺失、验收标准不清 |
| 业务验收争论多 | User Story 价值和边界不清 |
| Review 没有反馈 | 展示的不是可用增量 |
| Retrospective 没改善 | 没有形成具体行动项 |

# 11. 迁移复用层：可迁移到哪些场景？

## 11.1 底层模式

```text
复杂目标
↓
拆成用户价值单元
↓
检查质量
↓
进入短周期执行
↓
用完成标准验收
↓
统计真实完成量
↓
基于反馈调整下一轮
```

## 11.2 可迁移场景

| 场景 | 如何迁移 |
|---|---|
| Skill 设计 | 把 Skill 能力拆成用户故事，用 DoR / DoD 做质量门禁 |
| Agent 系统设计 | 把 Agent 能力拆成可验证的增量，不一次性做大系统 |
| 个人知识库建设 | 把知识库建设拆成小故事，如“能检索某类笔记” |
| Prompt 工程 | 把 Prompt 能力拆成输入识别、约束处理、输出稳定等故事 |
| 产品功能设计 | 用 Story Mapping 组织用户路径和 MVP |
| 团队项目管理 | 用 Sprint、DoD、Velocity 管理交付节奏 |

## 11.3 不变部分与可变部分

| 类型 | 内容 |
|---|---|
| 不变部分 | 小步交付、用户价值、清晰验收、反馈调整 |
| 可变部分 | Sprint 长度、故事点估算方式、DoR / DoD 细节、工具平台 |

# 13. 最终总结

| 结论 | 内容 |
|---|---|
| 敏捷开发的主线 | 从用户价值出发，小步交付，持续反馈 |
| Scrum 的本质 | 复杂产品开发的迭代协作框架 |
| User Story 的本质 | 以用户价值为中心的轻量需求单元 |
| Story Mapping 的本质 | 先看完整用户路径，再定义 MVP 和版本路线 |
| Story Splitting 的本质 | 把大故事拆成小而完整的价值切片 |
| INVEST 的本质 | 用户故事质量检查清单 |
| DoR 的本质 | 进入 Sprint 前的入口质量门禁 |
| DoD 的本质 | 离开 Sprint 前的出口质量门禁 |
| Velocity 的本质 | 团队真实完成量的统计，不是 KPI |
| 最重要的理解 | 敏捷不是“更快写代码”，而是“更快获得反馈，并用反馈修正方向”。 |

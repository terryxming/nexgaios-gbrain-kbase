# 知识入口与字段规则

版本：v0.2.2

生命周期：Draft

日期：2026-06-22

适用范围：知识入口、全层 frontmatter 字段契约、字段枚举、YAML 序列化、knowledge_id、非 Markdown 材料包、来源说明、domain 扩展、正式 GBrain DB 同步状态语义、连续性记忆字段语义

## 1. 入口类型

| 入口 | 第一落点 | 说明 |
| --- | --- | --- |
| Obsidian 网页剪藏 | `00 - raw/00 - inbox` | 剪藏插件直接生成 Markdown。 |
| 用户手写 | `00 - raw/00 - inbox` | 用户手写、整理、导入材料先进入 inbox。 |
| Agent 回写 | DB continuity raw -> `00 - raw/00 - inbox` | 先进入连续性记忆层，解决跨设备、跨线程、跨时间续作，再物化为本地 Markdown。 |
| 系统导入 | `00 - raw/00 - inbox` | 批量导入、迁移和外部系统导入先进入 inbox。 |

## 2. 入口约束

所有入口都必须进入 inbox 接收流程。

禁止绕过 inbox 直接进入领域 raw 目录。

禁止绕过 raw 直接生成 active wiki。

Agent 回写进入 DB continuity raw 后，仍必须物化到本地 inbox，不能只存在于 DB。

用户手写、Obsidian 网页剪藏和系统导入默认不写 DB continuity raw。

DB continuity raw 只解决 Agent 续作连续性，不等同于正式 GBrain DB 同步。

非 Markdown 材料必须与同名 Markdown 说明卡组成材料包。

## 3. 全层 Frontmatter 契约

所有 raw、wiki、schema Markdown frontmatter 全量展示以下 16 个字段。

字段顺序固定：

```yaml
knowledge_id:
title:
knowledge_layer:
lifecycle_status:
source:
captured_at:
domain:
tags: []
wiki_page_type:
compile_status:
compiled_to: []
trust_level:
gbrain_db_sync_status:
gbrain_db_sync_error:
memory_type:
continuity_db_status:
```

字段地图：

| 字段 | 含义 | 值类型 | 值生成位置 |
| --- | --- | --- | --- |
| `knowledge_id` | 稳定知识 ID。 | 固定格式字符串 | 入口、修复流程或 DB 返回。 |
| `title` | 知识标题。 | 字符串 | 入口或修复流程。 |
| `knowledge_layer` | 知识所属层级。 | 枚举 | 当前流水线阶段。 |
| `lifecycle_status` | 知识生命周期状态。 | 枚举 | 当前流水线阶段。 |
| `source` | 知识来源。 | 固定格式字符串 | 入口流程。 |
| `captured_at` | 知识进入系统的时间。 | 时间字符串 | 入口或修复流程。 |
| `domain` | 领域归属。 | 枚举 | inbox 路由或 schema 规则。 |
| `tags` | 正文主题标签。 | 字符串列表 | inbox 路由、wiki 编译或人工复核。 |
| `wiki_page_type` | wiki 页面形态。 | 枚举 | wiki 编译流程。 |
| `compile_status` | raw 编译状态。 | 枚举 | raw 编译流程。 |
| `compiled_to` | raw 被编译到哪些 wiki。 | 字符串列表 | raw 编译流程。 |
| `trust_level` | 可信等级。 | 枚举 | 当前流水线阶段。 |
| `gbrain_db_sync_status` | 正式 GBrain DB 同步状态。 | 枚举 | GitHub 后的正式同步流程。 |
| `gbrain_db_sync_error` | 正式 GBrain DB 同步失败原因。 | 字符串 | GitHub 后的正式同步流程。 |
| `memory_type` | 记忆内容类型。 | 枚举 | Agent 回写入口、字段修复流程或记忆提取流程。 |
| `continuity_db_status` | DB continuity raw 状态。 | 枚举 | Agent 回写入口或补偿流程。 |

字段地图外字段必须删除或迁移。

## 4. knowledge_id 生成规则

`knowledge_id` 是知识对象的稳定 ID，不是路径、标题或 Git commit。

格式：

```text
raw-<12位hash>
wiki-<12位hash>
schema-<12位hash>
continuity-<12位hash>
```

生成来源：

| 场景 | 生成规则 |
| --- | --- |
| Agent 回写 raw | 优先使用 DB continuity raw 返回的稳定 ID；连续性对象建议使用 `continuity-<12位hash>`。 |
| 网页剪藏 raw | 由本地修复流程生成 `raw-<12位hash>`。 |
| 用户写入 raw | 由本地修复流程生成 `raw-<12位hash>`。 |
| 系统导入 raw | 有外部稳定 ID 时映射为 `raw-<12位hash>`；否则由本地修复流程生成。 |
| wiki 页面 | 页面首次创建时生成 `wiki-<12位hash>`，不随标题变化。 |
| schema 规则 | 规则文件首次创建时生成 `schema-<12位hash>`，不随文件名变化。 |

唯一性要求：

1. `knowledge_id` 一旦生成，不因文件移动、重命名、标题修改而改变。
2. 发现重复 ID 时，后创建者必须重新生成。
3. 说明卡和对应非 Markdown 原文件共享同一个 `knowledge_id`。
4. raw 编译成 wiki 后，raw 和 wiki 不共享同一个 `knowledge_id`；关系通过 `compiled_to` 和正文 `依据来源` 表达。
5. `continuity-` 前缀只识别连续性对象身份，不表达状态。
6. 无法安全生成时写 `unknown`，并阻断后续自动路由。

本地修复流程生成 hash 时，输入至少包含标准化后的相对路径、标题、捕获时间和正文指纹。

## 6. Frontmatter YAML 序列化规则

字段顺序必须固定为第 3 章的 16 字段顺序。

字符串默认使用单引号。

单引号内部使用两个单引号转义。

数组统一使用 block list。

空数组写：

```yaml
tags: []
compiled_to: []
```

frontmatter 禁止多行字符串。

URL、冒号、井号、方括号、反斜杠必须作为字符串处理。

字段值未知时写：

```text
unknown
```

字段不适用时写：

```text
not_applicable
```

字段地图外字段必须删除或迁移，不保留。

合格示例：

```yaml
knowledge_id: 'raw-52f05a33a746'
title: 'Agent 资料提炼与 raw 回写 SOP'
knowledge_layer: 'raw'
lifecycle_status: 'inbox'
source: 'Agent回写：当前对话'
captured_at: '2026-06-21 20:43'
domain: 'unknown'
tags:
  - 'agent回写'
  - 'GBrain'
wiki_page_type: 'not_applicable'
compile_status: '未编译'
compiled_to: []
trust_level: 'raw'
gbrain_db_sync_status: 'excluded'
gbrain_db_sync_error: 'not_applicable'
memory_type: 'working'
continuity_db_status: 'active'
```

## 7. source 规则

`source` 固定格式：

```text
来源类型：来源引用
```

来源类型只允许：

```text
用户写入
Agent回写
网页剪藏
系统导入
unknown
not_applicable
```

示例：

```text
Agent回写：当前对话
网页剪藏：https://example.com/article
用户写入：Obsidian手写
系统导入：bulk-import-2026-06-22
unknown：unknown
not_applicable：not_applicable
```

`source` 用于人类快速识别来源，不承载完整机器追溯。

完整追溯写入正文 `来源说明` 或 `回写来源`。

## 8. 来源说明正文结构

除正式 wiki 页面已使用 `依据来源` 外，raw 和非 Markdown 说明卡正文必须包含 `来源说明`。

标准结构：

```markdown
## 来源说明

| 字段 | 值 |
| --- | --- |
| 来源类型 |  |
| 来源引用 |  |
| 捕获时间 |  |
| 原始作者 |  |
| 原始链接/会话/文件 |  |
```

网页剪藏必须记录 URL、网页标题、作者和剪藏时间。

用户手写至少记录：

```text
用户写入：Obsidian手写
```

Agent 回写按 `07 - Agent 回写闭环规则.md` 的 `回写来源` 结构执行。

非 Markdown 说明卡必须记录关联文件。

## 9. 非 Markdown 材料包规则

PDF、图片、截图、Office 文档等非 Markdown 原文件必须有同名 Markdown 说明卡。

原文件和说明卡共享同一个 `knowledge_id`。

说明卡正文必须包含：

```markdown
## 来源说明

## 关联文件

## 原始内容说明
```

`关联文件` 必须写明原文件名、相对路径、文件类型和文件数量。

移动、路由、归档时，说明卡和原文件必须作为材料包一起移动。

原文件缺失时，说明卡不得进入 wiki 编译，只能进入待处理。

说明卡缺失时，非 Markdown 原文件不得进入自动路由。

## 10. captured_at 规则

`captured_at` 固定格式：

```text
YYYY-MM-DD HH:mm
```

优先级：

1. 来源系统提供的捕获时间。
2. Markdown 已有合法时间。
3. 文件创建时间。
4. 文件修改时间。
5. 修复流程当前时间。

时间未知时写 `unknown`，并在正文 `来源说明` 写明原因。

## 11. tags 规则

`tags` 由当前流水线阶段生成。

inbox 阶段以 `02 - inbox 接收与路由分发规则.md` 为准。

所有 tag 必须是字符串。

单个 tag 不允许空格。

英文多词 tag 使用连字符。

## 12. compiled_to 规则

`compiled_to` 必须指向 wiki 页面路径、wiki 链接或稳定 ID。

raw 尚未编译时写：

```yaml
compiled_to: []
```

wiki 和 schema 层不适用时写：

```yaml
compiled_to: []
```

## 13. gbrain_db_sync_status 分层语义

`gbrain_db_sync_status` 只表示正式 GBrain DB 同步状态。

正式 GBrain DB 同步是指材料通过本地知识生产流水线和 GitHub 版本层后，被同步进入 GBrain DB 查询服务层。

`gbrain_db_sync_status` 不得用于表达 DB continuity raw 是否已写入。

| knowledge_layer | `pending` 含义 | `synced` 含义 |
| --- | --- | --- |
| raw | 已进入领域 raw 目录，等待正式 DB 同步。 | 已作为 raw 层材料进入正式 GBrain DB。 |
| wiki | 已进入 wiki candidate 或 active，等待正式 DB 同步。 | 已进入 GBrain DB 正式可检索知识。 |
| schema | 规则文件等待正式 DB 同步。 | 已进入 GBrain DB 规则知识，可作为规则依据。 |

其他状态含义：

| 状态 | 含义 |
| --- | --- |
| `pending` | 在正式同步范围内，尚未同步或待同步。 |
| `failed` | 同步失败，原因写入 `gbrain_db_sync_error`。 |
| `excluded` | 按规则排除，不参与同步。 |
| `not_applicable` | 当前层级不适用。 |

`00 - raw/00 - inbox` 阶段默认写 `excluded`，因为 inbox 不参与正式 GBrain DB 同步。

材料从 inbox 路由到领域 raw 目录后，若进入正式同步范围，后续规则必须将 `gbrain_db_sync_status` 重置为 `pending`。

Agent 回写是否已经进入 DB continuity raw，只能由 `continuity_db_status` 表达。

## 14. memory_type 与 continuity_db_status 规则

两个连续性字段的判断标准是：缺少该字段会导致机器做错动作。

`memory_type` 用于判断材料的记忆内容类型，不表达生命周期或编译阶段：

| 值 | 含义 |
| --- | --- |
| `not_applicable` | 当前材料不适用记忆内容类型判断。 |
| `working` | 工作记忆，用于续作、任务交接、当前状态和待办上下文。 |
| `episodic` | 情节记忆，用于项目经历、事故复盘、决策过程、踩坑案例和历史处置记录。 |
| `semantic` | 语义记忆，用于事实、概念、对象、定义、结论和用户已确认知识。 |
| `procedural` | 程序记忆，用于流程、SOP、规则、操作方法和可复用行为模式。 |
| `preference` | 偏好记忆，用于用户偏好、协作习惯、表达风格和长期个性化设定。 |
| `unknown` | 理论上应该判断记忆内容类型，但当前无法判断。 |

上一版中混合生命周期、场景形态和长期候选状态的三类枚举不再作为 `memory_type` 值。短期性由 `continuity_db_status` 与运行层状态表达；场景性归入 `episodic`；长期候选由 `lifecycle_status`、`compile_status`、`trust_level` 和 wiki 编译结果表达。

`continuity_db_status` 用于判断 DB continuity raw 的状态：

| 值 | 含义 |
| --- | --- |
| `not_applicable` | 当前材料没有对应 DB continuity raw。 |
| `active` | 对应 DB continuity raw 当前有效，可用于 Agent 续作查询。 |
| `superseded` | 对应 DB continuity raw 已被本地流水线后续产物替代。 |
| `failed` | 写入或物化 DB continuity raw 失败。 |
| `unknown` | 理论上应存在 DB continuity raw 状态，但当前无法判断。 |

`unknown` 和 `not_applicable` 必须严格区分。

用户手写、Obsidian 网页剪藏和系统导入在无法判断具体记忆内容类型时默认写：

```yaml
memory_type: 'not_applicable'
continuity_db_status: 'not_applicable'
```

Agent 连续性入口材料必须尽量写出明确值。

只有 Agent 连续性入口材料在本应知道记忆类型或连续性状态但无法判断时，才允许写 `unknown`。

关键规则：

1. `knowledge_id` 前缀负责识别连续性对象。
2. `memory_type` 负责判断记忆内容类型。
3. `continuity_db_status` 负责判断 DB continuity raw 状态。
4. `gbrain_db_sync_status` 只负责正式 GBrain DB 同步状态。

## 15. domain 扩展协议

新增 domain 必须同步以下位置：

1. `00 - raw` 顶层领域目录。
2. `01 - wiki` 主领域目录。
3. 本规则的 `domain` 枚举。
4. inbox dry-run 脚本的 `$AllowedDomain`。
5. raw / wiki / schema 模板。
6. 主手册活动规则清单或影响面映射。
7. 规则变更日志。

禁止只在某个文件里临时新增 domain。

新增 domain 前必须先确认该领域是否是长期知识域，而不是一次性项目、临时任务或 tag。

## 16. gbrain_db_sync_error 规则

`gbrain_db_sync_error` 无错误时写：

```text
not_applicable
```

同步失败时必须写清错误原因。

错误原因应包含：

1. 失败阶段。
2. 错误类型。
3. 可执行的下一步。

# Agent Memory 记忆模块｜第 35–46 章

## 阶段定位

第 35–46 章进入 Memory 的质量保障、安全、隐私、治理、成本和产品化阶段。

第 0–34 章解决的是：Memory 是什么、怎么设计、怎么接入 Agent Loop、怎么和 RAG/LLM Wiki/Graph 组合。

第 35–46 章解决的是：Memory 做出来之后，如何判断它是否真的有效、安全、可控、可维护、可产品化。

| 范围 | 主题 | 核心问题 |
|---|---|---|
| 第 35–40 章 | Memory Eval 与测试 | 如何判断 Memory 是否提升 Agent 质量 |
| 第 41–46 章 | 安全、隐私与产品化 | 如何让 Memory 可控、可信、低成本、可被用户理解 |

# 第 35 章｜Memory Eval 总框架

## 35.1 本章一句话

Memory Eval 的目标不是证明“系统有记忆”，而是证明“记忆让 Agent 在正确场景下变得更好，并且没有引入明显副作用”。

## 35.2 为什么 Memory 必须评估

| 如果不评估 | 可能后果 |
|---|---|
| 不知道记忆是否被正确写入 | 重要偏好漏记，垃圾信息乱记 |
| 不知道记忆是否被正确检索 | 该用的没用，不该用的被注入 |
| 不知道记忆是否影响输出 | 记忆只是存了，但没有产生行为变化 |
| 不知道记忆是否带来污染 | 旧信息、无关信息干扰当前任务 |
| 不知道记忆是否安全 | 敏感信息被保存或泄露 |

## 35.3 Memory Eval 的五层结构

| 层级 | 评估对象 | 核心问题 |
|---|---|---|
| 1. Write Eval | 写入质量 | 该记的是否记了，不该记的是否没记 |
| 2. Retrieval Eval | 检索质量 | 当前任务是否取回正确记忆 |
| 3. Injection Eval | 注入质量 | 记忆是否以正确方式进入上下文 |
| 4. Behavior Eval | 行为质量 | Agent 输出是否因为记忆变得更好 |
| 5. Safety Eval | 安全质量 | 是否避免隐私泄露、越权、错误延续 |

## 35.4 Memory Eval 和普通 LLM Eval 的区别

| 普通 LLM Eval | Memory Eval |
|---|---|
| 主要评估单次回答 | 评估跨轮次、跨会话连续性 |
| 看回答是否正确 | 看是否正确使用历史信息 |
| 输入相对独立 | 输入依赖历史上下文 |
| 输出是核心 | 写入、检索、注入、更新全是评估对象 |

## 35.5 最小 Eval 集

一个最小可用的 Memory Eval Suite 至少包含：

| 测试集 | 目的 |
|---|---|
| 写入测试 | 判断哪些内容被写入 |
| 漏召回测试 | 该用的记忆有没有取回 |
| 误召回测试 | 不相关记忆有没有混入 |
| 冲突测试 | 新旧信息冲突时是否正确处理 |
| 隐私测试 | 敏感信息是否被不当保存或输出 |
| 端到端测试 | 有 Memory 是否比无 Memory 更好 |

## 35.6 本章掌握标准

你能把 Memory Eval 拆成：

```text
写入质量
检索质量
注入质量
行为质量
安全质量
```

# 第 36 章｜写入质量测试

## 36.1 本章一句话

写入质量测试要回答：哪些信息应该进入 Memory，哪些信息不应该进入 Memory。

## 36.2 写入测试的核心指标

| 指标 | 含义 | 反面问题 |
|---|---|---|
| Write Recall | 该记的信息是否记了 | 漏记 |
| Write Precision | 记下的信息是否真的该记 | 乱记 |
| Type Accuracy | 类型是否正确 | 把项目状态当用户偏好 |
| Scope Accuracy | 范围是否正确 | 把临时会话记成长期偏好 |
| Sensitivity Control | 敏感信息是否被正确处理 | 保存不该保存的信息 |

## 36.3 写入测试用例模板

| 字段 | 示例 |
|---|---|
| 输入对话 | “以后讲 Agent 工程时，请用中文，并多用表格。” |
| 期望写入 | 用户偏好：中文、表格化 |
| 不应写入 | 当前语气、一次性闲聊 |
| 记忆类型 | User Preference |
| 作用范围 | user-level |
| 置信度 | 高 |

## 36.4 正向测试

| 输入 | 期望结果 |
|---|---|
| “以后我的工作语言是中文。” | 写入长期用户偏好 |
| “这个项目的目录结构固定为 raw/schema/views。” | 写入项目记忆 |
| “上次 Codex 没跑完就说完成。” | 写入失败记忆 |

## 36.5 反向测试

| 输入 | 期望结果 |
|---|---|
| “今天先用中文回答。” | 不一定写入长期偏好 |
| “我现在有点累。” | 不写入长期记忆 |
| “随便举个例子。” | 不写入长期事实 |

## 36.6 写入测试的常见失败

| 失败 | 说明 |
|---|---|
| 过度写入 | 把所有细节都保存 |
| 欠写入 | 明确长期偏好没保存 |
| 错误抽象 | 从一次性要求中泛化出长期规则 |
| 错误作用域 | 把项目偏好写成全局用户偏好 |
| 缺少来源 | 后续无法解释这条记忆怎么来的 |

## 36.7 本章掌握标准

你能设计测试用例验证：

```text
该记的记住
不该记的不记
记忆类型正确
记忆作用域正确
敏感信息不乱记
```

# 第 37 章｜检索质量测试

## 37.1 本章一句话

检索质量测试要回答：当前任务需要的记忆是否被取回，不相关记忆是否被挡住。

## 37.2 检索质量指标

| 指标 | 含义 |
|---|---|
| Recall | 需要的记忆是否被召回 |
| Precision | 召回的记忆是否相关 |
| Noise Rate | 无关记忆比例 |
| Scope Accuracy | 是否只取当前用户/项目/会话相关记忆 |
| Freshness | 是否优先取新记忆 |
| Conflict Awareness | 是否识别冲突记忆 |

## 37.3 检索测试用例模板

| 字段 | 示例 |
|---|---|
| 当前任务 | “继续讲 Memory 第 35–46 章。” |
| 应召回 | 用户偏好中文、已完成第 24–34 章、需要生成 Markdown 文件 |
| 不应召回 | Amazon 广告投放、儿童卡拉 OK 图片提示词 |
| 检索范围 | user + project |
| 预期结果 | 只注入课程相关记忆 |

## 37.4 漏召回测试

| 场景 | 应召回内容 |
|---|---|
| 用户说“继续” | 上次进度 |
| 用户说“按之前格式” | 之前确认的输出格式 |
| 用户说“沉淀到知识库” | Markdown 文件沉淀偏好 |

## 37.5 误召回测试

| 场景 | 不应召回内容 |
|---|---|
| Memory 课程讲解 | Amazon 广告数据 |
| Codex 工程任务 | 儿童广告图片创意 |
| 文件沉淀任务 | 无关生活闲聊 |

## 37.6 检索失败的主要原因

| 原因 | 说明 |
|---|---|
| 没有 scope filter | 跨用户、跨项目污染 |
| 只靠向量相似度 | 语义相似但任务无关 |
| 没有时间权重 | 旧记忆压过新记忆 |
| 没有状态过滤 | archived/deprecated 记忆被取回 |
| 没有 rerank | 初召回质量不可控 |

## 37.7 本章掌握标准

你能为 Memory Retriever 设计：

```text
召回率测试
准确率测试
噪音测试
作用域测试
新旧信息测试
```

# 第 38 章｜注入质量测试

## 38.1 本章一句话

注入质量测试要回答：检索到的记忆是否以正确、简洁、无污染的方式进入模型上下文。

## 38.2 注入质量指标

| 指标 | 含义 |
|---|---|
| Relevance | 注入内容是否与当前任务相关 |
| Compactness | 是否足够简洁 |
| Structure | 是否分类清晰 |
| Priority | 是否体现当前指令优先 |
| Non-contamination | 是否避免无关记忆污染 |
| Traceability | 是否能追踪来源 |

## 38.3 好的注入块

```text
<task_relevant_memory>
用户偏好：
- 中文输出。
- 高信息密度、表格化、MECE。

项目状态：
- Agent Memory 课程已完成第 24–34 章。
- 当前进入第 35–46 章。

当前任务：
- 先完整输出课程内容。
- 再整理成 Markdown 文件。
</task_relevant_memory>
```

## 38.4 差的注入块

```text
用户之前聊过 Agent、Memory、Amazon、图片、ASIN、敏捷开发、Codex、LLM Wiki，都可以作为背景。
```

问题：

| 问题 | 说明 |
|---|---|
| 过宽 | 与当前任务无关内容过多 |
| 无分类 | 模型不知道哪些是偏好、哪些是项目状态 |
| 无优先级 | 当前任务约束不突出 |
| 易污染 | 可能把广告或图片任务混入课程讲解 |

## 38.5 注入测试方法

| 测试 | 目的 |
|---|---|
| 对比测试 | 同一任务，有记忆注入 vs 无记忆注入 |
| 消融测试 | 去掉某条记忆，看输出是否变差 |
| 噪音测试 | 故意加入无关记忆，看系统是否过滤 |
| 冲突测试 | 加入旧偏好，看当前指令是否优先 |

## 38.6 本章掌握标准

你能判断一个 memory context block 是否：

```text
短
准
分类清晰
优先级明确
不污染当前任务
```

# 第 39 章｜冲突与遗忘测试

## 39.1 本章一句话

冲突与遗忘测试要回答：当新旧记忆不一致、记忆过期、用户要求删除时，系统是否能正确处理。

## 39.2 为什么这是高风险环节

| 问题 | 后果 |
|---|---|
| 旧偏好不更新 | Agent 按过时方式工作 |
| 旧项目状态不归档 | 当前进度混乱 |
| 错误记忆不删除 | 错误长期延续 |
| 敏感信息不清除 | 隐私风险 |
| 冲突记忆同时活跃 | Agent 输出摇摆 |

## 39.3 冲突类型

| 类型 | 示例 | 处理方式 |
|---|---|---|
| 偏好冲突 | 过去喜欢长文，现在讨厌长文 | 新明确偏好覆盖旧偏好 |
| 项目状态冲突 | 已完成第 24–34 章，但旧状态仍显示第 10–23 章 | 更新 current_stage |
| 事实冲突 | 产品价格变化 | 标记旧事实过期 |
| 权限冲突 | 某记忆不应被某 Agent 读取 | 按 access policy 阻断 |
| 敏感信息冲突 | 用户要求删除 | 删除或脱敏 |

## 39.4 遗忘测试用例

| 输入 | 期望行为 |
|---|---|
| “以后不要按那个旧格式了。” | deprecated 旧格式记忆 |
| “删除你记住的这个信息。” | 删除对应记忆 |
| “这只是本次任务要求。” | 不升级为长期记忆 |
| “这个项目已经结束。” | 项目状态归档 |

## 39.5 遗忘机制测试点

| 测试点 | 判断标准 |
|---|---|
| Delete | 是否彻底不再召回 |
| Archive | 是否不参与默认检索 |
| Deprecate | 是否标记为过时 |
| Decay | 是否随时间降低权重 |
| Scope Restrict | 是否限制可见范围 |

## 39.6 本章掌握标准

你能测试：

```text
新记忆是否覆盖旧记忆
旧记忆是否不再活跃
用户删除请求是否生效
过期信息是否降权或归档
```

# 第 40 章｜端到端 Agent Memory Eval

## 40.1 本章一句话

端到端 Memory Eval 要回答：加入 Memory 后，Agent 的真实任务成功率是否提高。

## 40.2 为什么需要端到端测试

局部测试只能证明某个模块工作了，不能证明整体效果变好。

| 局部模块通过 | 仍可能整体失败的原因 |
|---|---|
| 写入正确 | 检索没取回来 |
| 检索正确 | 注入太长导致污染 |
| 注入正确 | Agent 没有使用记忆 |
| 单轮正确 | 多轮后状态错乱 |
| 当前任务正确 | 长期使用成本过高 |

## 40.3 端到端测试结构

```text
测试任务
  ↓
准备历史记忆
  ↓
执行 Agent
  ↓
观察写入 / 检索 / 注入 / 输出
  ↓
对比无 Memory 基线
  ↓
评分
```

## 40.4 推荐测试场景

| 场景 | 测什么 |
|---|---|
| 继续型任务 | 是否记得上次进度 |
| 偏好继承任务 | 是否继承用户格式偏好 |
| 项目状态任务 | 是否按当前阶段执行 |
| 失败规避任务 | 是否避免过去错误 |
| 多 Agent 交接任务 | handoff memory 是否有效 |
| 删除记忆任务 | 是否真的不再使用旧记忆 |

## 40.5 端到端评分表

| 维度 | 分数 | 说明 |
|---|---:|---|
| 任务完成度 | 0–5 | 是否完成当前任务 |
| 记忆使用准确性 | 0–5 | 是否使用正确记忆 |
| 记忆噪音控制 | 0–5 | 是否避免无关记忆 |
| 状态连续性 | 0–5 | 是否继承项目进度 |
| 用户偏好继承 | 0–5 | 是否符合长期偏好 |
| 安全合规 | 0–5 | 是否避免敏感信息问题 |

## 40.6 A/B 测试

| 版本 | 说明 |
|---|---|
| A：无 Memory | 只用当前输入 |
| B：有 Memory | 读取用户、项目、流程、失败记忆 |

如果 B 没有明显优于 A，说明 Memory 系统可能只是增加复杂度，没有创造价值。

## 40.7 本章掌握标准

你能设计一个端到端测试，判断：

```text
有 Memory 的 Agent 是否比无 Memory 的 Agent 更连续、更准确、更少重复错误。
```

# 第 41 章｜Memory 隐私边界

## 41.1 本章一句话

Memory 的隐私边界是：只保存完成任务所必要、用户可理解、可控制、可删除的信息。

## 41.2 为什么 Memory 隐私风险更高

| 普通对话 | Memory 系统 |
|---|---|
| 信息主要用于当前回答 | 信息可能长期保存 |
| 用户可能以为只是临时说说 | 系统可能把它转成长期记忆 |
| 影响一次输出 | 影响未来多次输出 |
| 删除成本较低 | 需要定位、删除、停止召回 |

## 41.3 敏感信息类型

| 类型 | 示例 | 默认策略 |
|---|---|---|
| 身份信息 | 身份证、护照、精确地址 | 不保存或加密保存 |
| 财务信息 | 银行卡、税务、收入 | 不保存，除非业务必要 |
| 健康信息 | 诊断、病史、心理状态 | 高敏，默认不保存 |
| 政治/宗教等属性 | 政治观点、宗教身份 | 高敏，默认不保存 |
| 账号凭证 | 密码、API key、token | 禁止保存明文 |
| 商业机密 | 未公开战略、客户名单 | 权限隔离、最小化保存 |

OWASP LLM Top 10 将敏感信息披露列为大模型应用的重要风险之一，并强调个人信息、财务信息、专有算法和其他敏感数据都可能通过输出或应用上下文泄露。

## 41.4 隐私边界原则

| 原则 | 说明 |
|---|---|
| 最小化 | 只记必要信息 |
| 明确目的 | 记忆必须服务明确任务 |
| 用户可控 | 用户能查看、修改、删除 |
| 作用域隔离 | 用户、项目、组织之间隔离 |
| 默认不保存敏感信息 | 除非用户明确要求且业务必要 |
| 可审计 | 记录来源、时间、用途 |

## 41.5 本章掌握标准

你能判断：

```text
哪些信息不该记
哪些信息只能短期使用
哪些信息必须加权限和删除机制
```

# 第 42 章｜用户授权与可控记忆

## 42.1 本章一句话

用户可控记忆的核心是：用户应该知道系统记住了什么、为什么记、能改什么、能删什么。

## 42.2 用户应该拥有的控制权

| 权利 | 产品功能 |
|---|---|
| 查看 | 查看当前已保存记忆 |
| 修改 | 编辑错误或过时记忆 |
| 删除 | 删除指定记忆 |
| 暂停 | 暂停记忆写入 |
| 导出 | 导出自己的记忆数据 |
| 解释 | 查看某条记忆的来源和用途 |

## 42.3 授权等级

| 等级 | 说明 | 适合场景 |
|---|---|---|
| 自动短期记忆 | 当前会话内自动使用 | 会话上下文 |
| 自动长期记忆 | 系统判断后自动保存 | 低敏、明确偏好 |
| 用户确认后长期保存 | 保存前请求确认 | 重要偏好、项目规范 |
| 禁止保存 | 明确不进入长期记忆 | 敏感信息、一次性内容 |

## 42.4 记忆可解释模板

| 字段 | 示例 |
|---|---|
| 我记住了什么 | 用户偏好中文、表格化讲解 |
| 为什么记住 | 用户多次明确要求该输出风格 |
| 用在哪里 | 教学、知识沉淀、Agent 工程讲解 |
| 如何修改 | 用户可直接说“以后不要这样” |
| 如何删除 | 用户可直接要求删除该记忆 |

## 42.5 常见错误

| 错误 | 后果 |
|---|---|
| 用户不知道系统记了什么 | 不信任 |
| 用户无法修改错误记忆 | 错误持续影响体验 |
| 删除后仍被召回 | 严重信任损失 |
| 保存目的不清楚 | 用户感到被过度跟踪 |

## 42.6 本章掌握标准

你能设计一个 Memory 控制面板，至少支持：

```text
查看
修改
删除
暂停
解释来源
```

# 第 43 章｜Memory 权限隔离

## 43.1 本章一句话

Memory 权限隔离的核心是：正确的人、正确的 Agent、在正确场景下，只能访问必要记忆。

## 43.2 为什么需要权限隔离

| 场景 | 风险 |
|---|---|
| 多用户系统 | A 用户记忆被 B 用户召回 |
| 多项目系统 | 项目 A 的机密进入项目 B |
| 多 Agent 系统 | 下游 Agent 访问不该看的私有记忆 |
| 企业系统 | 员工权限、部门权限、客户权限混乱 |
| 工具调用 | 工具把私有记忆发到外部系统 |

## 43.3 权限维度

| 维度 | 示例 |
|---|---|
| User Scope | 只属于某个用户 |
| Project Scope | 只属于某个项目 |
| Org Scope | 组织级共享 |
| Agent Scope | 只有某类 Agent 可访问 |
| Tool Scope | 是否允许工具读取 |
| Sensitivity Level | public/internal/private/sensitive |

## 43.4 权限策略模板

```yaml
access_policy:
  memory_id: mem_001
  owner_user_id: user_123
  project_id: agent_memory_course
  readable_by:
    - teacher_agent
    - knowledge_manager_agent
  writable_by:
    - memory_manager_agent
  tool_access:
    external_tools: false
  sensitivity: private
```

## 43.5 权限测试

| 测试 | 目的 |
|---|---|
| 跨用户测试 | 防止用户间污染 |
| 跨项目测试 | 防止项目间污染 |
| 跨 Agent 测试 | 防止私有记忆泄露 |
| 工具外发测试 | 防止外部工具拿到敏感记忆 |
| 删除后访问测试 | 防止已删记忆继续可见 |

## 43.6 本章掌握标准

你能为 Memory 设计：

```text
用户隔离
项目隔离
Agent 隔离
工具访问限制
敏感等级
```

# 第 44 章｜Memory 可解释性

## 44.1 本章一句话

Memory 可解释性的目标是：让用户和开发者知道 Agent 为什么记得、为什么取回、为什么使用某条记忆。

## 44.2 为什么需要可解释性

| 没有解释 | 后果 |
|---|---|
| 用户不知道为什么被个性化 | 感到突兀或被监控 |
| 开发者不知道为什么输出跑偏 | 难以调试 |
| 记忆来源不清 | 无法修正错误记忆 |
| 检索过程不可见 | 难以评估召回质量 |
| 使用行为不可追踪 | 无法归因质量问题 |

## 44.3 可解释性需要记录什么

| 信息 | 作用 |
|---|---|
| memory_id | 追踪具体记忆 |
| source | 记忆来源 |
| created_at / updated_at | 判断新旧 |
| retrieval_query | 为什么被检索 |
| relevance_score | 为什么排名靠前 |
| injection_position | 放进了哪里 |
| usage_trace | 是否影响最终输出 |

OpenAI Agents SDK tracing 会记录 agent run 中的 LLM generation、tool call、handoff、guardrail 和自定义事件，用于调试、可视化和生产监控；这类 tracing 思路可以迁移到 Memory 的读写和使用链路中。

## 44.4 面向用户的解释

| 用户问题 | 系统应能回答 |
|---|---|
| 你为什么知道我喜欢表格？ | 因为你多次要求知识讲解要高信息密度和表格化 |
| 你为什么按这个格式输出？ | 因为这是当前项目之前确认的知识库沉淀格式 |
| 这条记忆可以删吗？ | 可以，删除后不再用于未来回答 |

## 44.5 面向开发者的解释

```text
Memory Trace:
- retrieved_memory_ids: [mem_user_style_001, mem_project_stage_006]
- reason: current task is memory course continuation
- injected: true
- output_influence: formatting, chapter continuity
```

## 44.6 本章掌握标准

你能设计 Memory Trace，使系统回答：

```text
这条记忆从哪里来
为什么被取回
放到了哪里
是否影响了输出
如何删除或更新
```

# 第 45 章｜Memory 成本控制

## 45.1 本章一句话

Memory 成本控制的核心是：不要为了“更聪明”而让系统在存储、检索、Token、延迟、维护上付出不可接受的代价。

## 45.2 Memory 的成本来源

| 成本 | 说明 |
|---|---|
| 存储成本 | 长期保存记忆、日志、向量索引 |
| 检索成本 | 查询数据库、向量库、图数据库 |
| Token 成本 | 注入上下文会增加输入长度 |
| 延迟成本 | 检索、rerank、压缩都增加响应时间 |
| 维护成本 | schema、policy、eval、权限管理 |
| 安全成本 | 加密、审计、删除、权限隔离 |

## 45.3 成本控制原则

| 原则 | 说明 |
|---|---|
| 少写 | 不值得复用的信息不写 |
| 精取 | 只检索当前任务相关记忆 |
| 短注入 | 注入摘要，不注入长原文 |
| 分层存储 | 热记忆、温记忆、冷记忆分层 |
| 延迟加载 | 需要时才检索深层记忆 |
| 定期清理 | 归档、降权、删除低价值记忆 |

## 45.4 热 / 温 / 冷记忆

| 层级 | 内容 | 使用方式 |
|---|---|---|
| Hot Memory | 高频偏好、当前项目状态 | 默认检索 |
| Warm Memory | 近期任务经验、流程规则 | 按任务检索 |
| Cold Memory | 历史记录、归档项目 | 需要时检索 |

## 45.5 Token 控制

| 方法 | 说明 |
|---|---|
| Top-K 限制 | 只注入前 K 条高价值记忆 |
| Summary 注入 | 注入压缩版，而不是原文 |
| Type Filter | 当前只取某类记忆 |
| Scope Filter | 当前只取某项目记忆 |
| Conflict Filter | 旧版、冲突、低可信记忆不注入 |

## 45.6 成本收益判断

一条记忆值得保存和使用，应该满足：

```text
长期收益 > 存储成本 + 检索成本 + Token 成本 + 风险成本
```

## 45.7 本章掌握标准

你能为 Memory 系统设计：

```text
写入限制
检索限制
注入长度限制
冷热分层
定期清理机制
```

# 第 46 章｜Memory 产品体验设计

## 46.1 本章一句话

Memory 的产品价值不是让用户知道系统“存了数据”，而是让用户明显感到：Agent 更懂我、更连续、更少重复、更可控。

## 46.2 用户感知到的 Memory 价值

| 用户感知 | 产品表现 |
|---|---|
| 更懂我 | 自动继承语言、格式、工作习惯 |
| 不重复问 | 不反复询问已确认背景 |
| 项目不断线 | 知道上次做到哪一章、哪一步 |
| 更少犯错 | 避免重复过去的失败 |
| 可控 | 能查看、修改、删除记忆 |
| 可信 | 能解释为什么记住、为什么使用 |

## 46.3 Memory 产品功能模块

| 功能 | 说明 |
|---|---|
| Memory List | 用户可查看已保存记忆 |
| Memory Detail | 查看来源、时间、用途、作用域 |
| Edit Memory | 修改错误记忆 |
| Delete Memory | 删除指定记忆 |
| Pause Memory | 暂停自动写入 |
| Memory Suggestions | 系统建议保存，用户确认 |
| Memory Trace | 显示当前回答使用了哪些记忆 |

## 46.4 产品化分级

| 级别 | 能力 |
|---|---|
| L1 隐式记忆 | 系统自动使用，用户感知弱 |
| L2 可查看记忆 | 用户能看到系统记住什么 |
| L3 可编辑记忆 | 用户能修改和删除 |
| L4 可解释记忆 | 用户知道为什么记、怎么用 |
| L5 可治理记忆 | 支持权限、审计、团队管理 |

## 46.5 个人 Agent 的最小产品设计

| 模块 | 最小实现 |
|---|---|
| 用户偏好 | 保存语言、格式、教学偏好 |
| 项目状态 | 保存当前项目阶段 |
| 输出规范 | 保存知识库沉淀格式 |
| 失败经验 | 保存常见质量问题 |
| 控制方式 | 用户可用自然语言修改/删除 |

## 46.6 企业 Agent 的产品设计重点

| 模块 | 重点 |
|---|---|
| 权限 | 用户、团队、部门隔离 |
| 审计 | 谁写入、谁读取、谁修改 |
| 合规 | 敏感信息和数据保留策略 |
| 成本 | 控制检索和上下文成本 |
| 可解释 | 支持问题追踪和责任归因 |
| 运维 | 监控、告警、回滚 |

## 46.7 常见错误

| 错误 | 后果 |
|---|---|
| 只做后端 Memory，不做用户控制 | 用户不信任 |
| 记忆表现太隐晦 | 用户感受不到价值 |
| 记忆表现太强 | 用户觉得被监控 |
| 无法解释来源 | 错误记忆无法修正 |
| 删除不彻底 | 信任崩塌 |

## 46.8 本章掌握标准

你能设计一个 Memory 产品体验，使用户感到：

```text
它记得有用信息
它不会乱记敏感信息
它能解释为什么记得
我能修改和删除它
它确实提高了工作效率
```

# Agent 工程化与产品化｜第 61–68 章：真实项目实战

> 适合沉淀位置：`llm-wiki/40_projects/agent-engineering-productization/` 或 `llm-wiki/50_skills/agent-system-design/`

---

## 本阶段目标

第 61–68 章的目标不是继续讲概念，而是把前面已经学过的内容落到真实项目中：

| 前面学过的模块 | 在本阶段如何落地 |
|---|---|
| 最小 Agent | 每个项目都要定义 Goal / Instruction / Context / Tool / Output / Quality Criteria |
| 工程化模块 | 每个项目都要设计流程、状态、异常、人工确认、版本 |
| Tool / MCP | 每个项目都要判断需要哪些工具、权限如何控制 |
| Context / Memory | 每个项目都要设计当前上下文、长期记忆、知识库 |
| Test / Eval / Trace | 每个项目都要定义测试、评估、运行记录 |
| 产品化 | 每个项目都要定义用户、场景、指标、MVP |
| 多 Agent 架构 | 复杂项目要拆角色、交接、审核和平台能力 |

核心结论：

```text
真实项目实战
= 业务场景
+ Agent 设计
+ 工程实现
+ 工具接入
+ 上下文系统
+ 质量评估
+ 产品指标
+ 可沉淀资产
```

# 第 61 章｜知识管理 Agent

## 61.1 一句话结论

**知识管理 Agent 的目标不是“总结内容”，而是把对话、资料、经验和方法论转化成可检索、可复用、可维护的知识资产。**

它非常适合个人 `llm-wiki`、Obsidian、团队知识库、项目复盘库。

---

## 61.2 典型场景

| 场景 | 用户原始需求 | Agent 交付物 |
|---|---|---|
| 对话沉淀 | 把这轮对话整理成 `.md` | Markdown 知识文件 |
| 课程沉淀 | 把第 61–68 章沉淀到知识库 | 结构化课程笔记 |
| 案例复盘 | 把一次广告诊断过程沉淀 | 案例复盘文档 |
| 方法论提炼 | 从多轮讨论中提炼框架 | 可复用方法模板 |
| 文件归档 | 把杂乱资料整理进知识库目录 | 分类后的知识文件 |

---

## 61.3 最小 Agent 设计

| 模块 | 设计 |
|---|---|
| Goal | 把输入内容整理成可长期沉淀的 Markdown 知识资产 |
| User | 知识工作者、运营人员、产品经理、工程师 |
| Input | 对话、文档、网页摘录、课程内容、项目复盘 |
| Context | 用户知识库结构、命名规则、已有分类、输出偏好 |
| Tools | 读取对话、读取文件、生成 Markdown、写入文件 |
| Output | `.md` 文件 + 标题 + 分类建议 + 关联标签 |
| Quality | 结构清晰、可检索、可复用、去除闲聊、保留方法 |

---

## 61.4 工作流设计

```text
接收内容
↓
识别主题
↓
判断知识类型
↓
提炼核心结论
↓
重组结构
↓
补充表格、模板、误区、实践题
↓
生成 Markdown
↓
质量检查
↓
输出下载链接或写入知识库
```

---

## 61.5 知识类型识别

| 类型 | 处理方式 |
|---|---|
| 概念解释 | 输出定义、边界、例子、误区 |
| 方法论 | 输出步骤、原则、模板、检查清单 |
| 项目复盘 | 输出背景、过程、问题、决策、经验 |
| 课程章节 | 输出学习目标、章节内容、实践作业 |
| 操作 SOP | 输出触发条件、步骤、注意事项、验收标准 |
| 案例分析 | 输出事实、判断、证据、动作、复盘 |

|---|---|

## 61.8 质量评估

| 维度 | 通过标准 |
|---|---|
| 结构 | 标题层级清晰，能快速扫描 |
| 信息密度 | 去除闲聊，保留关键知识 |
| 可复用 | 有模板、步骤、检查清单 |
| 可检索 | 标题、标签、关键词清楚 |
| 可维护 | 有版本、更新时间、关联链接 |
| 不失真 | 不改变原始观点和决策 |

---

## 61.9 常见误区

| 误区 | 正确做法 |
|---|---|
| 把对话原样搬进知识库 | 应提炼成知识结构 |
| 只做总结 | 要沉淀概念、方法、模板、误区 |
| 不做分类 | 要给出目录建议和标签 |
| 不保留决策背景 | 项目知识要保留为什么这样做 |
| 不区分 raw 和 notes | 原始资料和整理笔记要分开 |

# 第 62 章｜电商广告分析 Agent

## 62.1 一句话结论

**电商广告分析 Agent 的目标不是解释广告指标，而是基于数据证据诊断广告表现变化，并输出可执行优化动作。**

它适合 Amazon Ads、Google Ads、Meta Ads、TikTok Ads 等广告数据分析场景。

---

## 62.2 典型用户任务

| 用户说法 | Agent 应理解为 |
|---|---|
| ACOS 为什么升高？ | 诊断成本效率恶化原因 |
| CTR 为什么下降？ | 诊断曝光流量与点击吸引力问题 |
| CVR 为什么变差？ | 诊断转化链路、流量相关性、页面问题 |
| CPC 涨了怎么办？ | 判断竞价、预算、placement、竞争变化 |
| 哪些词该否？ | 基于花费、点击、订单、相关性判断浪费流量 |
| 哪些词值得加预算？ | 找高转化、高相关、高增量对象 |

---

## 62.3 最小 Agent 设计

| 模块 | 设计 |
|---|---|
| Goal | 诊断广告指标变化并输出优化建议 |
| Input | 广告报表、搜索词报表、时间区间、目标 ASIN / campaign |
| Context | 商品定位、价格、核心关键词、历史操作、促销、库存 |
| Tools | 表格读取、指标计算、异常检测、报告生成 |
| Output | 根因表、证据表、优化动作表、风险提示 |
| Quality | 结论有数据证据，动作落到具体对象 |

---

## 62.4 核心指标链路

```text
ACOS = Spend / Sales

ACOS 升高常见路径：
1. CPC 上升 → 点击成本变贵
2. CVR 下降 → 点击不能转化
3. AOV / 售价下降 → 每单销售额下降
4. 流量结构变化 → 低相关词占比提高
5. 预算或 placement 变化 → 流量进入低效位置
```

---

## 62.5 工作流设计

```text
读取报表
↓
检查字段完整性
↓
统一时间区间
↓
计算 CTR / CVR / CPC / ACOS / Spend / Sales / Orders
↓
做同比或环比对比
↓
定位变化最大的 campaign / ad group / keyword / search term
↓
拆解根因
↓
输出建议动作
↓
标记风险和待验证信息
```

---

## 62.6 根因诊断表

| 现象 | 可能根因 | 需要证据 | 建议动作 |
|---|---|---|---|
| ACOS 升高 + CPC 升高 | 竞价变贵、placement 变化 | CPC、placement、bid 变化 | 降低低效对象 bid，检查 ToS 溢价 |
| ACOS 升高 + CVR 下降 | 转化问题、流量不准 | CVR、搜索词、Listing 变化 | 否低相关词，检查页面转化 |
| Spend 增长但 Orders 不增 | 浪费流量增加 | search term spend/order | 否词、降 bid、收紧匹配 |
| CTR 下降 | 主图、标题、价格、排名变化 | impression/click/CTR | 优化创意或检查流量来源 |
| Sales 下降但 CTR 稳定 | 转化链路问题 | CVR、价格、库存、评价 | 检查 Listing、价格、评价、库存 |

---

## 62.7 输出模板

```markdown
# 广告诊断报告

## 1. 总结结论
- 核心问题：
- 主要根因：
- 优先处理对象：

## 2. 指标变化
| 指标 | 上期 | 本期 | 变化 | 判断 |
|---|---:|---:|---:|---|

## 3. 根因拆解
| 根因 | 证据 | 影响 | 置信度 |
|---|---|---|---|

## 4. 优化动作
| 动作 | 对象 | 理由 | 风险 | 优先级 |
|---|---|---|---|---|

## 5. 需要补充的数据
| 数据 | 用途 |
|---|---|
```

---

## 62.8 Guardrail

| 风险 | 护栏 |
|---|---|
| 编造数据 | 没有报表就只能给分析框架，不能给结论 |
| 误改广告 | 默认只输出建议，不自动执行 |
| 错误否词 | 否词建议必须附搜索词证据 |
| 过度降 bid | 核心词降 bid 必须提示曝光风险 |
| 时间区间错误 | 必须显示当前周期和对比周期 |

---

## 62.9 评估标准

| 维度 | 标准 |
|---|---|
| 准确性 | 指标计算正确 |
| 证据性 | 每个根因有数据支持 |
| 可执行性 | 建议落到 campaign / keyword / search term |
| 风险性 | 高风险动作有提示 |
| 业务价值 | 能帮助用户决定下一步优化 |

# 第 63 章｜Listing 优化 Agent

## 63.1 一句话结论

**Listing 优化 Agent 的目标不是“写更好听的文案”，而是基于关键词、用户场景、竞品、转化障碍和平台规则，生成能提升理解、点击和转化的 Listing 优化方案。**

---

## 63.2 典型任务

| 任务 | 输出 |
|---|---|
| 优化标题 | 新标题 + 关键词覆盖说明 |
| 优化五点 | 五点 bullet + FABE 逻辑 |
| 优化 A+ 文案 | 模块标题、正文、卖点结构 |
| 优化主图/副图文案 | 画面信息架构 + 文案 |
| 分析竞品差异 | 差异化卖点表 |
| 关键词语义布局 | 核心词、场景词、属性词分层 |

---

## 63.3 最小 Agent 设计

| 模块 | 设计 |
|---|---|
| Goal | 输出可执行的 Listing 优化方案 |
| Input | 当前 Listing、目标关键词、竞品、产品卖点、目标用户 |
| Context | 平台规则、类目习惯、用户评价、广告搜索词、品牌调性 |
| Tools | 关键词分析、竞品整理、文案生成、合规检查 |
| Output | 标题、五点、A+ 模块、图片信息架构、优化理由 |
| Quality | 不泛泛而谈，能体现产品差异化和转化逻辑 |

---

## 63.4 Listing 优化四层模型

| 层级 | 目标 | 示例 |
|---|---|---|
| 搜索层 | 覆盖关键词 | karaoke machine、with lyrics display |
| 理解层 | 让平台和用户理解产品 | built-in screen、wireless microphones |
| 转化层 | 解决购买理由 | family party、kids singing、home entertainment |
| 差异层 | 与竞品拉开差距 | AR effects、story music videos、progress world |

---

## 63.5 工作流设计

```text
读取当前 Listing
↓
识别目标关键词和语义簇
↓
提炼核心用户和购买场景
↓
分析产品差异化卖点
↓
分析竞品表达缺口
↓
生成优化方案
↓
检查关键词覆盖、可读性、合规性
↓
输出多个版本并说明适用场景
```

---

## 63.6 FABE 输出结构

| 模块 | 说明 | 示例 |
|---|---|---|
| Feature | 产品有什么 | Built-in lyrics display |
| Advantage | 比普通产品好在哪 | No need for extra screen or phone |
| Benefit | 用户得到什么 | Easier family karaoke setup |
| Evidence | 凭什么相信 | 2 wireless mics, TV connection, karaoke app |

---

## 63.7 输出模板

```markdown
# Listing 优化方案

## 1. 当前问题诊断
| 问题 | 影响 | 证据 |
|---|---|---|

## 2. 关键词布局
| 类型 | 关键词 | 放置位置 |
|---|---|---|

## 3. 标题方案
### 版本 A：搜索覆盖型
### 版本 B：转化表达型
### 版本 C：场景差异型

## 4. 五点优化
| Bullet | 核心卖点 | 用户收益 | 关键词 |
|---|---|---|---|

## 5. A+ 模块建议
| 模块 | 标题 | 正文方向 | 画面建议 |
|---|---|---|---|

## 6. 风险提示
- 平台合规风险：
- 信息过载风险：
- 关键词堆砌风险：
```

---

## 63.8 常见误区

| 误区 | 正确做法 |
|---|---|
| 只堆关键词 | 关键词要服务用户理解和转化 |
| 文案太泛 | 必须体现产品独有卖点 |
| 只写功能 | 要写功能带来的购买理由 |
| 不看竞品 | 差异化必须来自对比 |
| 忽略平台规则 | 不夸大、不违规、不绝对化 |

# 第 64 章｜客服回复 Agent

## 64.1 一句话结论

**客服回复 Agent 的目标不是自动替人聊天，而是在安全边界内识别问题、查找事实、生成可审核的回复草稿，并在高风险场景转人工。**

---

## 64.2 典型场景

| 场景 | Agent 任务 |
|---|---|
| 激活账号 | 识别需求，要求用户提供 SN，生成说明 |
| 物流咨询 | 查询订单状态，生成回复草稿 |
| 退换货 | 根据政策判断可处理方式 |
| 产品使用问题 | 提供步骤化排查 |
| 投诉升级 | 识别高风险情绪和转人工 |
| 赔偿请求 | 不自动承诺，交给人工确认 |

---

## 64.3 最小 Agent 设计

| 模块 | 设计 |
|---|---|
| Goal | 生成准确、礼貌、安全、可审核的客服回复草稿 |
| Input | 客户消息、订单信息、产品信息、政策 |
| Context | 售后 SOP、保修政策、物流信息、历史沟通 |
| Tools | 读取订单、读取政策、创建草稿、标记工单 |
| Output | 回复草稿 + 问题分类 + 风险等级 + 下一步 |
| Quality | 不乱承诺、不编造、不越权、不泄露隐私 |

---

## 64.4 工作流设计

```text
读取客户消息
↓
识别问题类型
↓
判断风险等级
↓
读取必要订单 / 政策 / 产品资料
↓
生成回复草稿
↓
检查是否有不当承诺或敏感信息
↓
低风险：输出草稿
高风险：转人工或请求确认
```

---

## 64.5 风险分级

| 等级 | 场景 | 处理方式 |
|---|---|---|
| L1 低风险 | 使用说明、普通咨询 | 可自动生成草稿 |
| L2 中风险 | 物流延迟、激活问题 | 草稿 + 需要信息补充 |
| L3 高风险 | 退款、赔偿、差评威胁 | 草稿 + 人工确认 |
| L4 极高风险 | 法律威胁、安全事故、平台投诉 | 立即转人工 |

---

## 64.6 回复输出模板

```markdown
# 客服回复建议

## 1. 问题分类
- 类型：
- 风险等级：

## 2. 需要核实的信息
| 信息 | 是否已有 | 用途 |
|---|---|---|

## 3. 回复草稿
Hi [Customer Name],

...

## 4. 不应承诺内容
- 不承诺：
- 不应提及：

## 5. 下一步
- 发送草稿 / 修改 / 转人工
```

---

## 64.7 安全护栏

| 风险 | 护栏 |
|---|---|
| 承诺赔偿 | 未授权不得承诺退款、补偿、折扣 |
| 泄露隐私 | 不输出不必要的订单或个人信息 |
| 误发邮件 | 默认只创建草稿，发送前确认 |
| 情绪升级 | 识别愤怒、威胁、投诉，转人工 |
| 政策不确定 | 缺少政策时标记不确定，不编造 |

OWASP 将 Prompt Injection、敏感信息泄露、过度代理权等列为 LLM 应用重要风险；客服 Agent 因为会处理私有订单、邮件和对外沟通，必须采用最小权限、草稿优先和人工确认机制。

# 第 65 章｜代码开发 Agent

## 65.1 一句话结论

**代码开发 Agent 的目标不是“直接让 AI 改代码”，而是围绕 PRD、代码库、测试、差异审查和 PR 流程，形成可追踪、可回滚、可验证的工程交付链路。**

---

## 65.2 典型任务

| 任务 | Agent 交付 |
|---|---|
| 读 PRD | 需求理解和疑点列表 |
| 拆任务 | 开发任务清单 |
| 改代码 | 小步提交的代码变更 |
| 跑测试 | 测试结果和失败原因 |
| 生成 diff | 变更摘要和风险 |
| 写 PR | PR 标题、描述、测试说明 |
| 复盘失败 | 失败原因、修复策略 |

---

## 65.3 最小 Agent 设计

| 模块 | 设计 |
|---|---|
| Goal | 基于需求完成可验证的代码变更 |
| Input | PRD、仓库代码、约束、目标文件 |
| Context | README、架构说明、测试规范、已有代码风格 |
| Tools | 读文件、搜索、编辑、运行测试、生成 diff |
| Output | 代码变更、测试结果、PR 说明、风险提示 |
| Quality | 小步变更、可测试、可回滚、不虚假宣称完成 |

---

## 65.4 工作流设计

```text
读取 PRD
↓
提炼需求和验收标准
↓
扫描代码结构
↓
制定变更计划
↓
小步修改代码
↓
运行测试或静态检查
↓
读取失败结果并修复
↓
生成 diff 摘要
↓
输出 PR 说明和风险
```

---

## 65.5 代码 Agent 的高风险点

| 风险 | 护栏 |
|---|---|
| 未完成就说完成 | 必须基于实际 diff / test 结果报告 |
| 大范围乱改 | 限制变更范围，小步提交 |
| 绕过测试 | 测试失败必须说明，不得隐瞒 |
| 破坏架构 | 先读项目规范和已有模式 |
| 删除重要代码 | 删除类操作需明确理由和确认 |
| 误改配置 | 环境变量、部署配置必须谨慎 |

## 65.7 推荐工程原则

| 原则 | 说明 |
|---|---|
| 先读再改 | 不理解结构不动代码 |
| 小步修改 | 降低回滚成本 |
| 测试驱动 | 改完必须验证 |
| diff 透明 | 用户必须能看懂改了什么 |
| 不虚假完成 | 没跑测试就不能说测试通过 |
| 可回滚 | 保留变更边界 |

LangGraph 的 durable execution 思路适合长任务：在关键点保存进度，失败后可以从中断处恢复，而不是从头再来。这对代码 Agent 的长流程执行也有参考价值。

# 第 66 章｜多 Agent 项目交付系统

## 66.1 一句话结论

**多 Agent 项目交付系统，是把产品、工程、测试、文档、审查等角色拆成多个协作 Agent，用统一状态、交接协议、评估和人工确认完成复杂项目。**

---

## 66.2 适合什么项目

| 项目 | 是否适合 |
|---|---|
| 单个简单脚本 | 不适合，单 Agent 或 Workflow 即可 |
| 小型文档整理 | 不适合多 Agent |
| 复杂 Skill 创建 | 适合 |
| 完整产品功能开发 | 适合 |
| PRD → 设计 → 开发 → 测试 → 文档 | 适合 |
| 多业务角色协作项目 | 适合 |

---

## 66.3 推荐角色拆分

| Agent | 职责 |
|---|---|
| Product Agent | 理解需求、定义边界、写 PRD |
| Architect Agent | 设计系统结构、模块、接口 |
| Engineer Agent | 实现代码、脚本、配置 |
| QA Agent | 设计测试、执行检查、发现缺陷 |
| Security Agent | 检查权限、安全、敏感信息 |
| Docs Agent | 生成使用文档、变更记录 |
| Reviewer Agent | 审查最终交付质量 |
| Orchestrator | 管理流程、状态、交接和汇总 |

---

## 66.4 架构模式

```text
用户需求
↓
Orchestrator
├─ Product Agent：需求澄清与 PRD
├─ Architect Agent：系统设计
├─ Engineer Agent：实现
├─ QA Agent：测试
├─ Security Agent：安全审查
├─ Docs Agent：文档

```

# 第 67 章｜Agent 产品 PRD

**Agent 产品 PRD 不是普通功能需求文档，而是要同时定义用户场景、Agent 行为、工具权限、上下文来源、失败处理、评估标准和产品指标。**

---

## 67.2 Agent PRD 和普通 PRD 的区别

| 维度 | 普通 PRD | Agent PRD |
|---|---|---|
| 功能 | 页面、按钮、流程 | 任务、行为、工具、上下文 |
| 输出 | 确定性结果 | 概率性结果 + 评估 |
| 风险 | 交互错误、系统 bug | 幻觉、越权、泄露、误执行 |
| 测试 | 功能测试 | Test + Eval + Trace |
| 权限 | 用户权限 | 用户 + Agent + Tool 权限 |
| 迭代 | 功能版本 | Prompt / Model / Tool / Eval 版本 |

# 第 68 章｜Agent 上线检查清单

## 68.1 一句话结论

**Agent 上线不是“功能能跑就上线”，而是必须通过场景、输入、工具、权限、质量、安全、成本、可观测性和运营检查。**

---

## 68.2 上线前总检查

| 类别 | 检查项 | 是否通过 |
|---|---|---|
| 场景 | 是否有明确用户和高价值任务 |  |
| 范围 | 是否定义处理什么、不处理什么 |  |
| 输入 | 是否明确必要输入和缺失处理 |  |
| 上下文 | 是否使用正确、可信、最新资料 |  |
| 工具 | 是否有 Tool Schema 和错误处理 |  |
| 权限 | 是否遵循最小权限 |  |
| 输出 | 是否结构化、可复用、可检查 |  |
| Test | 是否覆盖正确触发、误触发、失败输入 |  |
| Eval | 是否评估结果质量和过程质量 |  |
| Trace | 是否能追踪每一步 |  |
| Guardrail | 是否有输入、工具、输出护栏 |  |
| 安全 | 是否防 Prompt Injection、泄露、越权 |  |
| 成本 | 是否有限流、预算、最大步骤数 |  |
| 人工确认 | 高风险动作是否暂停确认 |  |
| 回滚 | 是否可撤销或回滚 |  |
| Owner | 是否有负责人和维护机制 |  |

---

## 68.3 MVP 上线最低标准

| 模块 | 最低要求 |
|---|---|
| 场景 | 单一高价值任务 |
| 用户 | 明确一个目标用户 |
| 工具 | 只接必要工具，优先只读 |
| 输出 | 固定模板 |
| 质量 | 至少 10–30 个典型测试案例 |
| 安全 | 高风险动作不自动执行 |
| Trace | 能看到输入、工具、输出、错误 |
| 反馈 | 用户可标记好/坏/需修改 |
| 版本 | 有 changelog |

## 68.6 上线原则

|---|---|

## 2. 八个核心结论

| 序号 | 核心结论 |
|---|---|
| 1 | 知识管理 Agent 的重点是把内容沉淀成长期可复用知识资产，不是简单总结 |
| 2 | 广告分析 Agent 必须基于数据证据输出根因和动作，不能凭经验猜 |
| 3 | Listing 优化 Agent 要同时服务搜索理解、用户转化和差异化表达 |
| 4 | 客服 Agent 应默认草稿优先，高风险场景必须转人工或确认 |
| 5 | 代码开发 Agent 必须围绕 PRD、diff、test 和 PR 流程交付，不能虚假完成 |
| 6 | 多 Agent 项目系统适合复杂交付，但必须有共享状态、Handoff 和质量门禁 |
| 7 | Agent PRD 必须定义行为、工具、上下文、权限、评估和指标，不只是功能说明 |
| 8 | Agent 上线必须通过场景、质量、安全、成本、Trace、Owner 和回滚检查 |

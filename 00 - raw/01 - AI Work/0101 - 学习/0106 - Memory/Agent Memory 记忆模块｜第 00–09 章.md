# Agent Memory 记忆模块｜第 0–9 章学习讲义

> 适用目标：从“听过 Memory”进入“能理解、能判断、能使用”的阶段。  
> 范围：第 0–9 章，只覆盖概念边界、类型系统和基础判断，不进入完整工程实现。

---

## 第 0 章｜为什么 Agent 需要 Memory

### 0.1 本章一句话

Agent 需要 Memory，不是为了“记住所有聊天内容”，而是为了在未来任务中保留关键上下文、用户偏好、项目状态和历史经验。

### 0.2 没有 Memory 的 Agent 像什么？

没有 Memory 的 Agent，本质上像一个“每次都重新入职的人”：

| 现象 | 具体表现 | 对用户的伤害 |
|---|---|---|
| 不记得你是谁 | 每次都问工作语言、格式偏好、项目背景 | 沟通成本高 |
| 不记得项目进展 | 不知道上次做到哪一章、哪一步、哪个版本 | 项目连续性差 |
| 不记得历史判断 | 之前验证过的假设又重新分析 | 重复劳动 |
| 不记得失败经验 | 上次踩过的坑还会再踩 | 质量不稳定 |
| 不记得约束条件 | 输出风格、目录结构、格式要求反复漂移 | 工程不可控 |

### 0.3 Memory 解决的核心问题

| 核心问题 | Memory 的作用 |
|---|---|
| 上下文窗口有限 | 把关键历史信息压缩成可复用记忆 |
| Agent 默认无状态 | 让 Agent 跨轮次、跨会话保持连续性 |
| 用户偏好稳定 | 自动继承长期偏好，减少重复确认 |
| 项目周期很长 | 保留项目状态、决策记录、阶段进度 |
| 工程任务复杂 | 沉淀流程、规范、失败经验和质量门禁 |

### 0.4 费曼解释

Memory 就像“项目笔记 + 个人档案 + 经验手册”的组合。

- 项目笔记：记录做到哪里。
- 个人档案：记录你喜欢什么格式、语言、风格。
- 经验手册：记录哪些方法有效，哪些坑不要再踩。

所以，Memory 不是“仓库”，而是 Agent 的“连续工作能力”。

### 0.5 本章掌握标准

你能清楚说出：

> Agent Memory 的价值不是保存历史，而是让历史在未来任务中产生作用。

## 第 1 章｜Memory 的本质

### 1.1 本章一句话

Memory 的本质是：把历史信息加工成未来可用的上下文资产。

### 1.2 Memory 不是原始记录

| 层级 | 是什么 | 是否等于 Memory |
|---|---|---|
| 原始对话 | 用户和 AI 的完整聊天记录 | 否 |
| 工具日志 | 工具调用、运行结果、错误信息 | 否 |
| 摘要 | 对历史内容的压缩 | 可能是 Memory 的中间形态 |
| 结构化事实 | 被提炼后的偏好、状态、经验 | 是 Memory 的常见形态 |
| 可检索经验 | 能在未来任务中被取回并使用 | 是高质量 Memory |

### 1.3 Memory 的公式

```text
Memory = 历史信息 × 选择性提炼 × 结构化表达 × 未来可用性
```

其中最重要的是“未来可用性”。

如果一条信息以后不会影响判断、输出、决策或执行，就不应该进入长期记忆。

### 1.4 一条好记忆的标准

| 标准 | 说明 | 示例 |
|---|---|---|
| 可复用 | 未来会再次用到 | 用户喜欢 MECE + 费曼教学法 |
| 可判断 | 能影响选择 | 用户工作语言是中文 |
| 可执行 | 能改变 Agent 行为 | 输出知识沉淀时使用 Markdown 文件 |
| 可更新 | 后续可以修正 | 用户当前项目阶段变更 |
| 有边界 | 不无限泛化 | “本课程用中文”不等于“所有任务都只能中文” |

### 1.5 反例：什么不是好记忆

| 信息 | 不适合作为长期 Memory 的原因 |
|---|---|
| 用户今天喝了咖啡 | 没有长期复用价值 |
| 某次临时任务的随机细节 | 过期快，容易污染未来上下文 |
| 未验证的模型猜测 | 可能形成错误记忆 |
| 一整段聊天原文 | 太长、噪音多、不可控 |

### 1.6 本章掌握标准

你能判断：

> 什么信息只是历史记录，什么信息才值得沉淀为 Memory。

## 第 2 章｜Memory 的知识边界

### 2.1 本章一句话

Memory 负责“历史信息如何变成未来可用上下文”，不负责替代模型能力、数据库、RAG 或 Prompt。

### 2.2 Memory 负责什么

| 模块 | Memory 负责的问题 |
|---|---|
| 捕获 | 哪些对话、工具结果、文件内容可能有价值 |
| 提炼 | 从原始内容中抽取什么信息 |
| 结构化 | 用什么字段表达这条记忆 |
| 存储 | 放在哪里，按什么范围隔离 |
| 检索 | 当前任务需要哪些记忆 |
| 注入 | 放进 Prompt 或上下文的哪个位置 |
| 更新 | 新旧信息冲突时如何处理 |
| 遗忘 | 什么信息应该过期、删除或降权 |
| 评估 | 记忆有没有提高任务质量 |

### 2.3 Memory 不负责什么

| 对象 | 为什么不是 Memory 的职责 |
|---|---|
| 模型参数知识 | 属于训练阶段，不是运行时记忆 |
| 单轮 Prompt | 只影响当前任务，不负责跨会话保留 |
| 完整聊天记录 | 是原材料，不是可用记忆 |
| 数据库本身 | 数据库只是存储工具 |
| RAG 知识库 | RAG 主要解决外部知识召回 |
| Cache 缓存 | 主要解决效率，不解决认知连续性 |
| Fine-tuning | 改模型能力，不是运行时上下文管理 |

### 2.4 Memory 的边界判断法

判断一个东西是不是 Agent Memory，可以问 4 个问题：

| 问题 | 如果答案是“是” |
|---|---|
| 它是否来自过去的交互、任务或经验？ | 可能是 Memory |
| 它是否会影响未来的判断或输出？ | 更可能是 Memory |
| 它是否需要被检索、更新或遗忘？ | 基本属于 Memory 系统 |
| 它是否只是临时缓存或原始日志？ | 不一定是 Memory |

### 2.5 本章掌握标准

你能说清楚：

> Memory 是运行时上下文管理系统的一部分，而不是数据库、RAG、聊天记录或模型训练的同义词。

## 第 3 章｜Memory vs 相邻概念

### 3.1 本章一句话

Memory 和相邻概念的区别，主要看它解决的是“当前输入”“外部知识”“历史状态”“效率缓存”还是“长期经验”。

### 3.2 总对比表

| 概念 | 解决的问题 | 是否跨会话 | 是否需要主动提炼 | 与 Memory 的关系 |
|---|---|---:|---:|---|
| Context Window | 当前可见信息容量 | 否 | 否 | Memory 最终要注入这里 |
| Chat History | 历史聊天记录 | 可跨 | 否 | Memory 的原材料 |
| Short-term Memory | 当前线程/任务状态 | 通常线程内 | 可选 | Memory 的短期层 |
| Long-term Memory | 长期偏好、事实、经验 | 是 | 是 | Memory 的长期层 |
| RAG | 外部知识检索 | 是 | 通常需要索引 | 可与 Memory 结合 |
| User Profile | 用户稳定属性和偏好 | 是 | 是 | Long-term Memory 的一种 |
| Agent State | 当前执行状态 | 任务内为主 | 是 | 与短期记忆高度相关 |
| Cache | 复用计算结果 | 可跨 | 否 | 解决效率，不解决经验 |
| Trace / Logs | 记录执行过程 | 是 | 否 | 可用于提炼 Memory |
| Fine-tuning | 改变模型参数能力 | 是 | 训练前处理 | 和 Memory 是不同层 |

### 3.3 Memory vs Chat History

| 维度 | Chat History | Memory |
|---|---|---|
| 形态 | 原始对话 | 提炼后的可用信息 |
| 噪音 | 高 | 应该低 |
| 是否结构化 | 通常否 | 最好是 |
| 未来复用 | 不稳定 | 明确面向复用 |
| 示例 | “用户说了很多关于课程的内容” | “用户希望课程按 MECE + 费曼法组织” |

### 3.4 Memory vs RAG

| 维度 | RAG | Memory |
|---|---|---|
| 核心对象 | 外部知识 | 用户、任务、项目、经验 |
| 典型内容 | 文档、论文、知识库 | 偏好、状态、历史决策 |
| 主要问题 | 找到相关知识 | 保持连续性和个性化 |
| 示例 | 检索 LangGraph 文档 | 记住用户正在学 Memory 第 0–9 章 |

### 3.5 Memory vs Agent State

| 维度 | Agent State | Memory |
|---|---|---|
| 时间范围 | 当前任务/运行过程 | 可短期，也可长期 |
| 内容 | 当前步骤、变量、工具结果 | 未来可复用的历史信息 |
| 生命周期 | 任务结束后可能消失 | 可跨任务保留 |
| 示例 | 当前执行到第 3 步 | 用户长期要求输出 Markdown 文件 |

### 3.6 本章掌握标准

你能判断：

> 当前问题需要的是 RAG、State、History、Cache，还是 Memory。

## 第 4 章｜Memory 的价值与代价

### 4.1 本章一句话

Memory 能提高连续性和个性化，但也会带来污染、冲突、隐私、成本和复杂度。

### 4.2 Memory 的主要价值

| 价值 | 具体表现 |
|---|---|
| 连续性 | 不用每次重新解释背景 |
| 个性化 | 自动适配用户偏好和工作方式 |
| 项目继承 | 记住阶段、决策、规范和约束 |
| 经验复用 | 把成功模式和失败教训迁移到未来任务 |
| 多轮效率 | 减少重复沟通和重复分析 |
| 质量稳定 | 避免输出风格、标准、流程反复漂移 |

### 4.3 Memory 的主要代价

| 代价 | 风险 |
|---|---|
| 上下文污染 | 取回无关记忆，干扰当前判断 |
| 错误延续 | 错误记忆被长期使用 |
| 旧信息冲突 | 用户偏好或项目状态变化后未更新 |
| Token 成本 | 注入过多记忆导致上下文膨胀 |
| 检索延迟 | 增加系统响应时间 |
| 隐私风险 | 长期保存用户敏感信息 |
| 工程复杂度 | 需要 schema、policy、eval、权限管理 |

### 4.4 记忆的收益判断

一条记忆值得保存，通常要满足至少一个条件：

| 条件 | 示例 |
|---|---|
| 高频复用 | 用户长期喜欢表格化输出 |
| 高影响决策 | 某个项目必须遵循固定目录结构 |
| 高纠错价值 | 某类任务经常发生质量漂移 |
| 高个性化价值 | 用户有明确工作语言和知识沉淀格式 |
| 高项目连续性 | 当前课程已进入第 0–9 章 |

### 4.5 费曼解释

Memory 像一个“助理的笔记本”。

- 笔记太少：助理每次都忘事。
- 笔记太多：助理翻半天，还可能翻到旧信息。
- 笔记错误：助理会一直按错的方式做事。

所以 Memory 的目标不是“多记”，而是“记对、取准、会更新”。

### 4.6 本章掌握标准

你能说出：

> Memory 是收益和风险同时存在的系统，不是越多越好。

## 第 5 章｜Working Memory

### 5.1 本章一句话

Working Memory 是 Agent 当前正在思考和操作的临时信息，类似人的“脑中草稿纸”。

### 5.2 Working Memory 解决什么问题

| 问题 | Working Memory 的作用 |
|---|---|
| 当前任务目标是什么 | 保存当前用户意图 |
| 当前做到哪一步 | 保存中间推理和计划进度 |
| 工具刚返回什么结果 | 暂存工具输出 |
| 下一步该做什么 | 支撑短期决策 |
| 当前有哪些约束 | 防止执行中偏离要求 |

### 5.3 Working Memory 的特点

| 特点 | 说明 |
|---|---|
| 生命周期短 | 通常只在当前推理或当前任务中有效 |
| 强相关 | 只和眼前任务有关 |
| 容易变化 | 每一步执行后都可能更新 |
| 不一定持久化 | 很多 working memory 不需要写入数据库 |
| 主要服务推理 | 帮助模型完成当前任务 |

### 5.4 示例

用户说：

```text
先讲第 0–9 章，然后整理成 .md 文件。
```

Working Memory 应该记住：

| 当前临时信息 | 用途 |
|---|---|
| 任务分两步 | 防止直接只做文件或只讲课 |
| 必须先输出第 0–9 章 | 确保顺序正确 |
| 输出后生成 Markdown 文件 | 完成知识沉淀 |
| 内容主题是 Agent Memory | 限定范围 |

### 5.5 Working Memory 和长期记忆的区别

| 维度 | Working Memory | Long-term Memory |
|---|---|---|
| 作用时间 | 当前任务 | 跨会话、跨任务 |
| 是否持久 | 通常不持久 | 持久 |
| 信息密度 | 只服务眼前步骤 | 面向未来复用 |
| 示例 | “现在要讲第 0–9 章” | “用户喜欢把课程沉淀成 Markdown 文件” |

### 5.6 本章掌握标准

你能判断：

> 哪些信息只需要放在当前任务里，不应该写入长期 Memory。

## 第 6 章｜Short-term Memory

### 6.1 本章一句话

Short-term Memory 是当前会话或当前任务线程内的记忆，用来维持多轮对话和任务连续性。

### 6.2 Short-term Memory 的典型内容

| 内容 | 示例 |
|---|---|
| 最近几轮对话 | 用户刚刚确认课程大纲 |
| 当前任务目标 | 进入第 0–9 章 |
| 当前任务约束 | 不要进入后续章节 |
| 临时偏好 | 本次输出需要表格多一点 |
| 工具结果 | 刚生成的文件路径 |
| 中间决策 | 先讲义，后导出 Markdown |

### 6.3 Short-term Memory 的作用范围

| 范围 | 说明 |
|---|---|
| Turn-level | 单轮推理内有效 |
| Session-level | 当前会话内有效 |
| Thread-level | 当前线程内有效 |
| Task-level | 当前任务完成前有效 |

LangGraph 文档中也把短期记忆描述为线程或会话内的记忆，用于让应用记住单个 thread/conversation 中的先前交互；长期记忆则用于跨会话保留信息。

### 6.4 Short-term Memory 常见实现方式

| 实现方式 | 说明 |
|---|---|
| 最近 N 轮消息 | 保留最近对话 |
| Sliding Window | 滑动窗口，超过长度就丢弃旧内容 |
| Summary Memory | 把早期对话压缩成摘要 |
| Session Store | 按 session_id 存储历史 |
| Checkpoint | 保存 Agent 当前状态，支持恢复 |

OpenAI Agents SDK 的 Sessions 机制就是一种会话级记忆：它会在每次运行前取回指定 session 的历史，并在运行后保存新消息，从而减少手动管理对话历史的工作。

### 6.5 Short-term Memory 的常见风险

| 风险 | 说明 |
|---|---|
| 过长 | 历史太多导致上下文膨胀 |
| 噪音 | 无关对话混入当前任务 |
| 过期 | 前面的临时约束已经不适用 |
| 丢失 | 截断时丢掉关键约束 |
| 混线 | 不同任务线程的记忆混在一起 |

### 6.6 本章掌握标准

你能设计一个基本策略：

> 当前会话保留哪些内容、什么时候摘要、什么时候丢弃、什么时候升级为长期记忆。

## 第 7 章｜Long-term Memory

### 7.1 本章一句话

Long-term Memory 是跨会话、跨任务保留的稳定信息，用来支持长期个性化、项目连续性和经验复用。

### 7.2 Long-term Memory 适合记什么

| 类型 | 示例 |
|---|---|
| 用户长期偏好 | 用户工作语言是中文 |
| 输出格式偏好 | 用户喜欢表格化、高信息密度的 Markdown |
| 项目背景 | 用户正在构建 LLM Wiki 和 Agent 工程体系 |
| 长期产品信息 | 用户运营儿童卡拉 OK 机 |
| 稳定方法论 | 用户偏好 MECE + 费曼教学法 |
| 失败经验 | Codex 执行时可能绕过 PRD，需要质量门禁 |
| 长期目标 | 用户希望构建高质量 Agent 系统 |

### 7.3 Long-term Memory 不适合记什么

| 信息 | 原因 |
|---|---|
| 一次性任务细节 | 未来复用价值低 |
| 临时情绪 | 容易过期且不稳定 |
| 未确认猜测 | 容易形成错误记忆 |
| 敏感信息 | 需要明确授权和严格边界 |
| 完整对话全文 | 噪音高、成本高、难检索 |

### 7.4 Long-term Memory 的关键字段

| 字段 | 作用 |
|---|---|
| memory_id | 唯一标识 |
| type | 记忆类型：偏好、事实、项目、流程等 |
| content | 记忆内容 |
| scope | 用户级、项目级、组织级等 |
| source | 来源 |
| created_at | 创建时间 |
| updated_at | 更新时间 |
| confidence | 可信度 |
| ttl | 有效期 |
| access_policy | 访问权限 |

### 7.5 Long-term Memory 的工程重点

| 重点 | 说明 |
|---|---|
| 写入要谨慎 | 不能把临时信息都长期保存 |
| 检索要精准 | 当前任务只取相关记忆 |
| 更新要明确 | 新旧信息冲突时要有规则 |
| 遗忘要可控 | 旧信息、错误信息、敏感信息要能删除 |
| 用户要可控 | 用户应能查看、修改、删除重要记忆 |

LangGraph 的长期记忆文档提到，长期记忆可以用 namespace 组织，并以 JSON 文档形式存储，常见 namespace 会包含用户、组织等标识，以便隔离和管理记忆。

### 7.6 本章掌握标准

你能判断：

> 哪些信息应该升级为长期记忆，哪些只应该停留在短期上下文中。

## 第 8 章｜Episodic / Semantic / Procedural Memory

### 8.1 本章一句话

从认知结构看，Memory 可以分为事件记忆、语义记忆和流程记忆；这三类决定了 Agent 到底是在记“发生过什么”“知道什么”还是“怎么做”。

### 8.2 三类记忆总表

| 类型 | 中文名 | 记什么 | Agent 中的例子 |
|---|---|---|---|
| Episodic Memory | 事件记忆 | 发生过的事情 | 上次用户完成了 Skill 第 8–11 章 |
| Semantic Memory | 语义记忆 | 稳定事实和知识 | 用户的产品是儿童卡拉 OK 机 |
| Procedural Memory | 流程记忆 | 做事方法和步骤 | 输出知识库文档时先结构化再导出 .md |

### 8.3 Episodic Memory：事件记忆

事件记忆关注“过去发生了什么”。

| 字段 | 示例 |
|---|---|
| 事件 | 用户确认了 Memory 课程大纲 |
| 时间 | 2026-05-21 |
| 场景 | Agent Memory 学习路线图 |
| 结果 | 进入第 0–9 章 |
| 后续影响 | 后续教学应从第 0 章开始 |

适合场景：

| 场景 | 用途 |
|---|---|
| 学习进度 | 记住学到哪一章 |
| 项目记录 | 记住上次完成了什么 |
| 决策记录 | 记住为什么选择某个方案 |
| 复盘分析 | 找回历史错误和成功经验 |

### 8.4 Semantic Memory：语义记忆

语义记忆关注“稳定事实是什么”。

| 事实 | 示例 |
|---|---|
| 用户目标 | 构建高质量 Agent 工程 |
| 用户偏好 | 喜欢 MECE、表格化、费曼式解释 |
| 项目实体 | LLM Wiki、Skill、Agent、Codex |
| 产品背景 | 儿童卡拉 OK 机，目标用户 3–12 岁儿童 |

语义记忆适合长期保存，但必须注意更新。比如产品价格、平台规则、市场数据可能会变化，不适合当作永远不变的事实。

### 8.5 Procedural Memory：流程记忆

流程记忆关注“以后遇到类似任务应该怎么做”。

| 流程 | 示例 |
|---|---|
| 课程设计流程 | 先边界，再概念，再原理，再工程，再实战 |
| 知识沉淀流程 | 先提炼结构，再生成 Markdown 文件 |
| Agent 质量流程 | 先读 PRD，再执行，再按 eval 检查 |
| 广告分析流程 | 先分解指标，再判断 CTR/CVR/ACOS 关系 |

流程记忆通常比单个事件更有迁移价值。

### 8.6 三类记忆的判断法

| 问题 | 对应记忆 |
|---|---|
| 过去发生了什么？ | Episodic |
| 现在稳定事实是什么？ | Semantic |
| 以后应该怎么做？ | Procedural |

### 8.7 本章掌握标准

你能把一段历史内容拆成：

```text
事件记忆：发生了什么
语义记忆：形成了什么稳定事实
流程记忆：沉淀了什么可复用方法
```

## 第 9 章｜User / Project / Tool / Failure Memory

### 9.1 本章一句话

从 Agent 工程落地看，最常用的记忆不是抽象心理学分类，而是用户记忆、项目记忆、工具记忆和失败记忆。

### 9.2 User Memory：用户记忆

用户记忆记录用户长期稳定的偏好、目标、工作方式和背景。

| 记忆内容 | 示例 |
|---|---|
| 工作语言 | 中文 |
| 结构偏好 | MECE、表格化、高信息密度 |
| 教学偏好 | 费曼教学法，先路线图后逐章讲解 |
| 输出偏好 | 经常需要 Markdown 文件沉淀到 LLM Wiki |
| 长期目标 | 构建高质量 Agent 工程体系 |

注意：用户记忆不应该随便记录敏感信息，尤其是身份、健康、政治、宗教、精确地址等内容，除非用户明确要求保存。

### 9.3 Project Memory：项目记忆

项目记忆记录某个项目的长期状态、结构、规范和决策。

| 记忆内容 | 示例 |
|---|---|
| 项目目标 | 构建个人 LLM Wiki 知识库 |
| 当前阶段 | Memory 课程进入第 0–9 章 |
| 目录规范 | 按模块、章节、方法论沉淀 |
| 决策记录 | 先课程学习，再沉淀为 Markdown 文件 |
| 质量标准 | 信息密度高、避免长篇堆叠 |

Project Memory 对 Codex、知识管理 Agent、课程 Agent 特别重要。

### 9.4 Tool Memory：工具记忆

工具记忆记录工具使用经验、限制和适用边界。

| 记忆内容 | 示例 |
|---|---|
| 工具适用场景 | 图片编辑用图像工具，文件导出用脚本 |
| 工具限制 | 某些工具不能后台异步执行 |
| 常见失败 | 文件路径、格式、权限问题 |
| 最佳实践 | 生成文件后必须提供下载链接 |

Tool Memory 的价值是减少重复试错，让 Agent 更稳定地使用工具。

### 9.5 Failure Memory：失败记忆

失败记忆记录过去出现过的问题、原因和预防措施。

| 失败类型 | 示例 | 未来预防 |
|---|---|---|
| 需求漂移 | AI 未按 PRD 执行 | 增加质量门禁 |
| 过早宣称完成 | 实际未完成却说完成 | 强制验收清单 |
| 格式不稳定 | 输出不符合知识库格式 | 使用模板 |
| 记忆污染 | 取回无关旧信息 | 增加检索过滤 |
| 错误延续 | 旧结论未更新 | 加冲突检测 |

Failure Memory 是高质量 Agent 的关键，因为它能让系统“越用越少犯同类错误”。

### 9.6 四类工程记忆的优先级

| 优先级 | 类型 | 为什么重要 |
|---:|---|---|
| 1 | Project Memory | 直接影响当前任务连续性 |
| 2 | User Memory | 影响长期个性化体验 |
| 3 | Failure Memory | 影响质量稳定性 |
| 4 | Tool Memory | 影响执行效率和工具可靠性 |

### 9.7 本章掌握标准

你能在设计 Agent 时明确：

```text
这个 Agent 需要记住哪些用户信息？
需要记住哪些项目状态？
需要记住哪些工具经验？
需要记住哪些失败教训？
```

## 第 0–9 章总结

### 一张总表

| 章节 | 核心问题 | 你需要掌握的判断 |
|---:|---|---|
| 0 | 为什么需要 Memory | Memory 解决 Agent 无状态问题 |
| 1 | Memory 本质是什么 | Memory 是未来可用的上下文资产 |
| 2 | Memory 边界是什么 | Memory 不等于历史、数据库、RAG |
| 3 | Memory 和相邻概念区别 | 能区分 History、RAG、State、Cache |
| 4 | Memory 价值与代价 | 记忆不是越多越好 |
| 5 | Working Memory | 当前推理临时信息 |
| 6 | Short-term Memory | 当前会话/线程内连续性 |
| 7 | Long-term Memory | 跨会话长期复用信息 |
| 8 | 三类认知记忆 | 事件、事实、流程 |
| 9 | 四类工程记忆 | 用户、项目、工具、失败 |

### 阶段性能力标准

学完第 0–9 章，你应该能做到：

| 能力 | 标准 |
|---|---|
| 概念判断 | 能解释 Memory 不是聊天记录 |
| 边界判断 | 能区分 Memory、RAG、State、Cache |
| 类型判断 | 能判断信息属于 working、short-term 还是 long-term |
| 结构判断 | 能区分 episodic、semantic、procedural |
| 工程判断 | 能识别 user、project、tool、failure memory |
| 风险判断 | 知道 Memory 会产生污染、冲突、隐私和成本问题 |

### 最重要的一句话

> Agent Memory 的核心不是“记住过去”，而是“让过去在未来正确地发挥作用”。

# Agent 工程化与产品化｜第 5–10 章：最小可用 Agent

> 阶段定位：本阶段不追求复杂架构，而是先掌握“一个 Agent 最小怎么成立”。  
> 核心目标：能从 0 设计一个可运行、可判断、可复用的最小 Agent。

---

## 本阶段学习目标

| 目标 | 具体表现 |
|---|---|
| 能拆结构 | 知道一个最小 Agent 由哪些模块构成 |
| 能写规则 | 能写出清晰、稳定、可执行的 Instruction |
| 能配上下文 | 知道 Agent 需要什么信息、不需要什么信息 |
| 能接工具 | 知道 Tool Use 的基本工作流 |
| 能控输出 | 能让 Agent 输出稳定格式 |
| 能闭环 | 能把输入、判断、执行、输出、反馈连起来 |

---

# 第 5 章｜最小 Agent 结构

## 5.1 一句话结论

**最小可用 Agent = 目标 + 指令 + 上下文 + 工具 + 输出 + 质量标准。**

如果缺少目标，它只是聊天。  
如果缺少工具，它只能建议。  
如果缺少输出标准，它难以交付。  
如果缺少质量标准，它无法判断做得好不好。

---

## 5.2 费曼解释

你可以把最小 Agent 想成一个“刚上岗的助理”。

你不能只对他说：

> 帮我做一下。

你至少要告诉他：

| 要素 | 人类助理版本 | Agent 版本 |
|---|---|---|
| 目标 | 这件事要完成什么 | Goal |
| 工作规则 | 先做什么、不能做什么 | Instruction |
| 资料 | 参考哪些文件和数据 | Context |
| 工具 | 可以用哪些系统和表格 | Tool |
| 交付物 | 最后交什么结果 | Output |
| 验收标准 | 怎么算做得好 | Eval / Quality Criteria |

所以，最小 Agent 不是“一个提示词”，而是一个小型任务系统。

---

## 5.3 最小 Agent 六件套

| 模块 | 作用 | 问题 |
|---|---|---|
| Goal | 明确任务目标 | 要完成什么？ |
| Instruction | 约束行为方式 | 应该怎么做？不能怎么做？ |
| Context | 提供判断资料 | 需要参考什么？ |
| Tool | 连接外部能力 | 能调用什么？ |
| Output | 定义交付格式 | 最后输出什么？ |
| Quality Criteria | 定义好坏标准 | 怎么判断结果合格？ |

OpenAI Agents SDK 把 Agent 描述为由 LLM 配合 instructions、tools，以及 handoffs、guardrails、structured outputs 等运行行为组成的应用构件。这个定义说明，现代 Agent 已经不是单一 prompt，而是由多种运行能力组合而成的应用单元。

参考：OpenAI Agents SDK 文档  
https://openai.github.io/openai-agents-python/agents/

---

## 5.4 最小 Agent 模板

```text
Agent 名称：
- 用一句话说明这个 Agent 是什么。

目标 Goal：
- 它要完成什么任务？
- 用户最终想获得什么结果？

输入 Input：
- 用户会提供什么？
- 系统会自动补充什么？

上下文 Context：
- 需要哪些资料？
- 哪些资料是必须的？
- 哪些资料不能使用？

工具 Tools：
- 可以调用哪些工具？
- 每个工具解决什么问题？
- 哪些工具调用前必须人工确认？

执行规则 Instruction：
- 先做什么？
- 后做什么？
- 遇到缺失信息怎么办？
- 遇到失败怎么办？
- 哪些情况必须停止？

输出 Output：
- 输出结构是什么？
- 需要表格、JSON、Markdown 还是操作建议？

质量标准 Quality Criteria：
- 准确性标准是什么？
- 完整性标准是什么？
- 可执行性标准是什么？
- 安全边界是什么？
```

---

## 5.5 例子：知识沉淀 Agent

| 模块 | 设计 |
|---|---|
| Goal | 把一段对话沉淀成可放入 llm-wiki 的 Markdown 知识文件 |
| Input | 对话内容、主题名称、知识库风格要求 |
| Context | 当前对话、历史课程大纲、用户知识库偏好 |
| Tools | 文件生成工具、Markdown 写入工具 |
| Instruction | 先提炼主题，再重组结构，再生成 Markdown 文件 |
| Output | `.md` 文件 + 下载链接 |
| Quality Criteria | 结构清晰、信息密度高、可复用、适合知识库长期沉淀 |

---

## 5.6 例子：Amazon 广告诊断 Agent

| 模块 | 设计 |
|---|---|
| Goal | 诊断广告表现变化原因，并输出优化动作 |
| Input | 广告报表、时间区间、目标 ASIN、目标关键词 |
| Context | 广告结构、历史表现、商品定位、竞品价格 |
| Tools | 表格读取、数据清洗、指标计算、图表生成 |
| Instruction | 先检查数据，再拆指标，再定位异常，再输出动作 |
| Output | 问题诊断表 + 优化建议 |
| Quality Criteria | 不泛泛而谈，必须指向 campaign / ad group / keyword / search term 层级 |

---

## 5.7 最小 Agent 的边界

| 能做 | 不一定能做 |
|---|---|
| 单一目标任务 | 大型复杂项目 |
| 明确输入输出 | 长周期不确定任务 |
| 简单工具调用 | 多系统复杂权限 |
| 基础判断 | 高风险自动决策 |
| 局部自动化 | 组织级平台治理 |

最小 Agent 的目标不是“一步到位”，而是建立可验证的最小闭环。

---

## 5.8 本章误区

| 误区 | 正确理解 |
|---|---|
| 最小 Agent = 一个 Prompt | Prompt 只是 Instruction 的一部分 |
| 先做复杂多 Agent | 先做单 Agent 闭环 |
| 工具越多越好 | 工具越多，错误路径越多 |
| 输出越自由越智能 | 输出越自由，越难评估和复用 |
| 没有质量标准也能上线 | 没有质量标准，就无法稳定迭代 |

# 第 6 章｜Instruction 设计

## 6.1 一句话结论

**Instruction 不是“提示词话术”，而是 Agent 的行为契约。**

它要告诉 Agent：

| 问题 | 说明 |
|---|---|
| 你是谁 | 角色定位 |
| 你要做什么 | 任务目标 |
| 你怎么做 | 执行步骤 |
| 你不能做什么 | 边界约束 |
| 你何时停下 | 停止条件 |
| 你如何输出 | 输出格式 |
| 你如何处理异常 | 失败策略 |

---

## 6.2 费曼解释

Instruction 就像你给助理写的“岗位说明书 + 操作手册”。

差的 Instruction：

```text
你是一个优秀的 Agent，请帮我处理任务。
```

问题是：

| 问题 | 结果 |
|---|---|
| 目标不清楚 | Agent 不知道交付什么 |
| 步骤不清楚 | Agent 容易跳步 |
| 边界不清楚 | Agent 容易做过头 |
| 输出不清楚 | 结果难以复用 |
| 异常不清楚 | 失败时容易编造 |

好的 Instruction：

```text
你是一个知识沉淀 Agent。
你的任务是把用户指定阶段的对话整理成 Markdown 文件。
你必须先提炼主题，再按知识库结构重组，再输出文件。
如果上下文不足，不要编造，应标记“缺失信息”。
输出必须包含：标题、摘要、核心概念、结构化表格、实践清单、误区、掌握标准。
```

---

## 6.3 Instruction 七段式结构

| 段落 | 作用 |
|---|---|
| 1. Role | 定义 Agent 身份 |
| 2. Goal | 定义任务目标 |
| 3. Scope | 定义处理边界 |
| 4. Process | 定义执行步骤 |
| 5. Tool Policy | 定义工具使用规则 |
| 6. Output Format | 定义输出格式 |
| 7. Failure Policy | 定义异常处理方式 |

---

## 6.4 Role：角色定义

差的写法：

```text
你是一个专家。
```

好的写法：

```text
你是一个 Amazon 广告诊断 Agent。
你擅长基于广告报表、Search Term Report、关键词表现和商品定位，
诊断 CTR、CVR、CPC、ACOS 变化原因，并输出可执行优化动作。
```

判断标准：

| 标准 | 说明 |
|---|---|
| 具体 | 不是泛泛的“专家” |
| 和任务相关 | 角色必须服务于目标 |
| 能约束行为 | 角色能影响判断方式 |
| 不堆头衔 | 不写无用形容词 |

---

## 6.5 Goal：目标定义

差的写法：

```text
帮我优化广告。
```

好的写法：

```text
你的目标是根据广告数据诊断 ACOS 上升的主要原因，
并按 campaign / ad group / keyword / search term 层级输出优化动作。
```

Goal 要回答：

| 问题 | 示例 |
|---|---|
| 最终结果是什么 | 输出诊断和动作 |
| 关注对象是什么 | 广告数据、关键词、ASIN |
| 判断维度是什么 | CTR、CVR、CPC、ACOS |
| 交付粒度是什么 | campaign / ad group / keyword |

---

## 6.6 Scope：边界定义

Scope 用来防止 Agent 做过头。

| 边界类型 | 示例 |
|---|---|
| 数据边界 | 只能基于用户提供的数据分析 |
| 时间边界 | 只能分析指定时间段 |
| 动作边界 | 不能自动修改广告，只能给建议 |
| 权限边界 | 未经确认不得调用写入工具 |
| 内容边界 | 不得编造不存在的数据 |
| 风险边界 | 高风险动作必须标记风险 |

示例：

```text
你不能假设不存在的数据。
如果缺少关键字段，必须先标记缺失字段。
你只能输出建议，不能直接执行调价、暂停广告或修改预算。
```

---

## 6.7 Process：执行步骤

Process 是最容易决定 Agent 稳定性的部分。

差的写法：

```text
分析数据并给建议。
```

好的写法：

```text
执行流程：
1. 先检查数据字段是否包含 impressions、clicks、spend、sales、orders、CPC、CTR、CVR、ACOS。
2. 对比当前周期和上一周期的指标变化。
3. 判断 ACOS 上升主要来自 CPC 上升、CVR 下降、流量结构变化，还是销售额下降。
4. 定位变化最大的 campaign、ad group、keyword 或 search term。
5. 输出原因、证据、建议动作和风险。
```

Process 的作用：

| 作用 | 说明 |
|---|---|
| 降低随机性 | Agent 不容易乱跳 |
| 提高可检查性 | 每一步都能被追踪 |
| 提高复用性 | 可以沉淀成 Skill |
| 方便评估 | 每一步都能设置质量标准 |

---

## 6.8 Tool Policy：工具使用规则

Tool Policy 要定义：

| 问题 | 示例 |
|---|---|
| 什么情况下用工具 | 需要读取文件时调用文件工具 |
| 什么情况下不用工具 | 用户只问概念时不调用工具 |
| 工具失败怎么办 | 重试一次，仍失败则报告失败原因 |
| 哪些工具需确认 | 写入、删除、发送、修改类工具 |
| 如何处理返回结果 | 必须基于工具返回，不得编造 |

OpenAI 的 function calling 文档描述了工具调用的基本流程：应用把可用工具提供给模型，模型返回工具调用，应用侧执行代码，再把工具结果交回模型，模型最终响应或继续调用工具。

参考：OpenAI Function Calling 文档  
https://developers.openai.com/api/docs/guides/function-calling

---

## 6.9 Output Format：输出格式

输出格式要稳定，否则后续无法复用。

常见输出类型：

| 类型 | 适用场景 |
|---|---|
| Markdown | 知识沉淀、报告、说明文档 |
| 表格 | 对比、诊断、清单 |
| JSON | 程序读取、自动化流程 |
| Checklist | 验收、执行、审核 |
| PRD | 产品需求 |
| SQL / Code | 技术执行 |

示例：

```text
输出必须包含：
1. 结论摘要
2. 关键数据证据
3. 根因判断
4. 优先级排序
5. 建议动作
6. 风险提示
7. 下一步检查项
```

---

## 6.10 Failure Policy：异常处理

失败策略比“聪明回答”更重要。

| 异常 | 应对方式 |
|---|---|
| 缺少数据 | 标记缺失字段，不编造 |
| 工具失败 | 重试，仍失败则说明失败点 |
| 任务不清楚 | 先提出最少必要问题 |
| 风险过高 | 暂停并请求确认 |
| 输出不满足格式 | 自动修正格式 |
| 判断不确定 | 明确标记不确定性和所需证据 |

## 6.12 本章误区

| 误区 | 正确理解 |
|---|---|
| Instruction 越长越好 | 越清楚越好，不是越长越好 |
| 多写形容词能提升效果 | 结构化约束比形容词更有用 |
| 只写角色就够了 | 角色只是七段式的一段 |
| 不需要写失败策略 | 失败策略决定真实可用性 |
| 输出格式靠模型自己理解 | 必须明确格式和字段 |

# 第 7 章｜Context 设计

## 7.1 一句话结论

**Context 不是“资料越多越好”，而是“让 Agent 在当前任务中拿到刚好够用、可信、相关、结构化的信息”。**

---

## 7.2 费曼解释

人做判断时，也不是资料越多越好。

如果你让一个运营助理分析广告，但给他：

- 一堆无关聊天记录
- 过期报表
- 没有时间范围的数据
- 不同 ASIN 混在一起的数据
- 没有字段说明的表格

他很可能会误判。

Agent 也是一样。

> **Context 的质量，决定 Agent 判断的上限。**

---

## 7.3 Context 的四个核心问题

| 问题 | 说明 |
|---|---|
| 有什么 | Agent 能拿到哪些信息 |
| 对不对 | 信息是否可信、最新、准确 |
| 用不用 | 信息是否和当前任务相关 |
| 怎么放 | 信息如何组织给 Agent |

---

## 7.4 Context 分层

| 层级 | 内容 | 示例 |
|---|---|---|
| 用户输入 | 用户当前说的话 | “帮我分析 ACOS 为什么升高” |
| 任务上下文 | 当前任务目标和约束 | 时间区间、ASIN、广告类型 |
| 数据上下文 | 文件、表格、数据库 | 广告报表、搜索词报告 |
| 业务上下文 | 商品、用户、市场背景 | 产品价格、竞品、核心关键词 |
| 历史上下文 | 之前的决策和动作 | 上周调价、否词、换图 |
| 规则上下文 | SOP、规范、策略 | 广告优化规则、品牌口径 |
| 用户偏好 | 用户长期偏好 | 喜欢表格化、高信息密度 |
| 系统上下文 | 工具、权限、环境 | 可用工具、不可执行动作 |

---

## 7.5 Context 不是 Memory

| 对比项 | Context | Memory |
|---|---|---|
| 时间范围 | 当前任务需要的信息 | 跨任务长期保留的信息 |
| 生命周期 | 短期 | 长期 |
| 目标 | 帮助完成当前判断 | 让系统越来越了解用户和项目 |
| 风险 | 上下文污染 | 记忆过期、记错、越权 |
| 例子 | 当前广告报表 | 用户长期做 Amazon 广告、喜欢 Markdown 沉淀 |

简单理解：

```text
Context = 当前这次任务要看的资料
Memory = 以后还可能继续用的背景
```

---

## 7.6 Context 选择原则

| 原则 | 说明 |
|---|---|
| 相关性 | 只放和当前任务有关的信息 |
| 充分性 | 关键判断资料不能缺 |
| 可信性 | 优先使用来源可靠的信息 |
| 新鲜度 | 对动态信息要确认时间 |
| 结构化 | 尽量用表格、字段、清单组织 |
| 可追溯 | 关键结论最好能追溯来源 |
| 权限最小化 | 不给不必要的敏感信息 |

---

## 7.7 Context 优先级

| 优先级 | 信息类型 | 说明 |
|---|---|---|
| P0 | 用户当前明确提供的信息 | 最高优先级 |
| P1 | 当前任务直接需要的数据 | 报表、文件、输入参数 |
| P2 | 项目长期背景 | 商品定位、历史策略 |
| P3 | 通用知识 | 方法论、行业经验 |
| P4 | 推测信息 | 必须标记为假设，不能当事实 |

---

## 7.8 Context 包装模板

```text
任务目标：
- 当前要完成什么？

用户输入：
- 用户原话或结构化任务说明。

关键约束：
- 时间、范围、格式、禁止事项。

可用资料：
- 文件、报表、历史记录、知识库片段。

业务背景：
- 和任务有关的商品、市场、项目背景。

工具能力：
- 当前可调用哪些工具？

输出要求：
- 结果要以什么形式交付？

不确定信息：
- 哪些信息缺失？
- 哪些结论只是推测？
```

---

## 7.9 例子：广告诊断 Context 包

```text
任务目标：
- 诊断 ACOS 上升原因，并输出优化动作。

用户输入：
- “1/26–1/29 广告 CVR 变差，帮我分析原因。”

关键约束：
- 只分析指定时间段。
- 不自动修改广告。
- 必须输出数据证据。

可用资料：
- SP 广告报表
- Search Term Report
- campaign 设置
- ASIN 当前价格
- 竞品价格区间

业务背景：
- 产品是高价位 karaoke machine，售价约 $249.99。
- 竞品常见价格约 $60–100。
- 核心关键词包括 karaoke machine、karaoke machine for adults。

输出要求：
- 输出诊断表、根因排序、动作建议。

不确定信息：
- 如果缺少展示、点击、花费、订单、销售额字段，需要先标记。
```

---

## 7.10 Context 常见错误

| 错误 | 后果 |
|---|---|
| 把所有资料都塞进去 | Agent 分不清重点 |
| 不说明时间范围 | 容易用错数据 |
| 不区分事实和假设 | 容易产生伪结论 |
| 不标记来源 | 结论不可追溯 |
| 不说明业务背景 | Agent 输出泛泛建议 |
| 不清理过期信息 | 用旧规则做新判断 |
| 把敏感信息随便给 Agent | 增加泄露风险 |

# 第 8 章｜Tool Use 基础

## 8.1 一句话结论

**Tool Use 是让 Agent 从“只会说”变成“能做事”的关键能力。**

但是：

> 模型不直接执行工具。  
> 模型选择工具并生成参数。  
> 应用程序负责真正执行工具。  
> 工具结果再交回模型继续判断。

---

## 8.2 费曼解释

Agent 像助理，Tool 像工具箱。

| 工具 | 对应能力 |
|---|---|
| 浏览器 | 搜索资料 |
| Excel | 分析表格 |
| Gmail | 读取和发送邮件 |
| Calendar | 创建日程 |
| 文件系统 | 读取、生成、移动文件 |
| 数据库 | 查询业务数据 |
| 脚本 | 批量处理文件或数据 |

如果 Agent 没有工具，它只能给建议。

如果 Agent 有工具，它可以执行真实动作。

---

## 8.3 Tool Use 的基本流程

OpenAI function calling 文档描述的工具调用流程可以简化为五步：

```text
1. 应用把可用工具和工具 schema 提供给模型
2. 模型判断是否需要调用工具
3. 模型生成工具名称和参数
4. 应用程序执行真实工具
5. 应用把工具结果交回模型，模型继续响应或再次调用工具
```

参考：OpenAI Function Calling 文档  
https://developers.openai.com/api/docs/guides/function-calling

---

## 8.4 Tool 的最小结构

| 字段 | 作用 |
|---|---|
| name | 工具名称 |
| description | 工具用途说明 |
| input_schema | 输入参数结构 |
| output_schema | 返回结果结构 |
| permission | 权限等级 |
| error_policy | 失败处理规则 |
| confirmation_policy | 是否需要人工确认 |

示例：

```json
{
  "name": "read_ad_report",
  "description": "读取 Amazon 广告报表并返回结构化指标数据",
  "input_schema": {
    "file_path": "string",
    "date_range": "string",
    "asin": "string"
  },
  "output_schema": {
    "campaigns": "array",
    "summary_metrics": "object",
    "missing_fields": "array"
  },
  "permission": "read_only",
  "confirmation_policy": "not_required"
}
```

---

## 8.5 工具分类

| 类型 | 示例 | 风险 |
|---|---|---|
| 读取型工具 | 读文件、查数据库、搜资料 | 低到中 |
| 计算型工具 | 指标计算、数据清洗 | 低 |
| 生成型工具 | 生成报告、生成文件 | 中 |
| 写入型工具 | 修改数据库、创建任务 | 中高 |
| 发送型工具 | 发邮件、发消息 | 高 |
| 删除型工具 | 删除文件、清空数据 | 高 |
| 支付/交易型工具 | 下单、扣款、调预算 | 极高 |

工具风险越高，越需要确认、日志、回滚和权限控制。

---

## 8.6 什么时候需要工具

| 情况 | 是否需要工具 |
|---|---|
| 解释一个稳定概念 | 通常不需要 |
| 读取用户文件 | 需要 |
| 查询最新信息 | 需要 |
| 计算复杂数据 | 需要 |
| 生成可下载文件 | 需要 |
| 修改外部系统 | 需要，而且需要权限控制 |
| 发送邮件或消息 | 需要，而且通常需要用户确认 |

---

## 8.7 Tool Schema 设计原则

| 原则 | 说明 |
|---|---|
| 名称清晰 | 工具名能表达用途 |
| 描述具体 | 告诉模型何时使用、何时不用 |
| 参数最少 | 只保留必要参数 |
| 类型明确 | string、number、boolean、array、object |
| 枚举优先 | 固定选项用 enum 降低错误 |
| 返回结构稳定 | 方便模型继续处理 |
| 错误可读 | 错误信息要能指导下一步 |
| 权限明确 | 区分读、写、删、发、支付 |

---

## 8.8 工具描述示例

差的描述：

```text
分析广告。
```

好的描述：

```text
读取 Amazon Sponsored Products 广告报表，
计算 impressions、clicks、CTR、CPC、spend、sales、orders、CVR、ACOS，
并按 campaign、ad group、keyword 聚合返回。
当用户要求分析广告表现、比较时间段或诊断 ACOS/CVR/CTR 变化时使用。
不要用于生成广告文案。
```

---

## 8.9 工具错误处理

| 错误类型 | 应对方式 |
|---|---|
| 参数缺失 | 要求补充或使用默认值 |
| 文件不存在 | 报告文件缺失 |
| 权限不足 | 请求授权或停止 |
| 返回为空 | 说明没有匹配数据 |
| 数据格式错误 | 标记字段问题 |
| 超时 | 重试或降级 |
| 高风险操作 | 暂停并请求确认 |

## 8.10 MCP 与 Tool Use 的关系

MCP 是连接外部系统的标准协议。MCP 官方文档将其描述为连接 AI 应用与外部数据源、工具和工作流的开放标准，并用 USB-C 类比它在 AI 应用中的作用。

参考：MCP 官方文档  
https://modelcontextprotocol.io/docs/getting-started/intro

简单理解：

```text
Tool Use = Agent 会使用工具
MCP = 工具接入的标准接口
```

MCP 规范中的 Tools 允许服务器暴露可被模型调用的工具，用于查询数据库、调用 API 或执行计算等外部动作。

参考：MCP Tools 文档  
https://modelcontextprotocol.io/specification/2025-06-18/server/tools

## 8.11 本章误区

| 误区 | 正确理解 |
|---|---|
| Tool 越多越好 | 工具越多，决策空间越大，错误也越多 |
| 模型直接执行工具 | 工具由应用侧执行 |
| 工具描述随便写 | 工具描述决定模型何时调用 |
| 所有工具都能自动执行 | 高风险工具必须确认 |
| 工具失败后让模型猜 | 失败必须显式处理 |

# 第 9 章｜输出结构化

## 9.1 一句话结论

**结构化输出，是让 Agent 从“看起来有道理”变成“可以被复用、检查、自动化处理”的关键。**

没有结构化输出，Agent 的结果很难：

| 问题 | 后果 |
|---|---|
| 被程序读取 | 无法进入自动化流程 |
| 被人快速判断 | 阅读成本高 |
| 被测试 | 无法做自动评估 |
| 被复用 | 难以进入知识库或工作流 |
| 被对比 | 多版本之间难比较 |

---

## 9.2 费曼解释

如果你让一个助理做广告诊断，他交给你一大段话：

> 这个广告可能是转化不好，也可能是点击质量下降，建议你优化一下……

你很难执行。

如果他交给你一张表：

| 问题 | 证据 | 根因 | 动作 | 优先级 |
|---|---|---|---|---|
| CVR 下降 | 从 5.5% 降到 3.8% | 流量不精准 | 否定低转化 search term | 高 |

这就可以直接进入执行。

---

## 9.3 常见结构化输出类型

| 类型 | 适合场景 |
|---|---|
| Markdown | 知识文档、报告、课程内容 |
| 表格 | 对比、诊断、清单、优先级 |
| JSON | 程序读取、工具链自动化 |
| YAML | 配置文件、规范文件 |
| Checklist | 验收、审核、执行 |
| XML | 特定系统集成 |
| CSV | 数据交换 |
| PRD 模板 | 产品需求文档 |
| SOP 模板 | 操作流程文档 |

---

## 9.4 结构化输出的三个层级

| 层级 | 表现 | 示例 |
|---|---|---|
| L1 文本结构化 | 标题、段落、列表 | Markdown |
| L2 字段结构化 | 固定字段、固定顺序 | 表格、模板 |
| L3 机器结构化 | 严格 schema | JSON Schema |

OpenAI Structured Outputs 文档说明，结构化输出可通过 function calling 或 `json_schema` 响应格式实现；function calling 适合把模型和应用功能连接起来，`json_schema` 适合让模型响应匹配指定结构。

参考：OpenAI Structured Outputs 文档  
https://developers.openai.com/api/docs/guides/structured-outputs

---

## 9.5 输出格式设计原则

| 原则 | 说明 |
|---|---|
| 和任务匹配 | 不同任务用不同格式 |
| 字段稳定 | 字段名不要每次变化 |
| 粒度合适 | 既不能太粗，也不能过碎 |
| 支持检查 | 每个结论最好有证据字段 |
| 支持执行 | 建议要能变成动作 |
| 支持复用 | 输出能进入下一步流程 |
| 支持异常 | 能表达缺失信息、不确定性和失败原因 |

# 第 10 章｜简单任务闭环

## 10.1 一句话结论

**简单任务闭环，就是让 Agent 能从用户输入开始，经过理解、取上下文、调用工具、生成结构化输出，最后让用户能判断结果好坏。**

最小闭环不是“回答一次”，而是：

```text
输入 → 判断 → 上下文 → 工具 → 输出 → 验收 → 反馈
```

---

## 10.2 费曼解释

一个不能闭环的 Agent，就像一个只会说“我建议你……”的助理。

一个能闭环的 Agent，至少应该做到：

| 步骤 | 表现 |
|---|---|
| 接任务 | 明确用户目标 |
| 查资料 | 拿到必要上下文 |
| 做判断 | 选择方法和步骤 |
| 执行动作 | 调用工具或生成文件 |
| 交结果 | 输出结构化交付物 |
| 可检查 | 用户知道好坏 |
| 可迭代 | 用户反馈后能修正 |

---

## 10.3 最小闭环结构

```text
1. 用户输入
   ↓
2. 任务识别
   ↓
3. 上下文组装
   ↓
4. 执行计划
   ↓
5. 工具调用
   ↓
6. 结果整合
   ↓
7. 结构化输出
   ↓
8. 质量检查
   ↓
9. 用户反馈
```

LangGraph 文档区分了 workflow 和 agent：workflow 通常有预设路径，agent 更动态，会决定自己的流程和工具使用。最小任务闭环可以理解为：先用 workflow 固定关键步骤，再在需要判断的节点引入 Agent。

参考：LangGraph Workflows and Agents 文档  
https://docs.langchain.com/oss/python/langgraph/workflows-agents

```text
- 用户知识库偏好
- 主题范围

执行计划：
1. 提炼主题
2. 重组结构
3. 删除闲聊
4. 保留概念、框架、例子、误区、实践题
5. 生成 Markdown 文件

- .md 文件

质量检查：
- 是否适合长期沉淀？
- 是否结构清晰？
- 是否有复用价值？
```

## 3. 本阶段核心判断

| 判断问题 | 说明 |
|---|---|
| 目标是否明确？ | 不明确就不能稳定执行 |
| 输入是否足够？ | 不足就不能可靠判断 |
| 上下文是否相关？ | 不相关会污染判断 |
| 工具是否必要？ | 不必要不要调用 |
| 工具是否安全？ | 高风险工具需要确认 |
| 输出是否结构化？ | 不结构化就难复用 |
| 是否有质量标准？ | 没有标准就无法评估 |
| 是否形成闭环？ | 没有反馈就无法迭代 |

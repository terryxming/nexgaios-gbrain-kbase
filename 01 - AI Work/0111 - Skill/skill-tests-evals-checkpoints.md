# Skill 中 Tests、Evals、Checkpoints 的区别

## 一句话总结

在 skill 设计里，可以这样理解：

> Tests 看“东西有没有坏”；Checkpoints 看“过程有没有走偏”；Evals 看“最终效果好不好”。

更精确一点：

| 概念 | 关注点 | 判断方式 | 典型问题 |
| --- | --- | --- | --- |
| Tests | 明确规则是否满足 | 硬检查，pass/fail | 这个 skill 的结构、脚本、资源有没有问题？ |
| Checkpoints | 执行过程是否走在正确路径上 | 阶段检查，状态确认 | agent 做到中途有没有漏关键步骤？ |
| Evals | 真实任务中的整体效果 | 质量评估、rubric、对比实验 | 这个 skill 是否真的让 agent 做得更好？ |

## Tests：硬检查

Tests 适合检查那些答案明确、边界清楚、可以自动判断的事情。

也可以把它理解成“断言判断”：

> 我提前写清楚一个必须成立的条件，程序跑完后检查它是否成立。

例如：

```python
assert skill_name == "pdf-editor"
assert "description" in frontmatter
assert script_exit_code == 0
assert output_file.exists()
```

在 skill 里，tests 常用来检查：

- `SKILL.md` frontmatter 是否合法
- `name` 是否符合命名规则
- `description` 是否存在且不为空
- 引用的 `references/xxx.md` 是否真实存在
- `scripts/` 里的脚本能否运行
- 给脚本一个固定输入，能否得到预期输出
- 输出文件、JSON 字段、表格列名等是否符合要求

Tests 的特点是：

- 结果通常是 pass/fail
- 适合放进 CI 或本地快速校验
- 稳定、可重复
- 主要抓“坏了”“漏了”“路径错了”“格式错了”

## Checkpoints：过程检查点

Checkpoints 是任务执行过程中的阶段性检查。

它不完全等同于 tests，也不完全等同于 evals。它更像是：

> 做到某一步时，停一下，看当前状态是否合理，再决定是否继续。

在 skill 里，checkpoints 常用来确认 agent 是否按正确路径推进。

例如：

- 是否正确触发了这个 skill
- 是否先读了 `SKILL.md`
- 是否读取了任务所需的 reference，而不是一股脑读所有文件
- 是否在编辑前检查现有代码或现有文档
- 是否在执行危险操作前停下来确认
- 是否在输出前做了验证
- 是否在长任务中阶段性总结当前发现

Checkpoints 的特点是：

- 更关注执行过程
- 适合长链路、多步骤、高风险任务
- 可以是人工检查，也可以部分自动化
- 主要抓“中途走偏”“漏关键步骤”“顺序错了”

## Evals：效果评估

Evals 用来判断 skill 在真实任务中是否真的有效。

它关注的不是某个文件有没有坏，而是：

> agent 用了这个 skill 之后，是否更准确、更稳定、更完整、更符合预期。

在 skill 里，evals 常用来检查：

- 用户提出真实请求时，agent 是否会触发正确 skill
- 触发 skill 后，agent 是否按 skill 的流程执行
- 最终输出是否满足任务目标
- 是否比没有 skill 时更好
- 是否比旧版 skill 更好
- 是否能处理相似但没有见过的新任务
- 是否减少了常见错误、漏步骤或无效操作

Evals 的结果不一定是简单的 pass/fail，可能是：

- 人工评分
- rubric 打分
- A/B 对比
- 多个测试任务的通过率
- 与旧版本输出的质量对比
- 是否命中关键行为或关键结论

Evals 的特点是：

- 更接近真实使用场景
- 可能有随机性
- 需要更丰富的样本
- 更适合评估复杂 skill、决策型 skill、工作流型 skill

## 生活类比

### 做菜

Tests：

- 烤箱是不是 180 度
- 有没有放盐
- 鸡肉中心温度是否达标

Checkpoints：

- 腌制是否完成
- 炒到一半是否需要尝咸淡
- 汤汁是否收到合适程度

Evals：

- 最后这道菜好不好吃
- 客人是否喜欢
- 是否愿意下次再点

### 写作文

Tests：

- 字数是否达标
- 有没有错别字
- 格式是否正确

Checkpoints：

- 提纲是否清楚
- 第一版是否跑题
- 论据是否支撑观点

Evals：

- 文章是否有说服力
- 结构是否自然
- 读者是否被打动

### 学车

Tests：

- 有没有系安全带
- 有没有打转向灯
- 有没有压线

Checkpoints：

- 上路前是否调整座椅和后视镜
- 变道前是否观察盲区
- 进路口前是否减速判断

Evals：

- 整体驾驶是否安全
- 是否平稳
- 是否能处理复杂路况

### 装修

Tests：

- 插座是否通电
- 水管是否漏水
- 门能不能正常关闭

Checkpoints：

- 水电改造完成后是否验收
- 贴砖前基层是否处理好
- 每个阶段是否和设计图一致

Evals：

- 住起来是否舒服
- 动线是否顺
- 风格是否协调
- 是否符合长期使用需求

## 放回 Skill 设计

### 一个 PDF 旋转 skill

Tests：

- 脚本能否运行
- 输入 PDF 是否能生成输出 PDF
- 旋转角度参数是否合法
- 输出页数是否和输入一致

Checkpoints：

- 通常不需要太复杂的 checkpoints
- 最多检查 agent 是否先确认输入文件和旋转方向

Evals：

- 用户说“帮我把这份 PDF 横过来”时，agent 是否能正确理解方向并调用脚本
- 输出文件是否真的符合用户意图

### 一个 GitHub PR review skill

Tests：

- GitHub 查询脚本能否运行
- diff 是否能正常读取
- 输出格式是否包含必要字段

Checkpoints：

- 是否先读 diff 再下结论
- 是否优先列风险和 bug
- 是否引用具体文件和行号
- 是否避免评论无关风格问题

Evals：

- 是否抓到了真实 bug
- 是否减少了误报
- review 是否对开发者有帮助
- 和没有 skill 时相比，问题质量是否更高

### 一个数据分析 skill

Tests：

- 数据能否读取
- 图表脚本能否运行
- 指标计算函数是否返回预期结果

Checkpoints：

- 是否先检查数据质量
- 是否确认指标口径
- 是否说明缺失值、异常值、时间范围
- 是否在给结论前做交叉验证

Evals：

- 分析结论是否正确
- 洞察是否有业务价值
- 建议是否可执行
- 是否能帮助用户做决策

## 是否三个都要有？

不一定。

更实用的判断是：

> Tests 是底线，Evals 看复杂度，Checkpoints 看任务链路长不长。

| Skill 类型 | Tests | Checkpoints | Evals |
| --- | --- | --- | --- |
| 简单格式或脚本型 skill | 应该有 | 通常可选 | 可选 |
| 多步骤 workflow skill | 应该有 | 建议有 | 建议有 |
| 会影响 agent 决策的 skill | 应该有 | 建议有 | 必须有 |
| 高风险输出 skill，例如法律、财务、医疗、生产发布 | 必须有 | 必须有 | 必须有 |
| 内部临时辅助 skill | 简单 tests 即可 | 可选 | 可选 |

## 设计判断框架

可以按三类失败来设计保障：

### 1. 东西坏了

用 tests 抓。

常见问题：

- 文件缺失
- YAML 写错
- 脚本跑不起来
- 输出格式不对
- 路径引用错误

### 2. 中途走偏了

用 checkpoints 抓。

常见问题：

- 没有先理解任务
- 没有读取关键上下文
- 步骤顺序错了
- 忘记验证
- 在高风险步骤前没有停下来

### 3. 最后不好用

用 evals 抓。

常见问题：

- 最终答案质量不高
- 没有解决真实用户问题
- 泛化能力差
- 比不用 skill 没有明显提升
- 输出虽然格式对，但决策价值低

## 最后的记忆方式

可以记成三句话：

> Tests：规则有没有错。

> Checkpoints：过程有没有走偏。

> Evals：结果好不好用。

或者更口语一点：

> Tests 管“零件没坏”；Checkpoints 管“路没走歪”；Evals 管“车开得好不好”。

6. 本次未实现新的生命周期扫描脚本；脚本实现前，由 Agent 按 10 v0.1.0 人工检查并在完成汇报中说明证据。
7. inbox dry-run 脚本、GBrain DB 同步执行器和三个模板已同步依赖声明到主手册 v0.4.7；后续如脚本实现新增契约，需要另行升版本并补变更日志。

影响范围：

- inbox 路由

兼容影响：需要迁移

涉及规则：

- 《00 - inbox》路由分发规则.md
- 自动化脚本/00 - inbox 校验与路由输入 dry-run.ps1

规则契约：

inbox 路由先直接执行无需用户决策的高置信度材料；Excel 审核表只包含需要用户决策的材料。审核表不再包含 `当前路径`、`source_type`、`当前 material_type`、`当前 tags`、`目标目录是否存在`、`动作建议` 等执行噪音字段。执行器必须用固定 inbox 路径和 `文件` 字段重建源路径，源文件不在 inbox 时必须跳过。

生成审核表的流程必须过滤掉可直接执行材料；执行审核表的流程必须按 `文件` 字段从 inbox 定位源文件，不再读取 `当前路径` 字段。

兼容影响：向后兼容

涉及规则：

- 《01 - wiki》目录与索引规则.md

新增 `01 - wiki` 目录与索引规则。wiki 目录只表达主知识领域，不表达 `candidate`、`active`、`deprecated`、`rejected` 等生命周期状态。`_index.md` 是总导航，`_meta` 只存放编译日志、冲突索引、晋升记录和待处理索引。治理索引是派生视图，不作为业务事实来源。

- 《01 - wiki》目录与索引规则.md
- 《01 - wiki》知识编译规则.md

规则契约：

Wiki 写入后必须执行落盘命名、页面结构、补丁污染和索引同步验收。`_index.md`、`_meta` 及其治理文件必须按规则命名；发现 `index.md` 或 `meta` 时必须立即修正。Markdown 正文中禁止残留补丁标记。新增 candidate 页面后必须同步 `_index.md`、`_meta/编译日志.md` 和 `_meta/待处理索引.md`。

后续执行 GBrain DB 同步时，先运行该脚本 dry-run。只有 dry-run 报告无阻断条件，且用户确认正式同步，才能执行 `-Execute`。不得绕过该脚本直接运行正式同步。

规则契约：

`06 - GBrain DB 同步与 MCP 查询规则.md` 升级为 v0.2.1。正式 GBrain DB 同步前，同步执行器必须校验同步计划中的目标文件是否存在、目标路径是否位于同步范围内、目标 Markdown 是否满足 14 字段 frontmatter 契约、`knowledge_layer` 是否为合法层级、`gbrain_db_sync_status` 是否为合法枚举。任一目标文件不满足前置校验时，正式同步不得执行，报告中必须标记 `sync_action: blocked` 和 `sync_status_after: failed`。

决策原因：

路径和 slug 正确不代表知识文件可以进入运行层。schema、raw 或 wiki 文件如果缺少 14 字段 frontmatter，会导致 GBrain DB 中无法区分层级、状态、来源和同步语义，后续 MCP 查询会把半成品误当作可用知识。

迁移要求：

所有待同步的 raw、wiki、schema Markdown 必须先回到“知识入口与字段契约”阶段补齐 14 字段。同步执行器不得为缺少 frontmatter 的文件直接写入 GBrain DB，也不得在目标文件没有可写 frontmatter 时强行回写 `failed` 状态。

# 第 55 章：实战一：内容清洁 Skill

## 55.1 项目目标

内容清洁 Skill 的目标是：

> 清理 Prompt、Skill 文件、工作流说明、课程大纲或 AI 生成文本中的废话、重复表达、迎合性语言、模糊约束和无意义包装，保留真正可执行、可维护、可复用的内容。

这个 Skill 适合解决一个常见问题：

```text
随着 Prompt、Skill、SOP 不断迭代，文件中会混入大量重复、空泛、迎合、无效的表达，导致指令越来越长，但执行质量没有提升。
```

## 55.2 适用场景

适合使用：

- 清理冗长 Prompt
- 清理 `SKILL.md`
- 清理工作流说明
- 清理课程大纲
- 清理产品需求草稿
- 清理 AI 输出中的废话
- 删除“你是世界顶级专家”这类低价值角色包装
- 删除重复的“请系统性、完整、全面”堆叠表达
- 清理互相冲突或没有验收标准的约束

不适合使用：

- 需要创意扩写的文案
- 需要保留原文风格的文学内容
- 法律、合同、协议原文的实质改写
- 用户明确要求逐字保留的内容
- 需要保留情绪表达的用户访谈原文

## 55.3 最终产出

这个 Skill 的输出应该包括：

```text
1. 清洁后的内容
2. 删除内容说明
3. 保留内容说明
4. 发现的冲突或歧义
5. 后续可优化建议
```

```text
│   ├── 噪音表达清单.md

├── assets/
```

## 55.5 中文化 `SKILL.md`

```markdown
---
name: content-cleaner
description: 当用户需要清理 Prompt、Skill 文件、工作流说明、课程大纲或 AI 生成文本中的废话、重复表达、迎合性语言、模糊约束和无效包装时使用。适用于“内容清洁”“提示词清理”“删除无意义表达”“精简 Skill 文件”“去掉废话但保留有效规则”等请求。不适用于需要文学润色、情绪表达保留、合同原文改写或用户明确要求逐字保留的内容。
---

# 目标

将输入内容清理为更简洁、更可执行、更可维护的版本。清理时保留任务目标、关键约束、输入输出要求、边界规则、安全要求和验收标准，删除重复、空泛、迎合、装饰性或不可执行的表达。

# 输入

用户可能提供：
- Prompt 原文
- `SKILL.md` 内容
- 工作流说明
- SOP 草稿
- 课程大纲
- AI 生成文本
- 需要清洁的任意中文说明文本

如果用户没有说明清洁强度，默认采用“中等清洁”：
- 删除明显无效表达
- 保留可能影响执行的规则
- 不改变原任务目标

# 处理流程

1. 识别输入内容的真实任务目标。
2. 标记无效内容，包括重复表达、迎合性语言、空泛形容词、无执行价值的角色包装。
3. 标记必须保留的内容，包括任务目标、输入要求、输出格式、边界、约束、测试和安全规则。
4. 检查是否存在互相冲突的要求。
5. 在不改变原意的前提下，重写为简洁、结构化、可执行版本。
6. 输出删除说明和保留说明。
7. 如果存在无法判断的歧义，单独列出，不自行编造。

# 输出格式

输出以下结构：

## 清洁后的内容

[给出清洁后的版本]

## 删除内容说明

| 删除类型 | 示例 | 删除原因 |
|---|---|---|

## 保留内容说明

| 保留类型 | 示例 | 保留原因 |
|---|---|---|

## 冲突或歧义

- [列出冲突、歧义或信息不足]

## 后续优化建议

- [给出可选优化方向]

# 质量标准

输出必须满足：
- 不改变原任务目标
- 不弱化安全要求
- 不删除关键约束
- 不保留无执行价值的夸张表达
- 不把内容改成完全不同的风格
- 清洁后更短、更清楚、更可维护
- 明确说明删除了什么、为什么删除

# 禁止事项

不要：
- 为了简短而删除安全边界
- 为了流畅而改写事实
- 把内容改成营销文案
- 删除用户明确要求保留的格式
- 用“更专业、更高级”等空泛词替代具体规则
- 在没有依据时新增任务目标

# 示例

## 正向示例

输入：
“你是世界顶级 AI 专家和课程设计大师，请系统性、完整、全面、深入地帮我整理这个内容，一定要高质量，不要敷衍。”

期望行为：
- 删除“世界顶级”“大师”“一定要高质量”等空泛表达
- 保留“系统性、完整、深入”中真正可执行的部分
- 转换为具体输出要求，例如结构、边界、示例、检查表

## 反向示例

输入：
“请保留以下用户访谈原文，不要改写，只标出可能的主题。”

期望行为：
- 不清理原文情绪表达
- 只做主题标注
- 不重写用户原话
```

## 55.7 验收标准

这个 Skill 合格的标准：

```text
1. 能稳定删除废话和重复表达
2. 能保留核心任务目标
3. 能保留安全、事实、边界规则
4. 能识别冲突或歧义
5. 能说明删除原因
6. 不会把清洁变成随意改写
7. 对 Prompt、Skill、SOP 都能适用
```

- 增加 Markdown 标题层级校验脚本
- 增加回归测试：防止误删安全规则

# 第 56 章：实战二：Amazon Review 分析 Skill

## 56.1 项目目标

Amazon Review 分析 Skill 的目标是：

> 从商品评论中提取用户痛点、好评驱动、购买动机、功能请求、使用场景、产品改进机会和文案角度。

这个 Skill 服务于电商运营、产品经理、市场人员和文案设计团队。

## 56.2 适用场景

适合使用：

- 分析 Amazon 商品评论
- 分析竞品评论
- 从评论中提取用户痛点
- 从好评中提取卖点表达
- 从差评中反推产品改进方向
- 为 A+、Listing、广告文案提供用户语言
- 将评论洞察转化为产品、文案、设计建议

不适合使用：

- 没有评论数据时虚构评论
- 把少量评论当成整个市场结论
- 替代正式市场调研
- 直接生成未经证据支持的合规宣称
- 抓取未经授权或不可访问的数据

## 56.3 最终产出

```text
1. 评论总体摘要
2. 评论主题分类表
3. 高频痛点
4. 好评驱动因素
5. 购买动机
6. 功能请求
7. 产品改进机会
8. 可用于文案的用户语言
9. 风险和证据限制
```

## 56.5 中文化 `SKILL.md`

```markdown
---
name: amazon-review-analyzer
description: 当用户提供 Amazon 商品评论、竞品评论、评论导出文件或评论文本，并希望分析用户痛点、好评驱动、购买动机、功能请求、使用场景和产品改进机会时使用。适用于“Review 分析”“评论洞察”“差评痛点”“竞品评论分析”“从评论反推卖点”等请求。不适用于没有评论数据时虚构评论、替代正式市场调研或生成未经证据支持的产品宣称。
---

# 目标

分析用户评论，提取可用于产品优化、运营决策、Listing 文案、A+ 内容和视觉设计的真实用户洞察。必须区分评论原始证据、分析推断和后续建议。

# 输入

必填：
- 评论文本、评论表格、评论导出文件，或用户粘贴的评论片段

可选：
- ASIN
- 产品名称
- 产品类目
- 目标市场
- 分析目标
- 竞品名称
- 重点关注维度

默认策略：
- 如果用户没有指定分析目标，默认分析：痛点、好评驱动、购买动机、功能请求、使用场景、产品机会。
- 如果评论数量较少，必须标注“样本有限，仅作为方向性参考”。
- 不得虚构评论内容。

# 处理流程

1. 识别产品类型、评论来源和样本规模。
2. 清理评论中的重复内容、无效符号和明显噪音。
3. 将评论按主题分类，例如音质、连接、安装、外观、耐用性、电池、兼容性、价格、售后、使用场景。
4. 提取正面信号：用户喜欢什么、为什么购买、什么场景满意。
5. 提取负面信号：用户抱怨什么、影响什么体验、是否重复出现。
6. 提取功能请求：用户希望增加、优化或改变什么。
7. 区分三类内容：评论证据、分析推断、建议动作。
8. 将洞察转化为产品优化建议、文案角度和设计表达方向。
9. 标注样本限制、证据不足和需要进一步验证的内容。

# 输出格式

## 一、总体摘要

- 评论样本规模：
- 总体倾向：
- 最值得关注的问题：
- 最有价值的机会：

## 二、评论主题分类表

| 主题 | 正面信号 | 负面信号 | 证据摘要 | 产品含义 | 建议动作 |
|---|---|---|---|---|---|

## 三、高频痛点

| 痛点 | 用户表现 | 可能影响 | 证据强度 | 优先级 |
|---|---|---|---|---|

## 四、好评驱动因素

| 驱动因素 | 用户喜欢的原因 | 可转化卖点 | 适合使用位置 |
|---|---|---|---|

## 五、功能请求

| 请求 | 来源表现 | 产品机会 | 是否需要验证 |
|---|---|---|---|

## 六、文案与视觉机会

| 洞察 | 可转化文案角度 | 可转化视觉表达 |
|---|---|---|

## 七、风险与限制

- [列出样本不足、数据偏差、证据不足、不可直接作为市场结论的内容]

# 质量标准

输出必须：
- 只基于用户提供或可验证的评论
- 明确区分证据、推断和建议
- 不虚构评论
- 不把少量评论当成市场定论
- 不生成未经验证的产品参数或认证
- 输出对运营、产品、文案、设计都有可用价值

# 禁止事项

不要：
- 编造评论内容
- 编造评论数量
- 编造产品参数
- 直接写“用户普遍认为”除非样本足够且证据明确
- 把推断当成事实
- 生成平台合规承诺
- 输出没有证据支撑的绝对化卖点

# 示例

## 正向示例

输入：
“以下是某款无线麦克风的 50 条评论，请分析用户痛点和卖点机会。”

期望行为：

- 提取正负信号
- 区分证据和推断
- 输出产品、文案、视觉建议

## 反向示例

输入：

期望行为：

- 可以提供评论分析框架
- 标注需要真实评论数据
```

# 第 60 章：实战六：团队级 Skill Library

## 60.1 项目目标

团队级 Skill Library 的目标是：

> 建设一个可分类、可发现、可测试、可治理、可迭代的团队 Skill 能力资产库。

这个项目不是写一个 Skill，而是建设一套 Skill 生产、管理和复用系统。

## 60.2 适用场景

适合使用：

- 团队已有多个 Skill
- 多人重复写相似 Prompt
- 需要统一文案、设计、研发、知识管理标准
- 需要让新人快速复用团队方法
- 需要建立 AI 工作流资产库
- 需要治理 Skill 安全、版本和权限

不适合使用：

- 只有一两个个人 Skill
- 任务还没有稳定流程
- 团队不准备维护
- 没有 owner 和治理机制
- 只想随便保存一些 Prompt

# 阶段十一总结

阶段十一的核心结论：

1. **实战项目是 Skill 工程化的落地入口。**  
   只有在真实任务中构建 Skill，才能暴露输入、输出、边界、测试和治理问题。

2. **每个 Skill 都应从明确场景出发。**  
   内容清洁、Review 分析、A+ 图片 brief、知识库沉淀、SDD/BDD/TDD 拆解，都有明确用户、输入、输出和验收标准。

3. **实战 Skill 必须包含反例。**  
   反例能防止误触发、越界执行和过度应用。

4. **输出契约决定 Skill 是否能进入工作流。**  
   如果输出不能直接被使用、测试或传递给下游 Skill，就还不是成熟 Skill。

5. **团队级 Skill Library 是能力资产化的最终载体。**  
   单个 Skill 解决任务，Skill Library 管理能力，治理机制保证长期可用。

阶段十一最重要的一句话：

> 实战 Skill 的价值，不在于写出一个漂亮的 `SKILL.md`，而在于它能在真实业务中稳定接收输入、执行流程、输出结果、通过测试，并被团队持续复用。

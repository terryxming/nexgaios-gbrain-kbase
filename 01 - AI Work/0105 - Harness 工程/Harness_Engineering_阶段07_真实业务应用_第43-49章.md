# 第 43 章：知识库沉淀 Agent Harness

## 43.1 本章核心

> 知识库沉淀 Agent 的目标，是把一次对话、一次课程、一次分析、一次决策，转成可长期检索、复用、迭代的知识资产。

它不是普通总结。

普通总结：

```text
把内容压缩一下，方便回看。
```

知识库沉淀：

```text
把内容重新结构化，放入稳定知识系统，未来可以被检索、引用、迁移和复用。
```

---

## 43.2 适用场景

| 场景 | 是否适合 |
|---|---:|
| 一轮系统课程讲解完成后 | 适合 |
| 一个方法论已经确认后 | 适合 |
| 一次复杂问题分析后 | 适合 |
| 一个项目规范已经形成后 | 适合 |
| 临时闲聊 | 不适合 |
| 信息还未确认 | 暂不适合 |
| 内容明显过期 | 需要先更新再沉淀 |

---

## 43.3 Harness 结构

| Harness 模块 | 设计 |
|---|---|
| Task Entry | 明确沉淀主题、范围、输出文件名 |
| Context | 当前对话、历史大纲、用户知识库结构、格式偏好 |
| Tool | Markdown 生成、文件写入、图片生成、目录建议 |
| State | 文件版本、阶段记录、变更记录 |
| Workflow | 提炼 → 分类 → 结构化 → 校验 → 生成文件 |
| Output | `.md` 文件、下载链接、摘要 |
| Quality Gate | 范围正确、结构清晰、可沉淀、格式合规 |
| Feedback | 用户修改意见进入模板和规则 |

---

## 43.4 标准工作流

```text
1. 识别沉淀主题
2. 确认覆盖范围
3. 提取已讲内容
4. 排除未讲内容和临时噪声
5. 按知识框架重组
6. 补充表格、结构图、自检题
7. 检查 Markdown 格式
8. 生成 .md 文件
9. 输出文件链接
10. 将用户反馈沉淀为模板改进
```

---

## 43.5 输出契约

| 输出项 | 要求 |
|---|---|
| 文件格式 | `.md` |
| 标题 | 清楚标明主题、阶段、章节 |
| 结构 | 总览、章节、核心概念、误区、自检、总结 |
| 语言 | 中文 |
| 信息密度 | 多表格、少长段落 |
| 范围 | 只覆盖指定内容，不扩展未讲章节 |
| 可迁移性 | 要能作为未来 Agent / Skill / 方法论参考 |
| 文件交付 | 提供下载链接 |

---

## 43.6 质量门禁

| Gate | 检查 |
|---|---|
| Scope Gate | 是否只覆盖指定阶段 |
| Structure Gate | 是否有稳定层级 |
| Density Gate | 是否避免大段散文 |
| Knowledge Gate | 是否从对话变成知识，而不是流水账 |
| Markdown Gate | 标题、表格、代码块是否合规 |
| Artifact Gate | 文件是否真实生成 |
| Reuse Gate | 是否适合未来复用 |

---

## 43.7 常见失败

| 失败 | 原因 | 修复 |
|---|---|---|
| 只是普通总结 | 没有知识结构 | 加知识框架和目录 |
| 内容过长 | 没有信息压缩 | 表格化、层级化 |
| 混入未讲内容 | 范围门禁缺失 | Scope Gate |
| 文件不能下载 | Artifact Gate 缺失 | 检查文件生成 |
| 不适合 Obsidian | 缺少标题、标签、链接结构 | 加知识库模板 |

---

## 43.8 最小 Prompt 模板

```text
请作为知识管理 Agent，将以下内容沉淀为 llm-wiki Markdown 文件。

要求：
1. 只覆盖指定范围；
2. 不扩展未讲内容；
3. 使用中文；
4. 多表格、少长段落；
5. 包含：背景、核心概念、结构图、章节内容、误区、自检、总结；
6. 生成 .md 文件并提供下载链接；
7. 如果内容不足，明确说明缺失部分。
```

---

## 43.9 本章掌握标准

| 能力 | 判断标准 |
|---|---|
| 区分总结和沉淀 | 能说清知识沉淀不是压缩复述 |
| 设计沉淀工作流 | 能从对话到 Markdown 文件 |
| 设计输出契约 | 能定义文件结构和格式 |
| 设计质量门禁 | 能防止范围错、格式乱、不可复用 |
| 迁移到 llm-wiki | 能形成长期知识资产 |

# 第 44 章：亚马逊广告分析 Agent Harness

## 44.1 本章核心

> 亚马逊广告分析 Agent 的目标，是把广告数据诊断从“凭经验判断”变成“按指标、场景、假设、动作、观察周期来分析”。

它不是简单回答：

```text
ACOS 高了，降 bid。
```

而是要判断：

```text
ACOS 高是 CTR 低、CVR 低、CPC 高、流量错配、位置变化、词根变化、竞品变化，还是 Listing 承接问题？
```

---

## 44.2 适用场景

| 场景 | 是否适合 |
|---|---:|
| CTR 下降原因诊断 | 适合 |
| CVR 下降原因诊断 | 适合 |
| ACOS 升高分析 | 适合 |
| CPC 上升但曝光不涨 | 适合 |
| SP / SB / SBV / SD 组合策略 | 适合 |
| 关键词自然排名波动 | 适合 |
| 类目节点变化后流量异常 | 适合 |
| 没有数据，只问“怎么投” | 可给框架，但不能下结论 |

---

## 44.3 Harness 结构

| Harness 模块 | 设计 |
|---|---|
| Task Entry | 明确分析目标、时间范围、指标、广告类型 |
| Context | Search Term Report、广告报表、自然排名、ABA、Listing、竞品 |
| Tool | 表格清洗、指标计算、趋势对比、图表 |
| State | 历史调整记录、观察周期、前后对比 |
| Workflow | 数据确认 → 指标拆解 → 假设诊断 → 动作建议 → 风险说明 |
| Output | 诊断报告、行动清单、观察指标 |
| Quality Gate | 数据范围清楚、假设可验证、动作可执行 |
| Feedback | 实际结果反哺诊断规则 |

---

## 44.4 核心输入

| 输入 | 作用 |
|---|---|
| 时间范围 | 判断前后变化 |
| 广告类型 | SP / SB / SBV / SD / DSP |
| 关键词 | 判断 root term / long-tail |
| 曝光 | 判断流量是否变化 |
| 点击 | 判断 CTR |
| 花费 | 判断成本 |
| CPC | 判断竞争和出价 |
| 订单 | 判断转化 |
| CVR | 判断承接质量 |
| ACOS | 判断效率 |
| Placement | 判断 ToS / ROS / PP 变化 |
| 自然排名 | 判断广告与自然流量联动 |
| Listing 变化 | 判断承接页影响 |
| 竞品变化 | 判断外部环境 |

---

## 44.5 标准诊断框架

```text
1. 明确问题：CTR / CVR / CPC / ACOS / ranking 哪个变了
2. 做时间对比：当前周期 vs 基准周期
3. 拆指标链路：曝光 → 点击 → CPC → CVR → 订单 → ACOS
4. 拆流量来源：SP / SB / SBV / SD / Organic / DSP
5. 拆位置：Top of Search / Rest of Search / Product Pages
6. 拆词：root keyword / long-tail / irrelevant terms
7. 拆人群：高意图 / 中意图 / 泛流量
8. 提出假设
9. 给出动作
10. 定义观察周期和判断标准
```

---

## 44.6 诊断输出模板

| 模块 | 内容 |
|---|---|
| 现象 | 哪些指标变了，变了多少 |
| 数据范围 | 对比周期、广告类型、关键词范围 |
| 核心判断 | 最可能的 1–3 个原因 |
| 证据 | 指标链路和数据依据 |
| 可验证假设 | 如何证明原因是否成立 |
| 建议动作 | bid、placement、keyword、否词、预算、Listing |
| 风险 | 动作可能造成的副作用 |
| 观察周期 | 观察 3 天、7 天、14 天等 |
| 下一步 | 如果指标继续恶化怎么办 |

---

## 44.7 常见诊断模式

| 现象 | 可能原因 | 动作 |
|---|---|---|
| CTR 下降 | 展位变差、创意不匹配、词不准 | 看 placement、搜索词、主图 |
| CVR 下降 | 流量变泛、Listing 承接弱、竞品促销 | 查搜索词、竞品价、优惠、评论 |
| CPC 上升 | 竞争加剧、bid 过高、ToS 抢位 | 拆 placement CPC |
| ACOS 升高 | CPC 升 + CVR 降 | 同时控 CPC 和优化流量质量 |
| 曝光不涨 | relevance 不足、预算限制、bid 不够到位 | 查 impression share 和相关性 |
| 点击涨订单不涨 | 流量低意图或 Listing 承接差 | 否词、拆词、优化详情页 |
| root term 排名跌 | 类目、转化、销售速度、竞品变化 | 拉 root term 转化和销量信号 |

---

## 44.8 质量门禁

| Gate | 检查 |
|---|---|
| Data Gate | 是否说明数据周期和来源 |
| Metric Gate | 是否拆 CTR / CVR / CPC / ACOS |
| Traffic Gate | 是否拆广告类型和搜索词 |
| Placement Gate | 是否拆 ToS / ROS / PP |
| Hypothesis Gate | 是否给出可验证假设 |
| Action Gate | 是否给出可执行动作 |
| Risk Gate | 是否说明副作用 |
| Review Gate | 是否定义观察周期 |

---

## 44.9 常见失败

| 失败 | 原因 | 修复 |
|---|---|---|
| 只说降 bid | 没拆原因 | 先拆 CPC / CVR / CTR |
| 没有时间对比 | 无法判断趋势 | 固定对比周期 |
| 混合所有广告类型 | 归因混乱 | 分 SP / SB / SBV / SD |
| 只看 ACOS | 指标太后置 | 从曝光链路往下拆 |
| 给动作无风险 | 可能伤排名或流量 | 加风险和观察周期 |
| 忽略 Listing | 广告不能单独解释 CVR | 加承接页检查 |

---

## 44.10 本章掌握标准

| 能力 | 判断标准 |
|---|---|
| 设计广告诊断入口 | 能明确目标、周期、广告类型、指标 |
| 拆指标链路 | 能从曝光到 ACOS 分析 |
| 提出可验证假设 | 不给无法验证的判断 |
| 输出动作清单 | 动作具体到 bid、词、placement、预算 |
| 建立复盘闭环 | 能用结果反哺规则 |

# 第 45 章：A+ 文案生成 Agent Harness

## 45.1 本章核心

> A+ 文案生成 Agent 的目标，是稳定产出短、准、强、差异化、符合父母购买理由的转化型文案。

它不是普通写作 Agent。

普通写作：

```text
写得顺、写得好看。
```

A+ 转化文案：

```text
让用户快速理解差异化卖点，并产生购买理由。
```

---

## 45.2 适用场景

| 场景 | 是否适合 |
|---|---:|
| 亚马逊 A+ 模块标题 | 适合 |
| A+ 图片正文短句 | 适合 |
| FABE 文案拆解 | 适合 |
| 儿童卡拉 OK 机卖点表达 | 适合 |
| Banner 文案 | 适合 |
| Listing 五点长描述 | 可适配 |
| 品牌故事长文 | 需要另设风格 |
| 纯翻译 | 不适合，应走翻译 Skill |

---

## 45.3 Harness 结构

| Harness 模块 | 设计 |
|---|---|
| Task Entry | 明确产品、用户、卖点、场景、输出数量 |
| Context | 产品功能、竞品缺口、目标人群、平台限制 |
| Tool | 文案模板、字符限制检查、禁用词检查 |
| State | 已确认口吻、历史优秀文案 |
| Workflow | 卖点识别 → 用户收益 → FABE → 短句压缩 → 质量检查 |
| Output | 英文标题、英文正文、中文标题、中文正文、FABE |
| Quality Gate | 差异化、短、具体、转化、合规 |
| Feedback | 用户偏好和高转化表达进入文案库 |

---

## 45.4 输入契约

| 输入 | 说明 |
|---|---|
| 产品 | 儿童卡拉 OK 机 / 成人卡拉 OK 机 |
| 使用对象 | 3–12 岁儿童、父母购买者等 |
| 核心卖点 | AR 特效、故事 MV、星球进度、歌词屏等 |
| 购买理由 | 陪伴、成长、复玩、家庭娱乐 |
| 输出位置 | A+ 大图、模块标题、Banner、五点 |
| 语言 | 英文 / 中文 / 双语 |
| 长度 | 标题短、正文短 |
| 禁止项 | 不夸大、不泛化、不像所有产品都能用 |
| 风格 | 强转化、父母视角、简洁 |

---

## 45.5 标准工作流

```text
1. 识别卖点本质
2. 判断目标用户和购买者
3. 拆 FABE：Feature / Advantage / Benefit / Evidence
4. 把功能语言转成用户收益
5. 压缩成短标题和短正文
6. 检查差异化是否足够
7. 检查是否过泛
8. 输出 3 个版本
9. 给出推荐版本
```

---

## 45.6 输出契约

推荐格式：

| 字段 | 说明 |
|---|---|
| 英文大标题 | 短、强、能放图上 |
| 英文正文 | 1–2 句，父母购买理由 |
| 中文大标题 | 方便理解 |
| 中文正文 | 解释策略 |
| FABE | 简洁拆解 |
| 适用画面 | 适合搭配什么图 |
| 推荐等级 | 推荐使用哪个版本 |

---

## 45.7 A+ 文案 Rubric

| 维度 | 检查问题 |
|---|---|
| 差异化 | 是否体现竞品没有的卖点 |
| 具体性 | 是否不是所有产品都能用 |
| 用户收益 | 是否站在父母和孩子角度 |
| 转化力度 | 是否让人产生购买理由 |
| 简洁性 | 是否适合图片阅读 |
| 可信度 | 是否不夸大承诺 |
| 英文自然度 | 是否像美国市场表达 |
| 画面适配 | 是否能配合视觉表达 |

---

## 45.8 常见失败

| 失败 | 表现 | 修复 |
|---|---|---|
| 太泛 | “Fun for Kids” | 加具体机制和场景 |
| 太长 | 图上读不完 | 压缩到标题 + 一句正文 |
| 太功能化 | 只说 built-in videos | 转成 real-life skills / replay motivation |
| 差异化弱 | 竞品也能说 | 强调 story videos、AR、progress world |
| 购买理由弱 | 孩子喜欢但父母不买单 | 加成长、陪伴、复玩 |
| 英文不自然 | 中式英语 | 用短促原生表达 |

---

## 45.9 示例：星球进度卖点

| 层 | 内容 |
|---|---|
| Feature | 孩子持续唱歌，星球 / 森林世界进度更新 |
| Advantage | 不只是唱完一首歌，而是看到世界逐步变美 |
| Benefit | 提供目标感、成就感、复玩动力 |
| Evidence | 20 首歌推动世界从荒芜到美丽 |

示例标题：

```text
Sing to Restore the Melody Forest
```

示例正文：

```text
Every song helps bring the world back to life, giving kids a clear goal and a reason to keep singing.
```

---

## 45.10 本章掌握标准

| 能力 | 判断标准 |
|---|---|
| 拆卖点本质 | 能区分功能、机制、收益 |
| 用 FABE 组织文案 | 能把卖点转成购买理由 |
| 控制短文案 | 能适配 A+ 图片阅读 |
| 防止泛化 | 能判断文案是否任何竞品都能用 |
| 建立文案质量门禁 | 能从差异化、转化、可信度评估 |

# 第 46 章：图片提示词 Agent Harness

## 46.1 本章核心

> 图片提示词 Agent 的目标，是把一个模糊视觉想法，转成可生成、可控、可迭代、适合商业用途的画面 brief。

它不是堆形容词。

低质量提示词：

```text
做一张高级感儿童卡拉 OK 广告图。
```

高质量提示词：

```text
主体是谁、在什么场景、做什么动作、画面构图、产品在哪里、光线如何、风格如何、比例如何、哪里预留文案、禁止什么。
```

---

## 46.2 适用场景

| 场景 | 是否适合 |
|---|---:|
| Amazon Banner 画面设计 | 适合 |
| A+ 模块图构思 | 适合 |
| 产品卖点视觉表达 | 适合 |
| 儿童卡拉 OK 使用场景 | 适合 |
| AR 特效画面 | 适合 |
| 插画风格转换 | 适合 |
| 精确修图 | 需要图像编辑工具配合 |
| 法律 / 医疗图像 | 需要额外审核 |

---

## 46.3 Harness 结构

| Harness 模块 | 设计 |
|---|---|

# 第 47 章：客服回复 Agent Harness

## 47.1 本章核心

> 客服回复 Agent 的目标，是生成事实准确、语气合适、风险可控、可发送前审查的平台 case / 售后回复。

## 47.3 Harness 结构

| Output | 英文回复、中文解释、风险提醒 |
| Quality Gate | 事实准确、语气专业、无越权承诺 |

## 47.4 输入契约

| 输入 | 示例 |
|---|---|
| 原始对方回复 | Amazon support 的原文 |

| 回复语言 | 英文 |
| 语气 | 礼貌、明确、专业 |
| 禁止项 | 不承认错误事实、不情绪化、不威胁 |

## 47.7 质量门禁

## 47.8 常见失败

# 第 48 章：需求澄清 Agent Harness

## 48.1 本章核心

> 需求澄清 Agent 的目标，是把一个模糊想法，转成清晰目标、范围边界、用户场景、验收标准和可执行任务。

它不是普通聊天，也不是直接执行。

用户常说：

```text
我想做一个高质量 skill。
我想做一个 AI 工作流。
我想把这个系统工程化。
这个想法你帮我理清楚。
```

这些都不是可执行任务。  
需求澄清 Agent 要先把它转成 PRD / task brief。

---

## 48.2 Harness 结构

| Harness 模块 | 设计 |
|---|---|
| Task Entry | 识别这是模糊需求，不直接执行 |
| Context | 用户目标、历史偏好、项目约束、已有资料 |
| Tool | 访谈框架、PRD 模板、需求分类表 |
| State | 已确认问题、未确认问题、假设 |
| Workflow | 意图识别 → 问题拆解 → 边界确认 → 任务建模 → PRD |
| Output | 需求澄清文档、问题清单、执行计划 |
| Quality Gate | 是否清楚、完整、可执行、可验收 |
| Feedback | 用户确认后更新需求文档 |

---

## 48.3 需求澄清的核心框架

| 层 | 问题 |
|---|---|
| 背景 | 为什么要做 |
| 目标 | 想达成什么 |
| 用户 | 谁使用 |
| 场景 | 在什么情况下使用 |
| 输入 | 需要什么材料 |
| 输出 | 要交付什么 |
| 范围 | 做什么、不做什么 |
| 流程 | 怎么执行 |
| 标准 | 什么叫完成 |
| 风险 | 哪些地方可能失败 |
| 优先级 | 先做什么 |
| 迭代 | 后续如何升级 |

---

## 48.4 澄清不是一直问问题

低质量澄清：

```text
你想要什么？
你有什么要求？
你希望怎么做？
```

高质量澄清：

```text
我先基于现有信息建立假设；
列出已明确部分；
列出关键缺口；
只问会影响执行的 3–5 个问题；
低风险部分先推进。
```

---

## 48.5 输出契约

| 输出 | 内容 |
|---|---|
| 需求摘要 | 一句话说明用户真正想做什么 |
| 已明确内容 | 当前已经确定的事实 |
| 未明确内容 | 缺失的信息 |
| 关键问题 | 必须确认的问题 |
| 默认假设 | 如果不确认，按什么假设执行 |
| 任务拆解 | 拆成可执行任务 |
| 验收标准 | 什么叫完成 |
| 风险清单 | 可能失败的地方 |
| 下一步建议 | 先做什么 |

---

## 48.6 质量门禁

| Gate | 检查 |
|---|---|
| Goal Gate | 目标是否明确 |
| Scope Gate | 边界是否清楚 |
| User Gate | 用户和使用场景是否清楚 |
| Output Gate | 交付物是否明确 |
| DoD Gate | 完成标准是否可检查 |
| Risk Gate | 风险是否识别 |
| Action Gate | 是否能进入执行阶段 |

---

## 48.7 你的 complex-task-clarifier 场景

这个 Skill 的核心价值就是需求澄清 Harness。

它应该做：

```text
模糊想法
→ 需求识别
→ 上下文审计
→ 任务建模
→ 边界拆解
→ 风险识别
→ 验收标准
→ 需求文档
```

不应该做：

```text
用户一说想法
→ 直接开始实现
```

---

## 48.8 常见失败

| 失败 | 原因 | 修复 |
|---|---|---|
| 问太多 | 没有区分关键问题 | 只问影响结果的问题 |
| 太快执行 | 入口 Harness 缺失 | 先生成 task brief |
| 需求文档空泛 | 没有验收标准 | 加 DoD |
| 范围膨胀 | 不做什么没写清 | 加 out of scope |
| 没有下一步 | 不能执行 | 加任务拆解 |

---

## 48.9 本章掌握标准

| 能力 | 判断标准 |
|---|---|
| 识别模糊需求 | 不直接跳执行 |
| 设计澄清框架 | 能从背景到验收完整拆解 |
| 控制问题数量 | 只问关键问题 |
| 产出 PRD / task brief | 让需求可执行 |
| 防止范围膨胀 | 明确做与不做 |

# 第 49 章：Repo Reverse Engineering Harness

## 49.1 本章核心

> Repo Reverse Engineering Agent 的目标，是让 Agent 快速、系统、可验证地理解一个代码仓库，而不是随机浏览文件。

它解决的问题是：

```text
新项目不知道从哪里看；
目录结构复杂；
文档不完整；
代码和文档不一致；
Agent 读了很多文件但没有形成项目地图。
```

---

## 49.2 适用场景

| 场景 | 是否适合 |
|---|---:|
| 第一次接手代码仓库 | 适合 |
| Codex 修改前需要理解项目 | 适合 |
| 为项目生成 AGENTS.md | 适合 |
| 为 repo 写 README / architecture docs | 适合 |
| 分析技术债 | 适合 |
| 大型重构前 | 适合 |
| 只改一个小 typo | 不必完整逆向 |
| 无仓库访问权限 | 无法执行 |

---

## 49.3 Harness 结构

| Harness 模块 | 设计 |
|---|---|
| Task Entry | 明确逆向目标：理解、改造、写文档、找风险 |
| Context | README、目录结构、package、配置、测试、入口文件 |
| Tool | 文件搜索、代码阅读、依赖分析、测试命令 |
| State | 已读文件、模块地图、假设、疑问 |
| Workflow | 扫目录 → 找入口 → 找依赖 → 找核心模块 → 找测试 → 输出地图 |
| Output | 项目地图、模块说明、运行方式、风险、建议 |
| Quality Gate | 是否覆盖入口、核心模块、测试、构建、约束 |
| Feedback | 逆向结果更新 AGENTS.md / docs |

---

## 49.4 标准逆向工作流

```text
1. 读取 README / docs
2. 查看目录树
3. 识别技术栈和包管理器
4. 读取 package / config / build files
5. 找应用入口
6. 找核心模块
7. 找数据流和调用链
8. 找测试目录和测试命令
9. 找脚本和 CI
10. 找风险点和技术债
11. 输出项目地图
12. 给出 AGENTS.md 建议
```

---

## 49.5 逆向输出契约

| 输出 | 内容 |
|---|---|
| 项目一句话说明 | 这个仓库做什么 |
| 技术栈 | 语言、框架、包管理器 |
| 目录地图 | 每个关键目录的作用 |
| 入口文件 | 程序从哪里启动 |
| 核心模块 | 主要业务模块 |
| 数据流 | 输入、处理、输出 |
| 测试体系 | 测试位置、运行命令 |
| 构建和部署 | build、CI、release |
| Agent 规则建议 | AGENTS.md 应写什么 |
| 风险点 | 文档缺失、测试不足、耦合、过时依赖 |
| 下一步 | 适合先改哪里 |

---

## 49.6 质量门禁

| Gate | 检查 |
|---|---|
| Entry Gate | 是否找到入口 |
| Structure Gate | 是否解释目录结构 |
| Dependency Gate | 是否识别依赖和技术栈 |
| Test Gate | 是否找到测试命令 |
| CI Gate | 是否识别自动化检查 |
| Architecture Gate | 是否解释核心模块关系 |
| Risk Gate | 是否列出不确定点和风险 |
| Evidence Gate | 是否说明依据来自哪些文件 |
| Action Gate | 是否能支持后续 Coding Agent 工作 |

---

## 49.7 常见失败

| 失败 | 原因 | 修复 |
|---|---|---|
| 只看 README | README 可能过期 | 结合代码和配置 |
| 随机读文件 | 没有逆向路径 | 先目录、入口、依赖 |
| 没找测试 | 后续修改风险大 | Test Gate |
| 没输出地图 | 信息不可复用 | 生成目录和模块图 |
| 没说明不确定性 | 误导后续执行 | 标记 unknown |
| 不写 AGENTS.md 建议 | 没沉淀 | 转成 repo harness |

## 4. 阶段七最重要的判断

```text
真实业务 Agent 不能只靠“聪明回答”。

每个 Agent 都必须回答：

它解决什么业务问题？
需要什么输入？
能调用什么工具？
按什么流程执行？
输出什么交付物？
怎么判断质量？
失败后怎么修？
经验如何沉淀？
哪些动作必须人工确认？
```

## 5. 阶段七掌握标准

| 能力 | 判断标准 |
|---|---|
| 场景建模 | 能为不同业务场景设计 Agent Harness |
| 任务入口设计 | 能把业务请求转成可执行 task brief |
| 上下文设计 | 能识别业务所需资料 |
| 工作流设计 | 能拆出标准执行步骤 |
| 输出契约设计 | 能让交付物稳定 |
| 质量门禁设计 | 能防止假完成和低质量输出 |
| 风险控制 | 能识别需要人工审批的动作 |
| 沉淀能力 | 能把业务流程转成 Skill / 模板 / wiki |

# 需求挖掘与需求工作方法定位知识卡

> 适用位置：`llm-wiki / product-management / requirements-discovery /`
>
> 核心主题：需求挖掘、需求分析、需求定义、需求优先级、需求验证  
> 目标：建立一套可复用的需求工作方法定位地图，说明不同方法在需求挖掘、分析、定义、结构化、优先级、验证和交付表达中的位置。

---

## 1. 核心结论

需求相关方法分布在需求工作流的不同环节，需要按主要定位、使用环节和作用边界来管理。

可以按需求工作流中的主要环节来定位：

| 类型 | 核心作用 |
|---|---|
| 需求挖掘 | 找到真实用户、真实场景、真实问题、真实动机 |
| 需求分析 | 澄清问题、拆解原因、找到根因、明确边界 |
| 需求定义 | 把需求写成可执行、可验收的表达 |
| 需求优先级 | 判断哪些需求先做、哪些后做、哪些不做 |
| 需求验证 | 判断需求真实性和方案有效性 |

---

## 2. 费曼式理解

> **需求挖掘不是问用户“想要什么功能”，而是弄清楚：谁，在什么场景下，遇到什么问题，为什么现有方案不行，真正想完成什么，以及这个问题的解决价值。**

生活化类比：

| 表层现象 | 深层含义 |
|---|---|
| 用户说“想要一匹更快的马” | 表层需求 |
| 用户真正想要“更快到达目的地” | 真实需求 |
| 分析为什么现在慢 | 问题分析 |
| 判断汽车、自行车、地铁哪个更合适 | 方案探索 |
| 先做最小可用方案测试 | 需求验证 |

## 3. 常见方法定位表

| 方法 | 主要定位 | 使用环节 | 作用边界 |
|---|---|---|---|
| 5W2H | 问题澄清 / 场景拆解工具 | 需求分析 | 帮助补全需求的对象、场景、原因、方式和成本等上下文。 |
| 5 Why | 根因分析工具 | 需求分析 | 用于追问问题背后的原因，识别表层问题背后的根因。 |
| JTBD | 需求洞察 / 用户动机方法论 | 需求挖掘 | 关注用户“雇佣产品完成什么任务”，用于理解功能背后的真实动机。 |
| MECE | 结构化分析原则 | 需求结构化 | 用于把需求拆分得不重不漏，帮助形成清晰分类。 |
| SMART | 目标 / 需求表达标准 | 需求定义 | 用于把模糊目标转成具体、可衡量、可达成、相关、有时限的表达。 |
| MoSCoW | 需求优先级排序方法 | 需求优先级 | 用于判断 Must / Should / Could / Won’t。 |
| 用户旅程 | 场景化需求发现方法 | 需求挖掘 / 场景洞察 | 通过用户完成目标的全过程发现痛点和机会点。 |
| 反向定义法 | 边界澄清 / 反向约束方法 | 需求分析 / 需求定义 | 通过“不是什么、不做什么、不是谁的需求”澄清边界。 |

## 4. 常见误区

### 4.1 把需求工作链路中的不同环节混为一类

| 层级 | 核心问题 | 典型方法 |
|---|---|---|
| 需求挖掘 | 用户真实需要什么？ | 用户访谈、观察法、JTBD、用户旅程 |
| 需求分析 | 这个问题为什么存在？ | 5W2H、5 Why、鱼骨图、MECE |
| 需求定义 | 需求如何写成可执行表达？ | User Story、Use Case、SMART、验收标准 |
| 需求优先级 | 哪些需求值得先做？ | MoSCoW、Kano、RICE、价值-成本矩阵 |
| 需求验证 | 这个需求成立的证据是什么？ | MVP、原型测试、A/B Test、Fake Door Test |

---

### 4.2 把结构化、表达、排序工具混作挖掘方法

| 方法 | 更准确的角色 | 类比 |
|---|---|---|
| MECE | 整理工具 | 把一堆需求分类放进文件夹 |
| SMART | 表达标准 | 把模糊愿望写成可衡量目标 |
| MoSCoW | 排序工具 | 决定哪些必须做、哪些以后做 |

---

### 4.3 忽略用户研究方法

真正的需求挖掘不能只依赖内部推理和框架拆解，还需要进入真实用户场景。

常见用户研究方法：

| 方法 | 作用 |
|---|---|
| 用户访谈 | 理解用户动机、痛点、决策逻辑 |
| 观察法 | 看用户真实行为，而不是只听用户表达 |
| 情境访谈 | 在用户真实使用环境中理解任务流程 |
| 问卷调研 | 量化验证问题、偏好或人群特征的普遍性 |
| 焦点小组 | 观察多个用户对问题、概念、方案的讨论 |
| 可用性测试 | 验证用户理解和使用方案或原型的顺畅程度 |
| 竞品分析 | 从替代方案中发现未满足需求 |
| 数据分析 | 从行为数据中识别异常、流失、机会点 |
| 服务蓝图 | 连接前台体验和后台流程 |
| 同理心地图 | 梳理用户看到、听到、想到、痛苦和期待 |

## 5. 需求方法论地图

### 5.1 用户研究类：直接从用户那里挖需求

| 方法 | 核心问题 | 适合场景 |
|---|---|---|
| 用户访谈 | 用户为什么需要这个？ | 新需求探索、购买动机分析、痛点理解 |
| 情境访谈 / Contextual Inquiry | 用户在真实环境中怎么做？ | B端系统、工具产品、复杂工作流 |
| 用户观察 | 用户实际行为是什么？ | 用户说不清楚、行为和表达不一致 |
| 问卷调研 | 大量用户中存在哪些共性问题？ | 需求量化、偏好验证、样本规模较大时 |
| 焦点小组 | 用户如何讨论某个问题？ | 概念测试、早期方向探索 |
| 可用性测试 | 用户完成任务的顺畅程度如何？ | 原型、页面、功能流程优化 |

---

### 5.2 场景洞察类：从任务、旅程、触点中挖需求

| 方法 | 核心问题 | 价值 |
|---|---|---|
| JTBD | 用户雇佣产品完成什么任务？ | 找到功能背后的真实动机 |
| 用户旅程地图 | 用户完成目标经历了哪些步骤？ | 找到痛点、断点、机会点 |
| 服务蓝图 | 前台体验和后台服务如何配合？ | 适合服务型、复杂流程产品 |
| 同理心地图 | 用户看到、听到、想什么、痛什么？ | 帮助团队建立用户共识 |
| Persona 用户画像 | 目标用户是谁？ | 帮助聚焦核心人群 |
| Use Case 用例 | 用户在什么场景下如何使用？ | 适合功能设计、系统需求，也常用于需求表达和系统交互建模 |

---

### 5.3 问题定义类：从表层需求挖到真实问题

| 方法 | 核心问题 | 价值 |
|---|---|---|
| 5 Why | 为什么会有这个问题？ | 找根因 |
| 5W2H | 这个需求到底是什么？ | 澄清问题边界 |
| 鱼骨图 | 问题由哪些原因造成？ | 系统性归因 |
| 反向定义法 | 这不是什么？不做什么？ | 澄清边界，避免误解 |
| Problem Statement | 真实问题一句话是什么？ | 对齐团队认知 |
| Double Diamond | 先发现问题，再定义问题，再探索方案 | 防止过早进入解决方案 |

---

### 5.4 需求结构化类：把需求整理成系统

| 方法 | 作用 | 注意点 |
|---|---|---|
| MECE | 不重不漏地拆需求 | 是结构化原则，主要服务于需求分类和需求结构化 |
| 亲和图 / KJ法 | 把零散用户反馈归类 | 适合大量访谈材料 |
| 用户故事地图 | 按用户流程组织需求 | 适合产品路线图 |
| Impact Mapping | 从业务目标倒推用户行为和功能 | 连接业务目标与产品需求 |
| Opportunity Solution Tree | 连接目标、机会、方案、实验 | 适合产品发现和持续探索 |

---

### 5.5 需求优先级类：判断哪些需求值得先做

| 方法 | 核心逻辑 | 适合场景 |
|---|---|---|
| MoSCoW | Must / Should / Could / Won’t | 项目范围管理 |
| Kano | 基本型、期望型、兴奋型需求 | 判断功能对满意度的影响 |
| RICE | Reach × Impact × Confidence ÷ Effort | 产品路线图排序 |
| ICE | Impact × Confidence × Ease | 快速评估机会 |
| 价值-成本矩阵 | 高价值低成本优先 | 粗排需求 |
| WSJF | 延迟成本 ÷ 工作量 | 敏捷规模化、复杂项目 |

---

### 5.6 需求验证类：判断需求成立依据

| 方法 | 验证什么 | 适合场景 |
|---|---|---|
| MVP | 最小方案对需求成立的证明程度如何？ | 新产品、新功能 |
| 原型测试 | 用户理解程度和使用意愿如何？ | 低成本验证 |
| A/B Test | 哪个方案数据更好？ | 流量充足的线上产品 |
| Fake Door Test | 用户的点击、报名、兴趣表达有多强？ | 验证需求强度 |
| Smoke Test | 市场真实反应如何？ | 新业务、新卖点 |
| 访谈回访 | 方案对原问题的解决程度如何？ | 定性验证 |

---

### 5.7 需求表达类：把需求转成可交付材料

| 方法 | 核心作用 | 输出形式 |
|---|---|---|
| User Story | 从用户视角表达需求 | 作为某类用户，我想要……以便…… |
| Use Case | 描述用户和系统如何交互 | 用例名称、参与者、前置条件、主流程、异常流程 |
| SMART | 把目标写清楚 | 具体、可衡量、可达成、相关、有时限 |
| 验收标准 | 判断需求完成程度 | Given / When / Then |
| Definition of Done | 判断交付完成程度 | 开发、测试、文档、上线标准 |

## 6. 需求工作全链路

| 阶段 | 核心问题 | 推荐方法 | 产出物 |
|---|---|---|---|
| 1. 识别对象 | 谁有需求？ | Persona、用户分层、利益相关者分析 | 用户类型、目标人群 |
| 2. 进入场景 | 需求发生在哪里？ | 用户旅程、情境访谈、观察法 | 场景地图、行为路径 |
| 3. 发现问题 | 用户卡在哪里？ | 5W2H、5 Why、鱼骨图 | 问题列表、根因假设 |
| 4. 挖掘动机 | 用户真正想完成什么？ | JTBD、用户访谈、同理心地图 | 用户动机、任务定义 |
| 5. 结构化需求 | 需求之间是什么关系？ | MECE、亲和图、用户故事地图 | 需求分类、需求地图 |
| 6. 判断价值 | 哪些需求值得做？ | Kano、MoSCoW、RICE、价值成本矩阵 | 优先级列表 |
| 7. 验证真假 | 这个需求的真实性证据是什么？ | MVP、原型测试、A/B Test、Fake Door Test | 验证结果、学习结论 |
| 8. 转为交付 | 如何写成可执行需求？ | User Story、Use Case、SMART、验收标准 | PRD、用户故事、验收标准 |

## 7. 方法选择速查表

| 当前问题 | 优先使用的方法 |
|---|---|
| 用户说了一个功能，但不知道真实原因 | 5 Why、JTBD、用户访谈 |
| 需求很模糊，不知道怎么问清楚 | 5W2H |
| 需求很多，混乱无结构 | MECE、亲和图、用户故事地图 |
| 不知道目标用户是谁 | Persona、用户分层、利益相关者分析 |
| 不知道用户在哪个环节痛苦 | 用户旅程地图、服务蓝图 |
| 不知道哪个需求先做 | MoSCoW、Kano、RICE、价值-成本矩阵 |
| 不知道需求真实性 | MVP、原型测试、Fake Door Test |
| 目标很虚，不知道怎么衡量 | SMART |
| 需求边界不清楚 | 反向定义法、Problem Statement |
| 团队过早讨论方案 | Double Diamond、Opportunity Solution Tree |

## 10. 最小可复用模板

面对一个新需求时，可以按以下模板追问：

| 模块 | 问题 |
|---|---|
| 用户 | 这个需求来自谁？核心用户是谁？ |
| 场景 | 用户在什么场景下遇到这个问题？ |
| 问题 | 现在哪里不顺？卡点是什么？ |
| 原因 | 为什么会出现这个问题？ |
| 动机 | 用户真正想完成什么任务？ |
| 替代方案 | 用户现在用什么方式解决？为什么不满意？ |
| 价值 | 这个问题有多重要？影响多大？ |
| 优先级 | 现在必须做吗？不做会怎样？ |
| 验证 | 用什么最小实验证明它是真的？ |
| 交付 | 如果要做，怎么写成可执行需求？ |

## 11. 总结

| 重点 | 结论 |
|---|---|
| 需求挖掘的核心 | 找到真实问题、真实场景、真实动机 |
| 用户说的功能 | 通常只是表层表达，不等于真实需求 |
| JTBD / 用户旅程 | 主要用于需求挖掘和场景洞察 |
| 5W2H / 5 Why | 更常用于问题澄清和根因分析 |
| MECE / SMART / MoSCoW | 更常用于结构化、表达和优先级判断 |
| 好的需求流程 | 挖掘 → 分析 → 定义 → 优先级判断 → 验证 → 交付表达 |

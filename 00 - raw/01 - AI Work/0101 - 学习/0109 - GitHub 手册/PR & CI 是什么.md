# PR 和 CI 是什么

## 一句话解释

PR 是“改稿申请”。

CI 是“自动检查”。

在 GitHub 仓库里，不建议直接修改 `main` 主线。通常先在一个分支里修改，修改完成后创建 PR，请求把这些修改合并进 `main`。创建 PR 后，GitHub 会自动运行 CI，检查这次改动有没有问题。

## 生活化例子

可以把 GitHub 仓库想成一本正式出版的书。

- `main` 是正式版。
- 分支是草稿版。
- PR 是把草稿改动提交给正式版的“改稿申请”。
- CI 是出版社里的“自动校对员”。

完整过程像这样：

```text
你先不直接改正式出版的书
  ↓
你复制一份草稿出来修改
  ↓
改完后提交一份“改稿申请”
  ↓
自动校对员检查错别字、格式、遗漏、危险内容
  ↓
检查通过后
  ↓
你再决定是否把草稿合并进正式版
```

对应到 GitHub：

```text
main = 正式版
分支 = 草稿版
PR = 改稿申请
CI = 自动校对员
```

## PR 是什么

PR 的全称是 Pull Request，中文可以理解为“拉取请求”或“合并请求”。

更好理解的说法是：

> PR 是一份“请审核并合并”的改稿申请。

它的作用是：

- 展示这次改了哪些文件。
- 展示每个文件具体改了什么。
- 让人或工具在合并前检查问题。
- 通过检查后，再决定是否合并到 `main`。

## CI 是什么

CI 的全称是 Continuous Integration，中文叫“持续集成”。

更好理解的说法是：

> CI 是 GitHub 自动帮你跑的一套检查流程。

在创建 PR 后，CI 会自动检查：

- 文件结构是否正确。
- 文档是否更新。
- 有没有误提交 `.env`、缓存、大文件、临时文件。
- 被修改的 skill 是否能通过基础验证。
- 这次修改是否会触发发布。

CI 不是人工审核，它是自动检查员。

## PR 和 CI 的关系

PR 是你提交的改动申请。

CI 是 GitHub 对这份申请自动做的检查。

流程是：

```text
修改代码或文档
  ↓
提交到一个分支
  ↓
创建 PR
  ↓
CI 自动检查
  ↓
检查通过
  ↓
决定是否合并到 main
```

## 在 nexgaios-skills 仓库里的含义

在 `nexgaios-skills` 仓库里：

- PR 用来申请把某次 skill、文档、工具或模板改动合并进 `main`。
- CI 会自动检查这次改动是否安全。
- CI 还会自动评论说明这次影响了哪些 skill。
- 如果没有修改 skill 目录，就不会发布 skill。
- 如果修改了 skill，但没有修改该 skill 的 `skill.yaml` 版本号，也不会发布 skill。
- 只有某个 skill 的内容变化，并且该 skill 的 `skill.yaml` 版本号也变化，合并到 `main` 后才会发布该 skill。

## 常见邮件怎么看

如果收到 GitHub 邮件，看到类似：

```text
github-actions[bot] left a comment
```

这通常表示 GitHub Actions 自动评论了 PR。

如果邮件里写：

```text
没有直接修改任何 skill 目录。
```

说明这次没有改具体 skill，不会发布 skill。

如果邮件里写：

```text
共享文件变更
```

说明这次改的是仓库文档、工具、模板、CI 配置等共享内容。

```text
main 是正式版。

```

## 问题集

### 问题 1：CI 通过后会自动合并到 main 吗？合并后的分支还存在吗？

问题：

> 如果分支修改完之后创建 PR，如果 CI 检查没问题，就会自动合并到 `main` 对吗？那合并之后这条分支还存在吗？如果存在，那存在的意义是什么呢？如果不存在，是被自动删除了吗？

答案：

CI 检查通过，不等于自动合并到 `main`。

更准确的流程是：

```text
分支修改
  ↓
创建 PR
  ↓
CI 自动检查
  ↓
CI 通过
  ↓
仍然需要人点击 Merge，或者明确让工具执行合并
  ↓
合并进 main
```

CI 的作用是告诉你：

```text
这次改动通过了自动检查，可以考虑合并。
```

它不是在说：

```text
我已经帮你合并了。
```

除非仓库额外开启了 GitHub Auto-merge，或者配置了某个机器人规则，让 CI 通过后自动合并。默认情况下，CI 通过后仍然需要人工或工具明确执行合并。

合并 PR 后，分支通常仍然存在，除非仓库开启了自动删除分支，或者合并后手动删除。

合并前，分支的意义是承载这次改动，供 PR 检查和审核。

合并后，这条分支的主要价值会变小。它可以短期保留用于回看、继续追加相关修改，或者作为临时备份。但大多数情况下，合并后可以删除，因为正式版本已经进入 `main`。

最短记忆：

```text
CI 通过 ≠ 自动合并

CI 只是检查。
Merge 才是合并。

PR 合并后，分支通常可以删除。
main 才是正式版本。
```

生活化理解：

```text
分支 = 草稿纸
PR = 把草稿交给你审核
CI = 自动校对
Merge = 正式采用这份草稿
删除分支 = 草稿已经采用，草稿纸可以扔掉
```

### 问题 2：合并是整个分支替换 main，还是只合并差异？

问题：

> 合并是怎么个合并法？拿整个分支替换 `main`，还是寻找差异项，对差异项做合并？

答案：

合并是按差异合并，不是拿整个分支替换 `main`。

可以这样理解：

```text
main = 正式版
分支 = 从 main 复制出来的一份草稿
```

你在分支里改了一些文件。创建 PR 后，GitHub 会比较：

```text
分支 相对于 main 改了什么
```

合并时，GitHub 会把这些差异改动应用到 `main` 上。

简单例子：

一开始 `main` 是：

```text
A
B
C
```

你从 `main` 创建分支，然后在分支里改成：

```text
A
B 改成 B2
C
D 新增
```

PR 看到的差异是：

```text
B -> B2
新增 D
```

合并时，不是把整个分支覆盖 `main`，而是把差异应用到 `main`：

```text
main 变成：
A
B2
C
D
```

如果你开分支后，`main` 又被别人改了，GitHub 会尽量保留 `main` 后来的改动，并把你的分支差异也合进去。

如果你和 `main` 都改了同一个位置，GitHub 不知道该用谁的，就会产生冲突。冲突需要人工决定最终保留哪一版，或者把两边内容合成一个最终版本。

最短记忆：

```text
合并不是整包替换。
合并是把分支相对 main 的差异应用到 main。
```

生活化理解：

```text
main 是正式文档。
分支是草稿修改稿。
合并不是把整本文档换掉。
合并是把草稿里标红的修改点，应用到正式文档里。
```

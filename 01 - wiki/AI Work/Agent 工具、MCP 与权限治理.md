# Agent 工具、MCP 与权限治理

## 当前结论

1. Tool 是 Agent 连接真实系统的执行能力；没有 Tool，Agent 主要是在回答，有 Tool，Agent 才能查、算、写、发、改和执行。
2. 产品级 Agent 的工具设计不能只看功能层，还必须同时设计接口层和治理层。
3. 工具调用的基本分工是：模型负责判断，程序负责执行，系统负责记录，高风险动作由人确认。
4. Tool Schema 是工具调用的参数合同和安全边界，必须描述用途、输入、输出、权限、副作用、确认要求和错误策略。
5. MCP 的价值在于把工具、资源和提示词以标准协议暴露给 Agent，但 MCP 不能替代权限、审计、确认和回滚设计。
6. 工具不是越多越好；工具越多，选择错误、权限复杂、调试困难和安全风险都会上升。

## 适用范围

### 适用

- 设计 Agent 可调用工具、MCP Server、MCP Client 或外部系统接入。
- 判断某个工具是否应该开放给 Agent。
- 设计工具权限、确认、审计、错误处理和回滚策略。
- 评审 Tool Schema 是否足以支撑稳定调用和安全治理。

### 不适用

- 不替代具体 MCP SDK、OpenAI function calling、LangGraph tool node 的实现文档。
- 不直接定义某个生产系统的权限矩阵。
- 不适用于完全只读、无外部动作的普通问答场景。

## 工具治理框架

| 层级 | 核心问题 | 必须定义 |
| --- | --- | --- |
| 功能层 | 工具能做什么？ | 工具名称、用途、输入、输出。 |
| 接口层 | Agent 如何稳定调用？ | Schema、参数校验、返回结构、错误策略。 |
| 治理层 | 工具能不能安全使用？ | 权限、副作用、确认、审计、回滚。 |

## 工具风险分级

| 工具类型 | 示例 | 风险 |
| --- | --- | --- |
| 读取型 | 读文件、查报表、查订单 | 低到中 |
| 检索型 | 搜索网页、查知识库、向量检索 | 低到中 |
| 计算型 | 计算 CTR、CVR、ACOS | 低 |
| 生成型 | 生成 Markdown、代码、图片提示词 | 中 |
| 写入型 | 更新数据库、创建任务 | 中高 |
| 发送型 | 发邮件、发 Slack、提交表单 | 高 |
| 删除型 | 删除文件、清空记录 | 高 |
| 交易型 | 调预算、下单、退款 | 极高 |

## Tool Schema 最小字段

| 字段 | 用途 |
| --- | --- |
| `name` | 工具名称，必须动词开头、对象明确。 |
| `description` | 工具用途和使用边界。 |
| `input_schema` | 输入参数、类型、枚举和必填项。 |
| `output_schema` | 返回结构和可解释字段。 |
| `permission` | 权限等级。 |
| `side_effect` | 是否改变外部系统。 |
| `confirmation_required` | 是否需要人工确认。 |
| `error_policy` | 失败后如何重试、降级或停止。 |
| `examples` | 正确和错误调用示例。 |

# Tool Schema 设计

## 当前结论

1. Tool Schema 是 Agent 调用工具时的参数合同和安全边界，不只是 API 参数说明。
2. 好的 Tool Schema 必须同时描述工具用途、输入、输出、权限、副作用、确认要求、错误策略和调用示例。
3. Schema 设计越清楚，模型越容易在正确时机选择正确工具，并生成可执行参数。
4. 真实系统不能原样暴露给 Agent；API、脚本、数据库能力必须被封装成任务能力切片。
5. 写入、发送、删除、交易类工具的 Schema 必须显式声明副作用和人工确认要求。

## 适用范围

### 适用

- 为 Agent、Skill、MCP Server 设计可调用工具。
- 将 API、脚本、数据库查询封装成稳定工具。
- 评审工具是否足以支撑模型稳定调用。
- 设计工具权限、确认、错误处理和审计字段。

### 不适用

- 不替代具体编程语言中的函数签名。
- 不替代外部系统 API 文档。
- 不处理工具内部业务逻辑实现。

## 最小字段

| 字段 | 含义 |
| --- | --- |
| `name` | 工具名称，必须表达动作和对象。 |
| `description` | 工具用途、使用边界、何时使用、何时不用。 |
| `input_schema` | 输入参数、类型、枚举、必填项和约束。 |
| `output_schema` | 返回结构、字段含义和下游可用信息。 |
| `permission` | 权限等级，如只读、计算、写入、发送、删除、交易。 |
| `side_effect` | 是否会改变外部系统或生成外部影响。 |
| `confirmation_required` | 是否必须人工确认后执行。 |
| `error_policy` | 参数缺失、权限不足、超时、失败时如何处理。 |
| `examples` | 正确调用、错误调用和边界案例。 |

## 设计检查

| 检查项 | 合格标准 |
| --- | --- |
| 命名 | 名称能看出动作和对象。 |
| 描述 | 说明使用场景和禁止场景。 |
| 参数 | 必填参数明确，枚举优先。 |
| 返回 | 返回结构稳定，可供模型继续判断。 |
| 权限 | 区分读、算、写、发、删、交易。 |
| 副作用 | 会改变外部系统时必须显式声明。 |
| 确认 | 高风险动作必须要求人工确认。 |
| 错误 | 失败后不允许模型猜测成功。 |

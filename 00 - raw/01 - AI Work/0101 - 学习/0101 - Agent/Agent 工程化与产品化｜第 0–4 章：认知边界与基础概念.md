# Agent 工程化与产品化｜第 0–4 章：认知边界与基础概念

> 目标：建立 Agent 工程化与产品化的底层认知，先搞清楚 Agent 是什么、不是什么、和相邻概念的区别、有哪些产品形态、能力如何分层。  
> 定位：这是正式进入 Agent 工程化之前的“概念地基”。

## 第 0 章｜为什么需要 Agent 工程化与产品化

### 0.1 一句话结论

**Agent 工程化，是把“看起来聪明的 AI Demo”变成“稳定、可控、可复用、可上线的系统”。**

**Agent 产品化，是把“能运行的 Agent 能力”变成“真实用户愿意用、业务愿意投入、团队能持续迭代的产品”。**

---

### 0.2 费曼解释

你可以把 Agent 想成一个“会做事的实习生”。

只会聊天的 AI，像一个会回答问题的人。

Agent 更像一个可以接任务的人：

1. 先理解你要什么
2. 再判断需要哪些资料
3. 再决定要不要调用工具
4. 然后一步步执行
5. 最后把结果交付给你

但是，一个实习生不能直接放进公司核心流程里。

因为他可能会：

- 听错任务
- 拿错资料
- 调错工具
- 忽略边界
- 没做完就说做完了
- 出错后不知道怎么恢复
- 需要主管确认时却自动执行

所以，Agent 工程化要解决的是：

> 如何让这个“AI 实习生”变得可控、可测、可追踪、可复用。

Agent 产品化要解决的是：

> 如何让这个“AI 实习生”真正嵌入用户工作流，解决高价值任务，并持续产生业务价值。

---

### 0.3 为什么 Demo 和真实 Agent 产品差距很大

| 层级 | Demo 关注 | 产品级 Agent 关注 |
|---|---|---|
| 输入 | 用户随便问一句 | 输入结构、上下文完整性、权限边界 |
| 推理 | 模型自己想 | 流程约束、状态管理、工具选择 |
| 执行 | 调一个 API 或生成文本 | 多工具、多步骤、失败恢复、人工确认 |
| 输出 | 看起来像答案 | 结构化、可验证、可交付、可追踪 |
| 质量 | 凭感觉判断好不好 | 测试集、评估、日志、回归 |
| 安全 | 基本不考虑 | Prompt Injection、越权、数据泄露、误操作 |
| 运营 | 一次性演示 | 成本、延迟、稳定性、版本、监控 |

---

### 0.4 Agent 工程化解决什么问题

| 问题 | 工程化解决方式 |
|---|---|
| 结果不稳定 | 指令规范、结构化输出、评估集 |
| 过程不可控 | 状态机、流程编排、工具权限 |
| 出错不可追踪 | Trace、日志、运行记录 |
| 工具调用混乱 | Tool Schema、参数校验、错误处理 |
| 上下文污染 | Context Engineering、检索过滤、压缩 |
| 无法复用 | Skill、模板、模块化工程结构 |
| 无法上线 | 部署、监控、告警、回滚 |
| 安全风险高 | Guardrail、最小权限、人工确认 |

---

### 0.5 Agent 产品化解决什么问题

| 问题 | 产品化解决方式 |
|---|---|
| 用户不知道怎么用 | 明确入口、任务模板、引导式交互 |
| 用户不信任结果 | 解释来源、展示过程、支持预览和撤销 |
| 业务价值不清楚 | 定义节省时间、提升转化、降低错误等指标 |
| 使用场景太泛 | 聚焦高频、高价值、高痛点任务 |
| 用户不持续使用 | 嵌入工作流，形成任务闭环 |
| 团队难维护 | 产品指标、反馈机制、版本迭代 |

---

### 0.6 本章核心判断

一个 Agent 项目是否值得做，不看它“聪不聪明”，而看：

| 判断项 | 问题 |
|---|---|
| 任务价值 | 它解决的是不是高频、高成本、高痛点任务？ |
| 自动化空间 | 这个任务是否有可拆解、可判断、可执行的步骤？ |
| 工具可接入 | Agent 是否能拿到必要数据和执行工具？ |
| 风险可控制 | 出错后能否拦截、回滚、人工确认？ |
| 结果可评估 | 能否判断它做得好不好？ |
| 产品可嵌入 | 用户是否能自然地在工作流中使用它？ |

---

### 0.7 常见误区

| 误区 | 正确理解 |
|---|---|
| 会聊天就是 Agent | 聊天只是交互方式，不代表具备执行能力 |
| 调用工具就是 Agent | 工具调用只是 Agent 的一部分 |
| Agent 越自动越好 | 高风险任务需要人工确认 |
| Demo 能跑就是产品 | 产品需要稳定性、安全、指标和反馈 |
| 模型强就不需要工程 | 模型强只降低部分难度，不能替代系统设计 |

## 第 1 章｜Agent 的本质

### 1.1 一句话结论

**Agent 的本质不是“会聊天的 AI”，而是“以目标为中心，能基于上下文做判断，并通过工具或流程推进任务的智能执行单元”。**

OpenAI Agents SDK 把 Agent 描述为：由 LLM 配置 instructions、tools，并可配置 handoffs、guardrails、structured outputs 等运行行为的应用构件。这个定义说明，产品级 Agent 不是单独的模型，而是“模型 + 指令 + 工具 + 运行机制 + 安全约束”的组合系统。

---

### 1.2 费曼解释

普通 AI 像“问答老师”。

你问它：

> 帮我分析一下广告数据。

它可能直接给你一些泛泛建议。

Agent 像“能处理任务的运营助理”。

它会进一步做：

1. 问你要数据，或者自己去读取数据
2. 判断数据字段是否完整
3. 识别 CTR、CVR、ACOS、CPC 的变化
4. 找出异常项
5. 结合规则生成诊断
6. 输出优化动作
7. 必要时生成报告或发邮件

所以 Agent 的关键不是“说得像人”，而是：

> 它能不能围绕目标持续推进任务。

---

### 1.3 Agent 的最小结构

一个最小 Agent 至少包含 5 个部分：

| 模块 | 作用 | 简单理解 |
|---|---|---|
| Goal | 目标 | 要完成什么 |
| Instruction | 行为规则 | 应该怎么做、不该怎么做 |
| Context | 上下文 | 判断任务需要的信息 |
| Tool | 工具 | 能调用的外部能力 |
| Output | 输出 | 最后交付什么格式的结果 |

更完整的 Agent 还需要：

| 模块 | 作用 |
|---|---|
| State | 记录任务执行到哪里 |
| Memory | 记住长期有用的信息 |
| Guardrail | 防止错误、越权、危险输出 |
| Eval | 判断结果质量 |
| Trace | 追踪执行过程 |
| Handoff | 把任务交给其他 Agent 或人工 |

---

### 1.4 Agent 和“人类助理”的类比

| 人类助理 | Agent |
|---|---|
| 工作职责 | Role / Instruction |
| 手头资料 | Context |
| 可用工具 | Tools |
| 工作步骤 | Plan / Workflow |
| 当前进度 | State |
| 工作记录 | Trace / Log |
| 主管审核 | Human-in-the-loop |
| 工作标准 | Eval / Test |
| 权限范围 | Guardrail / Permission |

---

### 1.5 Agent 的核心闭环

```text
目标输入
  ↓
理解任务
  ↓
获取上下文
  ↓
规划步骤
  ↓
调用工具 / 执行动作
  ↓
观察结果
  ↓
修正计划
  ↓
输出结果
  ↓
等待反馈或继续执行
```

这就是 Agent 最核心的循环：

> 感知 → 理解 → 规划 → 执行 → 反馈。

---

### 1.6 Agent 不等于什么

| 不是 | 原因 |
|---|---|
| 不等于 Prompt | Prompt 是控制语言，Agent 是任务系统 |
| 不等于 Chatbot | Chatbot 可以只回答，不执行任务 |
| 不等于 RAG | RAG 解决知识检索，不负责完整任务执行 |
| 不等于 API 调用 | API 是工具，Agent 要决定何时、如何调用 |
| 不等于自动化脚本 | 脚本按固定逻辑跑，Agent 可以动态判断 |
| 不等于多 Agent | 单 Agent 也可以是 Agent，多 Agent 是高级形态 |

---

### 1.7 判断一个系统是不是 Agent

| 判断问题 | 是 Agent 的表现 |
|---|---|
| 是否有明确目标？ | 有任务目标，而不只是闲聊 |
| 是否能使用上下文？ | 会读取、筛选、利用资料 |
| 是否能选择动作？ | 能决定调用哪个工具或走哪个流程 |
| 是否能推进任务？ | 能从输入走到交付 |
| 是否能处理反馈？ | 能根据工具结果或用户反馈调整 |
| 是否有边界约束？ | 知道哪些不能做、何时要确认 |

如果一个系统只能回答问题，不能推进任务，它更接近 Chatbot。

如果一个系统能理解目标、调用工具、处理状态、交付结果，它才更接近 Agent。

## 第 2 章｜Agent、Prompt、Workflow、RAG、Tool、MCP 的区别

### 2.1 一句话结论

**这些概念不是同一层级的东西。Prompt 是指令层，RAG 是知识层，Tool 是执行层，Workflow 是流程层，MCP 是连接层，Agent 是把这些能力组合起来推进任务的执行单元。**

---

### 2.2 总体分层

| 层级 | 概念 | 解决的问题 |
|---|---|---|
| 指令层 | Prompt / Instruction | 告诉模型怎么做 |
| 知识层 | RAG / Context | 给模型补充信息 |
| 执行层 | Tool / API | 让模型能做现实动作 |
| 连接层 | MCP | 标准化连接外部工具和数据源 |
| 流程层 | Workflow | 规定任务执行步骤 |
| 智能执行层 | Agent | 基于目标动态判断并推进任务 |
| 产品层 | AI Product | 面向真实用户和业务价值交付 |

---

### 2.3 Agent vs Prompt

| 对比项 | Prompt | Agent |
|---|---|---|
| 本质 | 指令文本 | 智能执行单元 |
| 作用 | 约束模型输出 | 推进完整任务 |
| 是否调用工具 | 本身不调用 | 可以调用 |
| 是否管理状态 | 通常不管理 | 可以管理 |
| 是否可产品化 | 很难单独产品化 | 可以作为产品核心能力 |
| 典型例子 | “请按表格输出” | “读取数据、分析异常、生成报告” |

**判断方法：**

如果你只是在改模型说话方式，就是 Prompt。

如果你在设计任务目标、上下文、工具、状态、输出和评估，就是 Agent。

---

### 2.4 Agent vs Workflow

| 对比项 | Workflow | Agent |
|---|---|---|
| 路径 | 固定流程 | 动态路径 |
| 决策 | 主要由代码规则决定 | 可由模型参与判断 |
| 稳定性 | 高 | 取决于设计 |
| 灵活性 | 中低 | 高 |
| 适用任务 | 流程明确、规则稳定 | 任务复杂、信息不确定 |
| 风险 | 低 | 较高，需要护栏 |

LangGraph 文档中也区分了 workflows 和 agents：workflows 通常有预设的代码路径，按一定顺序执行；agents 更动态，会自行定义流程和工具使用。

**费曼解释：**

Workflow 像“流水线”。

- 第一步收集资料
- 第二步清洗数据
- 第三步生成报告
- 第四步发送邮件

Agent 像“项目助理”。

- 它知道目标
- 会判断资料够不够
- 会决定先查数据还是先问用户
- 会根据中间结果调整步骤

**关键判断：**

| 任务情况 | 适合方案 |
|---|---|
| 步骤稳定 | Workflow |
| 判断复杂 | Agent |
| 高风险 | Workflow + 人工确认 |
| 需要动态探索 | Agent |
| 需要强稳定交付 | Workflow 优先，Agent 局部嵌入 |

---

### 2.5 Agent vs RAG

| 对比项 | RAG | Agent |
|---|---|---|
| 本质 | 检索增强生成 | 任务执行系统 |
| 解决问题 | 模型不知道资料 | 模型要完成任务 |
| 主要能力 | 找资料、引用资料 | 判断、规划、执行、反馈 |
| 是否调用工具 | 可选 | 常见 |
| 是否能独立完成任务 | 通常不能 | 可以 |
| 典型例子 | 查知识库回答问题 | 查知识库、写报告、发邮件、建任务 |

**简单理解：**

RAG 是给 Agent 找资料的能力。

Agent 可以使用 RAG，但 RAG 本身不是 Agent。

---

### 2.6 Agent vs Tool

| 对比项 | Tool | Agent |
|---|---|---|
| 本质 | 外部能力 | 使用能力的人 |
| 作用 | 执行具体动作 | 判断何时调用什么工具 |
| 是否有目标 | 没有，只完成调用 | 有目标 |
| 是否会规划 | 不会 | 可以 |
| 是否会纠错 | 不会 | 可以根据结果调整 |
| 典型例子 | 搜索、计算器、数据库查询、发邮件 | 选择工具并完成任务闭环 |

**费曼解释：**

Tool 像“锤子、螺丝刀、Excel、浏览器”。

Agent 像“知道什么时候该用锤子、什么时候该用 Excel 的人”。

---

### 2.7 Agent vs MCP

| 对比项 | MCP | Agent |
|---|---|---|
| 本质 | 连接协议 / 标准 | 智能执行单元 |
| 作用 | 标准化接入工具、数据、工作流 | 使用这些能力完成任务 |
| 类比 | AI 应用的 USB-C | 使用 USB-C 外设的人 |
| 解决问题 | 工具和数据接入碎片化 | 任务理解与执行 |
| 是否直接代表智能 | 不代表 | 代表智能执行能力 |

MCP 官方文档把 MCP 定义为连接 AI 应用与外部系统的开放标准，可连接本地文件、数据库、搜索引擎、计算器、工作流等外部能力，并用 USB-C 类比它在 AI 应用中的标准连接作用。

**简单理解：**

MCP 是“插口标准”。

Tool 是“被插进来的设备”。

Agent 是“会使用这些设备完成任务的人”。

---

### 2.8 最重要的判断表

| 你现在想做什么 | 优先考虑 |
|---|---|
| 只是让模型回答更规范 | Prompt |
| 只是让模型查资料 | RAG |
| 只是让模型调用一个功能 | Tool Use |
| 流程固定、规则明确 | Workflow |
| 外部工具很多，需要统一接入 | MCP |
| 任务复杂、路径不确定、需要动态判断 | Agent |
| 多角色协作、任务很大 | Multi-Agent |
| 要给真实用户长期使用 | Agent Product |

## 第 3 章｜Agent 的产品形态

### 3.1 一句话结论

**Agent 产品不是只有聊天框。Agent 可以是 Chatbot、Copilot、Autopilot、Workflow Agent、Embedded Agent、Agent Platform 等不同形态。**

产品形态的选择，取决于：

1. 用户是否愿意主动对话
2. 任务是否需要人工确认
3. 风险是否可控
4. 执行路径是否固定
5. 是否嵌入已有工作流
6. 是否需要多人协作

---

### 3.2 形态一：Chatbot

| 维度 | 内容 |
|---|---|
| 简单理解 | 对话式问答助手 |
| 用户动作 | 用户问，AI 答 |
| AI 权限 | 低 |
| 自动化程度 | 低 |
| 风险 | 低 |
| 适合场景 | FAQ、知识问答、政策解释、学习辅导 |
| 不适合场景 | 多步骤执行、高风险操作、复杂业务闭环 |

**例子：**

- “解释一下 ACOS 是什么”
- “帮我理解状态机”
- “总结这篇文档”

Chatbot 是最容易做的 AI 产品形态，但它通常不是完整 Agent。

---

### 3.3 形态二：Copilot

| 维度 | 内容 |
|---|---|
| 简单理解 | 人类旁边的辅助驾驶 |
| 用户动作 | 人类主导，AI 辅助 |
| AI 权限 | 中低 |
| 自动化程度 | 中 |
| 风险 | 可控 |
| 适合场景 | 写文案、分析数据、生成代码、优化 Listing |
| 不适合场景 | 无人监督的高风险操作 |

**例子：**

- Amazon 广告分析 Copilot
- Listing 文案优化 Copilot
- 代码开发 Copilot
- 客服回复建议 Copilot

Copilot 的核心是：

> AI 提建议，人类做决定。

---

### 3.4 形态三：Autopilot

| 维度 | 内容 |
|---|---|
| 简单理解 | AI 自动驾驶 |
| 用户动作 | 用户设定目标，AI 自动执行 |
| AI 权限 | 高 |
| 自动化程度 | 高 |
| 风险 | 高 |
| 适合场景 | 低风险、规则清晰、可回滚任务 |
| 不适合场景 | 高金额、高法律风险、高品牌风险操作 |

**例子：**

- 自动生成日报并发送
- 自动归档文件
- 自动清洗报表
- 自动创建低风险任务卡片

Autopilot 的关键前提：

1. 权限必须最小化
2. 操作必须可追踪
3. 高风险动作必须确认
4. 结果必须可回滚
5. 错误必须可告警

---

### 3.5 形态四：Workflow Agent

| 维度 | 内容 |
|---|---|
| 简单理解 | 嵌入固定流程中的 Agent |
| 用户动作 | 用户启动流程或处理关键节点 |
| AI 权限 | 中 |
| 自动化程度 | 中高 |
| 风险 | 可控 |
| 适合场景 | 数据分析、内容生产、审核流程、知识沉淀 |
| 不适合场景 | 完全开放、无边界任务 |

**例子：**

你的“对话沉淀成 Markdown 文件”就是典型 Workflow Agent：

```text
读取对话
  ↓
提炼主题
  ↓
按知识框架重组
  ↓
生成 Markdown
  ↓
输出文件
  ↓
放入知识库
```

其中有些步骤可以固定，有些步骤需要 Agent 判断。

---

### 3.6 形态五：Embedded Agent

| 维度 | 内容 |
|---|---|
| 简单理解 | 嵌入已有软件界面里的 Agent |
| 用户动作 | 在原有系统中点击、选择、确认 |
| AI 权限 | 取决于系统 |
| 自动化程度 | 中 |
| 风险 | 较可控 |
| 适合场景 | CRM、ERP、广告后台、知识库、设计工具 |
| 不适合场景 | 完全脱离业务系统的泛聊天 |

**例子：**

- 在广告报表旁边出现“诊断异常”按钮
- 在客服工单旁边出现“生成回复”按钮
- 在 PRD 页面旁边出现“拆解开发任务”按钮

Embedded Agent 的优势是：

> 不改变用户原有工作流，只在关键节点增强能力。

---

### 3.7 形态六：Agent Platform

| 维度 | 内容 |
|---|---|
| 简单理解 | 多个 Agent、工具、知识库、权限和评估的统一平台 |
| 用户动作 | 选择或配置不同 Agent |
| AI 权限 | 分级管理 |
| 自动化程度 | 高 |
| 风险 | 需要系统治理 |
| 适合场景 | 团队级、组织级 AI 能力平台 |
| 不适合场景 | 单一小任务、早期验证项目 |

**例子：**

- 企业内部 AI 工作台
- 多 Agent 项目交付系统
- 个人 LLM-Wiki + 工具 + Skill 工作台

---

### 3.8 产品形态选择表

| 任务特征 | 推荐形态 |
|---|---|
| 只是问答 | Chatbot |
| 需要人判断 | Copilot |
| 低风险重复执行 | Autopilot |
| 流程明确但部分步骤需要智能判断 | Workflow Agent |
| 已有业务系统，需要局部增强 | Embedded Agent |
| 多任务、多角色、多工具统一管理 | Agent Platform |

## 第 4 章｜Agent 的能力分层

### 4.1 一句话结论

**Agent 的能力可以拆成五层：感知、理解、规划、执行、反馈。越往后越接近真实生产力，也越需要工程约束。**

```text
感知层：看见信息
理解层：判断含义
规划层：决定步骤
执行层：调用工具
反馈层：根据结果修正
```

---

### 4.2 第一层：感知层

| 维度 | 内容 |
|---|---|
| 核心问题 | Agent 能拿到什么信息？ |
| 输入来源 | 用户输入、文件、网页、数据库、截图、邮件、日历、API |
| 工程重点 | 数据接入、权限、格式解析、上下文选择 |
| 常见问题 | 拿不到资料、资料太多、资料过时、资料不可信 |

**例子：**

广告分析 Agent 需要感知：

- 广告报表
- Search Term Report
- Listing 信息
- 时间区间
- 目标关键词
- 历史优化动作

如果这些信息不完整，Agent 后面的分析就会失真。

---

### 4.3 第二层：理解层

| 维度 | 内容 |
|---|---|
| 核心问题 | Agent 是否理解任务和资料含义？ |
| 关键能力 | 意图识别、实体识别、约束理解、异常识别 |
| 工程重点 | Instruction、任务分类、上下文标注 |
| 常见问题 | 误解用户目标、忽略约束、把相关信息当无关信息 |

**例子：**

用户说：

> 帮我看看这个广告为什么 ACOS 升高。

Agent 不能只解释 ACOS。

它要理解：

- 用户要的是诊断，不是科普
- ACOS 升高可能来自 CPC 上升、CVR 下降、价格变化、流量结构变化
- 需要对比时间区间
- 需要输出可执行动作

---

### 4.4 第三层：规划层

| 维度 | 内容 |
|---|---|
| 核心问题 | Agent 该如何拆解任务？ |
| 关键能力 | 任务分解、路径选择、优先级判断、风险判断 |
| 工程重点 | Workflow、Planner、状态机、人工确认点 |
| 常见问题 | 跳步、漏步、任务膨胀、过度规划 |

**例子：**

广告诊断 Agent 可以这样规划：

```text
1. 检查数据字段是否完整
2. 对比核心指标变化
3. 判断 ACOS 上升主因
4. 定位到 campaign / ad group / keyword / search term
5. 判断是流量问题、转化问题还是竞价问题
6. 输出优化动作
7. 给出风险提示
```

规划层决定 Agent 是否像一个“有方法的人”。

---

### 4.5 第四层：执行层

| 维度 | 内容 |
|---|---|
| 核心问题 | Agent 能不能真的做事？ |
| 关键能力 | 工具调用、API 操作、文件生成、数据库查询、自动化脚本 |
| 工程重点 | Tool Schema、权限、参数校验、错误处理、幂等性 |
| 常见问题 | 调错工具、参数错误、重复执行、越权操作 |

OpenAI Agents 文档把 tools、MCP、guardrails、human review、results/state、observability 等作为 Agent 构建的重要部分；这说明执行层不是“能调用工具”就结束，还必须考虑工具语义、状态、审批、观测和评估。

---

### 4.6 第五层：反馈层

| 维度 | 内容 |
|---|---|
| 核心问题 | Agent 能否根据结果调整？ |
| 关键能力 | 观察工具返回、识别失败、重试、修正计划、请求人工确认 |
| 工程重点 | Trace、Eval、日志、状态恢复、Human-in-the-loop |
| 常见问题 | 工具失败后继续编造、没有重试、不知道何时停下 |

LangGraph 的 durable execution 强调在关键点保存流程进度，使流程可以暂停并从中断处恢复，尤其适合人工介入和长时间运行任务。这类能力对应的就是产品级 Agent 的反馈、暂停、恢复和可靠执行能力。

---

### 4.7 五层能力与工程模块对应表

| 能力层 | 对应工程模块 | 典型问题 |
|---|---|---|
| 感知 | 数据接入、Context、RAG、MCP | Agent 是否拿到正确资料？ |
| 理解 | Instruction、任务分类、意图识别 | Agent 是否理解用户真正目标？ |
| 规划 | Workflow、Planner、State Machine | Agent 是否知道下一步怎么做？ |
| 执行 | Tool、API、脚本、权限 | Agent 是否能安全做事？ |
| 反馈 | Trace、Eval、Retry、HITL | Agent 是否能纠错和改进？ |

---

### 4.8 Agent 成熟度模型

| 等级 | 能力 | 状态 |
|---|---|---|
| L0 | 只能聊天 | Chatbot |
| L1 | 能按指令输出 | Prompt Assistant |
| L2 | 能检索资料 | RAG Assistant |
| L3 | 能调用工具 | Tool-Using Agent |
| L4 | 能按流程执行任务 | Workflow Agent |
| L5 | 能动态规划并反馈修正 | Autonomous Agent |
| L6 | 多 Agent 协作、平台化治理 | Agent System |

## 第 0–4 章总复盘

### 1. 这一阶段解决什么问题

第 0–4 章不是教你马上写 Agent，而是先建立判断框架：


### 2. 五个核心结论

| 序号 | 核心结论 |
|---|---|
| 1 | Agent 的本质是围绕目标推进任务，而不是单纯聊天 |
| 2 | Prompt、RAG、Tool、Workflow、MCP 都是 Agent 系统中的不同层级能力 |
| 3 | 并不是所有 AI 自动化都应该做成 Agent，固定流程优先 Workflow |
| 4 | Agent 产品不一定是聊天框，更常见的高价值形态是 Copilot、Workflow Agent、Embedded Agent |
| 5 | Agent 的能力分层是感知、理解、规划、执行、反馈，越靠后越需要工程化约束 |

---

### 3. 本阶段判断题

| 问题 | 判断 |
|---|---|
| 只会回答问题的 AI 是完整 Agent 吗？ | 不是 |
| 调用一个 API 就是 Agent 吗？ | 不一定 |
| RAG 可以作为 Agent 的一部分吗？ | 可以 |
| Workflow 和 Agent 是互斥的吗？ | 不是，可以组合 |
| MCP 是 Agent 吗？ | 不是，它是连接协议 |
| Agent 越自动越好吗？ | 不是，风险越高越需要人工确认 |
| Agent 产品一定是聊天框吗？ | 不是 |
| 反馈层重要吗？ | 很重要，它决定 Agent 能否可靠进入生产环境 |

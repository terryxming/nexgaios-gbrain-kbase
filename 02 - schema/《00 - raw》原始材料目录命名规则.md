# 00 - raw 原始材料目录命名规则

版本：v0.2.0

生命周期：Draft

日期：2026-06-20

适用范围：`00 - raw`

## 1. 定位

`00 - raw` 是本地知识库的原始材料层。

它用于保存：

1. 未进入 wiki 编译的项目记录。
2. 一手资料。
3. 原始网页剪藏。
4. 原始附件和截图。
5. 历史方案文件。
6. 用户或 Agent 直接写入的未编译笔记。

`00 - raw` 不是最终 Wiki。

## 2. 顶层结构

`00 - raw` 下固定保留一个系统入口目录：

```text
00 - inbox
```

`00 - inbox` 是所有新进入、未处理、待判断资料的默认入口，不属于领域目录。

从 `01` 开始是领域目录。

领域目录格式：

```text
NN - 领域名
```

示例：

```text
01 - AI Work
02 - Amazon
03 - 认知管理
04 - 财务投资
05 - 设计
06 - 产品
```

## 3. 子分类结构

领域下的子分类使用四位编号。

子分类目录格式：

```text
NNMM - 子分类名
```

其中：

```text
NN = 父领域编号
MM = 该领域下的子分类排序编号
```

示例：

```text
01 - AI Work
  0101 - 学习
  0102 - 项目
  0103 - 行业资讯

02 - Amazon
  0201 - 学习
  0202 - 政策
  0203 - 资料
```

## 4. 学习分类

每个领域下必须有一个学习分类。

学习分类固定使用该领域的第一个子分类编号：

```text
NN01 - 学习
```

示例：

```text
01 - AI Work / 0101 - 学习
02 - Amazon / 0201 - 学习
03 - 认知管理 / 0301 - 学习
```

学习分类用于保存该领域的概念解释、术语说明、工具教程和学习笔记。

## 5. 编号规则

编号只表达排序，不表达永久知识 ID。

整理期指用户确认目录结构前的本地调整阶段。

正式同步指用户确认目录结构并准备同步到 GitHub 知识库、Supabase 或 GBrain DB 前的最后校验点。

规则：

1. `00 - inbox` 是系统入口目录，不属于领域。
2. `01` 起是领域目录。
3. 新增领域使用下一个可用两位编号。
4. 新增子分类使用该领域下一个可用四位编号。
5. 子分类编号前两位必须等于父领域编号。
6. 每个领域必须有 `NN01 - 学习`。
7. 整理期允许修正跳号和错位编号。
8. 正式同步前必须运行目录校验。
9. 正式同步后，禁止重排已有编号。
10. 正式同步后，只新增编号，不整体重排，不复用已废弃编号。
11. 正式同步后必须重排编号时，先建立迁移清单，再由用户确认。

## 7. Inbox 规则

`00 - inbox` 是 raw 的入口目录。

inbox 文件进入、frontmatter 检查、材料包处理、路由分发和用户确认规则，以以下文件为准：

```text
02 - schema/《00 - inbox》路由分发规则.md
```

本文件不重复定义 inbox 处理细则。

## 8. 编译状态规则

raw 材料是否已经进入 wiki 编译，以以下规则为准：

```text
02 - schema/《00 - raw》编译状态规则.md
```

本文件只规定 raw 目录命名和放置原则，不重复定义 raw frontmatter 编译状态字段。

## 9. 校验规则

正式同步前，目录校验至少检查：

1. 顶层领域目录是否符合 `NN - 领域名`。
2. 是否存在非 `00 - inbox` 的未编号顶层目录。
3. 子分类目录是否符合 `NNMM - 子分类名`。
4. 子分类编号前两位是否等于父领域编号。
5. 每个领域是否存在 `NN01 - 学习`。
6. 是否存在明显跳号。
7. 是否存在空领域目录。

校验工具只输出问题清单，不自动重命名、不自动移动文件。

## 10. 命名中文化规则

原则：

```text
目录名和文件名优先使用中文。
```

适用范围：

1. `00 - raw` 下的领域目录。
2. `00 - raw` 下的子分类目录。
3. `00 - raw` 下的知识文件。
4. `02 - schema` 下的规则文件。

保留例外：

```text
00 - raw
01 - wiki
02 - schema
```

这三个是系统层目录，保留 Karpathy LLM Wiki 的架构术语。

允许保留英文或英文缩写的情况：

1. 产品名：Amazon、GBrain、Supabase、GitHub、Obsidian。
2. 技术名：MCP、API、LLM、RAG、OAuth。
3. 常用工程缩写：PR、CI、CLI、DB。
4. 原始标题必须保留英文才能准确溯源的材料。

文件命名格式：

```text
中文主题 + 必要英文术语 + 可选日期
```

日期规则：

1. 政策、报告、阶段方案、外部资料快照必须带日期。
2. 概念笔记、术语说明、固定工具说明不要求日期。

示例：

```text
GBrain 商业化 SaaS 权限体系重构方案.md
Codex 自定义 MCP 服务的三种鉴权方式.md
Amazon Science：COSMO、Rufus、Alexa for Shopping、搜索推荐相关资料.md
```

不推荐：

```text
note1.md
final.md
new.md
temp.md
misc.md
```

## 11. 目录重命名安全规则

目录重命名不能只看目录名，还要看目录内容。

风险：

```text
如果目录下已有文件，直接交换目录名可能导致文件留在错误语义目录里。
```

正确原则：

```text
文件随所属领域目录和子分类目录移动。
```

示例：

整理前：

```text
06 - 产品
  0601 - 工具
  0602 - 学习
```

目标规则：

```text
06 - 产品
  0601 - 学习
  0602 - 工具
```

正确做法：

```text
旧 0601 - 工具 整体移动为 新 0602 - 工具
旧 0602 - 学习 整体移动为 新 0601 - 学习
```

错误做法：

```text
只交换目录名，但不确认文件是否跟随正确语义。
```

自动化要求：

1. 重命名前先统计目录下文件数和子目录数。
2. 有内容的目录必须按语义映射移动。
3. 目录交换必须使用中转目录，避免 Windows 大小写和同名冲突问题。
4. 自动化默认只报告计划。
5. 实际移动必须得到用户确认。
6. 移动后重新运行目录校验。

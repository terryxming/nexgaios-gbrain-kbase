# Agent Memory 记忆模块｜第 10–23 章

> 主题：Memory 底层原理与工程模块设计  
> 适用场景：Agent 工程、LLM 应用、个人知识库、Codex/Workflow/Skill 系统  
> 学习目标：理解 Memory 如何写入、检索、排序、注入、更新、遗忘，并能设计一个可落地的 Memory 模块。

---

## 第 10 章｜上下文窗口与记忆压缩

### 本章一句话

Memory 的第一个底层约束是：LLM 只能看到上下文窗口里的内容，所以历史信息必须被选择、压缩、检索后再进入上下文。

### 10.1 为什么不能把所有历史都塞进上下文

| 问题 | 说明 | 后果 |
|---|---|---|
| Token 成本 | 历史越长，输入成本越高 | 成本上升 |
| 延迟 | 上下文越长，推理越慢 | 响应变慢 |
| 噪音 | 历史中有大量无关信息 | 模型注意力被干扰 |
| 过期 | 旧信息可能已经失效 | 输出基于错误背景 |
| 冲突 | 新旧偏好可能不一致 | Agent 决策摇摆 |

### 10.2 记忆压缩的本质

```text
记忆压缩 = 从历史中保留未来有用的信息，丢弃未来无用的信息
```

压缩不是简单摘要，而是面向未来任务的选择。

| 普通摘要 | 记忆压缩 |
|---|---|
| 复述发生过什么 | 提炼未来会用到什么 |
| 关注完整性 | 关注可用性 |
| 适合阅读 | 适合 Agent 调用 |
| 可以较长 | 应该结构化、短、准 |

### 10.3 常见压缩方式

| 方式 | 适合场景 | 风险 |
|---|---|---|
| Sliding Window | 只保留最近 N 轮 | 早期重要信息会丢失 |
| Summary | 长对话压缩为摘要 | 可能损失细节 |
| Structured Extraction | 提取成字段 | 需要 schema 设计 |
| Episodic Log | 保留事件记录 | 检索复杂 |
| Semantic Fact | 提炼稳定事实 | 容易误把临时信息长期化 |
| Procedural Rule | 提炼流程规则 | 容易过度泛化 |

### 10.4 工程判断

| 问题 | 推荐策略 |
|---|---|
| 对话很短 | 保留原文即可 |
| 对话变长 | 最近内容 + 摘要 |
| 用户偏好稳定 | 提炼为长期语义记忆 |
| 项目状态重要 | 提炼为结构化 Project Memory |
| 工具执行失败 | 提炼为 Failure Memory |

### 本章掌握标准

你能判断：什么时候保留原始历史，什么时候摘要，什么时候提炼为结构化长期记忆。

## 第 11 章｜记忆写入原理

### 本章一句话

Memory 写入的核心不是“看到什么就记什么”，而是判断哪些信息值得在未来复用。

### 11.1 记忆写入流程

```text
原始输入 → 候选信息 → 价值判断 → 类型判断 → 结构化 → 写入 Memory Store
```

### 11.2 什么信息值得写入

| 标准 | 判断问题 | 示例 |
|---|---|---|
| 长期稳定 | 未来还会成立吗 | 用户工作语言是中文 |
| 高频复用 | 未来会反复用吗 | 用户喜欢表格化输出 |
| 影响决策 | 会改变 Agent 行为吗 | 用户希望先路线图后讲课 |
| 项目连续 | 和长期项目有关吗 | 当前课程进入第 10–23 章 |
| 纠错价值 | 能避免未来重复犯错吗 | Codex 容易绕过 PRD，需要验收清单 |

### 11.3 什么不应该写入

| 信息 | 原因 |
|---|---|
| 一次性临时偏好 | 过期快 |
| 未确认猜测 | 容易形成错误记忆 |
| 敏感信息 | 需要权限和最小化原则 |
| 无复用价值细节 | 增加噪音 |
| 完整长对话 | 成本高、难检索 |

### 11.4 写入时机

| 时机 | 说明 | 优点 | 风险 |
|---|---|---|---|
| 对话中即时写入 | 边聊边写 | 实时性强 | 容易误写 |
| 回合结束后写入 | 本轮任务完成后提炼 | 稳定性更高 | 延迟一点 |
| 阶段结束后写入 | 一组任务完成后沉淀 | 结构更清晰 | 可能漏掉细节 |
| 用户确认后写入 | 由用户授权保存 | 安全性高 | 交互成本高 |

### 11.5 写入策略模板

```yaml
write_policy:
  write_if:
    - stable_over_time
    - likely_reused
    - affects_future_behavior
    - user_confirmed_or_high_confidence
  do_not_write_if:
    - one_time_detail
    - sensitive_without_permission
    - speculative
    - already_covered_by_existing_memory
```

### 本章掌握标准

你能设计一套规则，判断一段对话中哪些内容应该写入 Memory，哪些不应该写入。

## 第 12 章｜记忆检索原理

### 本章一句话

Memory 检索的目标不是“找最多记忆”，而是为当前任务找最相关、最可信、最必要的记忆。

### 12.1 检索输入是什么

| 输入 | 示例 |
|---|---|
| 当前用户请求 | 继续讲第 10–23 章 |
| 当前任务类型 | 教学、知识沉淀 |
| 当前项目 | Agent Memory 课程 |
| 当前实体 | Memory、Agent、LLM Wiki |
| 当前约束 | 输出后生成 .md 文件 |

### 12.2 常见检索方式

| 检索方式 | 适合记忆 | 示例 |
|---|---|---|
| 精确匹配 | ID、用户、项目、章节 | project_id = memory_course |
| 关键词检索 | 术语明确的记忆 | “Memory Schema” |
| 语义检索 | 自然语言相似内容 | “用户喜欢什么教学方式” |
| 结构化过滤 | 类型、时间、范围 | type = preference |
| 混合检索 | 复杂 Agent 任务 | 关键词 + 向量 + 过滤 |

### 12.3 检索不是一次性动作

更真实的检索流程通常是：

```text
生成检索查询 → 初步召回 → 按 scope 过滤 → 按相关性排序 → 去重 → 压缩 → 注入上下文
```

### 12.4 检索失败类型

| 失败类型 | 表现 | 影响 |
|---|---|---|
| 漏召回 | 应该取的没取到 | Agent 重复提问或判断缺失 |
| 误召回 | 不该取的取回来了 | 上下文污染 |
| 旧记忆优先 | 新信息被旧信息覆盖 | 决策过时 |
| 范围混乱 | 取到其他项目记忆 | 输出跑偏 |
| 语义漂移 | 相似但不相关 | 误导模型 |

### 本章掌握标准

你能根据当前任务设计检索条件：按用户、项目、类型、时间、相关性筛选 Memory。

## 第 13 章｜记忆排序与相关性

### 本章一句话

Memory 排序决定哪些记忆优先进入上下文，因此它直接影响 Agent 的判断质量。

### 13.1 为什么检索后还要排序

检索只解决“候选记忆有哪些”，排序解决“哪些更应该被用”。

| 没有排序的后果 | 说明 |
|---|---|
| 无关记忆进入上下文 | 干扰推理 |
| 旧记忆压过新记忆 | 造成过时判断 |
| 低可信信息被使用 | 形成错误输出 |
| 重要约束被截断 | 输出不符合要求 |

### 13.2 排序维度

| 维度 | 含义 | 示例 |
|---|---|---|
| Relevance | 与当前任务相关度 | 当前是 Memory 课程，优先课程记忆 |
| Recency | 新近程度 | 最近确认的章节优先 |
| Confidence | 可信度 | 用户明确说过 > 模型推断 |
| Salience | 重要性 | 长期偏好优先于小细节 |
| Scope Match | 范围匹配 | 当前项目记忆优先于其他项目 |
| Freshness | 是否过期 | 旧价格、旧规则需降权 |
| Source Quality | 来源质量 | 用户明确指令优先于日志推断 |

### 13.3 一个简单排序公式

```text
Memory Score = 相关性 × 0.4 + 重要性 × 0.2 + 可信度 × 0.2 + 新近度 × 0.1 + 范围匹配 × 0.1
```

这不是固定公式，而是帮助理解排序逻辑。

### 13.4 排序后的动作

| 排序结果 | 动作 |
|---|---|
| 高相关、高可信 | 注入上下文 |
| 高相关、低可信 | 可使用，但标注不确定 |
| 低相关、高重要 | 保留候选，不注入 |
| 过期或冲突 | 触发更新或忽略 |
| 敏感信息 | 按权限策略处理 |

### 本章掌握标准

你能解释为什么不是所有检索到的记忆都应该进入上下文，并能设计基本排序标准。

## 第 14 章｜记忆注入原理

### 本章一句话

Memory 注入是把检索到的记忆放进模型上下文的过程，关键是位置、格式、优先级和数量。

### 14.1 注入位置

| 注入位置 | 适合内容 | 风险 |
|---|---|---|
| System / Developer 指令 | 高优先级长期规则 | 容易压制当前任务 |
| Context Block | 用户偏好、项目状态 | 需要结构清晰 |
| User Message 附加区 | 当前任务相关记忆 | 可能和用户原文混淆 |
| Tool Result | 检索返回的记忆 | 需要说明来源和可信度 |
| Hidden State | 框架内部状态 | 可控性依赖框架 |

### 14.2 注入格式

推荐结构：

```text
<relevant_memory>
- 用户偏好：喜欢 MECE + 表格化解释
- 当前项目：Agent Memory 课程
- 当前阶段：第 10–23 章
- 输出要求：完成讲义后生成 Markdown 文件
</relevant_memory>
```

### 14.3 好的注入标准

| 标准 | 说明 |
|---|---|
| 少而准 | 只注入当前任务需要的记忆 |
| 有分类 | 区分用户偏好、项目状态、约束 |
| 有边界 | 不把临时偏好泛化为永久规则 |
| 有优先级 | 关键约束放在更显眼位置 |
| 可追踪 | 最好知道记忆来源 |

### 14.4 错误注入示例

```text
用户以前聊过很多 Agent、Amazon、广告、图片、Codex，所以都注入上下文。
```

问题：范围过宽，严重污染当前任务。

### 14.5 正确注入示例

```text
当前任务相关记忆：
1. 用户正在学习 Agent Memory 课程。
2. 用户要求先讲第 10–23 章，再沉淀为 .md 文件。
3. 用户偏好中文、表格化、高信息密度、MECE 结构。
```

### 本章掌握标准

你能把检索到的记忆整理成清晰、短小、相关的上下文块，并知道放在哪里。

## 第 15 章｜记忆更新与遗忘

### 本章一句话

Memory 不是一次写入永久有效的东西，它必须能更新、合并、降权和删除。

### 15.1 为什么需要更新

| 原因 | 示例 |
|---|---|
| 用户偏好变化 | 之前喜欢长文，现在要求高密度表格 |
| 项目阶段变化 | 从课程大纲进入第 10–23 章 |
| 事实过期 | 价格、规则、工具能力变化 |
| 旧记忆错误 | 早期推断后来被否定 |
| 多条记忆重复 | 同一偏好被写了很多次 |

### 15.2 更新策略

| 策略 | 用法 |
|---|---|
| Replace | 新信息明确覆盖旧信息 |
| Merge | 多条记忆合并成更完整版本 |
| Append | 添加新事件，不覆盖旧事件 |
| Versioning | 保留历史版本，标记当前版本 |
| Decay | 随时间降权 |
| Confirm | 冲突时向用户确认 |

### 15.3 遗忘策略

| 策略 | 说明 |
|---|---|
| TTL 过期 | 到期自动失效 |
| 手动删除 | 用户要求删除 |
| 降权 | 不删除，但降低检索优先级 |
| 归档 | 从活跃记忆移到历史记录 |
| 敏感清除 | 对隐私信息强制删除或脱敏 |

### 15.4 冲突处理优先级

| 优先级 | 来源 |
|---:|---|
| 1 | 用户当前明确指令 |
| 2 | 用户最近确认的长期偏好 |
| 3 | 项目当前状态 |
| 4 | 历史记忆 |
| 5 | 模型推断 |

### 本章掌握标准

你能设计：当新旧记忆冲突时，系统应该覆盖、合并、降权、归档还是询问用户。

## 第 16 章｜Memory 系统总架构

### 本章一句话

一个工程化 Memory 系统不是一个数据库，而是一条从捕获到评估的完整流水线。

### 16.1 总架构

```text
输入层
  ↓
Memory Capture
  ↓
Memory Extractor
  ↓
Memory Classifier
  ↓
Memory Schema Manager
  ↓
Memory Store
  ↓
Memory Retriever
  ↓
Memory Ranker
  ↓
Memory Compressor
  ↓
Memory Injector
  ↓
Agent Reasoning / Tool Use
  ↓
Memory Updater / Forgetter
  ↓
Memory Eval / Observability
```

### 16.2 模块职责表

| 模块 | 负责什么 |
|---|---|
| Capture | 捕获候选历史信息 |
| Extractor | 提炼可记忆内容 |
| Classifier | 判断记忆类型 |
| Schema Manager | 统一字段结构 |
| Store | 持久化保存 |
| Retriever | 召回相关记忆 |
| Ranker | 排序和过滤 |
| Compressor | 压缩成上下文可用形式 |
| Injector | 注入 Prompt / Context |
| Updater | 更新和合并 |
| Forgetter | 删除、过期、降权 |
| Eval | 测试记忆是否有效 |
| Observability | 记录读写和调用轨迹 |

### 16.3 最小可行架构

```text
MVP Memory = schema + store + retrieval + injection + update policy
```

不要一开始就做复杂 Memory OS。先让 Agent 能稳定记住用户偏好、项目状态和任务经验。

### 本章掌握标准

你能画出一个 Memory 系统从写入到使用再到更新的完整链路。

## 第 17 章｜Memory Schema 设计

### 本章一句话

Schema 决定 Memory 能不能被稳定存储、检索、更新和评估。

### 17.1 为什么需要 Schema

| 没有 Schema | 有 Schema |
|---|---|
| 记忆像散文 | 记忆像结构化资产 |
| 难过滤 | 可按类型、时间、项目筛选 |
| 难更新 | 可精确覆盖字段 |
| 难评估 | 可做一致性和完整性测试 |

### 17.2 通用 Memory Schema

```json
{
  "memory_id": "mem_001",
  "type": "preference | fact | project_state | procedure | failure",
  "scope": "user | project | session | org",
  "subject": "记忆主体",
  "content": "记忆内容",
  "source": "user_explicit | assistant_inferred | tool_result | file",
  "confidence": 0.95,
  "created_at": "2026-05-21",
  "updated_at": "2026-05-21",
  "ttl": null,
  "tags": ["agent", "memory"],
  "status": "active | archived | deprecated",
  "access_policy": "private"
}
```

### 17.3 常见 Schema 类型

| 类型 | 必要字段 |
|---|---|
| User Preference | user_id、preference、scope、confidence |
| Project State | project_id、current_stage、decision、next_step |
| Tool Memory | tool_name、use_case、limitation、best_practice |
| Failure Memory | failure_case、root_cause、prevention_rule |
| Procedure Memory | trigger、steps、quality_gate |

### 17.4 Schema 设计原则

| 原则 | 说明 |
|---|---|
| 字段少而稳定 | 不要一开始设计过多字段 |
| 类型明确 | preference、fact、state 不要混在一起 |
| 可更新 | 必须有 updated_at、status |
| 可追踪 | 必须有 source |
| 可隔离 | 必须有 scope |
| 可评估 | 必须能判断是否完整、冲突、过期 |

### 本章掌握标准

你能为用户偏好、项目状态、工具经验、失败经验分别设计 Memory Schema。

## 第 18 章｜Memory Store 设计

### 本章一句话

Memory Store 不是只能用向量库，不同记忆类型应该选择不同存储方式。

### 18.1 常见 Store 类型

| Store | 适合内容 | 不适合内容 |
|---|---|---|
| 内存变量 | 当前运行状态 | 长期记忆 |
| KV Store | 简单偏好、配置 | 复杂关系 |
| SQL | 结构化事实、项目状态 | 模糊语义检索 |
| 文档数据库 | 半结构化 JSON | 强事务关系 |
| 向量数据库 | 语义相似检索 | 精确状态管理 |
| 图数据库 | 实体关系、因果网络 | 简单偏好 |
| 文件系统 | LLM Wiki、Markdown 知识沉淀 | 高频实时检索 |
| Event Log | 可回放历史 | 直接作为上下文 |

### 18.2 Store 选择原则

| 问题 | 推荐 Store |
|---|---|
| 我要记用户偏好 | KV / SQL / JSON |
| 我要记项目状态 | SQL / 文档数据库 |
| 我要检索历史对话片段 | 向量库 |
| 我要沉淀知识文档 | 文件系统 |
| 我要追踪执行过程 | Event Log |
| 我要管理实体关系 | 图数据库 |

### 18.3 不要滥用向量库

向量库适合“相似内容召回”，不适合所有 Memory。

| 记忆 | 是否适合向量库 |
|---|---|
| “用户工作语言是中文” | 不首选，结构化更好 |
| “找类似失败案例” | 适合 |
| “当前项目阶段是第 10–23 章” | 不首选，项目状态表更好 |
| “检索过去相关对话片段” | 适合 |

### 本章掌握标准

你能根据记忆类型选择合适 Store，而不是默认所有东西都进向量库。

## 第 19 章｜Memory Extractor

### 本章一句话

Memory Extractor 负责从原始对话、工具结果、文件和日志中提炼候选记忆。

### 19.1 Extractor 的输入

| 输入来源 | 示例 |
|---|---|
| 用户消息 | “以后都用中文讲” |
| AI 输出 | 已完成某阶段总结 |
| 工具结果 | 文件生成成功或失败 |
| 代码日志 | 构建失败原因 |
| 项目文档 | PRD、规范、目录结构 |
| 评估结果 | 哪类输出不合格 |

### 19.2 Extractor 的输出

```json
{
  "candidate_memory": "用户偏好中文教学，并喜欢 MECE + 表格化结构",
  "type": "preference",
  "scope": "user",
  "confidence": 0.95,
  "reason": "用户多次明确提出类似要求"
}
```

### 19.3 Extractor 的核心判断

| 判断 | 问题 |
|---|---|
| 是否值得记 | 未来是否会复用 |
| 记忆类型 | 是偏好、事实、状态、流程还是失败经验 |
| 作用范围 | 用户级、项目级、会话级 |
| 可信度 | 是用户明确说的，还是模型推断的 |
| 是否敏感 | 是否需要用户授权 |

### 19.4 Extractor 常见错误

| 错误 | 例子 |
|---|---|
| 过度提取 | 把一次性要求当长期偏好 |
| 欠提取 | 漏掉明确长期偏好 |
| 错误归类 | 把项目状态当用户偏好 |
| 缺少来源 | 不知道记忆从哪里来 |
| 没有置信度 | 所有记忆都同等可信 |

### 本章掌握标准

你能从一段对话中抽取候选记忆，并标注类型、范围、来源和可信度。

---

## 第 20 章｜Memory Retriever

### 本章一句话

Retriever 负责根据当前任务，从 Memory Store 中取回可能有用的记忆。

### 20.1 Retriever 的输入

| 输入 | 示例 |
|---|---|
| 当前任务 | 讲解第 10–23 章 |
| 用户 ID | 当前用户 |
| 项目 ID | Agent Memory 课程 |
| 任务类型 | 教学 + 知识沉淀 |
| 检索范围 | user + project memory |

### 20.2 Retriever 的输出

| 输出 | 示例 |
|---|---|
| 用户偏好 | 中文、表格化、MECE |
| 项目状态 | 已完成第 0–9 章 |
| 当前目标 | 进入第 10–23 章 |
| 输出要求 | 完成后生成 Markdown 文件 |

### 20.3 Retriever 策略

| 策略 | 用法 |
|---|---|
| Scope-first | 先限制用户、项目、会话范围 |
| Type-first | 先按记忆类型筛选 |
| Query-based | 用当前任务生成检索 query |
| Hybrid | 结构化过滤 + 语义检索 |
| Rerank | 初召回后再排序 |

### 20.4 Retriever 的质量指标

| 指标 | 含义 |
|---|---|
| Recall | 应该取回的是否取回 |
| Precision | 取回的是否相关 |
| Noise Rate | 无关记忆比例 |
| Freshness | 是否使用了最新记忆 |
| Scope Accuracy | 是否取错用户或项目 |

### 本章掌握标准

你能为一个 Agent 设计 Memory 检索策略，并知道如何测试召回质量。

---

## 第 21 章｜Memory Injector

### 本章一句话

Injector 负责把取回的记忆转换成模型能正确理解和使用的上下文。

### 21.1 Injector 不是简单拼接

错误做法：

```text
把检索到的所有记忆直接拼到 prompt 前面。
```

正确做法：

```text
筛选 → 分类 → 压缩 → 标注优先级 → 注入到合适位置
```

### 21.2 Injector 输入输出

| 阶段 | 内容 |
|---|---|
| 输入 | 经过排序的候选记忆 |
| 处理 | 去重、压缩、结构化、标记来源 |
| 输出 | 可放入 Prompt 的 memory context block |

### 21.3 推荐注入模板

```text
<task_relevant_memory>
用户偏好：
- 使用中文。
- 喜欢 MECE、表格化、高信息密度表达。

项目状态：
- Agent Memory 课程已完成第 0–9 章。
- 当前进入第 10–23 章。

当前任务约束：
- 先完整输出课程内容。
- 再整理成 Markdown 文件。
</task_relevant_memory>
```

### 21.4 Injector 的风险

| 风险 | 说明 |
|---|---|
| 注入太多 | 上下文污染 |
| 注入太少 | 缺少关键背景 |
| 格式混乱 | 模型无法区分记忆和指令 |
| 优先级错误 | 旧记忆压过当前指令 |
| 无来源 | 难以审计和修正 |

### 本章掌握标准

你能把检索结果转换成简洁、分类明确、优先级清晰的上下文块。

---

## 第 22 章｜Memory Updater

### 本章一句话

Updater 负责处理新旧记忆之间的关系：新增、覆盖、合并、去重、降权和归档。

### 22.1 Updater 的核心任务

| 任务 | 示例 |
|---|---|
| 新增 | 用户提出新的长期偏好 |
| 覆盖 | 用户明确否定旧偏好 |
| 合并 | 多条相似偏好合成一条 |
| 去重 | 避免重复保存相同记忆 |
| 降权 | 旧信息不删除但减少使用 |
| 归档 | 项目阶段结束后转入历史 |

### 22.2 更新判断矩阵

| 新信息 vs 旧信息 | 动作 |
|---|---|
| 完全一致 | 不新增，只更新最后确认时间 |
| 部分补充 | Merge |
| 明确冲突 | Replace 或请求确认 |
| 临时例外 | 保留旧记忆，当前任务局部覆盖 |
| 信息过期 | Deprecated / Archived |

### 22.3 示例

旧记忆：

```text
用户喜欢详细长文解释。
```

新信息：

```text
长文读起来特别累，希望信息密度高，尽量表格化。
```

更新后：

```text
用户偏好高信息密度、表格化、结构化表达；不喜欢大篇长文堆叠。
```

### 22.4 Updater 的关键字段

| 字段 | 作用 |
|---|---|
| updated_at | 记录更新时间 |
| previous_value | 保留旧版本 |
| status | active / archived / deprecated |
| confidence | 更新可信度 |
| conflict_reason | 冲突原因 |
| update_source | 更新来源 |

### 本章掌握标准

你能设计一个新旧记忆冲突时的更新规则，而不是让 Memory 越积越乱。

---

## 第 23 章｜Memory Forgetter

### 本章一句话

Forgetter 负责让 Memory 系统保持干净、安全、不过期；没有遗忘机制的 Memory 最终会变成污染源。

### 23.1 为什么遗忘很重要

| 问题 | 后果 |
|---|---|
| 旧信息不删除 | Agent 使用过期背景 |
| 错误信息不清理 | 错误持续影响输出 |
| 临时信息长期保存 | 记忆库噪音变多 |
| 敏感信息长期保留 | 隐私和合规风险 |
| 重复记忆过多 | 检索质量下降 |

### 23.2 遗忘不是只有删除

| 遗忘方式 | 说明 |
|---|---|
| Delete | 彻底删除 |
| Expire | 到期失效 |
| Archive | 归档，不参与默认检索 |
| Deprecate | 标记过时 |
| Decay | 随时间降低权重 |
| Mask | 脱敏处理 |
| Scope Restrict | 缩小可见范围 |

### 23.3 什么时候应该遗忘

| 条件 | 动作 |
|---|---|
| 用户明确要求删除 | Delete |
| 信息超过有效期 | Expire |
| 被新信息覆盖 | Deprecate / Archive |
| 敏感且无必要保留 | Delete / Mask |
| 长期未被使用 | Decay |
| 多条重复 | Merge 后删除重复项 |

### 23.4 Forgetting Policy 示例

```yaml
forgetting_policy:
  delete_if:
    - user_requests_deletion
    - sensitive_without_valid_reason
    - confirmed_false
  archive_if:
    - project_phase_completed
    - replaced_by_newer_memory
  decay_if:
    - not_used_for_long_time
    - low_confidence
  never_delete_without_review:
    - compliance_required_records
    - audit_logs
```

### 本章掌握标准

你能设计一套 Memory 遗忘规则，让系统既能长期学习，又不会被旧信息和错误信息污染。

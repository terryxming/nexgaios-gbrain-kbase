# Sprint、SDD、TDD、BDD 在 AI 工程中的协作关系

## 1. 核心结论

Sprint、SDD、TDD、BDD 不是同一层级的概念，而是分别作用于项目交付、需求规格、业务验收和代码质量的不同方法。

在 AI 工程、Agent 工程、AI 编码工具使用场景中，它们可以形成一套完整的协作链路：

> **Sprint 管交付节奏，SDD 管规格约束，BDD 管业务行为，TDD 管代码正确性。**

对应关系如下：

| 概念 | 主要作用 | 核心理解 |
|---|---|---|
| Sprint | 项目节奏管理 | 在固定周期内交付一批可验证成果 |
| SDD | 规格驱动开发 | 先明确需求、边界、设计和验收标准，再进入实现 |
| BDD | 行为驱动开发 | 用业务场景描述系统应该如何表现 |
| TDD | 测试驱动开发 | 先写测试，再写代码，用测试约束实现 |

在 AI 编程中，这四者的价值会被进一步放大。原因是 AI 容易出现自作主张、需求漂移、上下文遗忘、代码看似可用但不符合业务目标等问题。Sprint、SDD、BDD、TDD 可以共同构成对 AI 的约束系统。

## 2. Sprint：交付节奏管理

## 2.1 定义

Sprint 来自敏捷开发和 Scrum，是一种固定周期的迭代交付机制。

一个 Sprint 通常持续一到四周，在这一周期内，团队围绕明确目标完成一批可验证、可交付的工作成果。

## 2.2 核心作用

Sprint 解决的问题是：

> **这一轮要交付什么，什么时候交付，交付到什么程度。**

它关注的是节奏，而不是具体代码实现。

## 2.3 常见输出物

Sprint 通常会产出：

- Sprint Goal：本轮迭代目标
- Sprint Backlog：本轮任务列表
- Increment：本轮完成的可交付增量
- Sprint Review：迭代评审结果
- Retrospective：迭代复盘结论

## 2.4 示例

```text
Sprint 周期：2 周

Sprint Goal：
完成 A11 电子说明书后台上传与用户端预览功能。

本轮交付内容：
1. 后台 PDF 上传入口
2. PDF 文件格式校验
3. 文件大小限制
4. 用户端说明书预览
5. 历史版本保留
6. 基础测试通过
```

Sprint 不负责描述每一个接口、字段和测试细节，而是负责定义本轮工作的交付范围和节奏。

## 3. SDD：规格驱动开发

## 3.1 定义

在 AI 工程语境中，SDD 通常指 **Spec-Driven Development，规格驱动开发**。

它强调在编码之前，先写清楚功能规格，包括需求背景、目标、范围、系统设计、约束条件、验收标准和任务拆解。

SDD 的核心思想是：

> **先写清楚规格，再让 AI 或工程师实现。**

## 3.2 与传统 SDD 的区别

传统软件工程中的 SDD 有时指 **Software Design Document，软件设计文档**，更偏架构和设计说明。

AI 语境下的 SDD 更偏向 **Spec-Driven Development**，即以规格文档作为 AI 编码和工程实现的核心上下文。

两者并不冲突，但关注点不同：

| 类型 | 侧重点 |
|---|---|
| Software Design Document | 系统设计、架构说明、技术方案 |
| Spec-Driven Development | 需求规格、实现边界、AI 执行约束、验收标准 |

## 3.3 SDD 解决的问题

AI 编程中常见问题包括：

- AI 自作主张扩展需求
- AI 忘记业务边界
- AI 只追求代码能跑，不关注业务是否正确
- AI 修改代码时破坏已有功能
- 多轮对话后上下文漂移
- 功能完成后难以验收

SDD 的作用是为 AI 提供明确、稳定、可执行的规格上下文。

## 3.4 SDD 的典型结构

一个较完整的 SDD 通常包括：

```text
1. 背景
   为什么要做这个功能？

2. 目标
   该功能解决什么问题？

3. 范围
   包含什么，不包含什么？

4. 用户流程
   用户如何使用该功能？

5. 系统设计
   涉及哪些模块、接口、数据结构？

6. 约束条件
   有哪些性能、安全、权限、兼容性要求？

7. 验收标准
   做到什么程度才算完成？

8. 任务拆解
   如何拆成可执行的开发任务？
```

## 3.5 示例

```text
功能名称：A11 电子说明书上传与预览

背景：
运营人员需要在后台上传电子说明书，用户可以在产品页面查看最新说明书。

目标：
实现 PDF 说明书上传、版本管理和用户端预览。

范围：
包含：
- 后台上传 PDF
- 文件格式校验
- 文件大小校验
- 版本号管理
- 用户端预览
- 旧版本保留

不包含：
- 多语言自动翻译
- OCR 识别
- PDF 在线编辑

约束：
- 仅支持 PDF 格式
- 文件最大 20MB
- 上传后保留历史版本
- 用户端默认展示最新版本

验收标准：
- 上传非 PDF 时系统提示错误
- 上传超过 20MB 文件时系统拒绝
- 新版本发布后用户端展示最新版本
- 后台可以查看历史版本
```

## 4. BDD：行为驱动开发

## 4.1 定义

BDD 是 **Behavior-Driven Development，行为驱动开发**。

BDD 关注系统在具体业务场景下应该表现出什么行为。它通常使用业务人员、产品经理、测试人员和开发人员都能理解的语言描述功能。

BDD 的核心思想是：

> **用用户行为场景定义系统是否正确。**

## 4.2 BDD 的常见格式

BDD 常使用 Gherkin 语法：

```gherkin
Given 前置条件
When 用户动作
Then 系统结果
```

含义如下：

| 关键词 | 含义 |
|---|---|
| Given | 在什么前置条件下 |
| When | 用户执行什么动作 |
| Then | 系统应该产生什么结果 |
| And | 补充条件或补充结果 |

## 4.3 BDD 解决的问题

BDD 主要解决业务理解不一致的问题。

在没有 BDD 的情况下，产品、开发、测试、AI 可能对同一个功能产生不同理解。例如“上传成功”到底意味着文件保存成功、用户端可见、还是生成了版本记录，可能并不清楚。

BDD 通过场景化表达，让业务行为变得可讨论、可验证、可自动化。

## 4.4 示例

```gherkin
Feature: A11 电子说明书上传

Scenario: 上传合法 PDF 文件
  Given 运营人员已经登录后台
  When 上传一个 10MB 的 PDF 文件
  Then 系统应保存该文件
  And 用户端应显示最新说明书
```

```gherkin
Scenario: 上传超大文件
  Given 运营人员已经登录后台
  When 上传一个 30MB 的 PDF 文件
  Then 系统应拒绝上传
  And 页面提示“文件不能超过 20MB”
```

```gherkin
Scenario: 上传非 PDF 文件
  Given 运营人员已经登录后台
  When 上传一个 JPG 文件
  Then 系统应拒绝上传
  And 页面提示“仅支持 PDF 格式”
```

## 4.5 BDD 在 AI 工程中的价值

BDD 可以把模糊的业务需求转化为 AI 可执行、可验证的行为约束。

在 AI 编码时，可以把 BDD 场景直接交给 AI，让 AI 根据场景生成：

- 自动化测试
- 接口测试
- 端到端测试
- 页面交互逻辑
- 异常处理逻辑
- 验收清单

BDD 的重点不是代码细节，而是业务行为是否符合预期。

## 5. TDD：测试驱动开发

## 5.1 定义

TDD 是 **Test-Driven Development，测试驱动开发**。

TDD 的核心流程是：

> **先写失败测试，再写最少代码让测试通过，最后重构代码。**

经典流程通常称为：

```text
Red → Green → Refactor
```

含义如下：

| 阶段 | 含义 |
|---|---|
| Red | 先写一个失败的测试 |
| Green | 写最少代码让测试通过 |
| Refactor | 在测试保护下优化代码结构 |

## 5.2 TDD 解决的问题

TDD 主要解决代码正确性问题。

它让代码不是凭感觉写出来，而是在测试约束下逐步实现。

TDD 能降低以下风险：

- 核心逻辑错误
- 边界条件遗漏
- 异常场景未处理
- 修改代码时引入回归问题
- AI 生成代码看似正确但实际不符合预期

## 5.3 示例

需求：

```text
订单金额大于等于 100 美元时免运费。
```

TDD 会先写测试：

```text
测试 1：订单金额为 99 美元时，运费不为 0
测试 2：订单金额为 100 美元时，运费为 0
测试 3：订单金额为 150 美元时，运费为 0
```

再实现对应逻辑。

## 5.4 在说明书上传功能中的 TDD 示例

```text
测试 1：isPdf(file) 对 PDF 文件返回 true
测试 2：isPdf(file) 对 JPG 文件返回 false
测试 3：validateFileSize(10MB) 返回 true
测试 4：validateFileSize(30MB) 返回 false
测试 5：getLatestManual(productId) 返回最新版本说明书
测试 6：createManualVersion(file) 会生成新的版本记录
```

这些测试用于约束代码实现，避免 AI 只完成表面功能，而忽略关键边界。

## 5.5 TDD 的适用场景

TDD 适合：

- 核心业务逻辑
- 价格计算
- 权限判断
- 数据转换
- API 返回逻辑
- 文件校验
- 状态机逻辑
- Agent 工具调用结果校验

TDD 不太适合：

- 纯视觉探索
- 一次性草稿页面
- 需求高度不确定的原型
- 只用于沟通方向的低保真 Demo

## 6. 四者之间的协作关系

## 6.1 层级关系

四者可以按层级理解：

```text
Sprint：这一轮什么时候交付？
  ↓
SDD：这一轮到底要做什么？边界是什么？设计是什么？
  ↓
BDD：从用户视角，什么行为才算正确？
  ↓
TDD：从代码视角，每个函数和模块怎么保证正确？
```

## 6.2 分工关系

| 层级 | 方法 | 关注点 |
|---|---|---|
| 项目管理层 | Sprint | 交付节奏、迭代目标、任务范围 |
| 需求设计层 | SDD | 规格、边界、系统设计、任务拆解 |
| 业务验收层 | BDD | 用户行为、业务场景、验收条件 |
| 代码实现层 | TDD | 单元测试、接口测试、代码正确性 |

## 6.3 协作逻辑

四者的理想协作方式是：

1. **Sprint 定义本轮目标**  
   明确本轮要交付什么功能。

2. **SDD 写清楚规格**  
   明确需求、边界、约束、系统设计和验收标准。

3. **BDD 描述业务行为**  
   将需求转化为用户可理解的场景。

4. **TDD 拆解代码测试**  
   将业务规则转化为代码级测试。

5. **AI 或工程师实现功能**  
   在规格、场景和测试约束下完成代码。

6. **Sprint Review 验收结果**  
   检查范围、行为、测试和交付物是否符合预期。

## 7. AI 编程中的推荐工作流

## 7.1 不推荐方式

不推荐直接向 AI 提出宽泛需求：

```text
帮我做一个说明书上传功能。
```

这种方式会导致 AI 自行猜测大量细节，例如：

- 是否需要权限控制
- 是否限制文件格式
- 是否限制文件大小
- 是否保留历史版本
- 用户端是否需要预览
- 错误提示如何展示
- 是否需要测试
- 是否需要日志

宽泛需求容易让 AI 进入自由发挥状态，生成结果可能看似完整，但并不一定符合业务目标。

## 7.2 推荐方式

推荐使用结构化流程：

```text
1. Sprint Goal
   明确本轮交付目标。

2. SDD
   写清楚需求、范围、边界、架构、接口、数据、约束和验收标准。

3. BDD
   写清楚用户行为场景和业务验收条件。

4. TDD
   写清楚单元测试、接口测试、异常测试和回归测试。

5. AI Implementation
   让 AI 按任务拆解逐步实现，而不是一次性生成所有代码。

6. CI / Review
   跑测试、代码审查、验收、复盘。
```

## 7.3 AI 执行提示词结构

可以将任务交给 AI 时使用以下结构：

```text
你现在是资深工程师，请严格根据以下材料实现功能：

一、Sprint Goal
[填写本轮目标]

二、SDD
[填写功能规格、边界、设计和验收标准]

三、BDD
[填写 Given / When / Then 场景]

四、TDD
[填写需要先生成或通过的测试]

执行要求：
1. 不允许超出 SDD 范围自由发挥。
2. 先确认任务拆解，再逐项实现。
3. 先补充或生成测试，再写实现代码。
4. 每完成一个任务，说明对应通过了哪些 BDD 场景和 TDD 测试。
5. 如果发现规格冲突，先指出冲突，不要自行假设。
```

## 8. 类比理解

可以用“装修房子”理解四者关系：

| 软件开发概念 | 装修类比 |
|---|---|
| Sprint | 这一期装修周期，例如两周完成厨房 |
| SDD | 施工图纸，包含尺寸、材料、水电、边界 |
| BDD | 用户验收，例如打开水龙头要有热水，开关要控制对应灯 |
| TDD | 工人自检，例如电压是否正常，水管是否漏水，插座是否通电 |

对应关系：

- 没有 Sprint，项目没有节奏，容易无限拖延。
- 没有 SDD，AI 就像没有图纸的施工队。
- 没有 BDD，做完后可能发现不是业务真正想要的结果。
- 没有 TDD，表面看起来完成了，但内部逻辑可能存在大量隐患。

## 9. 四者对 AI 的约束价值

| 方法 | 对 AI 的约束价值 |
|---|---|
| Sprint | 防止任务无限扩散，控制交付节奏 |
| SDD | 防止 AI 偏离需求，明确实现边界 |
| BDD | 防止 AI 只写功能表面，不符合真实业务行为 |
| TDD | 防止 AI 生成看似正确但没有验证的代码 |

在 AI 工程中，这四者共同构成一套从计划到验收的控制系统：

```text
计划约束：Sprint
规格约束：SDD
行为约束：BDD
代码约束：TDD
```

AI 的优势是执行速度快、生成能力强，但缺点是容易缺乏稳定边界。Sprint、SDD、BDD、TDD 的作用不是降低 AI 效率，而是把 AI 的生成能力放进可管理、可验证、可复盘的工程流程中。

## 10. 实践建议

## 10.1 轻量项目

适合小功能、低风险需求。

推荐组合：

```text
轻量 SDD + BDD 验收场景 + 少量关键 TDD 测试
```

适用场景：

- 内部工具
- 小型管理后台
- 简单页面功能
- 低风险自动化脚本

## 10.2 中型项目

适合业务功能较完整、需要多人协作的项目。

推荐组合：

```text
Sprint + 完整 SDD + BDD 场景 + TDD 测试
```

适用场景：

- 订单系统
- 内容管理系统
- 会员激活系统
- 设备后台管理系统
- AI Agent 工作流

## 10.3 高风险项目

适合支付、权限、安全、数据一致性要求较高的项目。

推荐组合：

```text
Sprint + 严格 SDD + BDD + TDD + 集成测试 + 回归测试 + 代码审查
```

适用场景：

- 支付系统
- 权限系统
- 用户数据系统
- 生产环境自动化 Agent
- 关键业务后台

## 12. 总结

Sprint、SDD、BDD、TDD 可以共同构成 AI 工程中的完整协作框架。

- **Sprint** 负责回答：这一轮什么时候交付？
- **SDD** 负责回答：到底要做什么，边界在哪里？
- **BDD** 负责回答：从用户行为看，什么结果才算正确？
- **TDD** 负责回答：从代码实现看，如何证明逻辑正确？

在 AI 编程时代，工程管理的重点不再只是让人写代码，而是让 AI 在清晰规格、明确场景和可靠测试的约束下完成实现。

因此，合理的协作关系可以概括为：

> **Sprint 定方向，SDD 定规格，BDD 定行为，TDD 定质量。**

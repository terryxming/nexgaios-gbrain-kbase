# Codex SOP 生成工作流

日期：2026-06-16

页面类型：Agent Playbook

适用范围：NexGaios 内部所有由 Codex 参与的 SOP 生成、流程梳理、岗位操作手册、检查清单、培训材料、交接文档、复盘沉淀和流程更新任务。

## 1. 这页是什么

这页定义 Codex 在生成 SOP 时如何使用 GBrain 作为第二大脑。

SOP 不是把步骤写得很多，也不是把经验整理成一篇文章。真正有价值的 SOP 必须能让团队成员按步骤执行、减少沟通成本、降低新人上手难度、避免关键动作遗漏，并且能随着业务变化持续更新。

本工作流的目标是让 Codex 生成的 SOP 从一开始就具备可执行性、可培训性、可检查性和可维护性。

## 2. 适用场景

只要任务符合下面任一条件，Codex 都应该进入本工作流：

1. 把 Terry 或团队成员的经验整理成标准流程。
2. 为 Amazon 运营、产品方案、代码交付、知识库维护、权限管理、客户交付等场景生成 SOP。
3. 把一次性操作整理成可复用步骤。
4. 把交接文档转成新人可执行手册。
5. 把混乱流程拆成职责、输入、步骤、输出和检查项。
6. 把复盘结果沉淀成以后可以复用的操作规范。
7. 更新已有 SOP，使它符合当前工具、人员和业务状态。

## 3. 生成 SOP 前必须检索的第二大脑内容

Codex 生成 SOP 前，应优先从 GBrain 检索以下页面或目录。若页面还不存在，应明确标记缺口，并建议后续补齐。

### 3.1 公司和组织背景

优先检索：

1. `company/context`
2. `company/strategy`
3. `company/org`
4. `company/roles`
5. `company/terry-principles`

目的：

1. 理解这个 SOP 服务哪个业务目标。
2. 理解谁负责、谁审批、谁执行、谁验收。
3. 避免把流程写成脱离组织现实的通用模板。

### 3.2 当前业务和项目状态

优先检索：

1. `projects/current`
2. `decisions`
3. `handoffs`
4. `retrospectives`
5. `agent-playbooks`

目的：

1. 确认当前项目阶段和优先级。
2. 找到历史决策和交接信息。
3. 避免 SOP 和当前真实执行方式冲突。

### 3.3 具体业务目录

根据任务类型优先检索：

1. Amazon 业务：`amazon`
2. 产品方案：`product`
3. 代码交付：`engineering`
4. GBrain 运维：`aiwork/gbrain`
5. 权限管理：`aiwork/gbrain/permission-system`
6. 客户交付：`customer-success`
7. 内容沉淀：`knowledge`

目的：

1. 获取业务事实、现有流程和关键限制。
2. 确认已有模板和过去的执行经验。
3. 让 SOP 从公司真实经验中长出来，而不是从通用知识里生成。

### 3.4 模板和标准

优先检索：

1. `templates/sop`
2. `templates/checklists`
3. `templates/handoff`
4. `templates/review`
5. `standards`

目的：

1. 保持 SOP 的结构一致。
2. 让团队看到同类文档时能快速理解。
3. 降低后续维护成本。

## 4. 生成 SOP 前必须回答的问题

Codex 在开始写 SOP 前，至少要明确：

1. SOP 解决什么问题？
2. 谁是执行人？
3. 谁是负责人？
4. 谁负责审批或验收？
5. 触发条件是什么？
6. 输入资料是什么？
7. 输出物是什么？
8. 需要用到哪些工具、账号、权限或数据源？
9. 哪些动作有风险？
10. 哪些地方必须人工确认？
11. SOP 的更新频率是什么？
12. 生成后应该写回 GBrain 的哪个页面？

如果这些信息缺失，Codex 应先列出缺口。可以先生成草案，但必须明确标记“待 Terry 或负责人确认”。

## 5. SOP 的标准结构

除非用户明确要求其他格式，Codex 生成 SOP 时默认使用下面结构。

### 5.1 基础信息

必须包含：

1. SOP 名称
2. 版本
3. 最近更新日期
4. 负责人
5. 执行人
6. 适用范围
7. 触发条件
8. 相关页面

### 5.2 目标

说明这个 SOP 要解决什么问题，以及完成后应该达到什么状态。

目标必须具体，不能只写“提高效率”或“规范流程”。应尽量写成：

1. 减少什么错误。
2. 让谁能独立完成什么任务。
3. 把什么时间从多少降低到多少。
4. 让什么输出物达到什么标准。

### 5.3 输入资料

列出执行 SOP 前必须准备的信息，包括：

1. 文件
2. 链接
3. 账号权限
4. 数据报表
5. 历史记录
6. 模板
7. 业务负责人确认的信息

### 5.4 操作步骤

步骤必须按真实执行顺序写。每一步应包含：

1. 做什么。
2. 在哪里做。
3. 用什么工具做。
4. 输入什么。
5. 输出什么。
6. 完成标准是什么。

如果某一步存在分支，应写成“如果...则...”。

### 5.5 检查清单

每个 SOP 必须有检查清单，用于执行完成后自查。

检查清单应使用可以判断对错的表达，避免模糊词。例如：

1. 已确认页面标题和 slug 正确。
2. 已读回远端内容并确认标题存在。
3. 已记录负责人和更新时间。
4. 已标记待确认事项。
5. 已把最终版本写回 GBrain。

### 5.6 异常情况

必须列出常见异常，包括：

1. 缺少权限。
2. 缺少数据。
3. 远端服务不可用。
4. 工具返回错误。
5. 业务信息冲突。
6. 执行结果和预期不一致。

每个异常都应包含处理方式和升级路径。

### 5.7 输出物

明确 SOP 执行完成后应该产出什么，包括：

1. 文档
2. 表格
3. 页面
4. 配置
5. 截图
6. 读回验证结果
7. 决策记录
8. 后续任务

### 5.8 维护规则

说明什么时候必须更新 SOP：

1. 工具变化。
2. 组织角色变化。
3. 权限变化。
4. 业务流程变化。
5. 出现重复错误。
6. Terry 或负责人做出新判断。
7. 已有 SOP 被验证不可执行。

## 6. SOP 质量标准

Codex 生成 SOP 时，应按以下标准自检。

### 6.1 可执行

一个没参与过讨论的人，应该能根据 SOP 独立完成任务。

如果还需要口头解释，说明 SOP 没写清楚。

### 6.2 可检查

每个关键步骤都应该有完成标准。不能只写“处理完成”，要写清楚完成后应该看到什么结果。

### 6.3 可培训

SOP 应适合新人学习。必要时可以加入背景解释，但不能把 SOP 写成散文。

### 6.4 可维护

SOP 必须有版本、负责人、更新时间和关联页面，方便后续更新。

### 6.5 可审计

涉及权限、账号、数据、客户、财务、平台规则或生产系统的 SOP，必须留下检查点和记录方式。

## 7. Codex 生成 SOP 时不能做什么

Codex 不得：

1. 编造公司流程。
2. 编造账号、权限、负责人或审批链路。
3. 把不确定的判断写成确定规则。
4. 隐藏风险。
5. 跳过验收标准。
6. 把一次性建议伪装成正式流程。
7. 在没有确认的情况下替 Terry 做商业决策。
8. 将敏感 token、密码、密钥写入 SOP 正文。

如果涉及密码、token 或密钥，SOP 只能写存放原则和操作方式，不能写明文。

## 8. 什么情况必须问 Terry 或负责人

以下情况必须先问，不得默认执行：

1. SOP 会改变现有岗位职责。
2. SOP 会改变权限分配。
3. SOP 会影响客户交付承诺。
4. SOP 会影响 Amazon 价格、广告预算、库存计划或 Listing 上线。
5. SOP 涉及正式对外发布内容。
6. SOP 涉及财务、法务、合规或平台政策判断。
7. SOP 和已有历史决策冲突。
8. SOP 会让团队采用新的工具或长期流程。

## 9. SOP 写回 GBrain 的规则

### 9.1 推荐写回位置

根据内容类型写回：

1. 通用 SOP：`sop`
2. Amazon SOP：`amazon/sop`
3. 产品 SOP：`product/sop`
4. 工程 SOP：`engineering/sop`
5. GBrain SOP：`aiwork/gbrain/sop`
6. 权限 SOP：`aiwork/gbrain/permission-system/sop`
7. 交接 SOP：`handoffs`
8. 模板：`templates/sop`

### 9.2 写回前必须确认

写回前必须确认：

1. slug 命名清晰。
2. 页面标题准确。
3. 版本和日期存在。
4. 没有明文密钥。
5. 待确认事项已标记。
6. 关联页面已列出。
7. 读回验证通过。

### 9.3 写回后必须做的事

写回后必须：

1. 用 `get_page` 读回。
2. 确认标题存在。
3. 确认关键章节存在。
4. 向 Terry 汇报 slug、保存位置和验证结果。

## 10. 标准执行模板

Codex 生成 SOP 时，应按这个节奏：

1. 判断任务属于哪个业务场景。
2. 检索 GBrain 中相关目录和历史记录。
3. 标记已知事实、缺失信息和待确认事项。
4. 明确执行人、负责人、输入、输出、工具和权限。
5. 生成 SOP 草案。
6. 加入检查清单和异常处理。
7. 标记需要 Terry 或负责人确认的内容。
8. 根据反馈修订。
9. 写回 GBrain。
10. 读回验证并汇报。

## 11. 当前第一版的边界

这份 playbook 是 Codex SOP 生成场景的第一版。它目前主要解决 SOP 的结构、生成原则、写回规则和质量标准。

它还不能替代：

1. Terry 对组织流程的最终判断。
2. 业务负责人对执行细节的确认。
3. 法务、财务、平台政策等专业判断。
4. 真实执行后的复盘和迭代。

后续需要补充：

1. SOP 标准模板页。
2. SOP 评审清单。
3. SOP 版本管理规则。
4. 新人培训 SOP 模板。
5. Amazon SOP 模板。
6. GBrain 运维 SOP 模板。
7. 权限管理 SOP 模板。

## 12. 本页的维护规则

当 NexGaios 的组织流程、工具链、权限系统、知识库结构或 Terry 的管理要求发生变化时，Codex 应更新本页。

当某个 SOP 在真实执行中出现重复问题时，Codex 应把问题写入对应复盘页面，并更新本页或相关 SOP 模板。

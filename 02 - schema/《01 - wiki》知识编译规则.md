# 01 - wiki 知识编译规则

版本：v0.7.0

生命周期：Draft

日期：2026-06-20

适用范围：`00 - raw` 到 `01 - wiki`

## 1. 规则边界

本规则只做：

1. 判断 raw 是否值得进入 wiki 编译。
2. 规定 wiki 页面形态。
3. 规定 candidate、active、deprecated、rejected 的使用边界。
4. 规定每次编译必须输出的判定结果。
5. 规定 wiki 页面写入、证据引用、链接和冲突处理方式。

本规则不做：

1. 不规定 raw 目录命名。
2. 不规定 inbox 路由分发。
3. 不直接同步 GitHub、Supabase 或 GBrain DB。
4. 不把 raw 原文复制成 wiki。
5. 不美化 Obsidian 图谱。

`raw` 保存原始材料和证据；`wiki` 保存可复用、可追溯、可更新的企业知识网络。

## 2. Raw 进入 Wiki 准入规则

Raw 满足以下条件中的两项及以上，才进入 wiki 编译：

1. 后续会被重复使用。
2. 可以更新已有 wiki 页面。
3. 可以形成稳定知识对象。
4. 会影响业务、项目或 Agent 执行。
5. 可以连接多个 raw、主题或页面。
6. 具备可追溯证据。

以下 raw 不进入 wiki：

1. 重复剪藏。
2. 一次性参考。
3. 没有结论的临时对话。
4. 没有复用价值的过程记录。
5. 已过期且没有追溯价值的材料。
6. 信息过碎，不能形成概念、流程、方案、决策、案例或主题。

进入 wiki 不是复制 raw，而是触发以下编译动作之一：

1. 新建 wiki 页面。
2. 更新已有 wiki 页面。
3. 补充证据。
4. 标记冲突。
5. 降级旧知识。
6. 仅保留 raw。

## 3. Wiki 页面形态规则

第一版 wiki 页面收敛为 10 类：

| 页面形态 | 用途 |
| --- | --- |
| 索引页 | 导航入口和知识地图。 |
| 对象页 | 概念、实体、工具、系统、产品、站点、SKU、政策对象。 |
| 主题页 | 跨资料综合一个主题。 |
| 项目页 | 项目主页和长期沉淀。 |
| 流程页 | 可重复执行流程，内部用 `process_level` 区分指南和 SOP。 |
| 决策页 | 方案取舍和理由。 |
| 政策页 | 外部政策、内部规范、平台规则。 |
| 模板页 | 可复用格式、提示词、文档模板。 |
| 案例页 | 成功案例、失败案例、复盘。 |
| 对比页 | 方案、工具、路径、产品对比。 |

编译时必须先判断 raw 应进入哪一种 wiki 页面形态。

已有 wiki 页面可以承载时，必须优先更新已有页面。

只有已有 wiki 无法承载时，才新建页面。

页面形态不是 raw 的 `material_type`，也不是目录分类。

## 4. Wiki 安全自动编译与晋升规则

Wiki 使用以下状态：

```text
candidate
active
deprecated
rejected
```

状态含义：

| 状态 | 含义 |
| --- | --- |
| candidate | 已从 raw 编译出来，但不能作为正式知识默认使用。 |
| active | 当前可用，允许用户和 Agent 默认使用。 |
| deprecated | 曾经有效，但已过期、被替代或被推翻。 |
| rejected | 确认不应进入 wiki 的候选知识。 |

Agent 可以自动生成 `candidate`。

以下内容禁止自动进入 `active`：

1. 证据不足。
2. 与已有 active 结论冲突。
3. 涉及权限、安全、财务、合规、商业决策或组织规则。
4. 会改变已有流程、规则或业务判断。
5. 无法追溯到 raw。

`candidate` 晋升为 `active` 必须满足：

1. raw 证据存在。
2. 知识对象清晰。
3. 不存在未解决冲突。
4. 不覆盖 active 结论。
5. 至少连接一个已有 active 页面。
6. 页面形态合法。
7. 证据来源可追溯。

## 5. Wiki 知识生命周期规则

Wiki 不是永久正确答案库，而是当前可用、可追溯、可更新、可降级的企业知识网络。

新 raw 进入时，只允许触发以下动作：

1. 只保留 raw。
2. 新建 candidate 页面。
3. 更新已有 candidate 页面。
4. 更新 active 页面。
5. 标记 active 页面为 deprecated。
6. 标记冲突，禁止自动覆盖。
7. 拒绝候选页面。

只有满足以下条件，才能自动更新 active：

1. 新 raw 与原 active 结论不冲突。
2. 更新是补充信息、事实修正、来源补强或表达优化。
3. 更新后不改变业务判断、权限规则、流程规则、财务判断或合规判断。
4. 更新内容能追溯到 raw。

当新 raw 证明旧 wiki 不再适用时，不删除旧 wiki，而是降级为 `deprecated`。

降级时必须记录：

```text
废弃原因
废弃时间
替代页面
依据来源
```

如果新 raw 与 active 冲突：

1. 禁止自动覆盖 active。
2. 在相关 wiki 页面写入冲突记录。
3. 新结论进入 candidate。
4. 等待更多证据或用户确认。

## 6. Wiki 编译输出契约规则

每次编译必须输出固定判定单。

判定单字段固定为：

```text
原始材料
编译决策
受影响 Wiki 页面
Wiki 页面形态
执行动作
目标状态
证据来源
是否冲突
判断依据
风险等级
是否需要用户确认
```

字段含义：

| 字段 | 含义 |
| --- | --- |
| 原始材料 | 本次处理的 raw 文件或材料包。 |
| 编译决策 | 进入 wiki、仅保留 raw、暂缓处理。 |
| 受影响 Wiki 页面 | 需要新建、更新、补证据、标冲突或降级的页面。 |
| Wiki 页面形态 | 索引页、对象页、主题页、项目页、流程页、决策页、政策页、模板页、案例页、对比页。 |
| 执行动作 | 新建、更新、补充证据、标记冲突、降级旧页面、拒绝候选、仅记录索引。 |
| 目标状态 | candidate、active、deprecated、rejected。 |
| 证据来源 | 支撑判断的 raw 文件。 |
| 是否冲突 | 是否与已有 active 页面冲突。 |
| 判断依据 | 具体证据，不允许写空泛判断。 |
| 风险等级 | 低、中、高。 |
| 是否需要用户确认 | 是、否。 |

风险等级规则：

| 风险等级 | 判断标准 |
| --- | --- |
| 低 | 补充事实、补充来源、优化表达，不改变结论。 |
| 中 | 形成新结论、更新流程、影响项目理解。 |
| 高 | 影响权限、安全、财务、合规、商业决策或组织规则。 |

高风险、冲突、降级 active、重大业务判断，必须设置为：

```text
是否需要用户确认：是
```

raw 编译完成后，必须同步更新 raw frontmatter 中的 `compile_status`。

raw 编译状态字段以以下规则为准：

```text
02 - schema/《00 - raw》编译状态规则.md
```

## 7. Wiki 页面写入规则

Wiki 写结论、结构、关系、适用范围和证据。

Raw 保存原文、细节和材料。

禁止把 raw 简单摘要成 wiki。

Wiki 页面最小结构：

```markdown
## 当前结论

## 适用范围

## 依据来源

## 相关链接
```

存在冲突时，必须增加：

```markdown
## 冲突与待确认
```

存在结论演化时，必须增加：

```markdown
## 变更记录
```

`当前结论` 必须可执行。

`适用范围` 必须写清楚适用场景和不适用场景。

`依据来源` 必须链接到 raw。

`相关链接` 只链接真实知识关系，不为凑数量建立链接。

## 8. Wiki 链接与网络规则

Wiki 链接只表达真实知识关系，不表达弱相关。

有效关系包括：

```text
上位知识
下位知识
前置流程
后续流程
同一项目
同一系统
同一决策
替代关系
冲突关系
证据来源
```

禁止因为以下原因建立链接：

1. 都提到了 AI。
2. 标题有相同词。
3. 同一天创建。
4. 同一个作者。
5. 同一个大目录。
6. 模型觉得可能相关。

链接必须写出关系类型。

错误写法：

```markdown
相关：[[Agent Memory]]
```

正确写法：

```markdown
- 前置概念：[[Agent Memory]]
- 后续流程：[[Inbox 路由分发]]
- 冲突页面：[[旧版知识编译方案]]
- 证据来源：[[某条 raw 文件]]
```

`wiki -> wiki` 是主知识网络。

`wiki -> raw` 是证据引用。

孤立页面只能保持 `candidate`，不能直接进入 `active`。

如果 A 替代 B，必须双向维护：

```markdown
A 页面：替代：[[B]]
B 页面：已被替代：[[A]]
```

## 9. Wiki Frontmatter 字段规则

Wiki frontmatter 只记录知识状态、页面形态、生命周期、风险和证据来源。

Wiki frontmatter 不记录 raw 材料属性，不重复记录目录路径。

### 9.1 必填字段

必填字段固定为：

```yaml
---
title:
status:
wiki_page_type:
created:
updated:
source_refs: []
risk_level:
tags: []
---
```

字段含义：

| 字段 | 含义 |
| --- | --- |
| `title` | Wiki 页面标题。 |
| `status` | Wiki 知识状态。只允许：`candidate`、`active`、`deprecated`、`rejected`。 |
| `wiki_page_type` | Wiki 页面形态。只允许：`索引页`、`对象页`、`主题页`、`项目页`、`流程页`、`决策页`、`政策页`、`模板页`、`案例页`、`对比页`。 |
| `created` | Wiki 页面创建时间。 |
| `updated` | Wiki 页面最后更新时间。 |
| `source_refs` | 支撑该页面结论的 raw 来源引用。 |
| `risk_level` | 该页面知识被错误使用时的风险等级。只允许：`低`、`中`、`高`。 |
| `tags` | 横向检索标签，用于补充主题词，不替代 Wiki 链接。 |

### 9.2 条件必填字段

条件必填字段只在满足条件时出现。

| 字段 | 含义 |
| --- | --- |
| `process_level` | 仅流程页使用。区分 `指南` 和 `SOP`。 |
| `promoted_at` | 仅 `active` 页面使用。记录晋升为 active 的时间。 |
| `promotion_basis` | 仅 `active` 页面使用。记录晋升依据。 |
| `deprecated_at` | 仅 `deprecated` 页面使用。记录降级为 deprecated 的时间。 |
| `deprecated_reason` | 仅 `deprecated` 页面使用。记录废弃原因。 |
| `replaced_by` | 仅 `deprecated` 页面使用。记录替代它的新 Wiki 页面。 |
| `rejected_reason` | 仅 `rejected` 页面使用。记录拒绝原因。 |

流程页必须增加：

```yaml
process_level:
```

允许值：

```text
指南
SOP
```

`status: active` 时必须增加：

```yaml
promoted_at:
promotion_basis:
```

`promotion_basis` 允许值：

```text
用户确认
多源证据
使用频率
稳定性
```

`status: deprecated` 时必须增加：

```yaml
deprecated_at:
deprecated_reason:
replaced_by:
```

`status: rejected` 时必须增加：

```yaml
rejected_reason:
```

### 9.3 禁止字段

第一版 wiki frontmatter 不使用以下字段：

```text
domain
source_type
material_type
confidence
owner
current_path
```

禁止原因：

| 字段 | 禁止原因 |
| --- | --- |
| `domain` | 容易和目录重复。 |
| `source_type` | 属于 raw 进入来源，不属于 wiki 知识属性。 |
| `material_type` | 属于 raw 材料类型，不属于 wiki 页面形态。 |
| `confidence` | 容易和 `status`、`risk_level` 混用。 |
| `owner` | 当前没有多人协作权责场景。 |
| `current_path` | 路径已经由文件位置表达，重复维护会出错。 |

## 10. Wiki 页面模板规则

新建 wiki 页面必须以以下模板为起点：

```text
02 - schema/模板/01 - wiki 页面模板.md
```

模板只提供页面骨架，不替代编译判断。

使用模板时必须执行：

1. 填写必填 frontmatter。
2. 按页面状态和页面形态保留条件字段。
3. 删除不适用的条件字段。
4. 删除模板说明区。
5. 删除不适用的正文小节。
6. 将 `依据来源` 指向 raw。
7. 将 `相关链接` 写成类型化链接。

禁止行为：

1. 不得把模板文件复制到 `01 - wiki` 后不填写字段。
2. 不得保留模板说明文字作为业务知识正文。
3. 不得用模板绕过 raw 准入、编译输出、生命周期和链接规则。

## 11. Wiki 写入执行验收规则

每次新建或更新 wiki 页面后，必须执行写入验收。

### 11.1 页面结构验收

必须检查：

1. frontmatter 必填字段是否齐全。
2. `status` 是否属于允许值。
3. `wiki_page_type` 是否属于 10 类页面形态。
4. `risk_level` 是否属于 `低 / 中 / 高`。
5. `source_refs` 是否指向 raw。
6. `相关链接` 是否使用类型化链接。
7. 页面是否保留了模板说明文字。

无真实冲突时，禁止保留空的 `冲突与待确认` 小节。

无结论演化时，禁止保留空的 `变更记录` 小节。

### 11.2 补丁污染验收

使用补丁工具写入多个文件后，必须扫描本次触达文件。

禁止页面正文中出现以下补丁标记：

```text
*** Add File
*** Update File
*** End Patch
```

如果发现补丁标记进入 Markdown 正文，必须立即停止后续写入，并按以下顺序处理：

1. 定位污染开始行。
2. 删除污染片段。
3. 将原本应写入的内容拆成独立补丁重新写入。
4. 重新扫描确认无补丁标记残留。
5. 再更新索引和治理记录。

### 11.3 多文件写入规则

一次写入多个 wiki 页面、索引和治理记录时，必须优先使用小补丁分批执行。

禁止把“新增文件”和“更新其他文件”的补丁内容作为 Markdown 正文写入同一个页面。

如果一次补丁只成功写入部分文件，必须先完成半写入修复，再继续后续动作。

### 11.4 索引同步验收

新增 candidate 页面后，必须检查：

1. `_index.md` 是否列出该 candidate。
2. `_meta/编译日志.md` 是否记录该编译动作。
3. `_meta/待处理索引.md` 是否记录待用户确认事项。
4. 如果页面引用了尚未创建的下位页面，必须在 `_meta/待处理索引.md` 记录补建事项。
5. 如果下位页面已经创建，必须移除或更新对应“尚未创建”的待处理事项。

### 11.5 Active 可用性验收

candidate 晋升为 active 后，必须执行 active 可用性验收。

active 可用性验收的细则以以下规则为准：

```text
02 - schema/《01 - wiki》质量验收规则.md
```

质量验收未通过时，禁止把页面作为 active 知识默认使用。

## 12. 后续同步要求

新增或修改 wiki 编译脚本、skill 或定时任务时，必须检查本规则是否需要同步。

修改以下内容时，必须同步更新本规则和规则变更日志：

1. 页面形态。
2. 状态含义。
3. 编译输出字段。
4. 风险等级。
5. active 晋升条件。
6. 冲突处理方式。
7. 页面写入结构。
8. 链接关系规则。
9. Wiki frontmatter 字段。
10. Wiki 页面模板。
11. Wiki 写入执行验收规则。
12. Active 可用性验收规则。
13. Wiki 质量验收规则。
14. Raw 编译状态同步规则。

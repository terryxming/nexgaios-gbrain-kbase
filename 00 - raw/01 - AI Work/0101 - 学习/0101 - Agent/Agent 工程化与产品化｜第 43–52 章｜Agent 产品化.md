# Agent 工程化与产品化｜第 43–52 章：Agent 产品化

本阶段目标：把 Agent 从“工程能力”升级成“真实用户愿意使用、业务愿意投入、团队能持续迭代的产品”。

| 能力目标 | 具体表现 |
|---|---|
| 能定义产品化 | 知道 Agent 产品化不是做一个聊天框，而是把能力嵌入真实任务 |
| 能选场景 | 能判断哪些场景适合做 Agent，哪些不适合 |
| 能建模任务 | 能把用户目标拆成任务、动作、判断点和交付物 |
| 能选形态 | 能判断做 Chatbot、Copilot、Workflow Agent、Autopilot 还是 Platform |
| 能设计交互 | 能设计确认、预览、修改、撤销、人工接管 |
| 能建立信任 | 能让用户知道 Agent 做了什么、为什么这么做、风险在哪里 |
| 能设计指标 | 能用完成率、采纳率、节省时间、复用率衡量价值 |
| 能评估商业价值 | 能判断一个 Agent 是否值得投入 |
| 能做 MVP | 能从最小可用产品开始迭代 |
| 能平台化 | 能从单点内部工具走向团队级 Agent 平台 |

---

# 第 43 章｜Agent 产品化总览

## 43.1 一句话结论

**Agent 产品化，是把 Agent 的理解、规划、工具调用和执行能力，包装成真实用户能稳定使用、能产生业务价值、能持续迭代的产品系统。**

Agent 工程化关注：

```text
能不能稳定做事
```

Agent 产品化关注：

```text
有没有人愿意用
用完有没有价值
能不能嵌入真实工作流
能不能长期运营和改进
```

OpenAI Agents 文档把 Agent 描述为能规划、调用工具、协作并保持足够状态以完成多步骤工作的应用。这说明 Agent 本身已经接近“任务应用”，但从应用能力到产品，还需要用户场景、交互、指标和运营体系。

---

## 43.2 费曼解释

工程化 Agent 像一个“训练好的员工”。

产品化 Agent 像一个“可交付的岗位系统”。

一个员工能力再强，如果：

- 用户不知道什么时候找它
- 找它之后不知道怎么提需求
- 它做了什么用户看不懂
- 结果不敢用
- 出错无法撤回
- 没有指标证明价值

那它就不是一个好产品。

所以 Agent 产品化不是问：

```text
这个 Agent 能做什么？
```

而是问：

```text
谁在什么场景下，为什么会持续使用它？
它解决了什么成本、效率、质量或增长问题？
```

---

## 43.3 Agent 产品化的核心转变

| 维度 | 工程视角 | 产品视角 |
|---|---|---|
| 起点 | 我能做一个 Agent | 用户有什么高价值任务 |
| 核心问题 | 能不能完成任务 | 用户愿不愿意用、敢不敢用 |
| 关注对象 | 模型、工具、流程 | 用户、场景、价值、信任 |
| 输出 | 功能可用 | 结果可采纳 |
| 质量标准 | 测试通过、调用成功 | 用户完成任务、节省时间、降低错误 |
| 迭代依据 | 技术缺陷 | 用户反馈 + 业务指标 |
| 成功标志 | Demo 能跑 | 用户持续使用并产生价值 |

---

## 43.4 Agent 产品化四件套

| 模块 | 作用 |
|---|---|
| 场景 | 明确用户在什么任务中使用 |
| 形态 | 决定 Agent 以什么方式出现 |
| 交互 | 设计用户如何下达、确认、修改、接管 |
| 指标 | 衡量它是否真的有价值 |

可以压缩成：

```text
Agent Product = 场景 × 形态 × 交互 × 指标
```

---

## 43.5 产品级 Agent 的基本标准

| 标准 | 问题 |
|---|---|
| 有明确用户 | 谁在用？ |
| 有高价值任务 | 解决什么痛点？ |
| 有稳定入口 | 用户从哪里启动？ |
| 有清晰输入 | 用户需要提供什么？ |
| 有可交付输出 | 最终得到什么？ |
| 有信任机制 | 用户为什么敢用？ |
| 有安全边界 | 哪些动作不能自动执行？ |
| 有效果指标 | 怎么证明有价值？ |
| 有反馈闭环 | 如何持续改进？ |

---

## 43.6 本章误区

| 误区 | 正确理解 |
|---|---|
| Agent 产品 = 聊天框 | 聊天框只是入口之一 |
| Agent 越自动越好 | 自动化程度要和风险匹配 |
| 用户会主动探索功能 | 产品要嵌入真实工作流 |
| 技术强就会有人用 | 用户只关心任务是否更快、更准、更省心 |
| 一开始就做平台 | 先从单点高价值场景验证 |
| 产品化只是 UI 包装 | 产品化包含场景、信任、指标、运营 |

# 第 44 章｜用户场景定义

## 44.1 一句话结论

**Agent 产品化的起点不是技术能力，而是高频、高价值、高痛点、可闭环的用户场景。**

一个场景是否值得做 Agent，先看：

```text
用户是谁
任务是什么
痛点有多强
频率有多高
结果是否可评估
风险是否可控制
是否能嵌入现有工作流
```

---

## 44.2 场景定义模板

```markdown
# Agent 场景定义

## 1. 用户是谁
- 角色：
- 能力水平：
- 使用频率：
- 当前工作方式：

## 2. 当前任务是什么
- 任务名称：
- 触发时机：
- 输入资料：
- 输出结果：

## 3. 当前痛点是什么
- 慢：
- 难：
- 容易错：
- 不稳定：
- 依赖专家：

## 4. Agent 介入点
- 在哪个节点介入？
- 替代什么动作？
- 增强什么判断？
- 输出什么结果？

## 5. 成功标准
- 节省多少时间？
- 提高什么质量？
- 降低什么错误？
- 产生什么业务结果？
```

---

## 44.3 好场景的特征

| 特征 | 说明 | 示例 |
|---|---|---|
| 高频 | 经常发生 | 每周广告诊断 |
| 高痛点 | 用户明显不想做 | 手工整理报表 |
| 有资料 | Agent 有上下文可用 | 报表、文档、历史记录 |
| 可拆解 | 任务能分步骤 | 读取、分析、总结、生成 |
| 可评估 | 能判断好坏 | 是否找到根因 |
| 风险可控 | 出错后可修改或撤销 | 先生成建议，不直接执行 |
| 能嵌入 | 不破坏原工作流 | 在报表旁生成诊断 |
| 有复用 | 多次任务结构相似 | 每个广告周期都可复用 |

---

## 44.4 不适合做 Agent 的场景

| 场景 | 原因 |
|---|---|
| 极低频任务 | 投入产出不高 |
| 规则完全固定 | 直接写 Workflow 或脚本更好 |
| 没有可用数据 | Agent 只能猜 |
| 结果无法评估 | 无法迭代质量 |
| 高风险且不可撤销 | 不适合自动执行 |
| 用户不愿授权 | Agent 拿不到工具和数据 |
| 需求非常模糊 | 先做需求澄清，不直接做 Agent |
| 只是为了“看起来 AI 化” | 没有真实价值 |

---

## 44.5 场景优先级矩阵

| 维度 | 低 | 高 |
|---|---|---|
| 任务频率 | 偶尔发生 | 每天/每周发生 |
| 任务价值 | 可有可无 | 影响成本、收入、质量 |
| 自动化空间 | 很难拆 | 可拆成稳定流程 |
| 数据可获得性 | 没资料 | 有文件/API/数据库 |
| 风险可控性 | 不可撤销 | 可预览、可确认、可回滚 |
| 用户接受度 | 不信任 | 愿意尝试、愿意反馈 |

优先做：

```text
高频 × 高痛点 × 有资料 × 可评估 × 风险可控
```

# 第 45 章｜用户任务建模

## 45.1 一句话结论

**用户任务建模，是把“用户想要的结果”拆成目标、输入、动作、判断、输出、验收和反馈。**

用户通常表达的是愿望：

```text
帮我优化广告
帮我整理知识
帮我做一个 Agent
帮我分析竞品
```

产品设计要把它变成任务模型：

```text
目标是什么
需要什么输入
分哪些步骤
哪些由系统做
哪些由 Agent 判断
哪些由用户确认
最终交付什么
如何判断好坏
```

---

## 45.2 用户目标、任务、动作的区别

| 概念 | 简单理解 | 示例 |
|---|---|---|
| 目标 | 用户想达成的结果 | 降低 ACOS |
| 任务 | 为达成目标要完成的一组工作 | 分析广告表现 |
| 动作 | 任务中的具体步骤 | 读取报表、计算 CVR、定位 keyword |
| 判断 | 需要智能参与的部分 | 判断 ACOS 上升主因 |
| 输出 | 用户拿到的交付物 | 诊断报告 + 优化动作 |
| 验收 | 判断是否完成 | 是否有证据、对象、动作、风险 |

---

## 45.3 任务建模七段式

| 段 | 问题 |
|---|---|
| Trigger | 用户什么时候启动任务？ |
| Input | 用户或系统提供什么信息？ |
| Context | Agent 需要参考什么？ |
| Process | 任务分几步？ |
| Decision | 哪些步骤需要判断？ |
| Output | 最终交付什么？ |
| Feedback | 用户如何确认、修改、采纳或拒绝？ |

---

## 45.4 示例：知识沉淀 Agent 任务模型

| 模块 | 内容 |
|---|---|
| Trigger | 用户说“把这段对话沉淀成 .md 文件” |
| Input | 当前对话、指定章节、知识库要求 |
| Context | 用户偏好、课程大纲、已有章节文件 |
| Process | 提炼主题 → 重组结构 → 删除闲聊 → 生成 Markdown |
| Decision | 哪些内容保留，哪些内容删除，如何分层 |
| Output | `.md` 文件 + 下载链接 |
| Feedback | 用户下载、放入 llm-wiki、要求重整或补充 |

---

## 45.5 示例：Amazon 广告诊断 Agent 任务模型

| 模块 | 内容 |
|---|---|
| Trigger | 用户发现 ACOS、CTR、CVR、CPC 异常 |
| Input | 广告报表、时间区间、ASIN、目标关键词 |
| Context | Listing 状态、价格、竞品、历史操作 |
| Process | 字段检查 → 指标计算 → 异常识别 → 根因判断 → 建议生成 |
| Decision | 判断是流量结构、转化、出价、预算还是竞争导致 |
| Output | 根因表、证据表、动作建议、风险提示 |
| Feedback | 用户采纳、调整、标记无效建议，进入后续迭代 |

# 第 46 章｜产品形态选择

## 46.1 一句话结论

**Agent 产品形态不是默认聊天框，而是根据任务频率、风险、复杂度、用户控制需求和系统嵌入程度来选择。**

常见形态：

| 形态 | 简单理解 |
|---|---|
| Chatbot | 对话问答 |
| Copilot | 辅助用户完成任务 |
| Workflow Agent | 嵌入固定流程中的智能节点 |
| Autopilot | 用户设定目标后自动执行 |
| Embedded Agent | 嵌入已有业务系统 |
| Dashboard Agent | 在看板中解释、诊断、建议 |
| Agent Platform | 多 Agent、多工具、多权限统一平台 |

---

## 46.2 形态选择表

| 任务特征 | 推荐形态 |
|---|---|
| 用户只是问概念 | Chatbot |
| 用户需要建议但自己决策 | Copilot |
| 流程稳定但部分步骤要判断 | Workflow Agent |
| 低风险重复任务 | Autopilot |
| 用户已经在某个系统工作 | Embedded Agent |
| 任务依赖数据监控和趋势 | Dashboard Agent |
| 多团队、多任务、多工具 | Agent Platform |

---

## 46.3 Chatbot

| 维度 | 内容 |
|---|---|
| 优点 | 上手快、灵活、开发成本低 |
| 缺点 | 入口泛、任务闭环弱、用户容易不知道怎么问 |
| 适合 | 学习、解释、FAQ、轻量总结 |
| 不适合 | 高风险执行、复杂业务流程 |

判断：

```text
如果用户主要想问问题，Chatbot 可以。
如果用户主要想完成任务，Chatbot 通常不够。
```

---

## 46.4 Copilot

| 维度 | 内容 |
|---|---|
| 优点 | 人主导，风险可控 |
| 缺点 | 自动化程度有限 |
| 适合 | 文案、数据分析、代码、客服建议、运营诊断 |
| 核心设计 | AI 提建议，人类确认和修改 |

示例：

- Listing 优化 Copilot
- Amazon 广告诊断 Copilot
- A+ 文案 Copilot
- Codex 开发 Copilot

---

## 46.5 Workflow Agent

| 维度 | 内容 |
|---|---|
| 优点 | 稳定、可控、适合产品化 |
| 缺点 | 灵活性不如开放 Agent |
| 适合 | 知识沉淀、报表诊断、工单处理、内容生产 |
| 核心设计 | 固定步骤 + 智能判断节点 |

推荐原则：

```text
能 Workflow 化的，不要全交给自由 Agent。
```

---

## 46.6 Autopilot

| 维度 | 内容 |
|---|---|
| 优点 | 效率高、自动化强 |
| 缺点 | 风险高、需要强治理 |
| 适合 | 低风险、规则清晰、可回滚任务 |
| 不适合 | 对外发送、支付、删除、改预算等高风险任务 |

Autopilot 必备条件：

| 条件 | 说明 |
|---|---|
| 明确目标 | 系统知道要优化什么 |
| 清晰边界 | 哪些动作不能做 |
| 可追踪 | 每一步有记录 |
| 可暂停 | 高风险点能停 |
| 可回滚 | 执行错误能恢复 |
| 可告警 | 异常能通知 |

---

## 46.7 Embedded Agent

| 维度 | 内容 |
|---|---|
| 优点 | 嵌入用户原工作流，使用阻力低 |
| 缺点 | 需要系统集成 |
| 适合 | CRM、ERP、广告后台、知识库、工单系统 |
| 核心设计 | 在用户已有任务节点增强能力 |

示例：

```text
广告报表页面旁边：
[诊断异常]
[生成优化建议]
[导出周报]
```

Embedded Agent 往往比独立聊天框更容易产品化，因为它不要求用户切换工作环境。

# 第 47 章｜交互设计

## 47.1 一句话结论

**Agent 交互设计的核心不是“让用户聊天”，而是让用户能清楚地下达目标、理解过程、确认风险、修改结果、撤销动作。**

Agent 交互必须解决五件事：

```text
怎么开始
怎么补充信息
怎么看过程
怎么确认动作
怎么修正结果
```

---

## 47.2 Agent 交互五段式

| 阶段 | 用户问题 | 产品设计 |
|---|---|---|
| Start | 我该怎么用？ | 入口、模板、示例任务 |
| Clarify | 它需要我补什么？ | 最少必要追问 |
| Preview | 它打算做什么？ | 计划预览、影响说明 |
| Confirm | 我是否同意执行？ | 确认、修改、取消 |
| Review | 结果能不能用？ | 证据、风险、编辑、反馈 |

---

## 47.3 好的启动入口

差的入口：

```text
请输入你的问题。
```

好的入口：

```text
你想让 Agent 做什么？

1. 诊断广告异常
2. 生成 Listing 优化建议
3. 整理对话成 Markdown
4. 生成客服回复草稿
5. 拆解 PRD 开发任务
```

入口设计原则：

| 原则 | 说明 |
|---|---|
| 任务化 | 不只是空白聊天框 |
| 模板化 | 给用户示例 |
| 场景化 | 按用户工作流分类 |
| 低认知负担 | 用户不用学 prompt |
| 可扩展 | 高级用户仍可自由输入 |

---

## 47.4 最少必要追问

Agent 不应该一遇到不确定就问很多问题。

应该区分：

| 信息类型 | 处理方式 |
|---|---|
| 关键缺失 | 必须追问 |
| 可默认 | 使用默认值并说明 |
| 可推断 | 标记为假设 |
| 无关信息 | 不追问 |

示例：

用户说：

```text
帮我分析广告为什么 ACOS 升高。
```

必要追问：

```text
请提供：
1. 当前周期广告报表；
2. 对比周期；
3. 希望分析的层级：账户 / campaign / keyword / search term。
```

不必要追问：

```text
你想要严肃风格还是轻松风格？
```

---

## 47.5 计划预览

在执行多步骤任务前，Agent 应该让用户看到计划。

```markdown
我将按以下步骤处理：

1. 检查数据字段是否完整
2. 计算 CTR、CVR、CPC、ACOS 变化
3. 定位变化最大的 campaign / keyword / search term
4. 判断 ACOS 上升主因
5. 输出建议动作和风险提示

不会执行：
- 不会直接修改广告
- 不会暂停 campaign
- 不会调整预算
```

计划预览的价值：

| 价值 | 说明 |
|---|---|
| 降低不确定性 | 用户知道 Agent 要做什么 |
| 建立信任 | 过程透明 |
| 便于纠偏 | 用户可提前修改方向 |
| 降低风险 | 高风险动作提前暴露 |

---

## 47.6 确认、修改、取消

高风险动作前，不要只问：

```text
是否继续？
```

应该给用户明确选项：

```markdown
即将创建一封客服邮件草稿：

| 项目 | 内容 |
|---|---|
| 收件人 | customer@example.com |
| 主题 | About your KaraFun activation |
| 动作 | 仅创建草稿，不发送 |

请选择：
1. 创建草稿
2. 修改内容
3. 取消
```

OpenAI 的 Guardrails and human review 文档说明，guardrails 可以做自动校验，human review 可以在敏感动作前暂停运行，等待人工批准或拒绝。这正是 Agent 产品交互中“先预览、再确认”的工程基础。

---

## 47.7 撤销与回滚

只要 Agent 能执行写入动作，就应该优先考虑撤销。

| 动作 | 撤销方式 |
|---|---|
| 生成草稿 | 删除草稿 |
| 创建任务 | 关闭或删除任务 |
| 修改文档 | 版本回滚 |
| 修改配置 | 恢复旧配置 |
| 发邮件 | 通常不可撤销，所以必须强确认 |
| 删除文件 | 进回收站，不直接永久删除 |

原则：

```text
不可撤销动作，确认等级必须更高。
```

# 第 48 章｜信任机制设计

## 48.1 一句话结论

**Agent 产品的信任不是靠“说自己很可靠”，而是靠透明过程、证据来源、风险提示、用户控制和稳定表现建立。**

Google People + AI Guidebook 的信任章节强调，AI 产品中的用户信任需要被拆解和校准；用户信任建立很慢，但当预期没有被满足时很容易丧失。

---

## 48.2 用户为什么不信任 Agent

| 原因 | 表现 |
|---|---|
| 不知道它怎么得出结论 | 结果看起来像黑箱 |
| 不知道它用了什么资料 | 担心依据错 |
| 不知道它会不会乱执行 | 担心越权 |
| 不知道它是否有风险 | 担心后果 |
| 之前被 AI 编造误导过 | 不敢直接采纳 |
| 输出太自信 | 反而降低信任 |
| 无法修改和撤销 | 不敢让它做事 |

---

## 48.3 信任机制六件套

| 机制 | 作用 |
|---|---|
| 证据 | 告诉用户结论依据 |
| 来源 | 告诉用户资料来自哪里 |
| 置信度 | 告诉用户判断把握有多大 |
| 风险提示 | 告诉用户可能有什么后果 |
| 用户控制 | 允许确认、修改、拒绝、撤销 |
| 稳定体验 | 多次使用表现一致 |

---

## 48.4 解释不是越多越好

差的解释：

```text
我根据复杂的多维数据综合判断，认为可能是流量结构、转化率、竞价环境等多个因素共同导致。
```

好的解释：

```markdown
ACOS 上升的主因更可能是 CVR 下降。

证据：
| 指标 | 上期 | 本期 | 变化 |
|---|---:|---:|---:|
| CPC | 3.11 | 3.18 | +2.3% |
| CVR | 5.5% | 3.8% | -30.9% |
| ACOS | 42% | 61% | +19pp |

判断：
- CPC 变化不大；
- CVR 明显下降；
- 因此优先检查 Listing 转化、流量相关性和搜索词质量。
```

信任来自：

```text
短解释 + 明证据 + 可行动
```

---

## 48.5 置信度设计

| 置信度 | 使用场景 | 输出方式 |
|---|---|---|
| High | 数据充分、证据一致 | 可以直接建议动作 |
| Medium | 数据基本支持，但有缺失 | 建议 + 需要验证 |
| Low | 数据不足或冲突 | 标记假设，不建议执行 |

示例：

```markdown
置信度：Medium

原因：
- 报表显示 CVR 明显下降；
- 但缺少 Listing 页面变化、价格变化和竞品促销数据；
- 因此当前判断是“优先假设”，不是最终结论。
```

---

## 48.6 信任与自动化程度

| 信任程度 | 适合自动化程度 |
|---|---|
| 低信任 | 只解释、只建议 |
| 中信任 | 生成草稿、生成方案 |
| 高信任 | 自动执行低风险任务 |
| 已验证长期可靠 | 可自动执行部分可回滚任务 |

原则：

```text
不要一开始就让 Agent 自动驾驶。
先做 Copilot，建立信任后再逐步自动化。
```

# 第 49 章｜指标体系

## 49.1 一句话结论

**Agent 产品指标要同时衡量：任务是否完成、用户是否采纳、质量是否稳定、效率是否提升、业务是否受益。**

不能只看：

```text
调用次数
聊天次数
生成次数
```

这些是使用量，不等于价值。

---

## 49.2 Agent 产品指标分层

| 层级 | 指标问题 | 示例指标 |
|---|---|---|
| 使用层 | 有没有人用 | DAU、WAU、任务启动数 |
| 完成层 | 任务有没有完成 | task completion rate |
| 采纳层 | 用户是否接受 | acceptance rate、edit rate |
| 质量层 | 结果是否好 | eval score、error rate |
| 效率层 | 是否节省时间 | time saved、cycle time reduction |
| 业务层 | 是否产生价值 | 成本下降、转化提升、错误减少 |
| 安全层 | 是否可控 | guardrail trigger rate、incident count |
| 运营层 | 是否稳定 | latency、cost per run、failure rate |

---

## 49.3 核心指标定义

| 指标 | 公式 / 含义 |
|---|---|
| Task Start Rate | 用户启动任务次数 |
| Task Completion Rate | 完成任务数 / 启动任务数 |
| User Acceptance Rate | 用户采纳结果数 / 完成任务数 |
| Edit Rate | 用户修改结果数 / 完成任务数 |
| Regeneration Rate | 用户要求重试数 / 完成任务数 |
| Escalation Rate | 转人工数 / 启动任务数 |
| Time Saved | 原流程耗时 - Agent 流程耗时 |
| Cost per Run | 单次任务平均模型 + 工具成本 |
| Failure Rate | 失败任务数 / 启动任务数 |
| Safety Incident Rate | 安全事故数 / 启动任务数 |

---

## 49.4 不同 Agent 的指标示例

### 知识沉淀 Agent

| 指标 | 意义 |
|---|---|
| 文件生成成功率 | 是否稳定生成 `.md` |
| 用户二次修改率 | 文件是否需要大量手改 |
| 入库率 | 用户是否真的放入知识库 |
| 复用率 | 之后是否被检索和引用 |
| 结构合格率 | 是否符合知识库模板 |

### 广告诊断 Agent

| 指标 | 意义 |
|---|---|
| 诊断完成率 | 是否能完成分析 |
| 根因命中率 | 是否找到真实主因 |
| 建议采纳率 | 用户是否执行建议 |
| 后续指标改善 | ACOS、CVR、CTR 是否改善 |
| 风险误判率 | 是否给出危险建议 |

### 客服回复 Agent

| 指标 | 意义 |
|---|---|
| 草稿生成率 | 是否能生成回复 |
| 人工编辑率 | 回复是否接近可用 |
| 首响时间下降 | 是否提高响应效率 |
| 客诉升级率 | 是否降低误回复风险 |
| 合规违规率 | 是否有不当承诺 |

---

## 49.5 指标误区

| 误区 | 正确理解 |
|---|---|
| 调用次数越多越好 | 可能只是用户反复失败 |
| 生成越长越好 | 长不等于有用 |
| 用户少修改就一定好 | 也可能用户没认真看 |
| 成本越低越好 | 不能牺牲关键质量 |
| 自动化率越高越好 | 高风险任务不能盲目自动化 |
| 只看平均值 | 要看失败案例和高分位延迟 |

OpenAI 的评估文档强调，生成式 AI 输出具有可变性，传统软件测试不足以覆盖 AI 系统，因此需要 evals 来测试生产环境中的模型和系统表现。指标体系必须和 eval、trace、用户反馈结合，而不是只看调用量。

# 第 50 章｜商业价值评估

## 50.1 一句话结论

**Agent 值不值得做，取决于它能否在效率、成本、质量、收入、风险或知识复用上产生明确收益。**

商业价值不是一句：

```text
AI 可以提效。
```

而是要量化：

```text
节省谁的时间？
减少什么错误？
提高什么质量？
带来什么收入？
降低什么风险？
沉淀什么能力？
```

---

## 50.2 Agent 价值六类

| 价值类型 | 说明 | 示例 |
|---|---|---|
| 效率价值 | 更快完成任务 | 报表分析从 2 小时降到 20 分钟 |
| 成本价值 | 减少人工或外包成本 | 自动生成初稿 |
| 质量价值 | 减少漏项、提高一致性 | 固定诊断框架 |
| 收入价值 | 提升转化、销售、留存 | 优化 Listing 和广告 |
| 风险价值 | 降低错误、合规、误操作 | 高风险动作确认 |
| 资产价值 | 沉淀知识、模板、流程 | 形成 Skill 和知识库 |

---

## 50.3 ROI 粗算模型

```text
Agent ROI
= 节省的人力成本
+ 增加的业务收益
+ 降低的错误损失
+ 沉淀的复用价值
- 开发成本
- 运行成本
- 维护成本
- 风险成本
```

---

## 50.4 简化评估表

| 问题 | 评分 1–5 |
|---|---:|
| 任务频率高吗？ |  |
| 单次任务耗时高吗？ |  |
| 用户痛点强吗？ |  |
| 数据是否可获得？ |  |
| 结果是否可评估？ |  |
| 自动化风险是否可控？ |  |
| 能否嵌入工作流？ |  |
| 是否能复用到多个场景？ |  |
| 是否能带来业务收益？ |  |
| 维护成本是否可接受？ |  |

建议：

| 总分 | 判断 |
|---:|---|
| 40–50 | 优先做 |
| 30–39 | 可以做 MVP 验证 |
| 20–29 | 谨慎，先缩小范围 |
| <20 | 暂不建议做 |

---

## 50.5 三类不值得做的 Agent

| 类型 | 原因 |
|---|---|
| 炫技型 Agent | 有技术展示，没有业务闭环 |
| 低频复杂 Agent | 开发成本高，使用次数低 |
| 高风险不可控 Agent | 出错损失大，难确认、难回滚 |

# 第 51 章｜MVP 与迭代

## 51.1 一句话结论

**Agent MVP 不是做一个“缩水版万能 Agent”，而是选择一个最小高价值任务，做出可用、可信、可评估的闭环。**

差的 MVP：

```text
做一个能帮我处理所有运营工作的 Agent。
```

好的 MVP：

```text
做一个能读取广告报表，诊断 ACOS 上升原因，并生成 Markdown 诊断报告的 Agent。
```

---

## 51.2 Agent MVP 的最小闭环

```text
单一用户
↓
单一高价值场景
↓
有限输入
↓
固定流程
↓
少量工具
↓
结构化输出
↓
人工确认
↓
质量评估
↓
用户反馈
```

---

## 51.3 MVP 范围控制

| 维度 | MVP 做法 |
|---|---|
| 用户 | 先服务一个角色 |
| 场景 | 先选一个高频任务 |
| 工具 | 先接 1–3 个必要工具 |
| 自动化 | 先做 Copilot，不急着 Autopilot |
| 输出 | 先固定模板 |
| 权限 | 先只读或草稿 |
| 评估 | 先建立 10–30 个典型案例 |
| 运营 | 先手动复盘失败案例 |

---

## 51.4 MVP 迭代路径

```text
V0：手动 Prompt
↓
V1：固定模板 + 人工复制资料
↓
V2：接入文件读取和结构化输出
↓
V3：接入工具和自动计算
↓
V4：加入测试、评估、Trace
↓
V5：嵌入真实工作流
↓
V6：部分低风险自动化
↓
V7：多场景复用和平台化
```

---

## 51.5 什么时候进入下一版本

| 进入下一版本的条件 | 说明 |
|---|---|
| 当前场景稳定 | 主要失败类型已明确 |
| 用户持续使用 | 不是一次性好奇 |
| 结果被采纳 | 用户愿意用输出 |
| 质量可评估 | 有测试集和评分标准 |
| 风险可控 | 没有高风险漏拦 |
| 成本可接受 | 单次运行成本合理 |
| 扩展需求明确 | 用户自然提出相邻任务 |

---

# 第 52 章｜从内部工具到产品平台

## 52.1 一句话结论

**Agent 产品化的长期路径，是从单点内部工具开始，逐步沉淀成工具库、Skill 库、评估体系、权限体系和团队级 Agent 平台。**

不要一开始就做平台。

推荐路径：

```text
单点任务 Agent
↓
稳定 Workflow Agent
↓
多个相邻场景复用
↓
工具库和 Skill 库
↓
统一权限、评估、Trace
↓
团队级 Agent 平台
```

---

## 52.2 单点工具阶段

| 特征 | 内容 |
|---|---|
| 目标 | 验证一个场景有没有价值 |
| 用户 | 少数核心用户 |
| 工具 | 少量工具 |
| 流程 | 半自动 |
| 指标 | 完成率、采纳率、节省时间 |
| 风险 | 可人工兜底 |

示例：

- 对话沉淀 Markdown Agent
- 广告 ACOS 诊断 Agent
- 客服回复草稿 Agent

---

## 52.3 多场景复用阶段

当多个 Agent 出现共用能力时，就要开始沉淀。

| 共用能力 | 可沉淀为 |
|---|---|
| 文件读取 | file tools |
| Markdown 生成 | report generator |
| 广告指标计算 | ads metrics script |
| 知识沉淀方法 | knowledge skill |
| 客服回复模板 | support reply skill |
| 质量评估 | eval cases |
| 权限确认 | approval workflow |

---

## 52.4 平台化阶段

Agent Platform 至少包含：

| 模块 | 作用 |
|---|---|
| Agent Registry | 管理有哪些 Agent |
| Tool Registry | 管理工具能力 |
| Skill Library | 管理可复用方法 |
| Context / Memory Layer | 管理上下文和记忆 |
| Permission System | 管理权限 |
| Eval System | 管理测试和评估 |
| Trace System | 追踪运行过程 |
| Feedback System | 收集用户反馈 |
| Deployment System | 发布、灰度、回滚 |
| Governance System | 安全、合规、审计 |

---

## 52.5 平台化风险

| 风险 | 表现 |
|---|---|
| 过早平台化 | 场景没验证，先做大架构 |
| 工具泛化过度 | 为未来假设设计太多能力 |
| 权限复杂化 | 没有统一治理 |
| 质量不可控 | Agent 多了但 eval 跟不上 |
| 用户入口混乱 | 用户不知道选哪个 Agent |
| 维护成本暴涨 | 每个 Agent 都有独立规则 |
| 没有 owner | 出问题没人负责 |

原则：

```text
先证明单点价值，再抽象平台能力。
```

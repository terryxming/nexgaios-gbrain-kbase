# 第 25 章｜Context Engineering：上下文工程

## 25.1 一句话结论

**Context Engineering 不是“把更多资料塞给模型”，而是“在正确时间，把正确、可信、相关、结构化的信息交给 Agent”。**

Agent 的判断质量主要取决于三件事：

| 因素 | 说明 |
|---|---|
| 模型能力 | 模型本身能不能理解和推理 |
| 指令质量 | Agent 是否知道自己该怎么做 |
| 上下文质量 | Agent 是否拿到了正确资料 |

很多 Agent 失败不是因为模型不够强，而是因为：

```text
上下文错了
上下文旧了
上下文太多了
上下文缺关键字段
上下文没有区分事实和假设
```

---

## 25.2 费曼解释

你让一个人帮你分析广告，但只给他说：

> ACOS 变高了，怎么办？

他只能泛泛建议。

如果你给他：

- 当前周期数据
- 对比周期数据
- campaign / ad group / keyword / search term 维度
- CTR、CVR、CPC、Spend、Sales、Orders
- 最近做过什么操作
- 商品价格是否变化
- 是否有促销、断货、换图、改类目

他才能做出更准确判断。

所以：

> **Context 不是背景资料堆积，而是任务判断所需的证据系统。**

---

## 25.3 State、Context、Memory 的区别

| 概念 | 简单理解 | 生命周期 | 示例 |
|---|---|---|---|
| State | 当前任务执行到哪里 | 当前任务内 | 已读取文件、正在计算指标、等待确认 |
| Context | 当前判断需要看什么 | 当前轮或当前任务 | 报表、用户输入、业务规则、参考文档 |
| Memory | 以后还可能继续用什么 | 跨会话、跨任务 | 用户偏好、项目规则、历史决策 |
| Knowledge Base | 可检索知识集合 | 长期 | SOP、文档、案例、产品资料 |
| RAG | 从知识库取相关资料的方法 | 运行时 | 根据问题检索相关片段 |

一句话区分：

```text
State = 任务进度
Context = 当前证据
Memory = 长期经验
Knowledge Base = 知识仓库
RAG = 从仓库取资料的方法
```

OpenAI Agents SDK 的 context 文档指出，context 对象是本地对象，可携带运行中的数据、依赖和辅助函数，不会直接发送给 LLM；这提醒我们要区分“应用本地上下文”和“真正注入模型的上下文”。

---

## 25.4 上下文工程的五个问题

| 问题 | 说明 |
|---|---|
| 取什么 | 当前任务需要哪些资料？ |
| 从哪取 | 用户输入、文件、数据库、知识库、工具返回？ |
| 怎么取 | 直接提供、检索、摘要、压缩、结构化？ |
| 怎么放 | 放在 prompt、tool result、state、memory 还是资源文件？ |
| 怎么控 | 如何避免过期、污染、泄露、冗余？ |

---

## 25.5 Context 的分层结构

| 层级 | 内容 | 示例 |
|---|---|---|
| L1 当前用户输入 | 用户这一轮明确说的话 | “分析 1/26–1/29 ACOS 为什么升高” |
| L2 任务定义 | 当前任务目标和输出要求 | 输出根因、证据、建议动作 |
| L3 执行状态 | 当前任务进度 | 已读取报表，等待字段校验 |
| L4 数据证据 | 当前判断所需数据 | 广告报表、Search Term Report |
| L5 业务背景 | 和任务相关的业务信息 | ASIN、价格、类目、竞品、关键词策略 |
| L6 方法规则 | SOP、分析框架、策略约束 | 广告诊断框架、否词规则 |
| L7 用户偏好 | 用户长期偏好的输出方式 | 中文、表格化、高信息密度、Markdown |
| L8 系统约束 | 工具、权限、安全边界 | 只能生成建议，不能直接改预算 |

---

## 25.6 好上下文的标准

| 标准 | 说明 |
|---|---|
| 相关 | 和当前任务直接相关 |
| 可信 | 来源可靠，可追溯 |
| 新鲜 | 没有过期 |
| 结构化 | 模型能快速理解 |
| 足够 | 支撑当前判断 |
| 克制 | 不塞无关内容 |
| 分层 | 区分事实、假设、规则、偏好 |
| 可更新 | 旧信息能被替换 |
| 可治理 | 能控制权限和敏感信息 |

---

## 25.8 本章误区

| 误区 | 正确理解 |
|---|---|
| 上下文越多越好 | 上下文越多，噪声越大 |
| 历史对话都要塞进去 | 只放当前任务相关内容 |
| 记忆就是完整聊天记录 | 记忆应该是提炼后的长期有效信息 |
| RAG 能自动解决上下文问题 | RAG 只是取资料方法，还需要筛选和组织 |
| Agent 自己会知道什么重要 | 工程上要明确上下文选择策略 |

# 第 26 章｜短期记忆

## 26.1 一句话结论

**短期记忆是 Agent 在当前任务或当前会话中保存的临时信息，用来保证多轮任务不断片、不重复、不丢步骤。**

短期记忆解决的是：

```text
这件事现在做到哪了？
刚才用户说了什么？
已经调用过哪些工具？
哪些信息已经确认？
下一步该做什么？
```

---

## 26.2 短期记忆和 State 的关系

| 对比项 | State | 短期记忆 |
|---|---|---|
| 重点 | 执行进度 | 当前任务相关信息 |
| 形式 | 状态字段、检查点 | 会话摘要、临时变量、工具结果 |
| 用途 | 控制流程 | 保持上下文连续 |
| 示例 | waiting_for_user_confirmation | 用户刚才选择了“只生成建议，不执行” |

简单理解：

```text
State 更像任务进度条
短期记忆更像当前工作台上的材料
```

OpenAI Agents SDK 的 Sessions 文档说明，sessions 可以维护多次 agent run 之间的会话历史，适合多轮对话；LangGraph 的 persistence 层会把 graph state 保存为 checkpoints，并支持 human-in-the-loop、会话记忆、time travel debugging 和容错执行。

---

## 26.3 短期记忆应该保存什么

| 类型 | 示例 |
|---|---|
| 当前任务目标 | 生成第 25–32 章课程并沉淀 Markdown |
| 用户约束 | 同样处理逻辑：先教学，后生成文件 |
| 当前步骤 | 正在讲解短期记忆 |
| 工具调用结果 | 已生成文件、已读取表格 |
| 临时计算结果 | 指标变化表 |
| 用户选择 | 用户确认使用方案 B |
| 待处理事项 | 还未生成最终 Markdown |
| 异常信息 | 文件生成失败，需要重试 |

---

## 26.4 短期记忆不应该保存什么

| 不应保存 | 原因 |
|---|---|
| 无关闲聊 | 污染判断 |
| 过期中间状态 | 误导后续步骤 |
| 敏感信息全文 | 增加泄露风险 |
| 巨量原始数据 | 浪费上下文 |
| 低价值工具日志 | 只保留摘要和关键结果 |
| 已经完成且无复用价值的临时细节 | 增加系统负担 |

---

## 26.5 短期记忆结构示例

```json
{
  "task_id": "agent_course_ch25_32",
  "goal": "teach chapters 25-32 and generate markdown file",
  "current_step": "chapter_26_short_term_memory",
  "completed_steps": [
    "chapter_25_context_engineering"
  ],
  "user_constraints": [
    "same logic as previous phases",
    "teach first, then create markdown file",
    "Chinese output",
    "suitable for llm-wiki"
  ],
  "pending_steps": [
    "chapters_27_32",
    "stage_recap",
    "markdown_generation"
  ],
  "temporary_results": {},
  "waiting_for_user": false
}
```

---

## 26.6 短期记忆的更新策略

| 策略 | 说明 |
|---|---|
| 每步更新 | 每完成一步更新 current_step |
| 关键节点保存 | 工具调用、用户确认、错误发生时保存 |
| 摘要优先 | 保存摘要，不保存所有原文 |
| 明确过期 | 任务结束后清理临时状态 |
| 可恢复 | 出错后能从 checkpoint 继续 |
| 可审计 | 关键动作有记录 |

---

## 26.7 本章误区

| 误区 | 正确理解 |
|---|---|
| 短期记忆就是聊天历史 | 聊天历史只是来源之一 |
| 所有工具结果都要保存 | 保存关键结果和摘要即可 |
| 短期记忆越完整越好 | 越完整不等于越有用 |
| 任务结束后还保留所有短期记忆 | 应该清理或提炼成长期记忆 |
| 短期记忆可以替代状态机 | 不能，状态机负责流程控制 |

# 第 27 章｜长期记忆

## 27.1 一句话结论

**长期记忆是跨任务、跨会话仍然有价值的信息，用来让 Agent 不需要每次从零开始理解用户、项目和业务规则。**

长期记忆回答的是：

```text
这个用户长期偏好什么？
这个项目长期规则是什么？
之前做过什么决定？
哪些经验以后还会复用？
```

OpenAI Cookbook 的 context personalization 示例展示了用结构化 state、hooks 和 context-injection logic 构建可演进的长期记忆模式；OpenAI 的 Sandbox Agents 文档也区分 session memory 和 sandbox memory：前者保存对话历史，后者把 workspace runs 中有用经验沉淀为未来可读取的文件。

---

## 27.2 长期记忆应该保存什么

| 类型 | 示例 |
|---|---|
| 用户偏好 | 喜欢中文、表格化、高信息密度、Markdown |
| 工作背景 | 用户长期做 Amazon 广告、Agent 工程化学习 |
| 项目规则 | llm-wiki 要适合长期沉淀 |
| 稳定业务信息 | 产品价格、核心关键词、目标人群 |
| 历史决策 | 某个 ASIN 关联 case、广告策略调整 |
| 常用模板 | 知识沉淀模板、FABE 文案模板 |
| 复用方法论 | 通用知识理解框架、Agent 设计框架 |
| 禁忌和边界 | 不要输出大篇散文、不要泛泛而谈 |

---

## 27.3 长期记忆不应该保存什么

| 不应保存 | 原因 |
|---|---|
| 短期临时任务状态 | 很快过期 |
| 一次性闲聊 | 无复用价值 |
| 未确认事实 | 可能污染未来判断 |
| 敏感隐私 | 除非用户明确要求且有必要 |
| 过细执行日志 | 成本高、价值低 |
| 旧版本错误规则 | 会误导后续任务 |
| 可从项目文件中检索的信息全文 | 可通过知识库读取，不必变成记忆 |

---

## 27.4 长期记忆的三种形态

| 形态 | 说明 | 示例 |
|---|---|---|
| Profile Memory | 用户画像型记忆 | 用户偏好、工作领域、输出风格 |
| Project Memory | 项目型记忆 | 项目目标、目录结构、历史决策 |
| Skill Memory | 能力型记忆 | 做某类任务的稳定方法、模板、评估标准 |

---

## 27.5 长期记忆写入原则

| 原则 | 说明 |
|---|---|
| 长期有效 | 未来数周或数月仍有价值 |
| 可复用 | 会影响未来类似任务 |
| 已确认 | 不是模型猜测 |
| 简洁 | 写成稳定事实，不写长篇记录 |
| 可更新 | 后续能覆盖旧信息 |
| 最小必要 | 不保存不必要隐私 |
| 来源清楚 | 知道来自用户明确表达还是历史交互 |

---

## 27.6 长期记忆写入模板

```markdown
# Memory Candidate

## 1. Memory Type
- Profile / Project / Skill / Preference / Business Rule

## 2. Content
- 需要保存的稳定信息：

## 3. Why Useful
- 未来会如何影响回答：

## 4. Validity
- 长期有效 / 项目内有效 / 临时有效

## 5. Sensitivity
- 普通 / 业务敏感 / 个人敏感

## 6. Update Policy
- 何时更新：
- 何时删除：
```

---

## 27.7 长期记忆和知识库的区别

| 对比项 | 长期记忆 | 知识库 |
|---|---|---|
| 内容粒度 | 短、关键、提炼后 | 可长、可结构化、可检索 |
| 作用 | 个性化和连续性 | 提供知识证据 |
| 更新方式 | 少量高价值写入 | 文档化维护 |
| 示例 | 用户喜欢表格化输出 | Agent 工程化课程文档 |
| 风险 | 记错会长期影响 | 检索错会影响当前任务 |

简单理解：

```text
长期记忆 = Agent 对你和项目的关键认识
知识库 = Agent 可以查阅的资料仓库
```

---

## 27.8 本章误区

| 误区 | 正确理解 |
|---|---|
| 所有历史都应该记住 | 只记长期有用信息 |
| 记忆越多越个性化 | 记忆越多越可能污染 |
| 记忆可以不更新 | 记忆会过期，需要治理 |
| 长期记忆可以替代知识库 | 不能，知识库承载完整资料 |
| 模型推测也可以写入记忆 | 不应该，必须基于用户明确或高置信证据 |

# 第 28 章｜项目记忆与用户画像

## 28.1 一句话结论

**项目记忆让 Agent 理解“这个项目一直在做什么”；用户画像让 Agent 理解“这个用户通常怎么工作”。两者必须分开管理。**

如果混在一起，会出现问题：

| 混淆类型 | 后果 |
|---|---|
| 把项目规则当用户偏好 | 换项目后仍然错误沿用 |
| 把用户偏好当项目事实 | 影响客观判断 |
| 把临时任务当长期项目状态 | 未来误判进度 |
| 把历史版本当当前规则 | 使用过期信息 |

---

## 28.2 用户画像包含什么

| 类型 | 示例 |
|---|---|
| 工作领域 | Amazon 运营、Agent 工程化、知识管理 |
| 输出偏好 | 中文、表格、结构化、少废话 |
| 学习偏好 | MECE、费曼、先大纲后讲解 |
| 文件偏好 | Markdown、llm-wiki、Obsidian |
| 常用任务 | 广告诊断、文案优化、知识沉淀、Skill 设计 |
| 质量偏好 | 高信息密度、可复用、能落地 |

用户画像不应该包含：

- 不必要个人隐私
- 短期情绪
- 一次性选择
- 未确认推测
- 敏感身份属性

---

## 28.3 项目记忆包含什么

| 类型 | 示例 |
|---|---|
| 项目目标 | 系统学习 Agent 工程化与产品化 |
| 当前阶段 | 已完成第 0–24 章 |
| 目录结构 | 按阶段生成 Markdown 文件 |
| 方法约定 | 先教学，再沉淀文件 |
| 版本记录 | 每阶段一个 `.md` 文件 |
| 项目决策 | 课程采用 MECE + 费曼方式 |
| 项目资产 | 已生成的 Markdown 文件 |
| 待办事项 | 下一阶段继续第 25–32 章 |

---

## 28.4 用户画像与项目记忆对比

| 对比项 | 用户画像 | 项目记忆 |
|---|---|---|
| 关注对象 | 用户本人长期偏好 | 某个项目长期状态 |
| 作用范围 | 多项目通用 | 当前项目内有效 |
| 示例 | 喜欢表格化 | 当前课程已讲到第 25–32 章 |
| 更新频率 | 较低 | 中等 |
| 风险 | 个性化错误 | 项目状态错误 |
| 管理方式 | Profile Memory | Project Memory / Project Docs |

---

## 28.5 项目记忆文件模板

```markdown
# Project Memory

## 1. Project Name
- Agent 工程化与产品化课程

## 2. Project Goal
- 系统掌握 Agent 工程化与产品化，提升创建高质量 Agent 工程的效率。

## 3. Learning Style
- MECE
- 费曼教学法
- 先大纲，后章节
- 每阶段沉淀 Markdown 文件

## 4. Completed Stages
| 阶段 | 章节 | 文件 |
|---|---|---|
| 阶段一 | 第 0–4 章 | ch00-04 |
| 阶段二 | 第 5–10 章 | ch05-10 |
| 阶段三 | 第 11–17 章 | ch11-17 |
| 阶段四 | 第 18–24 章 | ch18-24 |

## 5. Current Stage
- 第 25–32 章：状态、记忆与上下文工程

## 6. Decisions
- 每阶段都先完整教学，再输出 Markdown 文件。

## 7. Open Questions
- 是否需要最终汇总为完整总课程文件？
```

## 28.6 项目记忆更新策略

| 时间点 | 更新内容 |
|---|---|
| 阶段完成 | 更新 completed stages |
| 文件生成 | 记录文件名和主题 |
| 用户改变规则 | 更新学习风格或输出要求 |
| 项目目标变化 | 更新 project goal |
| 出现重要决策 | 写入 decisions |
| 规则废弃 | 移到 deprecated |

# 第 29 章｜RAG 与知识库

## 29.1 一句话结论

**RAG 是从知识库中检索相关资料并交给模型使用的方法；知识库是资料仓库。RAG 解决“模型不知道或不应凭记忆回答”的问题。**

RAG 不是 Agent，但 Agent 经常需要 RAG。

```text
Knowledge Base = 仓库
RAG = 取货方法
Agent = 拿到货后完成任务的人
```

---

## 29.2 什么时候需要 RAG

| 场景 | 是否需要 RAG |
|---|---|
| 问题依赖外部文档 | 需要 |
| 信息经常更新 | 需要 |
| 需要引用来源 | 需要 |
| 需要查长文档 | 需要 |
| 需要结合私有资料 | 需要 |
| 只是通用概念解释 | 不一定 |
| 用户已经提供完整上下文 | 不一定 |
| 规则简单且固定 | 不一定 |

MCP 官方文档把 MCP 定义为 AI 应用连接外部系统的开放标准，可连接数据源、工具和工作流；这也说明现代 Agent 的上下文不只来自 prompt，还会来自文件、数据库、搜索、工具返回和外部系统。

---

## 29.3 RAG 的基础流程

```text
用户问题
↓
查询改写
↓
检索知识库
↓
筛选相关片段
↓
重排和压缩
↓
注入上下文
↓
模型生成答案
↓
引用来源和校验
```

---

## 29.4 RAG 的关键模块

| 模块 | 作用 |
|---|---|
| 文档切分 | 把长文档拆成可检索片段 |
| 元数据 | 标记来源、时间、主题、权限 |
| 索引 | 让资料可被搜索 |
| 检索 | 根据问题找相关内容 |
| 重排 | 重新排序最相关片段 |
| 压缩 | 删除无关内容，只保留关键证据 |
| 引用 | 告诉用户依据来自哪里 |
| 评估 | 判断检索和回答质量 |

---

## 29.5 RAG 常见失败原因

| 失败点 | 后果 |
|---|---|
| 文档切分太碎 | 片段缺上下文 |
| 文档切分太大 | 噪声太多 |
| 缺少元数据 | 不知道资料是否过期 |
| 检索词不准 | 找不到正确资料 |
| 只靠向量相似度 | 找到语义近但事实错的内容 |
| 没有重排 | 低质量片段进入上下文 |
| 没有引用 | 结果不可验证 |
| 没有权限过滤 | 可能泄露资料 |
| 知识库过期 | 回答基于旧规则 |

---

## 29.6 RAG 不等于知识管理

| RAG | 知识管理 |
|---|---|
| 关注运行时取资料 | 关注长期组织知识 |
| 偏技术流程 | 偏知识结构和治理 |
| 解决“找到相关片段” | 解决“知识如何沉淀、更新、复用” |
| 依赖索引 | 依赖目录、标签、版本、规范 |
| 是工具 | 是体系 |

对 llm-wiki 来说：

```text
RAG 只是未来让 Agent 读取 llm-wiki 的一种方式
llm-wiki 本身更重要的是知识结构、文件规范、沉淀方法和维护机制
```

---

## 29.7 RAG 设计判断表

| 问题 | 判断 |
|---|---|
| 知识是否频繁变化？ | 是 → RAG |
| 是否需要引用来源？ | 是 → RAG |
| 是否依赖私有文档？ | 是 → RAG |
| 文档是否很多？ | 是 → RAG |
| 问题是否需要多文档综合？ | 是 → RAG + Agent |
| 是否只是固定流程？ | Workflow 可能更合适 |
| 是否需要工具执行？ | Agent + Tool，而不只是 RAG |

# 第 30 章｜文件系统型知识库

## 30.1 一句话结论

**文件系统型知识库，是用目录、文件、命名、元数据和链接，把知识组织成可读、可维护、可检索、可迁移的长期资产。**

它的核心不是“把文件存起来”，而是：

```text
让人能读
让 Agent 能找
让知识能更新
让经验能复用
```

---

## 30.2 为什么文件系统型知识库适合个人 Agent 系统

| 优点 | 说明 |
|---|---|
| 透明 | 文件在哪里、内容是什么，一眼能看 |
| 可控 | 不被平台锁死 |
| 可版本化 | 可以 Git 管理 |
| 可迁移 | Markdown 可跨工具迁移 |
| 可检索 | grep、全文搜索、RAG 都能用 |
| 可链接 | 支持 Wiki 链接、标签、反向链接 |
| 可审计 | 文件修改历史清楚 |
| 适合 Agent | Agent 可以按目录、文件、元数据读取 |

---

## 30.3 三层知识库结构

```text
llm-wiki/
├─ 00_inbox/
├─ 10_raw/
├─ 20_notes/
├─ 30_frameworks/
├─ 40_projects/
├─ 50_skills/
├─ 60_cases/
├─ 70_refs/
└─ 90_archives/
```

| 目录 | 作用 |
|---|---|
| 00_inbox | 临时收集，还没整理 |
| 10_raw | 原始资料、对话、网页摘录 |
| 20_notes | 整理后的知识笔记 |
| 30_frameworks | 可复用框架和方法论 |
| 40_projects | 项目型知识和过程记录 |
| 50_skills | Skill / Agent 能力包 |
| 60_cases | 案例、复盘、实战记录 |
| 70_refs | 外部参考资料 |
| 90_archives | 过期、废弃、历史版本 |

---

## 30.4 文件命名规范

推荐格式：

```text
主题-子主题-版本.md
```

示例：

```text
agent-engineering-productization-ch25-32-context-memory.md
amazon-ads-diagnosis-framework-v0.1.md
skill-design-standard-structure-v0.2.md
llm-wiki-three-layer-architecture-v0.1.md
```

命名原则：

| 原则 | 说明 |
|---|---|
| 英文小写 | 方便跨系统 |
| 使用连字符 | 避免空格兼容问题 |
| 主题明确 | 看文件名知道内容 |
| 可排序 | 章节、日期、版本可排序 |
| 可版本化 | 重要文件加 v0.1 / v1.0 |

---

## 30.5 Markdown 文件结构模板

```markdown
# 标题

## 1. 一句话结论

## 2. 适用场景

## 3. 核心概念

| 概念 | 简单理解 |
|---|---|

## 4. 结构框架

## 5. 操作方法

## 6. 常见误区

## 7. 示例

## 8. 可复用模板

## 9. 关联链接

## 10. 更新记录
```

## 30.6 元数据模板

```yaml
---
title: Agent 上下文工程
type: note
domain: agent-engineering
stage: ch25-32
status: draft
version: 0.1.0
created: 2026-05-22
updated: 2026-05-22
tags:
  - agent
  - context-engineering
  - memory
  - rag
---
```

元数据的价值：

| 价值 | 说明 |
|---|---|
| 可筛选 | 按 domain、type、status 查找 |
| 可治理 | 知道是否 draft / stable / deprecated |
| 可版本化 | 知道当前版本 |
| 可自动化 | Agent 可根据 metadata 读取或跳过文件 |
| 可迁移 | 未来 RAG、索引、网站生成都能用 |

## 30.7 文件系统知识库和 RAG 的关系

| 文件系统知识库 | RAG |
|---|---|
| 负责知识长期组织 | 负责运行时检索 |
| 人类可读 | 模型可读 |
| 有目录和命名 | 有索引和召回 |
| 有版本和状态 | 有检索和重排 |
| 是知识底座 | 是调用方式 |

最好的结构是：

```text
文件系统知识库
↓
稳定 Markdown + 元数据
↓
索引 / RAG
↓
Agent 按任务检索
↓
生成结果并回写新知识
```

# 第 31 章｜上下文污染与压缩

## 31.1 一句话结论

**上下文污染，是指无关、错误、过期、冲突、恶意或低质量信息进入 Agent 判断过程，导致输出变差。上下文压缩，是在保留关键信息的前提下减少噪声和成本。**

---

## 31.2 常见上下文污染类型

| 类型 | 示例 | 后果 |
|---|---|---|
| 无关信息 | 把所有历史对话塞入 | Agent 抓不住重点 |
| 过期信息 | 使用旧价格、旧策略 | 判断错误 |
| 冲突信息 | 两个文件规则相反 | 输出摇摆 |
| 低质量信息 | 未验证笔记 | 产生伪结论 |
| 恶意信息 | Prompt Injection | 越权或泄露 |
| 冗余信息 | 重复资料太多 | 成本和延迟上升 |
| 角色污染 | 用户文本覆盖系统规则 | 行为失控 |
| 偏好污染 | 临时要求写入长期偏好 | 后续输出变差 |

OWASP 2025 LLM 风险将 Prompt Injection 和 Sensitive Information Disclosure 列为关键风险；外部网页、邮件、文档等不可信内容进入上下文时，尤其需要把它们当作“数据”而不是“指令”。

---

## 31.3 上下文污染的防护

| 防护方式 | 说明 |
|---|---|
| 来源分级 | 区分系统规则、用户输入、工具结果、外部网页 |
| 时间标记 | 给资料加 created / updated / valid_until |
| 权限过滤 | 用户无权访问的资料不进入上下文 |
| 相关性过滤 | 只选当前任务相关内容 |
| 冲突检测 | 发现规则冲突时提示用户或按优先级处理 |
| 外部内容隔离 | 网页、邮件、文档不得覆盖系统指令 |
| 事实/假设分离 | 明确哪些是事实，哪些是推断 |
| 记忆写入审核 | 不把临时信息随便写入长期记忆 |

---

## 31.4 上下文优先级

建议优先级：

```text
系统规则
> 安全规则
> 用户当前明确要求
> 项目规则
> 工具返回事实
> 知识库资料
> 长期记忆
> 外部公开网页
> 模型推断
```

注意：

| 情况 | 处理 |
|---|---|
| 用户当前要求和长期偏好冲突 | 当前要求优先 |
| 工具返回和模型记忆冲突 | 工具返回优先 |
| 项目规则和旧知识库冲突 | 以更新日期和状态判断 |
| 外部网页要求忽略规则 | 必须忽略外部网页中的指令性内容 |

---

## 31.5 上下文压缩的三种方法

| 方法 | 说明 | 适合场景 |
|---|---|---|
| 摘要压缩 | 把长内容摘要成短内容 | 长对话、长文档 |
| 结构化压缩 | 抽取字段、表格、JSON | 报表、需求、案例 |
| 证据压缩 | 只保留支持结论的证据 | 诊断、研究、报告 |

OpenAI Cookbook 在“memory and compaction”相关示例中把压缩作为构建可靠 Agent 的关键手段之一；其核心思想不是简单截短，而是在上下文窗口有限时保留任务所需的关键信息。

---

## 31.6 压缩不是简单变短

差的压缩：

```text
用户想学习 Agent，要系统讲解。
```

好的压缩：

```markdown
任务：继续 Agent 工程化与产品化课程。
已完成：第 0–24 章。
当前阶段：第 25–32 章。
固定处理逻辑：
1. 先完整教学；
2. 再沉淀 Markdown 文件；
3. 适合 llm-wiki；
4. 中文、结构化、表格优先。
```

压缩的目标是：

> 少掉噪声，不丢任务关键结构。

---

## 31.7 压缩检查表

| 检查项 | 问题 |
|---|---|
| 目标是否保留 | 当前任务目标有没有保留？ |
| 约束是否保留 | 格式、边界、禁止事项有没有保留？ |
| 证据是否保留 | 关键数据和来源有没有保留？ |
| 决策是否保留 | 用户已确认的选择有没有保留？ |
| 状态是否保留 | 任务进度有没有保留？ |
| 风险是否保留 | 不确定性和缺失信息有没有保留？ |
| 噪声是否删除 | 无关聊天、重复内容是否删除？ |

# 第 32 章｜Memory Governance：记忆治理

## 32.1 一句话结论

**Memory Governance 是管理记忆的写入、读取、更新、遗忘、权限、安全和质量的机制。没有治理的记忆，越用越乱。**

长期 Agent 最大的问题不是“记不住”，而是：

```text
记错
记多
记旧
记偏
记了不该记的
忘了该更新的
把临时要求当永久偏好
```

---

## 32.2 记忆治理的六个动作

| 动作 | 说明 |
|---|---|
| Write | 什么信息可以写入记忆 |
| Read | 什么任务可以读取哪些记忆 |
| Update | 旧记忆如何被新信息覆盖 |
| Forget | 哪些记忆应该删除或失效 |
| Validate | 如何判断记忆是否可信 |
| Audit | 谁在什么时候写入或使用了记忆 |

---

## 32.3 记忆写入规则

| 问题 | 写入判断 |
|---|---|
| 是否长期有效？ | 是，才考虑写入 |
| 是否会影响未来回答？ | 是，才有价值 |
| 是否用户明确表达？ | 是，可信度更高 |
| 是否敏感？ | 敏感信息谨慎处理 |
| 是否已有相同记忆？ | 避免重复 |
| 是否和旧记忆冲突？ | 需要更新而不是追加 |
| 是否只是当前任务临时约束？ | 不写入长期记忆 |

---

## 32.4 记忆状态

| 状态 | 说明 |
|---|---|
| candidate | 候选记忆，待确认 |
| active | 当前有效 |
| stale | 可能过期 |
| deprecated | 已废弃但保留历史 |
| deleted | 已删除 |
| conflict | 与其他记忆冲突 |
| sensitive | 敏感，需要限制读取 |

---

## 32.5 记忆冲突处理

| 冲突类型 | 示例 | 处理 |
|---|---|---|
| 偏好冲突 | 以前要详细，现在要简洁 | 当前明确要求优先，更新偏好 |
| 项目冲突 | 旧目录结构 vs 新目录结构 | 以最新项目决策为准 |
| 事实冲突 | 旧价格 vs 新价格 | 以最新证据为准 |
| 权限冲突 | 某资料曾可用，现在不可用 | 权限优先 |
| 临时/长期冲突 | 今天要英文，不代表以后都英文 | 不写入长期偏好 |

---

## 32.6 记忆读取策略

| 策略 | 说明 |
|---|---|
| 最小读取 | 只读取当前任务需要的记忆 |
| 类型匹配 | 广告任务读取广告相关记忆 |
| 项目优先 | 当前项目记忆优先于通用偏好 |
| 当前输入优先 | 用户当前明确要求优先 |
| 敏感过滤 | 敏感记忆默认不暴露 |
| 冲突提示 | 发现冲突时说明依据 |

---

## 32.7 记忆治理模板

```markdown
# Memory Governance Policy

## 1. Memory Types
- User Preference
- Project Rule
- Business Fact
- Historical Decision
- Skill Pattern

## 2. Write Policy
| 类型 | 是否可写入 | 条件 |
|---|---|---|

## 3. Read Policy
| 任务类型 | 可读取记忆 |
|---|---|

## 4. Update Policy
| 冲突类型 | 更新规则 |
|---|---|

## 5. Forget Policy
| 情况 | 处理 |
|---|---|

## 6. Sensitivity Policy
| 信息类型 | 策略 |
|---|---|

## 7. Audit Policy
- 记录写入时间
- 记录来源
- 记录更新原因
- 记录废弃原因
```

---

## 32.8 本章误区

| 误区 | 正确理解 |
|---|---|
| 记忆越多越好 | 记忆越多越需要治理 |
| 用户说过就永久记住 | 只有长期有效才值得记 |
| 记忆不需要版本 | 记忆会过期、冲突、废弃 |
| 记忆可以替代项目文档 | 项目文档仍是知识主干 |
| 所有 Agent 都读所有记忆 | 必须按任务和权限最小读取 |
| 删除记忆不重要 | 遗忘是记忆治理的一部分 |

# 第 25–32 章总复盘

## 1. 本阶段核心公式

```text
Agent 判断质量
= 模型能力
+ 指令质量
+ 上下文质量
+ 记忆治理
```

```text
上下文工程
= 正确资料
+ 正确时间
+ 正确结构
+ 正确权限
+ 正确压缩
```

```text
长期可用 Agent
= State 管进度
+ Context 管当前证据
+ Memory 管长期经验
+ Knowledge Base 管系统知识
+ RAG 管运行时检索
+ Governance 管安全和质量
```

---

## 2. 八个核心结论

| 序号 | 核心结论 |
|---|---|
| 1 | Context 不是资料越多越好，而是当前任务刚好需要的证据 |
| 2 | State、Context、Memory、Knowledge Base、RAG 是五个不同概念 |
| 3 | 短期记忆服务当前任务，长期记忆服务跨任务复用 |
| 4 | 用户画像和项目记忆必须分开管理 |
| 5 | RAG 是取资料的方法，不是知识管理本身 |
| 6 | 文件系统型知识库适合作为个人 Agent 系统的长期知识底座 |
| 7 | 上下文污染会严重影响 Agent 判断，需要来源、权限、时间和优先级治理 |
| 8 | 记忆治理决定长期 Agent 会越来越好，还是越来越乱 |

---
